## Введение

**Архитектура программного обеспечения** — это фундаментальная организация системы, воплощённая в её компонентах, их связях друг с другом и с окружением, а также принципы, управляющие её проектированием и эволюцией (IEEE 1471).

Все архитектурные стили можно разделить на две большие группы:

| Группа | Суть | Коммуникация | Деплой |
|--------|------|-------------|--------|
| **Нераспределённые** | Всё приложение — один процесс (один артефакт) | Вызовы методов, без сети | Одна единица |
| **Распределённые** | Приложение состоит из независимых процессов/серверов | Сеть (HTTP, gRPC, очереди, P2P) | Независимые единицы |

Выбор архитектуры — одно из самых критичных решений в проекте. Ошибка стоит дорого, а «перепрыгнуть» из монолита в микросервисы без подготовки — ещё дороже.

# Часть I. Нераспределённые архитектуры

> **Нераспределённая архитектура** = всё приложение работает в рамках **одного процесса**, деплоится как **единый артефакт**.  
> Коммуникация между компонентами — вызовы методов, без сериализации и сетевых задержек.  
> Это не обязательно «монолит» в плохом смысле — модульный монолит и чистые архитектуры тоже сюда относятся.

---

## 1. Монолитная архитектура (Monolithic)

### Описание

Классический монолит — единое неделимое приложение: UI, бизнес-логика, доступ к данным собраны в один артефакт (`jar`/`war`/`exe`). Всё в одном репозитории, одна сборка, один деплой.

```
┌────────────────────────────────────────────┐
│              Единый процесс                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │    UI    │→ │ Business │→ │   Data   │  │
│  └──────────┘  └──────────┘  └──────────┘  │
│         ↓ прямое обращение к таблицам       │
│  ┌──────────────────────────────────────┐  │
│  │            Одна БД                     │  │
│  └──────────────────────────────────────┘  │
└────────────────────────────────────────────┘
```

### Характеристики

| Связность (coupling) | Высокая — любой модуль может вызвать любой |
|----------------------|---------------------------------------------|
| Единица деплоя | Всё приложение целиком |
| Коммуникация | Прямые вызовы методов в процессе |
| Данные | Чаще всего одна БД |

### Плюсы

- **Максимальная скорость на старте.** Один репозиторий, один CI/CD pipeline, один деплой. Идеально для MVP.
- **Простая отладка.** Stack trace без сетевых прыжков — видно весь путь запроса.
- **Простое тестирование.** End-to-end тест поднимает одно приложение и одну БД.
- **Производительность.** Нет сетевых задержек, сериализации/десериализации.
- **ACID-транзакции из коробки.** Никаких распределённых транзакций, саг, компенсаций.
- **Низкий порог входа.** Junior-разработчик быстро разбирается.

### Минусы

- **Big Ball of Mud.** При ~100k+ строк кода связанность растёт квадратично (каждый модуль потенциально связывается с каждым). Модули проникают друг в друга.
- **Масштабирование «всё или ничего».** Нельзя отмасштабировать только узкое место — приходится разворачивать ещё один экземпляр всего монолита.
- **Технологический lock-in.** Смена языка/фреймворка/БД = переписывание всего.
- **Медленный CI/CD.** Сборка огромного проекта — десятки минут.
- **Хрупкость.** Баг в email-рассылке может уронить всё приложение.

### Антипаттерн: Distributed Monolith

«Распределённый монолит» — система, формально разбитая на микросервисы, но с жёсткой связностью: сервисы деплоятся только вместе (независимый деплой невозможен), используют общую БД, синхронные цепочки вызовов без отказоустойчивости. Хуже классического монолита: та же связанность + сетевая неопределённость. Единственный способ избежать — внедрять микросервисы от реальной бизнес-потребности, а не «для галочки».

### Когда использовать

- **Стартапы и MVP.** Проверить гипотезу за 2–3 месяца.
- **Небольшие команды (1–5 разработчиков).**
- **Продукты с жёсткой консистентностью данных** (банковский учёт, бухгалтерия).

### Примеры

- Внутренний HR-портал — Spring Boot + Thymeleaf + PostgreSQL.
- E-commerce MVP для локального рынка — Django/Rails монолит с каталогом, корзиной, админкой.
- Банковский бэк-офис с ночным batch-процессингом.

## 2. Модульный монолит (Modular Monolith)

### Описание

Модульный монолит — это **монолит по деплою, но не по структуре**. Код организован в изолированные модули с чёткими границами и явными контрактами (API). Внутри процесса действуют те же принципы, что в микросервисах: «чужую БД не трогай, общайся через API».

```
┌──────────────────────────────────────────────┐
│                Единый процесс                  │
│                                               │
│  ┌──────────────────┐  ┌──────────────────┐   │
│  │    Module A       │  │    Module B       │   │
│  │  ┌──────┐ ┌────┐ │  │  ┌──────┐ ┌────┐ │   │
│  │  │Public│ │Data│ │  │  │Public│ │Data│ │   │
│  │  │ API  │ │    │ │  │  │ API  │ │    │ │   │
│  │  └──┬───┘ └──┬─┘ │  │  └──┬───┘ └──┬─┘ │   │
│  └─────┼────────┼───┘  └─────┼────────┼───┘   │
│        │    ┌───┴────────────┴─┐      │       │
│        └───→│  Module C (API)  │←─────┘       │
│             └──────────────────┘              │
│                                               │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐       │
│  │ Schema A│  │ Schema B│  │ Schema C│       │
│  └─────────┘  └─────────┘  └─────────┘       │
│          (одна физическая БД)                  │
└──────────────────────────────────────────────┘
```

**Ключевое правило**: модуль общается с другим модулем **только через его публичное API** (интерфейс сервиса). Прямые запросы в таблицы чужого модуля запрещены — это проверяется тестами (ArchUnit, NetArchTest).

### Плюсы (vs микросервисы)

- **Нет распределённой сложности.** ACID-транзакции, простой дебаг, никаких circuit breaker'ов.
- **Локальные вызовы** на порядок быстрее HTTP/gRPC.
- **Один деплой.** Не нужен Kubernetes, service mesh, распределённый трейсинг.
- **Лёгкий рефакторинг.** Переименовал метод — видно все места использования, компилятор проверит.

### Плюсы (vs классический монолит)

- **Чёткие границы.** Модули изолированы; код одного модуля не просачивается в другой.
- **Прямой путь к микросервисам.** Границы уже определены — при необходимости модуль «вытаскивается» в отдельный сервис.
- **Изолированное тестирование.** Модуль тестируется независимо, с моками вместо соседних модулей.

### Минусы

- **Требует дисциплины.** Без контроля границ (ArchUnit и code review) деградирует в классический монолит.
- **Общий технологический стек.** Нельзя написать один модуль на Go, другой на Kotlin.
- **Общий деплой.** Изменения в любом модуле требуют передеплоя всего приложения.

### Инструменты контроля границ

| Язык | Инструменты |
|------|-------------|
| Java | ArchUnit, Moduliths, jMolecules |
| .NET | NetArchTest |
| Kotlin | Konsist, ArchUnit |
| Python | import-linter, pytest-archunit |

### Когда использовать

- **«Золотая середина»** между монолитом и микросервисами.
- **Растущий продукт:** команда растёт с 5 до 20–30 человек.
- **Первый шаг миграции** от монолита к микросервисам: сначала наводим порядок внутри.

### Примеры

- Интернет-магазин среднего размера (каталог, заказы, оплата, доставка как модули).
- SaaS для управления проектами (пользователи, проекты, документы, биллинг).
- Финтех-продукт на стадии роста (.NET с выделенными bounded context'ами).

## 3. Слоевая архитектура (N-tier)

### Описание

Компоненты организуются по **горизонтальным слоям**: вышележащий слой зависит от нижележащего, но не наоборот. Самый распространённый стиль в энтерпрайзе.

```
┌─────────────────────────────────┐
│     Presentation Layer          │  ← Контроллеры, страницы (Spring MVC, ASP.NET)
├─────────────────────────────────┤
│     Business Logic Layer        │  ← Сервисы, use cases
├─────────────────────────────────┤
│     Data Access Layer           │  ← Репозитории, DAO, JPA/JDBC
├─────────────────────────────────┤
│     Database                    │  ← Сама СУБД
└─────────────────────────────────┘
```

Расширенные варианты (4-tier, 5-tier):
- Добавляется **Integration Layer** для внешних API
- Добавляется отдельный **Domain Layer** (сущности без зависимостей)

### Принципы

1. **Строгая направленность зависимостей:** сверху вниз.
2. **Закрытость слоёв:** запрос не может «перепрыгнуть» через слой (Presentation → Data минуя Business — запрещено).
3. В современных реализациях — **инверсия зависимостей** между Business и Data: Business определяет интерфейс репозитория, Data реализует.

### Плюсы

- **Интуитивно понятна.** Почти все бэкенд-разработчики знакомы с ней.
- **Чистое разделение ответственности.** Frontend, бизнес-логика, работа с БД — у разных людей.
- **Заменяемость слоёв.** Можно сменить БД, переписав только Data слой.

### Минусы

- **Искусственная «слоистость».** Простой CRUD проходит через 4–5 классов ради 10 строк SQL.
- **Каскадные изменения.** Добавление поля → миграция БД → Entity → DTO → View Model → UI.
- **«Протекание» бизнес-логики** в контроллеры или хранимые процедуры.
- **Не решает проблему масштабирования.** Система по-прежнему монолит.

### Когда использовать

- **Классический энтерпрайз:** банки, страховые, государственные системы — стабильность и предсказуемость.
- **CRUD-приложения:** «формы над таблицами».
- **Стабильные требования** (не меняются каждые две недели).

### Примеры

- Система документооборота — Spring MVC + Hibernate + Oracle.
- CRM для страховой компании — ASP.NET MVC + Entity Framework + MS SQL.

## 4. Микроядерная архитектура (Microkernel / Plug-in)

### Описание

Система состоит из минимального **ядра** и набора **плагинов**, расширяющих функциональность через стандартизированный Plugin API.

```
                    ┌─────────────────────┐
                    │     Plugin API       │  ← Контракт, через который плагины подключаются
                    └──────────┬──────────┘
          ┌────────────────────┼────────────────────┐
          │                    │                    │
    ┌─────┴─────┐        ┌─────┴─────┐        ┌─────┴─────┐
    │  Plugin A │        │  Plugin B │        │  Plugin C │
    │ (PDF)     │        │ (Export)  │        │ (Charts)  │
    └───────────┘        └───────────┘        └───────────┘
          │                    │                    │
          └────────────────────┼────────────────────┘
                               │
                        ┌──────┴──────┐
                        │    Core     │  ← Минимальная функциональность
                        │ (Minimal)   │
                        └─────────────┘
```

Ядро ничего не знает о плагинах; плагины знают о ядре и реализуют Plugin API.

### Плюсы

- **Расширяемость без изменения ядра.** Сторонние разработчики могут писать плагины.
- **Изоляция.** Сбой в плагине не рушит ядро (если плагины изолированы в classloader'ах/процессах).
- **Кастомизация.** Разным клиентам — разный набор плагинов.
- **Стабильное ядро.** Ядро можно сделать простым и почти не менять.

### Минусы

- **Сложность проектирования Plugin API.** API должно быть достаточно гибким для плагинов, о которых вы ещё не думали.
- **Версионирование API.** Обратная совместимость — постоянная боль.
- **Ограниченная функциональность.** Плагин может только то, что разрешает API.

### Когда использовать

- **Продукты с экосистемой расширений:** IDE, браузеры, CMS.
- **B2B-продукты с разными потребностями клиентов** (разные модули для разных индустрий).
- **Инфраструктурные инструменты:** CI/CD (Jenkins), системы сборки (Maven, Gradle), API-шлюзы (Kong).

### Примеры

- IntelliJ IDEA, VS Code — ядро + тысячи плагинов.
- Jenkins — ядро CI/CD + плагины (Git, Docker, Slack).
- Kong API Gateway — ядро + плагины (аутентификация, rate limiting).
- WordPress — ядро CMS + темы и плагины.

## 5. Компонентная архитектура (Component-Based)

### Описание

Система собирается из переиспользуемых, независимых, заменяемых компонентов. **Компонент** = единица композиции с явно определённым интерфейсом и скрытой реализацией.

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ Component│────→│ Component│────→│ Component│
│    A     │     │    B     │     │    C     │
│  ┌─────┐ │     │  ┌─────┐ │     │  ┌─────┐ │
│  │ API │ │     │  │ API │ │     │  │ API │ │
│  └─────┘ │     │  └─────┘ │     │  └─────┘ │
└──────────┘     └──────────┘     └──────────┘
```

Гранулярность: от класса (ООП) до модуля, npm/jar-пакета, Docker-контейнера.

### Характеристики компонента

| Характеристика | Описание |
|----------------|----------|
| **Инкапсуляция** | Реализация скрыта; доступ только через API |
| **Композируемость** | Компоненты собираются в более крупные композиции |
| **Заменяемость** | Компонент можно заменить альтернативной реализацией без изменения потребителей |
| **Версионируемость** | У каждого компонента — своя версия и жизненный цикл |
| **Явный контракт** | API компонента описан (интерфейс, зависимости, конфигурация) |

### Связь с другими архитектурами

Компонентная архитектура — **фундаментальный принцип**, а не отдельный конкурирующий стиль. Она лежит в основе большинства современных подходов:

| Архитектура | Где проявляется компонентный принцип |
|-------------|--------------------------------------|
| **Модульный монолит** | Модули — компоненты с явными API в одном процессе |
| **Микроядро** | Ядро + плагины — компоненты, подключаемые через Plugin API |
| **Микросервисы** | Каждый сервис — независимо деплоимый компонент |
| **Микрофронтенды** | Фронтенд-компоненты с независимым деплоем |

### Плюсы

- **Переиспользование.** Компонент пишется один раз, используется во многих местах.
- **Изоляция.** Изменения внутри компонента не ломают потребителей при стабильном API.
- **Параллельная разработка.** Команды разрабатывают компоненты независимо.
- **Тестируемость.** Каждый компонент тестируется изолированно (unit + контрактные тесты).

### Минусы

- **Dependency hell.** Несовместимые версии, конфликты транзитивных зависимостей.
- **Риск переабстрагирования** — компоненты мельчатся до бесконечности, каждая мелочь становится «переиспользуемым компонентом».
- **Накладные расходы** на прослойку абстракций и версионирование.
- **Сложность миграции API** — изменение публичного контракта требует координации всех потребителей.

### Инструменты

| Язык/Платформа | Компонентная модель | Управление зависимостями |
|----------------|---------------------|--------------------------|
| **Java** | Spring Beans, OSGi, JPMS (Java Platform Module System) | Maven, Gradle, BOM |
| **.NET** | NuGet packages, Assemblies | NuGet |
| **JavaScript/TypeScript** | npm packages, ES modules, Web Components | npm, yarn, pnpm |
| **Python** | pip packages, wheels | pip, Poetry, uv |
| **Фронтенд-фреймворки** | React/Vue/Svelte Components | npm, yarn |

### Когда использовать

Фактически **всегда** — это фундамент современной разработки. Любой Spring Bean, React-компонент, npm-пакет — это компонентная архитектура в действии. Вопрос не «использовать ли», а «насколько осознанно проектировать компоненты».

## 6. Чистые архитектуры (доменно-ориентированные)

**Чистые архитектуры** — это семейство архитектурных стилей, объединённых одним принципом: **доменная логика находится в центре и не зависит от инфраструктуры**. Зависимости направлены от инфраструктуры (внешние круги) к домену (внутренний круг), и никогда наоборот.

Три основные реализации: Гексагональная (Cockburn), Луковичная (Palermo), Чистая (Martin). Все они решают одну задачу — **изолировать бизнес-логику от фреймворков**.

---

### 6.1 Гексагональная архитектура (Hexagonal / Ports & Adapters)

**Автор:** Алистер Коберн (Alistair Cockburn), 2005.

**Ключевая идея:** домен в центре, внешний мир подключается через **порты** (интерфейсы) и **адаптеры** (реализации под конкретную технологию).

```
                       ┌──────────────────────────────┐
                       │          ДОМЕН                │
                       │   (Entities, Value Objects,    │
                       │    Domain Services)            │
                       └──────────┬───────────────────┘
                                  │
              ┌───────────────────┼───────────────────┐
              │                   │                   │
     ┌────────┴────────┐  ┌───────┴───────┐  ┌───────┴────────┐
     │  Driving Port   │  │  Driven Port  │  │  Driven Port   │
     │  (входящий)     │  │  (исходящий)  │  │  (исходящий)   │
     │  OrderService   │  │ OrderRepo     │  │ NotifySender   │
     └────────┬────────┘  └───────┬───────┘  └───────┬────────┘
              │                   │                   │
    ┌─────────┴─────────┐  ┌──────┴──────┐  ┌────────┴─────────┐
    │  Driving Adapter  │  │   Driven    │  │     Driven       │
    │  REST Controller  │  │   Adapter   │  │     Adapter      │
    │  gRPC Handler     │  │ JPA Repo    │  │  Kafka Producer  │
    └───────────────────┘  └─────────────┘  └──────────────────┘
```

- **Driving (первичные) порты** — входящие: интерфейсы, через которые внешний мир «ведёт» (drives) домен. Это API приложения для внешних потребителей (Use Cases, Application Services)
- **Driven (вторичные) порты** — исходящие: интерфейсы, через которые домен вызывает внешний мир. Домен определяет контракт, инфраструктура реализует (Repository, Gateway, MessageSender)
- **Адаптеры** — конкретные реализации портов под технологию (HTTP, JDBC, Kafka)

**Отличие от слоевой:** Слоевая архитектура ставит Data слой «внизу», и Business зависит от Data. В гексагональной — Data слой это адаптер сбоку, и он зависит от домена (через порт), а не наоборот.

### Плюсы

- **Домен не знает о технологиях.** Никаких `@Entity`, `@RestController` в бизнес-логике.
- **Тестируемость.** Домен тестируется в unit-тестах без БД, HTTP, очередей.
- **Заменяемость инфраструктуры.** Хотите переехать с PostgreSQL на MongoDB? Пишете новый адаптер — домен не трогаете.

### Минусы

- **Много boilerplate.** Порт + адаптер + маппинг DTO ↔ Domain для каждой интеграции.
- **Избыточна для простого CRUD.**

### Примеры

- Платёжный шлюз (правила маршрутизации, комиссии, фрод-скоринг — в домене, эквайеры — адаптеры).
- Страховая система (андеррайтинг в домене, UI и БД — адаптеры).

---

### 6.2 Луковичная архитектура (Onion Architecture)

**Автор:** Джеффри Палермо (Jeffrey Palermo), 2008.

**Ключевая идея:** развитие гексагональной — более детальная структура слоёв в виде концентрических колец. Каждый круг может зависеть только от более внутренних кругов.

```
┌──────────────────────────────────────────┐
│              Infrastructure               │  ← Самый внешний: БД, внешние API, UI
│  ┌──────────────────────────────────┐    │
│  │         Application Services      │    │  ← Use Cases, Application Logic
│  │  ┌──────────────────────────┐    │    │
│  │  │     Domain Services       │    │    │  ← Бизнес-правила, инварианты
│  │  │  ┌──────────────────┐    │    │    │
│  │  │  │   Domain Model   │    │    │    │  ← Самый центр: Entities, Value Objects
│  │  │  │  (Entities, VO)  │    │    │    │
│  │  │  └──────────────────┘    │    │    │
│  │  └──────────────────────────┘    │    │
│  └──────────────────────────────────┘    │
└──────────────────────────────────────────┘
```

**Слои (снаружи внутрь):**

| Слой | Содержит | Зависит от |
|------|----------|-----------|
| **Infrastructure** | JPA-репозитории, REST-контроллеры, Kafka-продюсеры, внешние HTTP-клиенты | Application, Domain |
| **Application Services** | Use Cases, DTO, Application Events, интерфейсы репозиториев (порты) | Domain |
| **Domain Services** | Бизнес-правила, затрагивающие несколько сущностей | Domain Model |
| **Domain Model** | Entities, Value Objects, Domain Events | Ничего |

**Ключевое правило:** весь поток управления направлен от внешних слоёв к центру. Внутренние слои определяют интерфейсы, внешние их реализуют. Это **инверсия зависимостей** на уровне архитектуры.

**Отличие от гексагональной:** Луковичная явно выделяет 4 концентрических слоя (а не просто «домен + адаптеры») и добавляет слой Application Services для use cases.

### Плюсы

- **Чёткая структура.** 4 слоя с понятной ответственностью — проще объяснить команде.
- **Домен полностью чист.** Никакой инфраструктуры в Domain Model/Domain Services.
- **Use Cases выделены явно.** Application Services — это конкретные сценарии использования, каждый в своём классе.

### Минусы

- Та же проблема boilerplate, что у гексагональной.
- 4 слоя могут быть избыточны для простых операций.

### Примеры

- Микросервис в .NET: Domain Model (Entities) → Application (Use Cases + MediatR) → Infrastructure (EF Core, Controllers).
- Типичный ASP.NET Core проект, следующий за рекомендациями Microsoft.

### 6.3 Чистая архитектура (Clean Architecture)

**Автор:** Роберт Мартин (Robert C. Martin), 2012.

**Ключевая идея:** обобщение всех «чистых» подходов (гексагональной, луковичной, DCI, BCE) в единую модель. Формулирует универсальное **правило зависимостей**: исходный код может зависеть только **внутрь**.

```
┌──────────────────────────────────────────┐
│         Frameworks & Drivers              │  ← Spring, Express, React, PostgreSQL driver
│  ┌──────────────────────────────────┐    │
│  │     Interface Adapters            │    │  ← Контроллеры, презентеры, репозитории
│  │  ┌──────────────────────────┐    │    │
│  │  │   Application Business    │    │    │  ← Use Cases (сценарии использования)
│  │  │   Rules (Use Cases)       │    │    │
│  │  │  ┌──────────────────┐    │    │    │
│  │  │  │   Enterprise     │    │    │    │  ← Entities (корпоративные бизнес-правила)
│  │  │  │   Business       │    │    │    │
│  │  │  │   Rules          │    │    │    │
│  │  │  └──────────────────┘    │    │    │
│  │  └──────────────────────────┘    │    │
│  └──────────────────────────────────┘    │
└──────────────────────────────────────────┘
```

**Слои (снаружи внутрь):**

1. **Frameworks & Drivers** — фреймворки, БД, веб-серверы. Здесь «живут» Spring Boot, Express, React.
2. **Interface Adapters** — контроллеры, презентеры, репозитории. Преобразуют данные между use cases и внешним миром.
3. **Application Business Rules (Use Cases)** — прикладные бизнес-правила; оркеструют сущности для конкретной задачи.
4. **Enterprise Business Rules (Entities)** — корпоративные бизнес-правила; чистые объекты, не зависящие ни от чего.

**Правило зависимостей:** всё указывает внутрь. `@RestController` (внешний круг) зависит от Use Case (внутренний), который зависит от Entity (самый внутренний). И никогда наоборот.

### Плюсы (в дополнение к общим для чистых архитектур)

- **Полная независимость от фреймворка.** Переезд со Spring на Quarkus не трогает бизнес-логику.
- **Независимость от UI.** Бизнес-логика не знает, будет ли UI браузером, мобильным приложением или CLI.
- **Независимость от БД.** Можно сменить СУБД без изменения домена.

### Минусы

- **Максимальный boilerplate.** Каждый слой требует своих моделей: JPA Entity → Domain Entity → DTO → JSON.
- **Высокий порог входа.** Понять, почему для сохранения в БД нужно 5 классов — требует смены парадигмы.

### 6.4 Сравнение чистых архитектур

| Аспект | Гексагональная | Луковичная | Clean Architecture |
|--------|:--------------:|:----------:|:------------------:|
| **Автор, год** | Cockburn, 2005 | Palermo, 2008 | R. Martin, 2012 |
| **Метафора** | Шестиугольник с портами | Луковица с 4 слоями | Концентрические круги |
| **Ключевой фокус** | Порты и адаптеры | Чёткая иерархия слоёв | Правило зависимостей |
| **Число слоёв** | 3 (Domain, Ports, Adapters) | 4 (Domain Model, Domain Services, Application, Infrastructure) | 4 (Entities, Use Cases, Adapters, Frameworks) |
| **Use Cases** | В домене или портах | В Application Services | Отдельный круг |
| **Совместимость** | Является частным случаем Clean Architecture | Является частным случаем Clean Architecture | Обобщает обе |

**Общее у всех трёх:**
- Домен не зависит от инфраструктуры
- Инверсия зависимостей: внутренние круги определяют интерфейсы, внешние реализуют
- Высокая тестируемость и заменяемость технологий
- Избыточны для простого CRUD

---

### 6.5 Когда применять чистые архитектуры

#### Когда использовать

- **Сложные долгоживущие системы** (10+ лет), где технологии гарантированно сменятся.
- **Домен с богатой бизнес-логикой** — когда CRUD-мышления недостаточно.
- **Высокие требования к тестируемости** (медицина, авионика, финтех).
- **Когда пишете платформу/фреймворк** — ваш код должен работать в разных окружениях.

#### Когда НЕ использовать

- **MVP / прототипы.**
- **Простые CRUD-приложения.**
- **Когда бизнес-логика — это «показать форму, сохранить в БД».**

#### Примеры

- Медицинская система управления пациентами.
- Финансовая платформа для трейдинга (ордера, портфели, риск-менеджмент).
- B2B SaaS с длинным жизненным циклом.

# Часть II. Распределённые архитектуры

> **Распределённая архитектура** = система состоит из нескольких независимых процессов, общающихся по сети.  
> Каждый процесс может быть развёрнут, отмасштабирован и обновлён независимо.  
> Плата за гибкость — сетевая неопределённость: задержки, таймауты, частичные сбои, eventual consistency.

---

## 7. Клиент-серверная архитектура (Client-Server)

### Описание

Одна из старейших распределённых архитектур: **сервер** предоставляет ресурсы и сервисы, **клиенты** запрашивают их по сети.

```
┌──────────┐     запрос      ┌──────────┐
│  Client  │ ──────────────→ │  Server  │
│          │ ←────────────── │          │
└──────────┘     ответ       └────┬─────┘
                             ┌───┴──────┐
                             │ Database │
                             └──────────┘
```

**Разновидности:**
- **Тонкий клиент** — только UI, логика на сервере (веб-приложения, Citrix)
- **Толстый клиент** — содержит бизнес-логику (IDE, мобильное приложение с офлайн-режимом)
- **Гибрид** — часть логики на клиенте, часть на сервере (SPA + REST API)

### Плюсы

- **Централизованное управление.** Данные, безопасность, логика — в одном месте.
- **Горизонтальное масштабирование** сервера (несколько экземпляров за балансировщиком).
- **Безопасность.** Данные не хранятся на клиенте (в тонком клиенте).

### Минусы

- **Единая точка отказа.** Упал сервер — система встала (решается кластеризацией).
- **Сетевая зависимость.** Задержки, потеря пакетов — нужны fallback и retry.
- **Обновление толстых клиентов** — боль при сотнях/тысячах установок.

### Когда использовать

- **Практически все веб-приложения** используют эту модель.
- **Корпоративные системы** с централизованными данными (ERP, CRM).
- **Мобильные приложения с бэкендом.**

### Примеры

- Gmail, Google Docs — тонкий клиент + мощный сервер.
- 1С:Предприятие — клиент-сервер с толстым и тонким клиентом.
- Интранет-портал — браузер + сервер приложений + БД.

---

## 8. Сервис-ориентированная архитектура (SOA)

### Описание

SOA — предшественник микросервисов (2000-е). Система собирается из сервисов, общающихся через **Enterprise Service Bus (ESB)** — центральную шину, обеспечивающую маршрутизацию, трансформацию форматов и оркестрацию.

```
                         ┌─────────────────────┐
    ┌────────────┐       │                     │       ┌────────────┐
    │  Service A │ ←────→│         ESB         │←────→ │  Service C │
    │  (CRM)     │       │  (Enterprise        │       │  (Billing)  │
    └────────────┘       │   Service Bus)      │       └────────────┘
                         │                     │
    ┌────────────┐       │  • Маршрутизация    │       ┌────────────┐
    │  Service B │ ←────→│  • Трансформация    │←────→ │  Service D │
    │  (Orders)  │       │  • Оркестрация      │       │(Warehouse) │
    └────────────┘       │  • Безопасность     │       └────────────┘
                         └─────────────────────┘
```

**Ключевые стандарты:** SOAP/WSDL, XML Schema, UDDI, WS-\* (WS-Security, WS-AtomicTransaction).

### Плюсы

- **Переиспользование сервисов** (сервис «Клиенты» один, а используют CRM, биллинг, колл-центр).
- **Гетерогенность** — сервисы на разных платформах, ESB всё свяжет.
- **WS-\* стандарты** покрывают безопасность, транзакции, надёжность.

### Минусы

- **ESB — дорогое узкое горло.** Центральный брокер, через который всё идёт.
- **Тяжеловесный SOAP/XML** — многословный формат, медленный парсинг.
- **Vendor lock-in** (IBM, Oracle, Tibco).
- **Waterfall-ориентированность:** «большое проектирование вверх» — месяцы анализа до первой строчки кода.

### Когда использовать

- **Крупные энтерпрайзы с гетерогенным ландшафтом** (мейнфреймы, SAP, Oracle EBS + современные приложения).
- **Строгие compliance-требования** (WS-\* дают аудит и безопасность из коробки).

### Примеры

- Интеграция бэк-офиса телеком-оператора (биллинг Amdocs + CRM Siebel + provisioning).
- Банковская интеграционная шина (core-banking + интернет-банк + партнёрские API).
- Система межведомственного обмена (СМЭВ в России, X-Road в Эстонии).

### SOA vs Микросервисы

| Аспект | SOA | Микросервисы |
|--------|-----|--------------|
| Коммуникация | ESB (smart pipe) | «Dumb pipes, smart endpoints» |
| Протокол | SOAP/XML | REST/JSON, gRPC |
| Данные | Каноническая модель предприятия | Каждый сервис со своей БД |
| Размер сервиса | Крупный (бизнес-функция) | Мелкий (bounded context) |
| Управление | Централизованное | Децентрализованное |

## 9. Микросервисная архитектура (Microservices)

### Описание

Приложение собирается из **независимо деплоимых**, слабосвязанных сервисов. Каждый сервис реализует конкретную бизнес-возможность и **владеет собственными данными**.

```
                    ┌──────────┐
                    │  API GW  │  ← Единая точка входа, аутентификация, роутинг
                    └────┬─────┘
          ┌──────────────┼──────────────┐
          │              │              │
    ┌─────┴─────┐  ┌─────┴─────┐  ┌─────┴─────┐
    │  Service  │  │  Service  │  │  Service  │
    │  Orders   │  │  Payment  │  │  Catalog  │
    │  ┌─────┐  │  │  ┌─────┐  │  │  ┌─────┐  │
    │  │ DB  │  │  │  │ DB  │  │  │  │ DB  │  │
    │  └─────┘  │  │  └─────┘  │  │  └─────┘  │
    └───────────┘  └───────────┘  └───────────┘
          │              │              │
          └──────────────┼──────────────┘
                         │
                  ┌──────┴──────┐
                  │  Message    │  ← Асинхронная коммуникация (Kafka, RabbitMQ)
                  │  Broker     │
                  └─────────────┘
```

### Характеристики

| Характеристика | Описание |
|----------------|----------|
| **Автономия** | Каждый сервис — отдельный артефакт, может быть на своём языке/стеке |
| **Database per Service** | Прямой доступ к БД другого сервиса запрещён |
| **Bounded Context** | Каждый сервис ответственен за один bounded context (DDD) |
| **Smart endpoints, dumb pipes** | Логика в сервисах, коммуникация через простые протоколы |
| **Децентрализованное управление** | Нет центрального оркестратора |

### Коммуникационные паттерны

**Синхронные:** REST/HTTP + JSON, gRPC, GraphQL Federation.  
**Асинхронные:** Message Broker (Kafka, RabbitMQ, AWS SQS/SNS).

### Паттерны отказоустойчивости

| Паттерн | Назначение |
|---------|------------|
| **Circuit Breaker** | При ошибках «разрывает цепь», давая сервису восстановиться |
| **Retry** | Повтор при transient-ошибках |
| **Bulkhead** | Изоляция ресурсов (thread pools) |
| **Timeout** | Ограничение ожидания |
| **Fallback** | Дефолтное поведение при недоступности (кэш, стаб) |

### Стратегии декомпозиции

Как разбить монолит на микросервисы — один из самых сложных вопросов. Основные подходы:

| Стратегия | Описание | Пример |
|-----------|----------|--------|
| **По бизнес-возможностям (Business Capability)** | Сервис = законченная бизнес-функция | Orders, Payment, Shipping, Catalog |
| **По Bounded Context (DDD)** | Сервис = один Bounded Context со своей ubiquitous-моделью | Customer Context, Order Context, Billing Context |
| **По subdomain (DDD)** | Core, Supporting, Generic subdomains — разный приоритет | Core — сами, Generic — готовое решение |
| **По типу нагрузки** | Чтение vs запись, OLTP vs OLAP, batch vs real-time | CQRS: Command Service + Query Service |
| **По rate of change** | Часто меняющиеся части vs стабильные | Экспериментальный recommendation engine отдельно от стабильного catalog |

На практике чаще всего комбинируют **Business Capability + Bounded Context** — это даёт и бизнес-ориентированность, и строгие границы данных.

### Плюсы

- **Независимое масштабирование** — масштабируется только тот сервис, что под нагрузкой.
- **Независимые деплои** — команды релизятся по своему расписанию.
- **Гетерогенность технологий** — аналитика на Python, high-load на Go, остальное на Kotlin.
- **Отказоустойчивость** — падение одного сервиса не валит систему.
- **Быстрый CI/CD** — сборка одного сервиса = минуты, а не часы.

### Минусы

- **Распределённая сложность.** Сеть ненадёжна: таймауты, дубликаты, частичные сбои.
- **Сложность отладки.** Трассировка через 10 сервисов требует Jaeger/Zipkin.
- **Eventual Consistency.** Вместо ACID — саги и компенсирующие транзакции.
- **Инфраструктурная сложность.** Kubernetes, service mesh, мониторинг, логирование — нужна Platform Engineering команда.
- **Накладные расходы на сеть.** Каждый вызов добавляет 5–20ms + сериализацию.

### Когда использовать

- **5+ команд**, работающих над одним продуктом.
- **>1000 RPS** с разными профилями нагрузки на разные компоненты.
- **Сложный продукт с естественными доменными границами** (e-commerce, fintech).

### Когда НЕ использовать

- **Стартап на 3 разработчиков.**
- **Простой CRUD.**
- **Отсутствие DevOps-культуры.**

### Примеры

- **Netflix** — ~1000 микросервисов для стриминга.
- **Uber** — сервисы для ride-hailing, payments, driver matching.
- **Spotify** — бэкенд для рекомендаций, плейлистов, социальных функций.

## 10. Событийно-ориентированная архитектура (Event-Driven)

### Описание

В EDA компоненты общаются через **события** — факты, произошедшие в прошлом. Продюсер публикует событие, не зная, кто его потребит. Консьюмеры реагируют асинхронно.

**Ключевая идея:** вместо императивного «сделай X» — реактивное «я сделал X, решайте сами».

```
  ┌──────────┐   Event    ┌──────────────┐
  │ Producer │ ─────────→ │   Message    │ ─┬─→ Consumer A
  │(Orders)  │            │   Broker     │  │
  └──────────┘            │  (Kafka)     │  └─→ Consumer B
                          └──────┬───────┘
                                 └───→ Consumer C (аналитика, аудит, уведомления)
```

### Модели доставки

| Модель | Суть | Инструменты |
|--------|------|-------------|
| **Pub/Sub** | Продюсер → Топик → все подписчики получают копию | Kafka, Google Pub/Sub |
| **Event Streaming** | События как append-only лог, консьюмеры со своим offset'ом | Kafka |
| **Message Queue** | Сообщение доставляется одному консьюмеру из пула | RabbitMQ, AWS SQS |

### Choreography vs Orchestration

- **Хореография** — нет координатора. Сервисы слушают события и реагируют.  
  `Order → [OrderPlaced] → Payment → [PaymentDone] → Warehouse`  
  _Плюсы:_ слабая связь. _Минусы:_ трудно понять end-to-end flow.
- **Оркестрация** — центральный Saga Orchestrator управляет процессом.  
  _Плюсы:_ явный flow. _Минусы:_ дополнительная связность.

### Плюсы

- **Слабая связанность.** Продюсер не знает о консьюмерах. Новые реакции добавляются без изменения продюсера.
- **Масштабируемость.** Продюсеры и консьюмеры масштабируются независимо.
- **Отказоустойчивость.** События в брокере; консьюмер падает и продолжает с того же offset'а.
- **Историчность.** Полная история событий — можно переиграть для аналитики/отладки.

### Минусы

- **Сложность отладки.** События нелинейны, многокомпонентны.
- **Eventual Consistency.** Данные между сервисами согласуются не мгновенно.
- **Версионирование схемы событий** — Avro/Protobuf + Schema Registry.
- **Дупликация** — at-least-once → нужна идемпотентность.

### Когда использовать

- **Асинхронные процессы** (заказ → оплата → доставка → уведомление).
- **IoT/телеметрия, clickstream, логирование.**
- **Микросервисы, которым нужна слабая связь.**

### Примеры

- **Uber** — триллионы событий в день через Kafka (поездки, оплаты, GPS).
- **Банковский фрод-мониторинг** — транзакции → real-time detection → алерты.
- **E-commerce** — события синхронизируют каталог, склад, биллинг.

## 11. Потоковая архитектура (Pipeline / Streaming)

### Описание

Обработка данных как последовательность шагов (фильтров), соединённых каналами (трубами). Каждый шаг получает данные, трансформирует и передаёт дальше.

```
[Source] → [Extract] → [Transform] → [Validate] → [Enrich] → [Load] → [Destination]
```

### Варианты

| Тип | Описание | Инструменты |
|-----|----------|-------------|
| **Batch Pipeline** | Периодическая обработка (ночной ETL) | Apache Spark, Spring Batch |
| **Streaming Pipeline** | Обработка событий в реальном времени | Kafka Streams, Apache Flink |
| **Workflow Pipeline** | Долгоиграющие процессы с состоянием, retry, компенсацией | Temporal, AWS Step Functions |

### Плюсы

- **Модульность.** Каждый шаг независим и переиспользуем.
- **Масштабируемость.** Узкое место распараллеливается горизонтально.
- **Back-pressure.** Streaming-фреймворки притормаживают продюсера, если консьюмер не успевает.

### Минусы

- **Сложность отладки.** Ошибка на шаге 5, а источник на шаге 1.
- **Латентность.** Пакетная обработка — не real-time; стриминг — задержки на каждом шаге.
- **Exactly-once семантика** сложна в реализации.

### Когда использовать

- **ETL/ELT** — извлечение, трансформация, загрузка в DWH.
- **Потоковые данные** — IoT, финансы, логи.
- **CI/CD пайплайны** — checkout → build → test → deploy.
- **Data Science** — сбор → очистка → обучение → деплой модели.

### Примеры

- Система фрод-мониторинга: транзакции → обогащение → фичи → ML-модель → алерт.
- Сбор логов: агенты → Kafka → нормализация → Elasticsearch → Grafana.

## 12. Пространственная архитектура (Space-Based)

### Описание

Устранение БД как узкого горла: данные распределяются по **processing units (PU)** в оперативной памяти. БД синхронизируется асинхронно — она вне горячего пути.

```
    ┌─────────────────────────────┐
    │       Load Balancer         │
    └─────┬───────┬───────┬───────┘
          │       │       │
    ┌─────┴─┐ ┌───┴───┐ ┌─┴───────┐
    │ PU-1  │ │ PU-2  │ │ PU-3    │
    │(App)  │ │(App)  │ │(App)    │  ← Приложение + данные в RAM
    │(Data) │ │(Data) │ │(Data)   │
    └──┬────┘ └───┬───┘ └───┬─────┘
       │          │         │
       └──────────┼─────────┘
                  │
    ┌─────────────┴──────────────┐
    │   Async Persistence (БД)   │  ← БД вне горячего пути
    └────────────────────────────┘
```

**Data Grid:** Hazelcast, Apache Ignite, Oracle Coherence — распределённое хранение в памяти.

### Плюсы

- **Экстремальная производительность** — микросекундные задержки (данные в RAM).
- **Линейное масштабирование** — добавил узел = добавил пропускную способность.
- **Отказоустойчивость** через репликацию между PU.

### Минусы

- **Сложность настройки** data grid (шардирование, split-brain).
- **Дорого** — всё в RAM.
- **Риск потери данных** при сбое до асинхронной записи в БД.

### Когда использовать

- **Высокочастотный трейдинг (HFT)** — микросекундные задержки.
- **ММО-игры** — состояние игроков и мира в памяти.
- **Аукционы, системы бронирования** — реальное время без ожидания БД.

### Примеры

- Трейдинговая платформа — состояние рынка в памяти.
- Система бронирования авиабилетов — доступность мест в RAM.

## 13. Серверлесс архитектура (Serverless)

### Описание

Разработчик не управляет серверами. Код выполняется в ответ на события, облачный провайдер динамически аллоцирует ресурсы. **Оплата за фактическое время исполнения.**

```
    ┌────────┐   HTTP    ┌──────────────┐
    │ Client │ ────────→ │  API Gateway │
    └────────┘           └──┬──┬──┬────┘
                            │  │  │
                   ┌────────┘  │  └────────┐
                   │           │           │
            ┌──────┴────┐ ┌───┴────┐ ┌────┴─────┐
            │  Lambda   │ │ Lambda │ │  Lambda  │
            │  Auth     │ │ Orders │ │  Catalog │
            └───────────┘ └────────┘ └──────────┘
```

### Модели

- **FaaS** (Function as a Service): AWS Lambda, Azure Functions, GCP Cloud Functions
- **BaaS** (Backend as a Service): Firebase, Auth0, Supabase

### Плюсы

- **Нулевое управление серверами.** Нет ОС для обновления, серверов для патча.
- **Авто-масштабирование от 0 до тысяч** (с cold start).
- **Оплата за использование** — нет затрат на idle.
- **Максимальный time-to-market** — загрузил функцию = продукт в проде.

### Минусы

- **Cold start** — сотни мс/секунды задержки при первом вызове (особенно Java/.NET).
- **Лимиты.** AWS Lambda — до 15 мин выполнения, до 10 GB памяти; Azure Functions — 10 мин (Consumption) / неограниченно (Premium); GCP Cloud Functions — 9 мин (gen 1) / 60 мин (gen 2). У каждого провайдера свои лимиты — проверяйте документацию.
- **Vendor lock-in** — у каждого облака свои API.
- **Дорого на постоянной высокой нагрузке** — reserved instances дешевле.

### Когда использовать

- **Эпизодическая/переменная нагрузка:** вебхуки, cron-задачи, обработка файлов.
- **MVP и прототипы** — бэкенд без DevOps за пару дней.
- **Не-critical-path операции:** ресайз картинок, отправка email, транскодирование.

### Примеры

- Обработка изображений: S3 триггерит Lambda → ресайз → сохранение.
- Чат-бот для Telegram на Lambda.
- Ежедневный ETL: Cloud Scheduler → Cloud Functions → экспорт в DWH.

## 14. Микрофронтенды (Micro Frontends)

### Описание

Микросервисный подход, применённый к фронтенду. Сайт собирается из независимых микро-приложений, каждое отвечает за бизнес-возможность и деплоится отдельной командой.

```
┌─────────────────────────────────────────────┐
│              Container App                   │
│  ┌─────────┐ ┌─────────┐ ┌──────────────┐   │
│  │ Team A  │ │ Team B  │ │ Team C       │   │
│  │ Catalog │ │ Cart    │ │ Checkout     │   │
│  │ (React) │ │ (Vue)   │ │ (Svelte)     │   │
│  └─────────┘ └─────────┘ └──────────────┘   │
│  Каждый микрофронтенд — отдельный артефакт   │
└─────────────────────────────────────────────┘
```

**Подходы к интеграции:**
- **Webpack Module Federation** (самый популярный)
- Single-SPA, qiankun
- Server-side composition (Tailor от Zalando)
- Web Components, iframes

### Плюсы

- **Независимые деплои.** Команды релизятся по своему расписанию.
- **Гетерогенность фреймворков** (React + Vue — ок при договорённостях).
- **Масштабирование команд** — 50+ фронтенд-разработчиков параллельно.

### Минусы

- **Сложность интеграции** (общие зависимости, роутинг, стейт).
- **Увеличенный размер бандла** (дублирование библиотек).
- **Стилизация** — конфликты CSS.
- **Разный UX** без единой дизайн-системы.

### Когда использовать

- **3+ фронтенд-команд** над одним продуктом.
- **Несколько бизнес-доменов в одном UI.**
- **Частые независимые деплои.**

### Примеры

- **Spotify Desktop** — каждая вкладка отдельный микрофронтенд.
- **IKEA** — каталог, корзина, чекаут разными командами.

## 15. Peer-to-Peer архитектура (P2P)

### Описание

Каждый узел — одновременно клиент и сервер. Нет центрального сервера; сеть децентрализована.

```
Централизованная:              P2P:
     ★ (Server)                    ★——★
    /|\                           /|\ /|\
   / | \                         ★—★—★—★
  ★  ★  ★                        \|/ \|/
                                   ★   ★
```

### Типы P2P сетей

| Тип | Описание | Примеры |
|-----|----------|---------|
| **Неструктурированные** | Поиск через flooding | Gnutella |
| **Структурированные (DHT)** | Каждый узел отвечает за часть ключей; поиск O(log N) | Chord, Kademlia (BitTorrent, Ethereum) |
| **Гибридные** | Центральный tracker для discovery, данные напрямую | BitTorrent |

### Плюсы

- **Отказоустойчивость** — нет единой точки отказа.
- **Масштабируемость** — каждый новый узел добавляет ресурсы.
- **Устойчивость к цензуре** — нет центрального контролирующего органа.

### Минусы

- **Сложность координации** — распределённый консенсус, репликация.
- **Безопасность** — зловредные узлы (Sybil attack, Eclipse attack).
- **Непредсказуемая доступность узлов** (churn).

### Когда использовать

- **Файлообмен** (BitTorrent, IPFS).
- **Блокчейн и криптовалюты** (Bitcoin, Ethereum).
- **Распределённые БД** (Cassandra — gossip протокол).
- **P2P CDN** (Streamroot).

### Примеры

- BitTorrent — крупнейшая P2P сеть обмена файлами.
- Bitcoin / Ethereum — P2P + консенсус (PoW/PoS).
- IPFS — распределённая файловая система (P2P + DHT).

## 16. Data Mesh (архитектура данных)

### Описание

**Data Mesh** (Zhamak Dehghani, 2019) — архитектурная парадигма управления данными в крупных организациях. В отличие от централизованного Data Lake, Data Mesh децентрализует владение данными по доменам — аналогично тому, как микросервисы децентрализуют прикладную логику.

```
┌──────────────────────────────────────────────────────────┐
│              Federated Governance                         │
│         (глобальные стандарты, политики, совместимость)    │
└──────┬──────────┬──────────┬──────────┬──────────────────┘
       │          │          │          │
┌──────┴──┐  ┌────┴───┐  ┌───┴────┐  ┌──┴───────┐
│ Domain  │  │ Domain │  │ Domain │  │  Domain   │
│  Team A │  │ Team B │  │ Team C │  │   Team D  │
│         │  │        │  │        │  │           │
│ • Owns  │  │• Owns  │  │• Owns  │  │ • Owns    │
│   data  │  │  data  │  │  data  │  │   data    │
│ • Publ. │  │• Publ. │  │• Publ. │  │ • Publ.   │
│   as    │  │  as    │  │  as    │  │   as      │
│  product│  │ product│  │ product│  │  product  │
└────┬────┘  └───┬────┘  └───┬────┘  └────┬──────┘
     │           │           │            │
     └───────────┴───────────┴────────────┘
                     │
            ┌────────┴──────────┐
            │ Self-Serve Data    │
            │ Infrastructure     │
            │ Platform           │
            └───────────────────┘
```

### Четыре принципа Data Mesh

| Принцип | Описание |
|---------|----------|
| **Domain-oriented ownership** | Владельцы домена (команды, создающие данные) владеют и предоставляют их — а не центральная data-команда |
| **Data as a Product** | Данные публикуются как продукт: discoverable, addressable, trustworthy, self-describing, interoperable, secure |
| **Self-serve data infrastructure** | Инфраструктурная платформа, позволяющая доменным командам публиковать и потреблять данные без центральной команды |
| **Federated governance** | Глобальные стандарты (схемы, качество, безопасность) при локальной автономии реализации |

### Связь с микросервисами

Data Mesh — это применение микросервисных принципов к данным: децентрализация, автономия, продуктное мышление. Микросервисы разбивают прикладную логику, Data Mesh — данные.

### Плюсы

- **Масштабируемость владения.** Нет bottleneck в виде центральной data-команды.
- **Близость данных к домену.** Данными владеют те, кто их понимает — доменные команды.
- **Качество данных.** Data Product подразумевает контракты, мониторинг, SLO.

### Минусы

- **Высокий организационный порог.** Требует зрелой инженерной культуры и сильной платформенной команды.
- **Дублирование данных.** Разные домены могут публиковать пересекающиеся данные.
- **Сложность федеративного управления.** Баланс между стандартизацией и автономией.

### Когда использовать

- **Крупная организация** с десятками доменных команд и огромным количеством данных.
- **Data Lake перестаёт справляться** — центральная команда стала бутылочным горлышком.
- **Уже есть микросервисы** — естественное продолжение децентрализации на данные.

### Примеры

- **Zalando** — один из пионеров Data Mesh в Европе.
- **Netflix** — децентрализованная платформа данных для аналитики и рекомендаций.
- **Intuit** — Data Mesh для финансовых данных с федеративным управлением.

## 17. Edge Computing (периферийные вычисления)

### Описание

Вычисления и обработка данных максимально приближены к источнику — на «границе» сети (edge), а не в централизованном облаке. Это снижает задержки, экономит трафик и позволяет работать офлайн.

```
               ┌────────────────────────────────────────┐
               │              Cloud / Core                │
               │  • Тяжёлая аналитика                     │
               │  • Долгосрочное хранение                 │
               │  • Глобальная координация               │
               └─────────────┬──────────────────────────┘
                             │ (редкая синхронизация)
          ┌──────────────────┼──────────────────┐
          │                  │                  │
   ┌──────┴──────┐   ┌───────┴──────┐   ┌──────┴──────┐
   │  Edge Node  │   │  Edge Node   │   │  Edge Node  │
   │ • Быстрый   │   │ • Быстрый    │   │ • Быстрый   │
   │   отклик    │   │   отклик     │   │   отклик    │
   │ • Лок.      │   │ • Лок.       │   │ • Лок.      │
   │   фильтрация│   │   агрегация  │   │   ML-инфер  │
   │ • Офлайн    │   │ • Офлайн     │   │ • Офлайн    │
   └──────┬──────┘   └───────┬──────┘   └──────┬──────┘
          │                  │                  │
    ┌─────┴─────┐      ┌─────┴─────┐      ┌─────┴─────┐
    │IoT-датчики│      │CDN-запросы│      │ 5G-базовая│
    │(фабрика)  │      │(пользоват)│      │  станция  │
    └───────────┘      └───────────┘      └───────────┘
```

### Варианты

| Тип | Описание | Примеры |
|-----|----------|---------|
| **CDN Edge** | Кэширование контента + serverless-функции на edge-узлах | Cloudflare Workers, Fastly Compute@Edge, AWS CloudFront Functions |
| **IoT Edge** | Обработка телеметрии на шлюзе/контроллере до отправки в облако | AWS IoT Greengrass, Azure IoT Edge |
| **5G MEC (Multi-access Edge)** | Вычисления на базовых станциях 5G для сверхнизких задержек | AWS Wavelength, GCP Mobile Edge Cloud |
| **On-premise Edge** | Вычисления на оборудовании заказчика для compliance/offline | Azure Stack Edge, AWS Outposts |

### Плюсы

- **Сверхнизкая задержка** (единицы миллисекунд вместо десятков/сотен до облака).
- **Экономия трафика** — в облако уходит только агрегированное/значимое.
- **Работа офлайн** — критично для производства, транспорта, ритейла.
- **Compliance/суверенитет** — данные обрабатываются локально, не покидая периметр.

### Минусы

- **Ограниченные ресурсы** — edge-оборудование слабее облачных серверов.
- **Гетерогенность** — десятки моделей устройств, разные процессорные архитектуры.
- **Сложность деплоя и обновлений** — тысячи edge-узлов с разной связностью.

### Когда использовать

- **IoT и Индустрия 4.0** — датчики, контроллеры, предиктивное обслуживание.
- **Автономные автомобили** — обработка лидара/камер в реальном времени.
- **Игровой стриминг, AR/VR** — минимальная задержка критична.
- **CDN с динамической логикой** — не только кэширование, но и вычисления на edge.

### Примеры

- Cloudflare Workers — JavaScript/WebAssembly на 300+ точках присутствия.
- AWS IoT Greengrass — ML-инференс на промышленных шлюзах.
- Netflix Open Connect — CDN-узлы в сетях провайдеров для стриминга.

## Заключение: как выбирать архитектуру

### Простой алгоритм

```
1. Оцените сложность домена
   ├── Простой CRUD → Монолит / Слоевая
   ├── Средняя сложность → Модульный монолит
   └── Сложный домен → Чистые архитектуры (гексагональная/clean) + DDD

2. Определите размер команды
   ├── 1–5 человек → Монолит
   ├── 5–20 человек → Модульный монолит
   ├── 20+ человек → Микросервисы
   └── 3+ фронтенд-команд → Микрофронтенды

3. Оцените требования к масштабированию
   ├── < 100 RPS → Монолит
   ├── 100–1000 RPS → Модульный монолит / Serverless
   ├── 1000+ RPS → Микросервисы
   ├── >100k RPS, low-latency → Space-Based
   └── IoT/геораспред. → Edge Computing

4. Оцените требования к time-to-market
   ├── Нужно вчера (MVP) → Монолит / Serverless
   └── Качество важнее скорости → Чистые архитектуры

5. Оцените бюджет и DevOps-зрелость
   ├── Нет DevOps → Монолит
   ├── Базовая автоматизация → Модульный монолит / Serverless
   └── Зрелая DevOps-культура → Микросервисы / Event-Driven / Data Mesh
```

### Эволюционный путь

```
Стартап             Рост                Scale-up             Enterprise
   │                  │                     │                    │
   ▼                  ▼                     ▼                    ▼
Монолит ────────→ Модульный ────────→ Микросервисы ──────→ Микросервисы
                  монолит                                    + Event-Driven
                                                             + CQRS
                                                             + Event Sourcing
                                                             + Data Mesh (данные)
```

Большинство успешных приложений останавливаются на этапе **модульного монолита** — и это нормально.

### Ключевые принципы

> **«Начинайте с монолита, но проектируйте его модульным. Вырастете — разберёте.»**  
> — Мартин Фаулер, паттерн «Monolith First»

> **«Архитектура — это то, что трудно изменить. Поэтому откладывайте архитектурные решения до последнего ответственного момента.»**  
> — Роберт Мартин, «Clean Architecture»

> **«Не микросервисы делают систему хорошей — хорошая архитектура делает систему хорошей.»**  
> — Саймон Браун

Не существует «лучшей» архитектуры. Есть архитектура, **адекватная контексту**: размеру команды, сложности домена, требованиям бизнеса, бюджету и технологической зрелости. Хороший архитектор знает все стили и выбирает ровно тот уровень сложности, который нужен сейчас — и позволит эволюционировать завтра.
