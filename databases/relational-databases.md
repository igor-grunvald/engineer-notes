## Введение

**Реляционная база данных (РСУБД)** — это система управления базами данных, основанная на реляционной модели, предложенной Эдгаром Коддом в 1970 году. Данные хранятся в таблицах (отношениях), состоящих из строк (кортежей) и столбцов (атрибутов). Для работы с данными используется язык SQL.

**Почему PostgreSQL?** Открытая РСУБД с широкой экосистемой, расширяемой архитектурой и строгим следованием стандартам SQL. Используется как в стартапах, так и в enterprise-решениях.

**Почему Oracle?** Enterprise-СУБД с возможностями, которых нет в других системах (RAC, Data Guard, продвинутое партицирование). Понимание Oracle полезно при работе в крупных корпорациях, где она исторически распространена.

## Часть I. Архитектура и внутреннее устройство

### 1.1. Физическая архитектура PostgreSQL: процессы и память

PostgreSQL использует модель «процесс на соединение» (process-per-connection) — в отличие от многопоточной модели MySQL. Каждый клиент получает выделенный серверный процесс.

```
┌──────────────────────────────────────────────────────────────┐
│                    PostgreSQL Server                          │
│                                                              │
│  ┌──────────────┐  ┌──────────────────────────────────────┐  │
│  │ postmaster   │  │        Shared Memory                  │  │
│  │  (главный    │  │  ┌─────────────────────────────────┐  │  │
│  │   процесс)   │  │  │  shared_buffers                 │  │  │
│  │              │  │  │  (буферный кэш страниц)         │  │  │
│  └──────┬───────┘  │  ├─────────────────────────────────┤  │  │
│         │          │  │  WAL buffers                    │  │  │
│  ┌──────┴───────┐  │  ├─────────────────────────────────┤  │  │
│  │ bgwriter     │  │  │  lock space (блокировки)        │  │  │
│  │ (фоновый     │  │  ├─────────────────────────────────┤  │  │
│  │  писатель)   │  │  │  pg_stat_activity / статистика  │  │  │
│  └──────────────┘  │  └─────────────────────────────────┘  │  │
│  ┌──────────────┐  └──────────────────────────────────────┘  │
│  │ wal writer   │                                            │
│  │ (писатель    │  ┌──────────────────────────────────────┐  │
│  │  WAL)        │  │     Backend Processes                 │  │
│  └──────────────┘  │  ┌─────────┐ ┌─────────┐ ┌────────┐  │  │
│  ┌──────────────┐  │  │backend 1│ │backend 2│ │backend N│  │  │
│  │ autovacuum   │  │  │(клиент 1)│ │(клиент 2)│ │(клиент N)│ │  │
│  │ workers      │  │  └─────────┘ └─────────┘ └────────┘  │  │
│  └──────────────┘  └──────────────────────────────────────┘  │
│  ┌──────────────┐                                            │
│  │ checkpointer │                                            │
│  │ (контрольные │                                            │
│  │  точки)      │                                            │
│  └──────────────┘                                            │
│  ┌──────────────┐                                            │
│  │ stats        │                                            │
│  │ collector    │                                            │
│  └──────────────┘                                            │
└──────────────────────────────────────────────────────────────┘
```

#### Основные процессы PostgreSQL

| Процесс | Назначение |
|---------|------------|
| **postmaster** | Главный процесс. Слушает входящие соединения, порождает backend-процессы. Если он упал — база лежит. |
| **backend process** | По одному на каждое клиентское соединение. Выполняет SQL-запросы, работает с буферным кэшем. |
| **bgwriter** (Background Writer) | Сбрасывает грязные страницы из `shared_buffers` на диск **до** того, как они понадобятся backend'ам. Снижает latency запросов. |
| **wal writer** | Сбрасывает WAL-записи из WAL-буфера на диск. Работает либо по таймауту (`wal_writer_delay`, по умолчанию 200 мс), либо при заполнении буфера. |
| **checkpointer** | Выполняет контрольные точки (checkpoint): сбрасывает **все** грязные страницы на диск и обновляет заголовок контрольной точки. |
| **autovacuum workers** | Автоматически запускает VACUUM и ANALYZE. Может быть несколько параллельных workers (`autovacuum_max_workers`). |
| **autovacuum launcher** | Планирует запуск autovacuum workers для таблиц, которые в этом нуждаются. |
| **WAL sender / WAL receiver** | Используются при потоковой репликации. Sender на мастере отправляет WAL, receiver на реплике принимает. |
| **stats collector** | Собирает статистику о работе сервера (pg_stat_* представления). |

#### Разделяемая память (Shared Memory)

- **shared_buffers** — основной буферный кэш. По умолчанию 128 МБ. На выделенном сервере обычно **25% RAM**.
- **WAL buffers** — буфер для WAL-записей перед сбросом на диск (`wal_buffers`, по умолчанию 1/32 от shared_buffers).
- **lock space** — разделяемая память для хранения блокировок.
- **pg_stat_activity** и другие статистические структуры.

#### Почему «процесс на соединение», а не потоки?

Историческое решение (PostgreSQL родом из академической среды 80-х, где fork был прост и понятен). Преимущества:
- Изоляция: крах одного backend'а не роняет остальных.
- Простота отладки.
- Отсутствие проблем с thread-safety в расширениях.

Недостаток: **дороговизна** создания соединения и потребление памяти. Именно поэтому всегда используют **пул соединений** (PgBouncer, Pgpool-II).

### 1.2. Страницы, кортежи и TOAST

#### Страница (Page / Block)

В PostgreSQL данные хранятся в виде **страниц** фиксированного размера — **8 КБ** по умолчанию (настраивается только при компиляции, `BLCKSZ`). Каждая таблица и индекс состоят из страниц.

```
┌────────────────────────────────────────────────────────────┐
│                  Страница (8 КБ)                            │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ PageHeaderData (24 байта)                             │  │
│  │  - LSN (Log Sequence Number) — позиция в WAL          │  │
│  │  - Контрольная сумма (если включена)                  │  │
│  │  - Флаги                                               │  │
│  │  - Указатель на начало свободного пространства         │  │
│  └──────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ Массив указателей на элементы (ItemIdData)            │  │
│  │  - Каждый по 4 байта                                  │  │
│  │  - Указывает на смещение кортежа внутри страницы       │  │
│  │  - Растёт сверху вниз                                 │  │
│  └──────────────────────────────────────────────────────┘  │
│                                                            │
│                      СВОБОДНОЕ ПРОСТРАНСТВО                 │
│                                                            │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ Данные кортежей (Tuple Data)                          │  │
│  │  - Сами строки таблицы                                │  │
│  │  - Растут снизу вверх                                 │  │
│  └──────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ Специальное пространство (Special Space)              │  │
│  │  - Для индексов (например, B-Tree метаданные)         │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────┘
```

#### Кортеж (Tuple) — строка таблицы

В PostgreSQL каждая строка хранится как **кортеж** на странице:

```
┌──────────────────────────────────────────────────────────┐
│                   Структура кортежа                        │
│  ┌────────────────────────────────────────────────────┐  │
│  │ HeapTupleHeaderData (27 байт)                       │  │
│  │  - xmin: ID транзакции, создавшей кортеж (4 байта)  │  │
│  │  - xmax: ID транзакции, удалившей/обновившей (4 б.) │  │
│  │  - cid (command id): номер команды в транзакции      │  │
│  │  - ctid: физическое расположение (block, offset)     │  │
│  │  - infomask: флаги (HAS_NULLS, HOT_UPDATED, ...)    │  │
│  │  - t_hoff: смещение до пользовательских данных       │  │
│  │    (учитывает NULL bitmap и alignment padding)       │  │
│  └────────────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────────────┐  │
│  │ NULL bitmap (опционально, если есть NULL столбцы)   │  │
│  └────────────────────────────────────────────────────┘  │
│  ┌────────────────────────────────────────────────────┐  │
│  │ Данные столбцов (сами значения)                      │  │
│  └────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────┘
```

Ключевые поля заголовка — `xmin` и `xmax` — это основа MVCC (см. раздел 1.3). Они определяют **видимость** кортежа для разных транзакций.

#### TOAST (The Oversized-Attribute Storage Technique)

Страница 8 КБ не может вместить строку больше ~2 КБ (потому что нужно место для других строк). Для больших значений (TEXT, BYTEA, JSONB и др.) PostgreSQL прозрачно применяет TOAST:

1. Значение сжимается (по умолчанию алгоритм pglz, можно LZ4).
2. Если после сжатия всё ещё велико — разбивается на чанки и хранится в отдельной TOAST-таблице (обычно `pg_toast.pg_toast_<OID>`).
3. В основной таблице хранится TOAST-указатель.

**Стратегии TOAST (на уровне столбца):**
- `PLAIN` — не применять TOAST.
- `EXTENDED` (по умолчанию) — сжать, потом вынести если нужно.
- `EXTERNAL` — вынести без сжатия (быстрее доступ, но больше места).
- `MAIN` — сжать, вынести в крайнем случае.

### 1.3. MVCC — Multiversion Concurrency Control

**MVCC** — это механизм, обеспечивающий конкурентный доступ к данным без избыточных блокировок. Вместо того чтобы блокировать строку при чтении, каждая транзакция видит **свой снимок (snapshot)** данных на момент начала транзакции.

#### Как это работает в PostgreSQL

В отличие от многих других СУБД (Oracle, MySQL/InnoDB), PostgreSQL **не имеет отдельного сегмента отката (rollback segment / undo log)**. Вместо этого он хранит **несколько версий строки прямо в таблице**.

```sql
-- Создаём таблицу
CREATE TABLE users (id SERIAL PRIMARY KEY, name TEXT, salary INT);

-- Вставляем строку транзакцией с XID=100
INSERT INTO users VALUES (1, 'Alice', 50000);

-- В таблице:
-- xmin=100 | xmax=0 | id=1 | name='Alice' | salary=50000
```

Когда транзакция (XID=100) вставляет строку, `xmin` устанавливается в 100, а `xmax` = 0 (означает «не удалена»).

```sql
-- Транзакция XID=200 обновляет зарплату
UPDATE users SET salary = 60000 WHERE id = 1;

-- Теперь в таблице ДВЕ версии строки:
-- xmin=100 | xmax=200 | id=1 | name='Alice' | salary=50000  (старая)
-- xmin=200 | xmax=0   | id=1 | name='Alice' | salary=60000  (новая)
```

**UPDATE в PostgreSQL = INSERT новой версии + пометить старую как устаревшую** (установить xmax). Это очень важно для понимания:
- UPDATE никогда не обновляет строку на месте!
- При интенсивных UPDATE таблица «раздувается» (bloat) — нужен VACUUM.

#### Снимок данных (Snapshot)

Каждая транзакция получает снимок при старте, который содержит:

```
Snapshot {
    xmin:      минимальный XID, который ещё активен (не завершён)
    xmax:      следующий XID, который будет выдан (верхняя граница)
    xip[]:     список активных XID'ов (тех, которые выполняются прямо сейчас)
}
```

**Правила видимости кортежа:**

Кортеж видим транзакции с XID = txid, если:

| Условие | Значение |
|---------|----------|
| `xmin < txid` и `xmin` не в `xip[]` | ✅ Видим (кортеж создан завершённой транзакцией) |
| `xmin = txid` | ✅ Видим (собственный кортеж транзакции) |
| `xmin > txid` или `xmin ∈ xip[]` | ❌ Не видим (создан будущей или активной транзакцией) |
| `xmax = 0` | Кортеж не удалён |
| `xmax ≠ 0, xmax < txid, xmax ∉ xip[]` | ❌ Удалённая версия (не видна) |
| `xmax ≠ 0, xmax ∈ xip[]` | ✅ Удалявшая транзакция ещё выполняется — кортеж видим |
| `xmax ≠ 0, xmax > txid` | ✅ Удалившая транзакция в будущем — кортеж пока видим |

#### Последствия MVCC в PostgreSQL

1. **Читатели не блокируют писателей, писатели не блокируют читателей.** Это свойство MVCC — одно из главных преимуществ PostgreSQL перед СУБД с блокировочными моделями.
2. **UPDATE создаёт мёртвые кортежи.** Требуется VACUUM для очистки.
3. **Длинные транзакции задерживают очистку.** Если транзакция висит часами, все мёртвые кортежи, созданные после её старта, не могут быть удалены.
4. **Нет undo-лога** — в отличие от Oracle и MySQL.

#### Сравнение с Oracle

| Аспект | PostgreSQL | Oracle |
|--------|-----------|--------|
| Механизм | Копирование строки в таблице | Undo-сегменты (rollback segments) |
| Старая версия | В основной таблице (как мёртвый кортеж) | В undo-табличном пространстве |
| Очистка | VACUUM | Автоматическая перезапись undo |
| Чтение старой версии | По xmin/xmax на странице | Реконструкция из undo |
| ORA-01555 | Отсутствует | Может возникнуть при нехватке undo |
| Раздувание таблиц | Присутствует (bloat) | Меньше подвержено |

### 1.4. WAL — Write-Ahead Log

**WAL** — это журнал предзаписи, фундаментальная технология, обеспечивающая:
- Восстановление после сбоя (crash recovery).
- Потоковую репликацию (streaming replication).
- Point-in-Time Recovery (PITR).

#### Принцип работы

1. **Прежде чем изменить страницу данных, запись об изменении пишется в WAL.**
2. WAL пишется **последовательно** на диск — это очень быстро (sequential write).
3. Изменённые страницы в `shared_buffers` сбрасываются на диск **позже** (checkpointer и bgwriter).
4. При сбое PostgreSQL читает WAL с последней контрольной точки и **повторяет** изменения (REDO).

```
Время ──────────────────────────────────────────────────────>

  BEGIN  ─── UPDATE ─── INSERT ─── COMMIT
    │          │          │          │
    ▼          ▼          ▼          ▼
  ┌───────────────────────────────────────────┐
  │              WAL (pg_wal/)                 │
  │  [запись] [запись] [запись] [COMMIT зап.] │
  └───────────────────────────────────────────┘
                    │
                    ▼ (асинхронно, позже)
  ┌───────────────────────────────────────────┐
  │         Страницы данных на диске           │
  │        (pgdata/base/<OID>/<relfilenode>)   │
  └───────────────────────────────────────────┘
```

#### Структура WAL

WAL состоит из **сегментов** (файлы по 16 МБ, настраивается через `--wal-segsize` при initdb). Сегменты имеют имена вида `000000010000000000000001`.

Каждая запись в WAL идентифицируется **LSN** (Log Sequence Number) — 64-битное смещение в WAL.

```
LSN = (номер_сегмента_в_старших_битах << 32) | смещение_внутри_сегмента
```

#### Уровни защиты WAL (`wal_level`)

| Уровень | Описание | Для чего |
|---------|----------|----------|
| `minimal` | Только для crash recovery. Минимум записей. | Тестовые стенды |
| `replica` (по умолчанию) | Достаточно для репликации + PITR | Репликация, бэкапы |
| `logical` | + логическая репликация (доп. информация о строках) | Логическая репликация, CDC |

#### Синхронизация WAL (`wal_sync_method`)

Параметр `wal_sync_method` определяет, как WAL сбрасывается на диск:
- `fsync` (по умолчанию в Linux) — системный вызов `fsync()`.
- `fdatasync` — быстрее, но не обновляет метаданные (размер файла и т.п.).
- `open_datasync` — флаг `O_DSYNC` при открытии.

#### Контрольная точка (Checkpoint)

**Checkpoint** — это момент, когда все грязные страницы из `shared_buffers` сбрасываются на диск. После checkpoint'а WAL до этого момента можно удалить (если не нужен для репликации).

Параметры:
- `checkpoint_timeout` — интервал между контрольными точками (по умолчанию 5 мин).
- `max_wal_size` — максимальный размер WAL до принудительного checkpoint'а (по умолчанию 1 ГБ).
- `checkpoint_completion_target` — доля времени между checkpoint'ами, за которую надо **растянуть** сброс страниц (по умолчанию 0.9 = 90% времени). Это сглаживает нагрузку на дисковую подсистему.

### 1.5. VACUUM, AUTOVACUUM и заморозка

**VACUUM** — одна из особенностей PostgreSQL (и одна из самых сложных для новичков). Так как UPDATE создаёт новые версии строк, мёртвые кортежи нужно **физически удалять**.

#### Что делает VACUUM

1. **Удаляет мёртвые кортежи** (те, чей xmax < любой активный XID) и освобождает место на странице.
2. **Обновляет карту видимости (Visibility Map)** — битовая карта, показывающая страницы без мёртвых кортежей. Ускоряет Index-Only Scan.
3. **Обновляет Free Space Map (FSM)** — карта свободного места для вставки новых строк.
4. **Замораживает старые XID'ы** (freeze) — предотвращает wraparound (см. раздел 2.7).
5. **VACUUM FULL** дополнительно физически сжимает таблицу (EXCLUSIVE LOCK!).

#### VACUUM vs VACUUM FULL

| | VACUUM | VACUUM FULL |
|---|---|---|
| Блокировка | Минимальная (SHARE UPDATE EXCLUSIVE) | EXCLUSIVE (полная блокировка) |
| Освобождение места ОС | ❌ Не возвращает (остаётся в таблице для будущих INSERT) | ✅ Возвращает |
| Производительность | Быстро, онлайн | Медленно, блокирует таблицу |
| Дефрагментация | Частичная | Полная |
| Использовать | Постоянно (AUTOVACUUM) | Только при острой необходимости в обслуживающее окно |

#### AUTOVACUUM

PostgreSQL автоматически запускает VACUUM, когда количество мёртвых кортежей превышает порог:

```ini
# postgresql.conf
autovacuum = on
autovacuum_max_workers = 3                    # количество параллельных workers
autovacuum_naptime = 1min                     # интервал проверки таблиц
autovacuum_vacuum_threshold = 50              # минимальное число мёртвых кортежей
autovacuum_vacuum_scale_factor = 0.2          # + 20% от размера таблицы
autovacuum_vacuum_cost_limit = 200            # лимит «стоимости» VACUUM
autovacuum_vacuum_cost_delay = 2ms            # задержка при превышении лимита
```

**Формула:** autovacuum запускается, когда:
```
мёртвых_кортежей >= autovacuum_vacuum_threshold 
                    + autovacuum_vacuum_scale_factor * количество_строк
```

#### Заморозка (FREEZE)

XID в PostgreSQL — 32-битный (4 миллиарда значений). Когда XID приближается к переполнению (wraparound), старые XID'ы «замораживаются» — помечаются специальным значением `FrozenTransactionId`, которое **всегда старше** любого обычного XID. Подробно проблема Transaction ID Wraparound и параметры заморозки рассмотрены в [разделе 2.7](#27-transaction-id-wraparound).

### 1.6. Буферный кэш (shared_buffers) и работа с диском

#### Как работает shared_buffers

`shared_buffers` — это область разделяемой памяти, в которой хранятся кэшированные страницы таблиц и индексов.

```
Запрос (SELECT * FROM users WHERE id = 42)
  │
  ▼
Backend процесс смотрит в shared_buffers:
  │
  ├── страница в кэше (cache hit) → читаем → возвращаем результат
  │
  └── страницы нет (cache miss)
        │
        ├── выделяем свободный буфер в shared_buffers
        │   (если нет свободного — вытесняем по clock-sweep алгоритму)
        │
        ├── читаем страницу с диска в буфер (8 КБ)
        │
        └── читаем данные из буфера → возвращаем результат
```

#### Алгоритм вытеснения: Clock Sweep

В отличие от LRU (используемого в MySQL), PostgreSQL использует алгоритм **Clock Sweep** (разновидность «часов» / second chance):

1. Каждый буфер имеет счётчик использования (usage count, 0–5).
2. При обращении к буферу счётчик увеличивается (до 5).
3. При поиске свободного буфера сканер идёт по кругу и уменьшает счётчик на 1.
4. Буферы со счётчиком 0 становятся кандидатами на вытеснение.

#### Двойная буферизация

Важный нюанс: **ОС также кэширует файлы** (page cache). Поэтому одна и та же страница может находиться и в `shared_buffers`, и в page cache ОС. Это приводит к двойной буферизации и неэффективному использованию памяти.

**Решение:** использовать `direct I/O` (начиная с PG 16 через `debug_io_direct`, в PG 17+ — стабильный `io_direct = data`), или уменьшить `shared_buffers` и положиться на page cache ОС (для рабочих нагрузок, где данные в основном читаются).

#### Параметры управления сбросом

Параметр `backend_flush_after` (по умолчанию 0 = отключён) заставляет backend-процесс сбрасывать грязные страницы на диск после записи указанного объёма. Это предотвращает накопление слишком большого объёма грязных данных у одного backend'а, из-за которого checkpoint может вызвать резкий всплеск I/O:

```ini
backend_flush_after = 512kB   # сбрасывать после каждых 512 КБ грязных страниц
```

Аналогично, `bgwriter_flush_after` контролирует то же для background writer (по умолчанию 512 КБ).

#### Прогрев кэша: pg_prewarm

После перезапуска PostgreSQL `shared_buffers` пуст, и первые запросы идут на диск. Расширение **pg_prewarm** позволяет загрузить таблицы и индексы в буферный кэш заранее:

```sql
CREATE EXTENSION pg_prewarm;

-- Загрузить таблицу в кэш
SELECT pg_prewarm('users');

-- Загрузить только индексы
SELECT pg_prewarm('orders', 'main');       -- основная таблица
SELECT pg_prewarm('orders', 'fsm');        -- Free Space Map
SELECT pg_prewarm('orders', 'vm');         -- Visibility Map

-- Автоматически прогревать после рестарта
-- (добавить в shared_preload_libraries и настроить)
-- pg_prewarm.autoprewarm = true
-- Список блоков сохраняется в pg_autoprewarm.blocks
```

#### effective_cache_size

Этот параметр **ничего не выделяет** — он сообщает планировщику, сколько памяти примерно доступно для кэширования (ОС + shared_buffers). Планировщик использует это значение для оценки стоимости Index Scan vs Seq Scan.

```ini
effective_cache_size = 12GB  # 75% RAM на выделенном сервере
```

#### work_mem

Память, доступная **одной операции** (сортировка, хеш-таблица, bitmap). Не на соединение! Один сложный запрос может использовать несколько `work_mem`.

```ini
work_mem = 64MB  # Умеренно-консервативное значение
```

Если запрос включает 3 сортировки и 2 хеш-таблицы — он может использовать до `5 × work_mem` памяти. А если 100 одновременных соединений выполняют такие запросы — это `5 × 100 × work_mem` потенциального потребления.

### 1.7. Планировщик и оптимизатор запросов

#### Этапы выполнения запроса

```
SQL-текст (строка)
      │
      ▼
┌──────────────┐
│   Parser     │  Синтаксический разбор → дерево разбора (parse tree)
└──────┬───────┘
      ▼
┌──────────────┐
│  Analyzer    │  Семантический анализ (существуют ли таблицы? права?)
└──────┬───────┘  → дерево запроса (query tree)
      ▼
┌──────────────┐
│  Planner     │  Оптимизатор генерирует возможные планы, выбирает самый дешёвый
└──────┬───────┘  → plan tree (план выполнения)
      ▼
┌──────────────┐
│  Executor    │  Выполняет план, возвращает результат
└──────────────┘
```

#### Как оптимизатор оценивает стоимость (Cost Model)

PostgreSQL использует **cost-based optimizer** (CBO). Каждая операция имеет вычисляемую стоимость:

```sql
EXPLAIN SELECT * FROM users WHERE id = 42;
-- Seq Scan on users  (cost=0.00..25.00 rows=1 width=40)
--   Filter: (id = 42)
```

Параметры стоимости (в `postgresql.conf`):

| Параметр | По умолчанию | Смысл |
|----------|:---:|--------|
| `seq_page_cost` | 1.0 | Стоимость чтения одной страницы при последовательном сканировании |
| `random_page_cost` | 4.0 | Стоимость чтения страницы при произвольном доступе (индекс) |
| `cpu_tuple_cost` | 0.01 | Стоимость обработки одной строки в CPU |
| `cpu_index_tuple_cost` | 0.005 | Стоимость обработки одной индексной записи |
| `cpu_operator_cost` | 0.0025 | Стоимость выполнения одного оператора (WHERE, JOIN) |

**Для SSD** рекомендуется `random_page_cost = 1.0–1.5` (произвольный доступ почти не медленнее последовательного).

#### Генетический оптимизатор запросов (GEQO)

Когда в запросе много JOIN'ов (по умолчанию > 12 таблиц, `geqo_threshold`), полный перебор всех планов становится экспоненциально дорогим. Включается **генетический алгоритм** (GEQO), который находит «достаточно хороший» план, не перебирая все варианты.

#### Параллельное выполнение запросов

PostgreSQL (начиная с 9.6) может распараллеливать чтение и обработку данных. **Какие операции могут быть параллельными:**

| Операция | Параллельная версия |
|----------|---------------------|
| Sequential Scan | **Parallel Seq Scan** |
| Aggregate (COUNT, SUM, AVG, ...) | **Partial Aggregate → Gather → Finalize Aggregate** |
| Hash Join | **Parallel Hash Join** (внешняя таблица сканируется параллельно) |
| B-Tree Index Scan | **Parallel Index Scan** (только для Index-Only Scan, PG 11+) |
| Bitmap Heap Scan | **Parallel Bitmap Heap Scan** |
| CREATE INDEX | **Parallel CREATE INDEX** (отдельный параметр) |
| VACUUM | **Параллельный VACUUM** (несколько workers) |
| Append (UNION ALL по партициям) | **Parallel Append** |

**Что НЕ может быть параллельным:**
- CTE (до PG 12 они всегда материализуются последовательно).
- Запросы, изменяющие данные (INSERT/UPDATE/DELETE — только чтение может быть параллельным).
- Курсоры и запросы с FOR UPDATE/SHARE.
- Запросы внутри функций (если функция не помечена как `PARALLEL SAFE`).

**Ключевые параметры (уже упомянуты в 1.6, но важны здесь):**
```ini
max_parallel_workers_per_gather = 2   # workers на один Gather-узел
max_parallel_workers = 8              # всего параллельных workers на сервере
min_parallel_table_scan_size = 8MB    # минимальный размер таблицы для параллелизма
parallel_setup_cost = 1000.0          # стоимость запуска параллельного worker'а
parallel_tuple_cost = 0.1             # стоимость передачи строки между worker'ами
```

**Планировщик выбирает параллельный план, если:**
1. Размер таблицы > `min_parallel_table_scan_size`.
2. `max_parallel_workers_per_gather > 0`.
3. Есть свободные параллельные workers.
4. Стоимость параллельного плана ниже последовательного.

Для принудительного включения/отключения:
```sql
SET parallel_setup_cost = 0;  -- сделать параллелизм «дешевле» для планировщика
SET max_parallel_workers_per_gather = 0;  -- отключить параллелизм для сессии
```

## Часть II. Транзакции

### 2.1. ACID: четыре свойства транзакций

**Транзакция** — это логическая единица работы, которая должна быть выполнена полностью или не выполнена вовсе. Гарантируется четырьмя свойствами ACID:

| Свойство | Расшифровка | Что гарантируется |
|----------|------------|-------------------|
| **A**tomicity | Атомарность | Транзакция либо выполняется целиком, либо полностью откатывается. Нельзя частично. |
| **C**onsistency | Согласованность | Транзакция переводит БД из одного согласованного состояния в другое. Все ограничения (constraints) соблюдены. |
| **I**solation | Изолированность | Параллельные транзакции не должны влиять друг на друга. Каждая видит согласованное состояние. |
| **D**urability | Долговечность | Зафиксированные изменения не теряются даже при сбое питания. |

#### Как PostgreSQL обеспечивает ACID

- **Atomicity**: через MVCC — при ROLLBACK все изменения, сделанные с `xmin = XID`, становятся невидимы (xmax не 0, а устаревшие для всех).
- **Consistency**: ограничения (CHECK, FK, UNIQUE), которые проверяются в конце транзакции (или немедленно, зависит от `DEFERRABLE`).
- **Isolation**: MVCC + блокировки. Каждая транзакция видит снимок.
- **Durability**: WAL. COMMIT не возвращает управление, пока WAL-запись о коммите не сброшена на диск (`fsync`).

---

### 2.2. Уровни изоляции по SQL-стандарту

SQL-стандарт определяет 4 уровня изоляции, различающихся тем, какие феномены допускаются:

| Уровень изоляции | Dirty Read | Non-repeatable Read | Phantom Read | Serialization Anomaly |
|:-----------------|:----------:|:-------------------:|:------------:|:---------------------:|
| **Read Uncommitted** | ✅ Да | ✅ Да | ✅ Да | ✅ Да |
| **Read Committed** | ❌ Нет | ✅ Да | ✅ Да | ✅ Да |
| **Repeatable Read** | ❌ Нет | ❌ Нет | ✅ Да | ✅ Да |
| **Serializable** | ❌ Нет | ❌ Нет | ❌ Нет | ❌ Нет |

#### Важно: реальное поведение отличается от стандарта!

**PostgreSQL:**
- Read Uncommitted **эквивалентен** Read Committed (грязных чтений нет никогда!).
- Repeatable Read в PG **не допускает фантомов** (это сильнее стандарта!).
- Serializable реализован через **Serializable Snapshot Isolation (SSI)**.

**Oracle:**
- Read Committed — уровень по умолчанию и основной (Dirty Read невозможен).
- Serializable — есть, но реализован через блокировки, может давать больше конфликтов.
- Repeatable Read в Oracle **отсутствует как отдельный уровень!**

### 2.3. Феномены параллельного выполнения

#### 2.3.1. Dirty Read (грязное чтение)

**Транзакция читает незафиксированные (незакоммиченные) данные другой транзакции.**

```
Время  TX1                    TX2
  │    UPDATE balance = 0     
  │    WHERE user_id = 1;     
  │                           SELECT balance ← прочитал 0 (ГРЯЗНО!)
  │    ROLLBACK;              COMMIT; ← зафиксировал невалидные данные
  ▼
```

**В PostgreSQL невозможно.** Даже на уровне Read Uncommitted.

#### 2.3.2. Non-repeatable Read (неповторяемое чтение)

**В рамках одной транзакции два одинаковых SELECT возвращают разные значения одной строки.**

```
Время  TX1                          TX2
  │    SELECT balance FROM users    
  │    WHERE id = 1; -- результат: 100
  │                                 UPDATE users SET balance = 200 WHERE id = 1;
  │                                 COMMIT;
  │    SELECT balance FROM users
  │    WHERE id = 1; -- результат: 200 (ДРУГОЙ!)
  ▼
```

**Допускается:** Read Committed. **Запрещён:** Repeatable Read, Serializable.

#### 2.3.3. Phantom Read (фантомное чтение)

**В рамках одной транзакции два одинаковых SELECT возвращают разное количество строк (новые строки «фантомно» появляются).**

```
Время  TX1                          TX2
  │    SELECT COUNT(*) FROM users   
  │    WHERE age > 18; -- результат: 5
  │                                 INSERT INTO users (age) VALUES (25);
  │                                 COMMIT;
  │    SELECT COUNT(*) FROM users
  │    WHERE age > 18; -- результат: 6 (ФАНТОМ!)
  ▼
```

**Допускается:** Read Committed (в стандарте), **но PostgreSQL Repeatable Read его не допускает!**

#### 2.3.4. Serialization Anomaly (аномалия сериализации)

**Результат параллельного выполнения транзакций невозможно получить при любом последовательном выполнении.**

```
-- Учёт посещений врачей двумя администраторами
-- Таблица: doctors(id, name, on_duty)

Время  TX1                                    TX2
  │    -- Администратор A:
  │    -- Хочет убедиться, что хотя бы 1 врач  
  │    -- дежурит, если нет — назначает дежурным
  │    SELECT COUNT(*) FROM doctors
  │    WHERE on_duty = true; -- результат: 1
  │                                           SELECT COUNT(*) FROM doctors
  │                                           WHERE on_duty = true; -- результат: 1
  │                                           UPDATE doctors SET on_duty = false
  │                                           WHERE name = 'Smith'; -- снимает
  │                                           COMMIT;
  │    -- Видит что был 1 дежурный, 
  │    -- но это уже устарело!
  │    UPDATE doctors SET on_duty = true
  │    WHERE name = 'Jones'; -- назначает другого
  │    COMMIT;
  │    -- ИТОГО: 0 дежурных врачей! Сериализации не было бы при любой последовательности.
  ▼
```

**Serializable** уровень предотвращает такие аномалии.

### 2.4. Реализация изоляции в PostgreSQL

#### Read Committed (по умолчанию)

Каждый оператор в транзакции видит **свой** снимок: снимок делается в начале каждого SQL-оператора, а не в начале транзакции.

```sql
BEGIN; -- транзакция началась
-- В этот момент снимка ещё нет

SELECT * FROM users WHERE id = 1; -- снимок №1 сделан ПРЯМО СЕЙЧАС
-- Другая транзакция обновила users и закоммитилась

SELECT * FROM users WHERE id = 1; -- снимок №2 — видит обновление!
```

Это означает, что два одинаковых SELECT в одной транзакции могут вернуть **разные** данные. Это и есть допущение Non-repeatable Read.

#### Repeatable Read

**Один снимок на всю транзакцию** — делается при первом обращении к данным (первый SELECT/INSERT/UPDATE/DELETE).

```sql
BEGIN ISOLATION LEVEL REPEATABLE READ;
-- Снимок будет сделан при первом запросе к данным

SELECT * FROM users WHERE id = 1; -- снимок сделан, заморожен
-- Другая транзакция обновила и закоммитилась

SELECT * FROM users WHERE id = 1; -- ТОТ ЖЕ снимок, изменения не видны!
```

**Конфликт обновлений:** Если две параллельные транзакции в Repeatable Read обновят одну и ту же строку, вторая получит ошибку:

```
ERROR: could not serialize access due to concurrent update
```

#### Serializable (SSI — Serializable Snapshot Isolation)

PostgreSQL реализует **Serializable Snapshot Isolation** (начиная с версии 9.1). Это не блокировочный подход (как S2PL в традиционных СУБД), а оптимистический:

1. Транзакции работают как в Repeatable Read (снимок).
2. Система отслеживает **зависимости чтения-записи** между транзакциями (через SIREAD-блокировки).
3. При коммите проверяется, не возник ли цикл зависимостей.
4. Если цикл обнаружен — одна из транзакций откатывается с ошибкой сериализации:

```
ERROR: could not serialize access due to read/write dependencies among transactions
DETAIL: Reason code: Canceled on identification as a pivot, during commit attempt.
HINT: The transaction might succeed if retried.
```

**Важно:** при использовании Serializable клиент должен быть готов к повторным попыткам (retry loop)!

#### Сравнение изоляции в PostgreSQL

| Уровень | Снимок | Non-repeatable Read | Phantom Read | Конфликты |
|---------|--------|:-------------------:|:------------:|-----------|
| Read Committed | На каждый оператор | ✅ Да | ✅ Да | Минимальные |
| Repeatable Read | На транзакцию | ❌ Нет | ❌ Нет (сильнее стандарта!) | Ошибка при конкурентном UPDATE |
| Serializable | На транзакцию + отслеживание зависимостей | ❌ Нет | ❌ Нет | Ошибка сериализации (retry!) |

---

### 2.5. Реализация изоляции в Oracle

Oracle использует другой подход. В Oracle:

- **Read Committed** (по умолчанию): снимок на каждый оператор, **НЕТ грязных чтений**.
- **Serializable**: снимок на транзакцию (как PG Repeatable Read). Отличие: при конфликте обновлений Oracle выдаёт ошибку только при коммите:

```
ORA-08177: can't serialize access for this transaction
```

- **Read-Only**: как Serializable, но без возможности изменений.

Oracle не имеет отдельного уровня Repeatable Read. Чтобы получить поведение как PG Repeatable Read, нужно использовать `SET TRANSACTION ISOLATION LEVEL SERIALIZABLE`.

**Ключевое отличие:** в Oracle читатель НИКОГДА не ждёт писателя (даже при Serializable), потому что всегда использует undo для реконструкции старых версий. В PostgreSQL это тоже так, но за счёт хранения старых версий прямо в таблице.

### 2.6. Распределённые транзакции и Two-Phase Commit (2PC)

**Two-Phase Commit (2PC)** — протокол, позволяющий выполнить транзакцию, затрагивающую несколько независимых ресурсов (разные базы, брокеры сообщений).

#### Фазы 2PC

```
Координатор                        Участник A          Участник B
    │                                  │                   │
    │── PREPARE TRANSACTION ──────────►│                   │
    │── PREPARE TRANSACTION ───────────┼──────────────────►│
    │                                  │                   │
    │        ◄── READY ────────────────│                   │
    │        ◄────────────────────── READY ────────────────│
    │                                                   
    │  Все сказали READY — первая фаза завершена          
    │                                                   
    │── COMMIT PREPARED ──────────────►│                   
    │── COMMIT PREPARED ───────────────┼──────────────────►│
    │                                                   
    │        ◄── COMMITTED ────────────│                   
    │        ◄──────────────────── COMMITTED ──────────────│
    ▼
```

1. **Фаза 1 (Prepare / Voting):** координатор спрашивает всех участников, готовы ли они закоммитить. Каждый участник должен ответить READY (готов) или ABORT (отказ). Ответив READY, участник **уже не может самостоятельно откатить** транзакцию.
2. **Фаза 2 (Commit / Decision):** если все сказали READY — координатор отправляет COMMIT. Если хоть один сказал ABORT — всем ROLLBACK.

#### 2PC в PostgreSQL

```sql
-- Подготовка транзакции (первая фаза)
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
PREPARE TRANSACTION 'tx_accounts_1';

-- Транзакция теперь в состоянии "prepared" — видна в pg_prepared_xacts
SELECT * FROM pg_prepared_xacts;

-- Завершение (вторая фаза)
COMMIT PREPARED 'tx_accounts_1';
-- или
ROLLBACK PREPARED 'tx_accounts_1';
```

**Важно:** prepared-транзакции хранятся на диске (в `pg_twophase/`) и переживают перезапуск сервера. Если координатор упал после PREPARE, транзакции останутся в «подвешенном» состоянии (in-doubt transactions). Нужен мониторинг и ручное разрешение.

### 2.7. Transaction ID Wraparound

#### Проблема

XID в PostgreSQL — 32-битное целое без знака (4,294,967,296 значений, от 0 до 2³²-1). Значения 0, 1, 2 зарезервированы. Половина значений (~2 миллиарда) считается «прошлым», половина — «будущим» относительно данного XID.

Это **циклическое** пространство: достигнув максимума, XID начинает отсчёт с 3 (wraparound). Если старая транзакция не была заморожена до того, как круг завершился, она внезапно начнёт выглядеть как **будущая** (с точки зрения новых XID'ов). Это катастрофа: целые таблицы перестанут быть видны.

#### Решение: заморозка (FREEZE)

Когда строка заморожена, её `xmin` заменяется на `FrozenTransactionId` (значение 2). Frozen-строки видны всем транзакциям всегда.

```sql
-- Ручная заморозка (обычно выполняется автоматически VACUUM)
VACUUM FREEZE;

-- Агрессивный VACUUM (замораживает даже не очень старые строки)
VACUUM (FREEZE ON, FREEZE_MIN_AGE 0) mytable;
```

**Критические параметры:**
```ini
vacuum_freeze_min_age = 50000000        # минимальный возраст для заморозки
vacuum_freeze_table_age = 150000000     # возраст таблицы для агрессивного VACUUM
autovacuum_freeze_max_age = 200000000   # если таблица старше — форсированный autovacuum
```

**Правило:** если `age(datfrozenxid)` приближается к 1 миллиарду — у вас проблемы с autovacuum.

### 2.8. Savepoints

Savepoint — точка внутри транзакции, до которой можно откатиться, не откатывая всю транзакцию.

```sql
BEGIN;
    INSERT INTO users VALUES (1, 'Alice');
    SAVEPOINT sp1;
    
    INSERT INTO users VALUES (2, 'Bob');   -- Ошибка в Bob? 
    -- Откат до sp1: Alice остаётся, Bob удаляется
    ROLLBACK TO sp1;
    
    INSERT INTO users VALUES (3, 'Charlie');
COMMIT; -- Остаются Alice и Charlie
```

**Внутренне:** savepoint — это маркер в цепочке команд транзакции. При `ROLLBACK TO sp` команды после маркера отбрасываются. При этом savepoint'ы, созданные после sp, также удаляются.

Savepoint'ы **не освобождают блокировки**, удерживаемые с момента savepoint'а!

### 2.9. Практические рекомендации по транзакциям

1. **По умолчанию используйте Read Committed.** Это уровень по умолчанию по объективным причинам — он даёт наилучший баланс производительности и корректности для большинства OLTP-нагрузок.

2. **Repeatable Read — для отчётов.** Когда нужно, чтобы данные в рамках одного отчёта были консистентны (например, несколько SELECT'ов для построения финансового отчёта).

3. **Serializable — только когда действительно нужно.** Готовьте retry-logic в приложении.

4. **Транзакции должны быть короткими.** Чем длиннее транзакция, тем:
   - Больше мёртвых кортежей накапливается (мешает VACUUM).
   - Дольше удерживаются блокировки.
   - Выше риск дедлоков и конфликтов сериализации.

5. **Не делайте SELECT перед UPDATE без причины.** Это не даёт «защиты» в MVCC-мире и только замедляет.

6. **Используйте SELECT ... FOR UPDATE** когда реально нужно заблокировать строку для последующего обновления (пессимистическая блокировка).

7. **Осторожно с длительными idle-in-transaction соединениями.** Настройте `idle_in_transaction_session_timeout`.

```sql
-- Найти долгие незавершённые транзакции
SELECT pid, now() - xact_start AS duration, query, state
FROM pg_stat_activity
WHERE state = 'idle in transaction'
  AND xact_start < now() - interval '5 minutes';
```

## Часть III. Блокировки

### 3.1. Типы блокировок в PostgreSQL

PostgreSQL имеет многоуровневую систему блокировок:

```
Уровень 1: Блокировки на уровне строк (Row-Level Locks)
           - хранятся прямо в заголовке кортежа (xmax + infomask)
           - легковесные, не требуют разделяемой памяти

Уровень 2: Блокировки на уровне таблиц (Table-Level Locks)
           - хранятся в разделяемой памяти (lock space)
           - совместимость регулируется матрицей конфликтов

Уровень 3: Advisory Locks
           - блокировки на уровне приложения
           - не связаны с конкретными таблицами/строками

Уровень 4: Тяжёлые блокировки объектов (Heavyweight Locks)
           - также в lock space, но для DDL-операций
```

---

### 3.2. Блокировки на уровне строк (Row-Level Locks)

В PostgreSQL **читатели НЕ блокируют строки**. Блокировка строки происходит только когда транзакция хочет **изменить** строку (UPDATE/DELETE) или явно запрашивает блокировку (SELECT ... FOR UPDATE).

#### Режимы блокировок строк

| Режим | SQL | Описание |
|-------|-----|----------|
| **FOR UPDATE** | `SELECT ... FOR UPDATE` | Эксклюзивная блокировка. Никто другой не может заблокировать. |
| **FOR NO KEY UPDATE** | `SELECT ... FOR NO KEY UPDATE` | Слабее FOR UPDATE. Разрешает FOR KEY SHARE. |
| **FOR SHARE** | `SELECT ... FOR SHARE` | Разделяемая блокировка. Разрешает чтение, запрещает UPDATE/DELETE. |
| **FOR KEY SHARE** | `SELECT ... FOR KEY SHARE` | Самая слабая. Запрещает только DELETE и UPDATE ключа. |

#### Матрица конфликтов строковых блокировок

| Запрашиваемая →<br>Удерживаемая ↓ | FOR UPDATE | FOR NO KEY UPDATE | FOR SHARE | FOR KEY SHARE |
|:---|:---:|:---:|:---:|:---:|
| **FOR UPDATE** | ❌ | ❌ | ❌ | ❌ |
| **FOR NO KEY UPDATE** | ❌ | ❌ | ❌ | ✅ |
| **FOR SHARE** | ❌ | ❌ | ✅ | ✅ |
| **FOR KEY SHARE** | ❌ | ✅ | ✅ | ✅ |

#### Как блокировка строки хранится

Блокировка строки в PostgreSQL неявная: она хранится в заголовке кортежа (`xmax` и биты в `infomask`). Нет отдельной «таблицы блокировок» для строк. Это очень эффективно.

Когда транзакция делает `SELECT ... FOR UPDATE`, она устанавливает `xmax` существующей версии кортежа равным своему XID и ставит флаги блокировки (LOCK_ONLY в infomask).

```sql
-- Транзакция 1
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
-- xmax строки установлен в XID транзакции 1

-- Транзакция 2
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE; -- БЛОКИРУЕТСЯ, ждёт TX1

-- Транзакция 2 может увидеть, кто её блокирует:
SELECT 
    blocked_locks.pid AS blocked_pid,
    blocked_activity.query AS blocked_query,
    blocking_locks.pid AS blocking_pid,
    blocking_activity.query AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity 
    ON blocked_activity.pid = blocked_locks.pid
JOIN pg_catalog.pg_locks blocking_locks 
    ON blocking_locks.locktype = blocked_locks.locktype
    AND blocking_locks.database IS NOT DISTINCT FROM blocked_locks.database
    AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
    AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
    AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
    AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
    AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
    AND blocking_locks.classid IS NOT DISTINCT FROM blocked_locks.classid
    AND blocking_locks.objid IS NOT DISTINCT FROM blocked_locks.objid
    AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
    AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity 
    ON blocking_activity.pid = blocking_locks.pid
WHERE NOT blocked_locks.granted;
```

#### SELECT ... FOR UPDATE с JOIN

```sql
-- Блокирует строки ТОЛЬКО из users (не из orders)
SELECT u.* 
FROM users u 
JOIN orders o ON u.id = o.user_id 
WHERE o.status = 'pending'
FOR UPDATE OF u;

-- Блокирует строки из нескольких таблиц
SELECT u.*, o.*
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.status = 'pending'
FOR UPDATE OF u, o;
```

### 3.3. Блокировки на уровне таблиц (Table-Level Locks)

Табличные блокировки — более грубые. Они автоматически захватываются DDL-командами и некоторыми DML. Их также можно запросить вручную (`LOCK TABLE`).

#### Режимы (от самого слабого к самому сильному)

| # | Режим | Пример автоматического захвата |
|---|-------|-------------------------------|
| 1 | **Access Share** | `SELECT` |
| 2 | **Row Share** | `SELECT FOR UPDATE/SHARE` |
| 3 | **Row Exclusive** | `INSERT`, `UPDATE`, `DELETE` |
| 4 | **Share Update Exclusive** | `VACUUM` (не FULL), `ANALYZE`, `CREATE INDEX CONCURRENTLY` |
| 5 | **Share** | `CREATE INDEX` (не CONCURRENTLY) |
| 6 | **Share Row Exclusive** | `CREATE TRIGGER`, некоторые формы `ALTER TABLE` |
| 7 | **Exclusive** | `REFRESH MATERIALIZED VIEW CONCURRENTLY` |
| 8 | **Access Exclusive** | `ALTER TABLE`, `DROP TABLE`, `TRUNCATE`, `VACUUM FULL`, `REINDEX` |

#### Матрица конфликтов

| Режим | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 |
|-------|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| Access Share (1) | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ❌ |
| Row Share (2) | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ |
| Row Exclusive (3) | ✅ | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ |
| Share Update Exclusive (4) | ✅ | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ |
| Share (5) | ✅ | ✅ | ❌ | ❌ | ✅ | ❌ | ❌ | ❌ |
| Share Row Exclusive (6) | ✅ | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| Exclusive (7) | ✅ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |
| Access Exclusive (8) | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ | ❌ |

**Важнейшее практическое следствие:**
- `SELECT` (Access Share) **конфликтует только** с Access Exclusive (ALTER/DROP/TRUNCATE).
- `INSERT/UPDATE/DELETE` (Row Exclusive) конфликтует с Share, Share Row Exclusive, Exclusive, Access Exclusive.
- `CREATE INDEX CONCURRENTLY` (Share Update Exclusive) **не блокирует** чтение и запись!
- `CREATE INDEX` (без CONCURRENTLY, режим Share) **блокирует** INSERT/UPDATE/DELETE!

---

### 3.4. Advisory Locks

Advisory Locks — это блокировки на уровне приложения. Они не привязаны к конкретным объектам БД. Полезны для координации распределённых процессов.

```sql
-- Блокировка по одному 64-битному ключу
SELECT pg_advisory_lock(42);

-- Блокировка по двум 32-битным ключам
SELECT pg_advisory_lock(1, 100);

-- Неблокирующая попытка
SELECT pg_try_advisory_lock(42); -- вернёт false если не может

-- Разделяемая (shared) блокировка
SELECT pg_advisory_lock_shared(42);

-- Освобождение
SELECT pg_advisory_unlock(42);

-- Освободить все advisory locks сессии
SELECT pg_advisory_unlock_all();
```

**Типичные сценарии:**
- Координация распределённых задач (только один worker обрабатывает задачу).
- Предотвращение одновременного выполнения миграций.
- Эмуляция распределённого мьютекса.

Advisory Locks **автоматически освобождаются** при завершении сессии или транзакции (если захвачены на уровне транзакции: `pg_advisory_xact_lock`).

---

### 3.5. Блокировки в Oracle: сравнение

| Аспект | PostgreSQL | Oracle |
|--------|-----------|--------|
| Блокировка читателем | ❌ Никогда (MVCC) | ❌ Никогда (MVCC + undo) |
| Блокировка строки при UPDATE | ✅ Да (xmax в кортеже) | ✅ Да (ITL в блоке + TX lock) |
| Эскалация блокировок | ❌ Нет! Не повышает блокировку до табличной | ❌ Нет! |
| SELECT ... FOR UPDATE | Есть, с разными режимами | Есть (`FOR UPDATE`, `FOR UPDATE NOWAIT`, `SKIP LOCKED`) |
| SKIP LOCKED | ✅ Да (PG 9.5+) | ✅ Да |
| Advisory Locks | ✅ Да | ✅ Через `DBMS_LOCK` |
| Табличные блокировки | Многоуровневая матрица конфликтов | Аналогично, через `LOCK TABLE` |

**Главное:** в обоих СУБД нет эскалации блокировок (в отличие от SQL Server, где это частая проблема).

### 3.6. Обнаружение и диагностика блокировок

#### Основные представления

```sql
-- Все текущие блокировки
SELECT 
    locktype, 
    relation::regclass AS table_name,
    mode,
    granted,
    pid,
    waitstart,
    now() - waitstart AS wait_duration
FROM pg_locks
WHERE NOT granted
ORDER BY waitstart;

-- Кто кого блокирует (рекурсивно, начиная с PG 8.4+)
SELECT 
    a.pid AS blocked_pid,
    unnest(pg_blocking_pids(a.pid)) AS blocking_pid,
    a.query AS blocked_query
FROM pg_stat_activity a
WHERE cardinality(pg_blocking_pids(a.pid)) > 0;

-- Активность с блокировками
SELECT 
    pid,
    usename,
    application_name,
    state,
    wait_event_type,
    wait_event,
    query,
    now() - xact_start AS xact_duration,
    now() - query_start AS query_duration
FROM pg_stat_activity
WHERE state != 'idle'
  AND pid != pg_backend_pid()
ORDER BY query_start;

-- Что именно ждёт сессия (PG 9.6+)
SELECT 
    pid, 
    wait_event_type, 
    wait_event
FROM pg_stat_activity
WHERE wait_event_type = 'Lock';
```

#### Типичные wait_event'ы по блокировкам

| wait_event_type | wait_event | Что происходит |
|:---|:---|:---|
| `Lock` | `relation` | Ждёт блокировку таблицы |
| `Lock` | `transactionid` | Ждёт завершения транзакции (UPDATE одной строки) |
| `Lock` | `tuple` | Ждёт блокировку конкретного кортежа |
| `Lock` | `extend` | Ждёт расширения отношения (добавления страниц) |

#### Параметр `lock_timeout`

Важнейший параметр, позволяющий ограничить время ожидания любой блокировки. В отличие от `statement_timeout`, который прерывает весь запрос, `lock_timeout` прерывает только ожидание блокировки:

```ini
lock_timeout = 2s  # ждать блокировку не более 2 секунд
```

```sql
-- На уровне сессии
SET lock_timeout = '5s';

-- На уровне транзакции
SET LOCAL lock_timeout = '1s';
SELECT * FROM users WHERE id = 1 FOR UPDATE;  -- ошибка если ждать >1с
```

**Отличие от `statement_timeout`:** `lock_timeout` отсчитывается только пока транзакция ждёт блокировку. Если блокировка получена, начинается отсчёт `statement_timeout` на выполнение самого запроса.

```ini
lock_timeout = 3s           # таймаут ожидания блокировки
statement_timeout = 30s     # таймаут выполнения запроса
```

#### Параметр `max_locks_per_transaction`

Определяет, сколько блокировок может удерживать одна транзакция. Разделяемая память для блокировок выделяется при старте сервера из расчёта:

```
max_locks_per_transaction × (max_connections + max_prepared_transactions)
```

```ini
max_locks_per_transaction = 64  # по умолчанию
```

Если транзакция затрагивает очень много строк (например, `UPDATE большой_таблицы`), она может исчерпать лимит блокировок. В логах появится:

```
ERROR: out of shared memory
HINT: You might need to increase max_locks_per_transaction.
```

**Повышайте**, если работаете с массивными обновлениями или имеете много партиций (каждое обращение к партиции требует блокировку).

#### Расширение pg_stat_statements

**pg_stat_statements** (нужно включить в shared_preload_libraries) — важнейшее расширение для мониторинга:

```sql
CREATE EXTENSION pg_stat_statements;

-- Топ-10 самых медленных запросов
SELECT 
    queryid,
    LEFT(query, 120) AS query_preview,
    calls,
    mean_exec_time::numeric(10,2) AS avg_ms,
    total_exec_time::numeric(10,2) AS total_ms,
    rows,
    shared_blks_hit,
    shared_blks_read,
    (shared_blks_read::float / NULLIF(shared_blks_hit + shared_blks_read, 0) * 100)::numeric(5,2) AS cache_miss_pct
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;
```

## Часть IV. Дедлоки

### 4.1. Как возникают дедлоки

**Deadlock (взаимная блокировка)** — ситуация, когда две (или более) транзакции ждут друг друга, и ни одна не может продолжиться.

Классический пример:

```
Время   TX1                                 TX2
  │     BEGIN;                             BEGIN;
  │     UPDATE users SET salary = 60000    
  │     WHERE id = 1;                      
  │     -- заблокирована строка id=1       UPDATE users SET salary = 70000
  │                                        WHERE id = 2;
  │                                        -- заблокирована строка id=2
  │     UPDATE users SET salary = 65000    
  │     WHERE id = 2;                      
  │     -- ЖДЁТ TX2 (строка id=2)          
  │                                        UPDATE users SET salary = 75000
  │                                        WHERE id = 1;
  │                                        -- ЖДЁТ TX1 (строка id=1)
  │                                        💥 ДЕДЛОК!
  ▼
```

#### Условия возникновения дедлока (условия Коффмана)

1. **Mutual Exclusion** (взаимное исключение): ресурс не может использоваться совместно.
2. **Hold and Wait** (удержание и ожидание): транзакция удерживает ресурсы и ждёт новые.
3. **No Preemption** (отсутствие принудительного изъятия): нельзя забрать ресурс силой.
4. **Circular Wait** (циклическое ожидание): T1 ждёт T2, T2 ждёт T1.

Все четыре условия выполняются в СУБД при параллельных транзакциях — отсюда и дедлоки.

---

### 4.2. Обнаружение дедлоков в PostgreSQL

PostgreSQL **не предотвращает** дедлоки — он их **обнаруживает** и разрешает.

#### Алгоритм обнаружения

1. При попытке захвата блокировки, если она недоступна, транзакция встаёт в очередь ожидания.
2. **Deadlock Detector** (deadlock_timeout) периодически строит **граф ожидания** (Wait-For Graph) и ищет циклы.
3. Если цикл найден — одна из транзакций в цикле принудительно откатывается.

```
Wait-For Graph:

    TX1 ──────────► TX2
     ▲               │
     │               │
     └───────────────┘
     
    Цикл: TX1 ждёт TX2, TX2 ждёт TX1 → ДЕДЛОК
```

#### Параметры

```ini
deadlock_timeout = 1s  # как часто проверять граф ожидания
log_lock_waits = on    # логировать долгие ожидания блокировок
log_lock_waits_min_duration = 0  # минимальная длительность ожидания для логирования
```

#### Что происходит при обнаружении

PostgreSQL выбирает одну из транзакций-участниц цикла и откатывает её с ошибкой:

```
ERROR: deadlock detected
DETAIL: Process 12345 waits for ShareLock on transaction 67890; 
        blocked by process 23456.
        Process 23456 waits for ShareLock on transaction 12345; 
        blocked by process 12345.
HINT: See server log for query details.
CONTEXT: while updating tuple (0,5) in relation "users"
```

**Пострадавшая транзакция откатывается полностью.** Все её изменения теряются, блокировки освобождаются, вторая транзакция продолжается.

---

### 4.3. Обнаружение дедлоков в Oracle

Oracle действует аналогично: периодически проверяет граф ожидания и откатывает одну из транзакций с ошибкой:

```
ORA-00060: deadlock detected while waiting for resource
```

Oracle также пишет детали в **alert log** и создаёт **trace file** с подробной информацией о deadlock graph.

---

### 4.4. Предотвращение дедлоков

Как разработчик, вы **не можете отключить** deadlock detection, но можете предотвращать дедлоки:

#### Стратегия 1: Всегда блокируйте ресурсы в одном порядке

```sql
-- ПЛОХО: разный порядок блокировки
-- TX1: блокирует users(id=1), потом accounts(id=1)
-- TX2: блокирует accounts(id=1), потом users(id=1) ← ДЕДЛОК!

-- ХОРОШО: всегда сначала users, потом accounts
-- Соглашение: все транзакции блокируют таблицы в алфавитном порядке
```

Или, для одной таблицы — всегда по возрастанию id:
```sql
-- Всегда сортируем id перед блокировкой
SELECT * FROM users WHERE id IN (5, 2, 8) ORDER BY id FOR UPDATE;
-- Блокирует в порядке: id=2, id=5, id=8
```

#### Стратегия 2: Уменьшайте длительность транзакций

Чем короче транзакция, тем меньше окно для дедлока. Выносите бизнес-логику за пределы транзакции, когда это возможно.

#### Стратегия 3: Используйте NOWAIT и SKIP LOCKED

```sql
-- Не ждать блокировку, сразу ошибка
SELECT * FROM users WHERE id = 1 FOR UPDATE NOWAIT;

-- Пропускать заблокированные строки (очереди задач!)
SELECT * FROM tasks WHERE status = 'pending' 
ORDER BY created_at 
FOR UPDATE SKIP LOCKED 
LIMIT 1;
```

`SKIP LOCKED` — идеален для реализации очередей в PostgreSQL.

#### Стратегия 4: Используйте явные табличные блокировки

В редких случаях, когда транзакция затрагивает множество таблиц и предотвратить дедлок сложно:

```sql
BEGIN;
LOCK TABLE users IN EXCLUSIVE MODE;
LOCK TABLE accounts IN EXCLUSIVE MODE;
-- Теперь никто не вмешается
UPDATE users SET ...;
UPDATE accounts SET ...;
COMMIT; -- Блокировки снимаются
```

Цена — серьёзное падение конкурентности.

#### Стратегия 5: Повтор при ошибке (retry)

Даже при всех предосторожностях дедлоки возможны. Приложение должно уметь их обрабатывать:

```java
// Псевдокод
int retries = 3;
while (retries-- > 0) {
    try {
        executeTransaction();
        break;
    } catch (DeadlockException e) {
        if (retries == 0) throw e;
        Thread.sleep(random(50, 200)); // exponential backoff
    }
}
```

### 4.5. Диагностика и логирование

#### Включение логирования блокировок в PostgreSQL

```ini
# postgresql.conf
log_lock_waits = on
deadlock_timeout = 1s
log_min_duration_statement = 1000  # запросы дольше 1 секунды
```

#### Лог дедлока в PostgreSQL

```
LOG:  process 12345 detected deadlock while waiting for 
      ShareLock on transaction 67890 after 1000.123 ms
DETAIL:  Process holding the lock: 23456.
         Wait-for graph: ...
LOG:  process 12345 still waiting for ShareLock on 
      transaction 67890 after 1000.123 ms
```

#### Мониторинг в реальном времени

```sql
-- Кто кого блокирует (удобно и просто)
SELECT 
    blocked.pid AS blocked_pid,
    blocked.query AS blocked_query,
    blocking.pid AS blocking_pid,
    blocking.query AS blocking_query,
    now() - blocked.query_start AS blocked_duration
FROM pg_stat_activity blocked
JOIN pg_stat_activity blocking 
    ON blocking.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE blocked.wait_event_type = 'Lock';
```

#### Oracle: мониторинг блокировок

```sql
-- Oracle: кто кого блокирует
SELECT 
    blocking_session,
    sid,
    serial#,
    wait_class,
    seconds_in_wait,
    sql_id
FROM v$session
WHERE blocking_session IS NOT NULL;
```

## Часть V. Индексы

### 5.1. Зачем нужны индексы: фундаментальные концепции

**Индекс** — это дополнительная структура данных, позволяющая быстро находить строки по определённым столбцам **без полного сканирования таблицы** (Seq Scan).

#### Аналогия

Таблица без индекса — книга без алфавитного указателя. Чтобы найти все упоминания слова «WAL», вам придётся прочитать книгу от корки до корки. Индекс — это тот самый указатель в конце книги: вы идёте в индекс, находите номера страниц и открываете только их.

#### Ключевые метрики индекса

| Метрика | Описание |
|---------|----------|
| **Селективность (Selectivity)** | Доля строк, удовлетворяющих условию. 0.01 = 1% строк. Чем ниже — тем лучше индекс. |
| **Cardinality** | Количество уникальных значений в столбце. Высокая кардинальность → хороший кандидат для B-Tree. |
| **Fillfactor** | Процент заполнения страницы при создании/перестройке индекса. |
| **Bloat** | Раздувание индекса из-за частых UPDATE/DELETE. |
| **Clustering factor** | Насколько физический порядок строк соответствует индексу. |

---

### 5.2. B-Tree — основной индекс

**B-Tree (Balanced Tree)** — тип индекса по умолчанию. Используется для операторов: `<`, `<=`, `=`, `>=`, `>`, `BETWEEN`, `IN`, `IS NULL`, `LIKE 'prefix%'`.

#### Внутренняя структура B-Tree в PostgreSQL

В PostgreSQL используется разновидность B-Tree — **B+Tree Лемана-Яо** (Lehman-Yao), оптимизированная для конкурентного доступа.

```
                    ┌───────────────┐
                    │  Корень (Root) │
                    │  [10, 20, 30]  │
                    └───┬───┬───┬───┘
            ┌───────────┘   │   └───────────┐
            ▼               ▼               ▼
    ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
    │ Внутр. узел   │ │ Внутр. узел   │ │ Внутр. узел   │
    │ [5, 7, 9]    │ │ [12, 15, 18] │ │ [22, 25, 28] │
    └──┬──┬──┬──┘   └──┬──┬──┬──┘   └──┬──┬──┬──┘
       │  │  │         │  │  │         │  │  │
       ▼  ▼  ▼         ▼  ▼  ▼         ▼  ▼  ▼
    ┌─────────────────────────────────────────────────┐
    │          Листовые страницы (Leaf Pages)          │
    │  Каждая содержит пары (ключ → ctid)              │
    │  ctid = (номер_блока, смещение_внутри_блока)     │
    │  Листья связаны в двусвязный список              │
    └─────────────────────────────────────────────────┘
```

**Ключевые особенности B-Tree в PostgreSQL:**

1. **Листья связаны в двусвязный список** — это позволяет эффективно выполнять ORDER BY и диапазонные сканирования (BETWEEN, >, <).
2. **Страницы по 8 КБ**, как и у таблиц.
3. Каждая запись в листе содержит **ключ + ctid** (физический указатель на строку).
4. **Нет ROWID** (как в Oracle) — используется ctid, который меняется при UPDATE.

#### Дедупликация индексов (PG 13+)

Начиная с PG 13, B-Tree индексы автоматически дедуплицируют повторяющиеся ключи в листовых страницах. Если в индексе много одинаковых значений (например, `status = 'active'` для 90% строк), вместо хранения каждого `(key, ctid)` отдельно, хранится один ключ и **список ctid'ов**:

```
Без дедупликации:  (active, ctid1), (active, ctid2), (active, ctid3), ...
С дедупликацией:   (active, [ctid1, ctid2, ctid3, ...])
```

Это даёт:
- **Меньший размер индекса** (иногда до 50% экономии).
- **Меньше page split'ов** при вставке дубликатов.
- **Более быстрый Sequential Scan по индексу.**

Дедупликация включена по умолчанию. Отключить:
```sql
CREATE INDEX idx_users_status ON users (status) WITH (deduplicate_items = off);
```

#### Операции с B-Tree

**Поиск:**
```
SELECT * FROM users WHERE id = 42;
-- 1. Начинаем с корня. 42 > 30 → идём в правое поддерево.
-- 2. Внутренний узел: 42 > 28 → идём вправо.
-- 3. Лист: находим (42 → ctid).
-- 4. По ctid читаем страницу таблицы → строку.
```

**Вставка:**
```
INSERT → поиск листа → вставка в лист → если лист полон → расщепление (split)
```

Расщепление страницы индекса (page split) — дорогая операция, особенно под нагрузкой. Вот почему важен fillfactor.

**Удаление:**
Строка помечается как удалённая в индексе. Физическое удаление происходит при VACUUM.

#### Параметры B-Tree

```sql
-- Создание с параметрами
CREATE INDEX idx_users_name ON users (name) 
WITH (fillfactor = 70);  -- 30% свободного места под будущие вставки

-- Перестройка индекса
REINDEX INDEX CONCURRENTLY idx_users_name;

-- Перестройка с новыми параметрами
ALTER INDEX idx_users_name SET (fillfactor = 70);
REINDEX INDEX CONCURRENTLY idx_users_name;
```

`fillfactor` для индекса (по умолчанию 90):
- **Высокий** (90-100): меньше места, быстрее чтение. Для read-only таблиц.
- **Низкий** (50-70): больше свободного места, меньше page split'ов при вставках. Для активно обновляемых таблиц.

#### Составной (композитный) индекс

```sql
CREATE INDEX idx_users_name_age ON users (name, age);
```

Порядок столбцов важен! Индекс работает для:
- `WHERE name = 'Alice'` ✅
- `WHERE name = 'Alice' AND age = 30` ✅
- `WHERE name = 'Alice' AND age > 25` ✅
- `WHERE age = 30` ❌ (не используется, если нет условия на name)
- `WHERE name > 'Alice' AND age = 30` ⚠️ (используется для name, но не фильтрует по age эффективно)

**Правило:** столбцы в порядке от наиболее селективных к наименее.

### 5.3. Hash-индексы

**Hash-индекс** хранит хеши ключей и используется только для оператора `=`.

```
Ключ → hash(ключ) → bucket → ctid
```

**Особенности в PostgreSQL:**
- До PG 10: не рекомендовались (не WAL-logged — ломались при сбое).
- Начиная с PG 10: полностью WAL-logged, рекомендуется для высокоселективных `=` запросов на больших таблицах.
- **Не поддерживают**: диапазонные запросы, ORDER BY. Уникальные ограничения (UNIQUE) также не поддерживаются — для них нужен B-Tree.
- Занимают меньше места, чем B-Tree (для одинаковых ключей).

```sql
CREATE INDEX idx_users_email_hash ON users USING hash (email);
```

**Когда использовать:** таблица очень большая, запросы только по точному совпадению (`=`), не нужны ORDER BY и диапазоны.

### 5.4. GiST — Generalized Search Tree

**GiST** — расширяемая инфраструктура для построения индексов. Это не конкретный индекс, а **фреймворк**, на основе которого можно строить индексы для разных типов данных.

#### Встроенные GiST-индексы

| Тип данных / применение | Операторы | Пример |
|--------------------------|-----------|--------|
| Геометрические типы | `&&` (пересечение), `@>` (содержит) | `WHERE geom && 'POINT(1 1)'::geometry` |
| Полнотекстовый поиск | `@@` (tsvector match) | `WHERE to_tsvector('russian', text) @@ to_tsquery('запрос')` |
| Диапазонные типы (range) | `&&` (overlaps), `@>` (contains) | `WHERE valid_period @> '2024-01-15'::date` |
| Сетевые адреса (inet) | `<<` (contains) | `WHERE ip << '192.168.1.0/24'` |

#### Как работает GiST

GiST-индекс хранит не точные значения, а **ограничивающие предикаты** (bounding predicates) для групп значений:

- Для геометрии: bounding boxes (прямоугольники).
- Для диапазонов: минимальные покрывающие диапазоны.
- Для текста: сигнатуры слов.

```sql
-- GiST для геоданных (PostGIS)
CREATE INDEX idx_locations ON places USING gist (location);

-- Запрос: найти места в радиусе
SELECT name FROM places 
WHERE ST_DWithin(location, ST_MakePoint(37.6, 55.7)::geography, 1000);
```

#### GiST vs GIN для полнотекстового поиска

| Характеристика | GiST | GIN |
|:---|:---|:---|
| Скорость поиска | Быстрее записи | Быстрее чтения |
| Скорость вставки/обновления | Быстрее | Медленнее (перестраивает posting lists) |
| Размер индекса | Меньше | Больше (в 2-3 раза) |
| Use case | Часто обновляемые данные | Часто читаемые данные |

### 5.5. GIN — Generalized Inverted Index

**GIN** — инвертированный индекс. Идеален для случаев, когда один ключ индекса указывает на **множество** строк.

#### Применение

| Тип данных | Операторы |
|------------|-----------|
| Массивы (`int[]`, `text[]`) | `@>` (contains), `&&` (overlaps) |
| JSONB | `@>`, `?`, `?|`, `?&` |
| Полнотекстовый поиск (`tsvector`) | `@@` |
| hstore | `@>`, `?` |
| pg_trgm (триграммы) | `LIKE '%text%'`, `ILIKE`, similarity |

#### Структура GIN

```
┌───────────────────────────────────────┐
│              GIN Index                 │
│                                       │
│  ┌─────────────────────────────────┐  │
│  │  Мета-страница                   │  │
│  └─────────────────────────────────┘  │
│  ┌─────────────────────────────────┐  │
│  │  B-Tree по уникальным ключам     │  │
│  │  (элементы массива / лексемы)    │  │
│  └──────────────┬──────────────────┘  │
│                 │                      │
│  ┌──────────────▼──────────────────┐  │
│  │  Posting Lists (списки ctid'ов) │  │
│  │  или Posting Tree (B-Tree)      │  │
│  └─────────────────────────────────┘  │
└───────────────────────────────────────┘
```

Каждый ключ (например, элемент массива) указывает на **список ctid'ов** строк, содержащих этот элемент.

#### Примеры

```sql
-- Индекс для поиска по JSONB
CREATE INDEX idx_products_attrs ON products USING gin (attributes jsonb_path_ops);

-- Теперь быстро работает:
SELECT * FROM products 
WHERE attributes @> '{"color": "red", "size": "XL"}';

-- Индекс для полнотекстового поиска
CREATE INDEX idx_articles_text ON articles 
USING gin (to_tsvector('russian', content));

-- Индекс для LIKE '%что-то%' (через триграммы)
CREATE EXTENSION pg_trgm;
CREATE INDEX idx_users_name_trgm ON users USING gin (name gin_trgm_ops);

-- Быстро работает:
SELECT * FROM users WHERE name ILIKE '%ванов%';
```

#### GIN и обновления

GIN имеет `fastupdate` параметр (по умолчанию включён). Вместо немедленного обновления posting lists новые записи копятся в отдельном буфере (pending list). Когда буфер достигает `gin_pending_list_limit`, содержимое переносится в основную структуру.

```sql
-- Отключить fastupdate (для read-only данных)
CREATE INDEX idx_large ON big_table USING gin (col) 
WITH (fastupdate = off);
```

### 5.6. BRIN — Block Range Index

**BRIN** — самый компактный индекс. Он не хранит каждое значение, а хранит **минимальное и максимальное значение для диапазона страниц** (по умолчанию 128 страниц = 1 МБ).

```
Таблица: страницы 0-127 → min=10, max=500
         страницы 128-255 → min=501, max=1000
         страницы 256-383 → min=1001, max=1500
         ...
```

#### Когда BRIN эффективен

- Данные **физически коррелированы** с индексируемым столбцом (например, временные ряды — данные вставляются по возрастанию `created_at`).
- Таблица **очень большая** (сотни ГБ и больше).
- Нужен **компактный** индекс (BRIN в сотни раз меньше B-Tree).

```sql
-- Идеальный сценарий: таблица логов, вставка по времени
CREATE INDEX idx_logs_created ON logs USING brin (created_at) 
WITH (pages_per_range = 32);

-- Запрос
SELECT * FROM logs 
WHERE created_at BETWEEN '2024-06-01' AND '2024-06-02';

-- BRIN: сканирует только те диапазоны, где min <= '2024-06-02' 
--       и max >= '2024-06-01'. Остальные пропускает.
```

#### Ограничения BRIN

- Не подходит для неупорядоченных данных (если значения разбросаны хаотично, будет сканировать почти всю таблицу).
- Не даёт точного позиционирования — только исключает заведомо не подходящие диапазоны.
- Для точного поиска одной строки всё равно хуже B-Tree (если данные не очень упорядочены).

### 5.7. SP-GiST — Space-Partitioned GiST

**SP-GiST** (Space-Partitioned Generalized Search Tree) — индекс для данных с естественной иерархией или пространственным разбиением. В отличие от GiST, который строит сбалансированное дерево, SP-GiST поддерживает **несбалансированные** структуры (quad-trees, k-d trees, radix trees).

#### Когда применять SP-GiST

| Тип данных | Пример |
|------------|--------|
| Геометрические типы (точки) | `point`, `box`, `polygon` |
| Сетевые адреса (inet/cidr) | Проверка вхождения в подсеть |
| Текстовые данные с радикс-деревом | Префиксный поиск через `SP-GiST text_ops` |
| Диапазонные типы | Как GiST, но для специфичных кластеризованных распределений |

#### Структура SP-GiST

В отличие от B-Tree (где все листья на одном уровне), SP-GiST разбивает пространство ключей рекурсивно. Например, для 2D-точек используется **quad-tree**: плоскость делится на 4 квадранта, каждый из которых делится дальше, пока в ячейке не останется мало точек.

```
┌──────────────────────────────────────────┐
│                SP-GiST                    │
│                                          │
│  ┌────────────────────────────────────┐  │
│  │   Корень: bounding box всей области │  │
│  │   [ (0,0) – (100,100) ]            │  │
│  └──────────────┬─────────────────────┘  │
│        ┌────────┼────────┬────────┐       │
│        ▼        ▼        ▼        ▼       │
│   ┌────────┐┌────────┐┌────────┐┌──────┐ │
│   │  NW    ││  NE    ││  SW    ││  SE  │ │
│   │ квадр. ││ квадр. ││ квадр. ││квадр.│ │
│   └───┬────┘└───┬────┘└───┬────┘└──┬───┘ │
│       │   ...   │   ...   │   ...  │ ... │
│       ▼         ▼         ▼        ▼     │
│    Листовые страницы с ctid'ами          │
└──────────────────────────────────────────┘
```

#### Примеры использования

```sql
-- SP-GiST для геометрических точек (поиск ближайших)
CREATE INDEX idx_places_point ON places USING spgist (location);

-- Поиск мест в прямоугольнике
SELECT * FROM places 
WHERE location <@ box '((37.0, 55.0), (38.0, 56.0))';

-- SP-GiST для inet/cidr: поиск IP в подсети
CREATE INDEX idx_ips ON ip_addresses USING spgist (ip);

SELECT * FROM ip_addresses 
WHERE ip <<= '192.168.0.0/16';  -- входит в подсеть

-- SP-GiST для текста с радикс-деревом (префиксный поиск)
CREATE INDEX idx_wiki_title ON wikipedia USING spgist (title);

-- Быстрый префиксный поиск:
SELECT * FROM wikipedia WHERE title LIKE 'Postgre%';
```

**Когда выбирать SP-GiST, а не GiST:**
- Данные имеют естественное иерархическое разбиение (точки в пространстве, IP-адреса).
- Структура данных неоднородна (например, кластеризация точек в разных областях карты).
- GiST в таких случаях может давать менее эффективное разбиение.

**Ограничения:** SP-GiST поддерживает только подмножество операторов по сравнению с GiST. Не все типы данных имеют SP-GiST opclass.

### 5.8. Bloom-индекс (расширение)

**Bloom-фильтр** — вероятностная структура данных. Индекс на основе bloom-фильтра (расширение `bloom`) позволяет индексировать **много столбцов одновременно** для поиска по точному совпадению, занимая значительно меньше места, чем отдельные B-Tree индексы.

```sql
CREATE EXTENSION bloom;

-- Bloom-индекс по трём столбцам
CREATE INDEX idx_users_bloom ON users USING bloom (name, email, phone)
WITH (length = 80, col1 = 2, col2 = 2, col3 = 2);

-- Быстро работает:
SELECT * FROM users WHERE name = 'Alice' AND email = 'alice@example.com';
SELECT * FROM users WHERE name = 'Alice' AND phone = '12345';
-- Любая комбинация столбцов в WHERE!
```

**Параметры:**
- `length` — длина каждого bloom-фильтра в сигнатурах (по умолчанию 80).
- `colN` — количество бит на столбец (чем больше, тем меньше ложных срабатываний, но больше размер).

**Когда использовать:**
- Много столбцов, по которым ищут в разных комбинациях.
- Дешёвле, чем создавать множество отдельных или составных B-Tree индексов.

**Ограничения:** только оператор `=`; возможны ложные срабатывания (false positives) — требуется перепроверка, но это происходит редко при правильной настройке.

### 5.9. Специальные типы индексов

#### 5.9.1. Частичный индекс (Partial Index)

Индекс только для подмножества строк (WHERE). Меньше размер, быстрее обновление.

```sql
-- Индекс только для активных пользователей (меньшинство)
CREATE INDEX idx_active_users ON users (email) 
WHERE status = 'active';

-- Планировщик использует этот индекс, только если WHERE совпадает:
SELECT * FROM users WHERE status = 'active' AND email = 'alice@example.com';
```

#### 5.9.2. Покрывающий индекс (Covering Index / INCLUDE)

Включает дополнительные столбцы, которые не участвуют в поиске, но доступны без обращения к таблице (Index-Only Scan).

```sql
-- Без INCLUDE: при Index-Only Scan нужна таблица для столбца name
CREATE INDEX idx_users_email_cover ON users (email) INCLUDE (name);

-- Index-Only Scan: email и name читаются прямо из индекса, без визита к таблице!
SELECT name FROM users WHERE email = 'alice@example.com';
```

**Важно:** Index-Only Scan возможен только если страница в visibility map помечена как «все кортежи видны всем» (all-visible). Иначе приходится заглядывать в таблицу.

#### 5.9.3. Функциональный индекс (Expression Index)

Индекс по выражению, а не по значению столбца.

```sql
-- Индекс по lower(email)
CREATE INDEX idx_users_lower_email ON users (lower(email));

-- Используется:
SELECT * FROM users WHERE lower(email) = 'alice@example.com';

-- Индекс по дате (из timestamp)
CREATE INDEX idx_orders_day ON orders ((created_at::date));

-- Индекс по JSON-полю
CREATE INDEX idx_products_color ON products ((attributes->>'color'));
```

#### 5.9.4. Уникальный индекс

```sql
-- Автоматически создаётся для PRIMARY KEY и UNIQUE constraint
CREATE UNIQUE INDEX idx_users_email_unique ON users (email);

-- Частичный уникальный индекс (уникальность только для активных)
CREATE UNIQUE INDEX idx_users_active_email 
ON users (email) WHERE status = 'active';
```

#### 5.9.5. Многостолбцовый vs несколько индексов

```sql
-- ОДИН составной индекс (name, age)
CREATE INDEX idx_name_age ON users (name, age);

-- ДВА отдельных индекса
CREATE INDEX idx_name ON users (name);
CREATE INDEX idx_age ON users (age);
```

Планировщик может комбинировать несколько индексов через **Bitmap Index Scan** (BitmapAnd, BitmapOr):

```sql
SELECT * FROM users WHERE name = 'Alice' AND age = 30;
-- Может использоваться: idx_name + idx_age → BitmapAnd → Bitmap Heap Scan
```

Но составной индекс всё равно эффективнее, если предикаты часто встречаются вместе.

### 5.10. Сравнение индексов PostgreSQL и Oracle

| Аспект | PostgreSQL | Oracle |
|--------|-----------|--------|
| B-Tree (основной) | B+Tree Лемана-Яо | B*Tree (разновидность B+Tree) |
| Hash Index | ✅ WAL-logged (PG 10+) | ✅ (для кластеров) |
| GiST | ✅ (геоданные, FTS, диапазоны) | ❌ |
| SP-GiST | ✅ (точки, IP, радикс-деревья) | ❌ |
| GIN | ✅ (JSONB, массивы, FTS, триграммы) | ❌ |
| BRIN | ✅ (временные ряды) | ❌ |
| Bloom | ✅ (расширение) | ❌ |
| Bitmap Index | ✅ Через Bitmap Heap Scan (динамически из B-Tree) | ✅ Отдельный тип (статический, для DW) |
| Partial Index | ✅ | ❌ (эмулируется через F.B.I. + NULL) |
| Covering Index | ✅ INCLUDE | ❌ (только IOT приблизительно) |
| Function-Based Index | ✅ Expression Index | ✅ Function-Based Index |
| Reverse Key Index | ❌ Нет встроенного | ✅ (для предотвращения hot block) |
| IOT (Index-Organized Table) | ❌ Нет (только через CLUSTER) | ✅ Таблица хранится как B-Tree |
| Bitmap Join Index | ❌ Нет | ✅ (для соединений в DW) |
| Domain Index | ❌ (но GiST/GIN расширяемы) | ✅ (Extensible Indexing) |
| Invisible Index | ❌ (можно через гипотетические) | ✅ |
| Index Deduplication | ✅ (PG 13+, автоматически для B-Tree) | ❌ |

**Oracle-специфичные индексы:**
- **IOT (Index-Organized Table)**: таблица физически хранится в B-Tree, нет отдельной кучи. Идеально для таблиц с частым поиском по первичному ключу.
- **Bitmap Index**: статический битовый индекс для DW, где обновления редки. Создаётся как `CREATE BITMAP INDEX ...`.
- **Reverse Key Index**: ключи индекса реверсируются (`12345` → `54321`), чтобы распределить вставки по разным блокам (предотвращает contention на правой границе индекса).

---

### 5.11. Стратегии индексирования и лучшие практики

#### 5.11.1. Какие столбцы индексировать

| Приоритет | Где создавать индекс |
|:---------:|----------------------|
| 🔴 Критично | PRIMARY KEY (автоматически), FOREIGN KEY |
| 🟠 Высокий | Столбцы в WHERE, JOIN, ORDER BY |
| 🟡 Средний | Столбцы с высокой селективностью в больших таблицах |
| 🟢 Низкий | Всё остальное (думать!) |

#### 5.11.2. Какие столбцы НЕ индексировать

- Низкая кардинальность (пол: M/F, булевы флаги). B-Tree будет неэффективен. Если очень нужно — используйте **Bitmap Index** (Oracle) или **Partial Index** (PG: `WHERE flag = true`).
- Часто обновляемые столбцы (индекс будет раздуваться, много page split'ов).
- Широкие текстовые столбцы (лучше GIN с триграммами или полнотекстовый индекс).

#### 5.11.3. Мониторинг «мёртвых» индексов

```sql
-- Неиспользуемые индексы (PG 9.0+)
SELECT 
    schemaname || '.' || relname AS table_name,
    indexrelname AS index_name,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size,
    idx_scan AS scans,
    idx_tup_read AS tuples_read,
    idx_tup_fetch AS tuples_fetched
FROM pg_stat_user_indexes
WHERE idx_scan = 0
  AND indexrelname NOT LIKE '%pkey%'
ORDER BY pg_relation_size(indexrelid) DESC;

-- Дублирующиеся индексы
SELECT 
    indrelid::regclass AS table_name,
    array_agg(indexrelid::regclass) AS duplicate_indexes
FROM pg_index
GROUP BY indrelid, indkey
HAVING count(*) > 1;
```

#### 5.11.4. Обслуживание индексов

```sql
-- Поиск раздутых индексов (Bloat)
-- Простой метод: сравнить фактический и ожидаемый размер
SELECT 
    schemaname || '.' || tblname AS table_name,
    idxname AS index_name,
    pg_size_pretty(pg_relation_size(idxid)) AS actual_size,
    pg_size_pretty(estimated_size) AS expected_size,
    ROUND(100.0 * (pg_relation_size(idxid) - estimated_size) 
          / NULLIF(pg_relation_size(idxid), 0), 1) AS bloat_pct
FROM (
    -- запрос для оценки bloat (упрощённый)
    ...
) sub
WHERE pg_relation_size(idxid) > estimated_size * 1.2
ORDER BY pg_relation_size(idxid) DESC;

-- Перестройка индекса (без блокировки!)
REINDEX INDEX CONCURRENTLY idx_users_name;

-- Перестройка всех индексов таблицы
REINDEX TABLE CONCURRENTLY users;
```

#### 5.11.5. Итоговая таблица выбора индекса

| Сценарий | Индекс |
|----------|--------|
| `=`, `>`, `<`, `BETWEEN`, ORDER BY, `LIKE 'prefix%'` | **B-Tree** |
| Только `=` на большом статическом справочнике | **Hash** |
| Геоданные, диапазоны, полнотекстовый (часто обновляется) | **GiST** |
| Геоточки, IP-адреса, префиксный поиск с естественной иерархией | **SP-GiST** |
| Полнотекстовый (read-only), массивы, JSONB, LIKE '%text%' | **GIN** |
| Огромные таблицы с временными рядами, физическая упорядоченность | **BRIN** |
| Много столбцов для `=` в любых комбинациях, экономия места | **Bloom** |
| Подмножество строк (статус = 'active') | **Partial Index** |
| Покрытие столбцов для Index-Only Scan | **B-Tree + INCLUDE** |
| Функция от столбца (lower, date_trunc) | **Expression Index** |

## Часть VI. EXPLAIN и планы выполнения

### 6.1. EXPLAIN в PostgreSQL

`EXPLAIN` показывает план выполнения запроса — как PostgreSQL планирует (или выполнил) запрос. Это **главный инструмент** для понимания производительности.

#### Синтаксис

```sql
EXPLAIN [ ( option [, ...] ) ] sql_statement;
EXPLAIN [ ANALYZE ] [ VERBOSE ] sql_statement;

-- Основные опции:
-- ANALYZE       — реально выполнить запрос и показать фактические метрики
-- VERBOSE       — больше деталей (список столбцов, имена схем)
-- COSTS         — показывать стоимость (по умолчанию ON)
-- BUFFERS       — показать использование буферов (только с ANALYZE)
-- TIMING        — замерять время (по умолчанию ON с ANALYZE)
-- SUMMARY       — показать итоговую сводку
-- FORMAT        — TEXT (по умолчанию), JSON, XML, YAML
```

#### Самый полезный вариант

```sql
EXPLAIN (ANALYZE, BUFFERS, TIMING, FORMAT TEXT) 
SELECT ...ваш запрос...;
```

**Внимание:** `ANALYZE` реально выполняет запрос! Для INSERT/UPDATE/DELETE используйте внутри транзакции с откатом:

```sql
BEGIN;
EXPLAIN (ANALYZE, BUFFERS) UPDATE users SET salary = salary * 1.1 WHERE department_id = 5;
ROLLBACK;
```

#### Чтение вывода EXPLAIN

```
Seq Scan on users  (cost=0.00..25.00 rows=1000 width=40)
                   (actual time=0.015..0.550 rows=1000 loops=1)
  Filter: (age > 30)
  Buffers: shared hit=25
```

Разбор по частям:

| Часть | Что означает |
|-------|-------------|
| `Seq Scan on users` | Тип операции: последовательное сканирование таблицы users |
| `cost=0.00..25.00` | Оценка стоимости: стартовая (получение первой строки) и общая (возврат всех строк). Единицы — условные. |
| `rows=1000` | Оценка планировщика: сколько строк вернёт (оценка!) |
| `width=40` | Оценка средней ширины строки в байтах |
| `actual time=0.015..0.550` | Реальное время: до первой строки и до последней (в мс) |
| `rows=1000` | Реальное количество строк |
| `loops=1` | Сколько раз выполнен этот узел (для вложенных циклов >1) |
| `Filter: (age > 30)` | Условие фильтрации |
| `Buffers: shared hit=25` | Чтений из буферного кэша (hit=25, без read — всё из кэша) |

#### Buffers: ключевой индикатор

```
Buffers: shared hit=100 read=5 dirtied=3 written=1
```

| Параметр | Смысл |
|----------|-------|
| `shared hit=N` | Страниц прочитано из shared_buffers (кэш) — хорошо |
| `shared read=N` | Страниц прочитано с диска — плохо (или первый запуск) |
| `shared dirtied=N` | Страниц изменено в кэше (для UPDATE/INSERT/DELETE) |
| `shared written=N` | Страниц сброшено на диск bgwriter'ом в процессе |
| `local hit/read` | Для временных таблиц (локальная память) |
| `temp read/written=N` | Использование временных файлов (сортировка на диске!) |

### 6.2. Типы узлов (Node Types) в плане выполнения

#### 6.2.1. Seq Scan (последовательное сканирование)

Чтение всей таблицы от начала до конца. Используется когда:
- Нет подходящего индекса.
- Планировщик решил, что индекс будет дороже (например, таблица маленькая, или условие возвращает >5-10% строк).

```sql
EXPLAIN SELECT * FROM users WHERE age > 10;
-- Seq Scan on users  (cost=0.00..25.00 rows=950 width=40)
--   Filter: (age > 10)
```

**Оптимизация:** Увеличить `work_mem` для параллельного Seq Scan, либо создать индекс.

#### 6.2.2. Index Scan

Использует индекс для поиска строк, затем читает таблицу для получения всех столбцов.

```sql
EXPLAIN SELECT * FROM users WHERE id = 42;
-- Index Scan using users_pkey on users  (cost=0.29..8.31 rows=1 width=40)
--   Index Cond: (id = 42)
```

**Когда:** высокая селективность (мало строк соответствует условию).

#### 6.2.3. Index Only Scan

Читает данные **только из индекса**, без обращения к таблице. Возможен, если:
- Все запрошенные столбцы есть в индексе (включая INCLUDE).
- Страница помечена как all-visible в visibility map.

```sql
EXPLAIN SELECT id, email FROM users WHERE email = 'alice@example.com';
-- Index Only Scan using idx_users_email on users  (...)
--   Index Cond: (email = 'alice@example.com')
--   Heap Fetches: 0  ← ноль обращений к таблице = идеально!
```

Если `Heap Fetches > 0` — страницы не all-visible, приходится проверять видимость в таблице. Решение: `VACUUM` таблицы.

#### 6.2.4. Bitmap Index Scan + Bitmap Heap Scan

Планировщик сканирует индекс и строит **битовую карту** страниц, которые нужно прочитать. Затем читает страницы в физическом порядке (это эффективнее случайного доступа).

```sql
EXPLAIN SELECT * FROM users WHERE age BETWEEN 30 AND 40;
-- Bitmap Heap Scan on users  (cost=... rows=... width=...)
--   Recheck Cond: ((age >= 30) AND (age <= 40))
--   ->  Bitmap Index Scan on idx_users_age  (cost=... rows=... width=0)
--         Index Cond: ((age >= 30) AND (age <= 40))
```

**Recheck Cond:** битовая карта работает на уровне страниц. Строка может быть на странице, но не удовлетворять условию — тогда она отбрасывается при перепроверке.

#### 6.2.5. Parallel Seq Scan

PostgreSQL (начиная с 9.6) может распараллелить сканирование. Несколько workers читают таблицу одновременно.

```sql
EXPLAIN SELECT count(*) FROM big_table;
-- Finalize Aggregate  (cost=...)
--   ->  Gather  (cost=...)
--         Workers Planned: 2
--         ->  Partial Aggregate  (cost=...)
--               ->  Parallel Seq Scan on big_table  (cost=...)
```

Параметры:
```ini
max_parallel_workers_per_gather = 2   # workers на один Gather узел
max_parallel_workers = 8              # всего параллельных workers
max_parallel_maintenance_workers = 2  # для CREATE INDEX, VACUUM
parallel_tuple_cost = 0.1
parallel_setup_cost = 1000.0
min_parallel_table_scan_size = 8MB    # минимальный размер таблицы
```

#### 6.2.6. Subquery Scan и CTE Scan

Сканирование результата подзапроса или CTE.

```sql
EXPLAIN WITH cte AS (SELECT * FROM users WHERE age > 30)
SELECT * FROM cte WHERE name LIKE 'A%';
-- CTE Scan on cte  (cost=...)
--   Filter: (name ~~ 'A%'::text)
--   CTE cte
--     ->  Seq Scan on users  (cost=...)
--           Filter: (age > 30)
```

**Важно:** в PostgreSQL (до версии 12) CTE всегда материализуется (вычисляется и сохраняется во временную область). PG 12+ ввела `MATERIALIZED / NOT MATERIALIZED` для контроля:

```sql
WITH cte AS NOT MATERIALIZED (SELECT ...) -- инлайнится как подзапрос
```

#### 6.2.7. Sort, Hash, Materialize

| Узел | Что делает | Применение |
|------|-----------|------------|
| **Sort** | Сортирует строки (в памяти или на диске). | `ORDER BY`, подготовка к Merge Join. |
| **Hash** | Строит хеш-таблицу в памяти. | Hash Join, Hash Aggregate. |
| **Materialize** | Сохраняет результат во временную область. | Повторное использование результата подзапроса. |

### 6.3. Методы соединения (Join Methods)

#### 6.3.1. Nested Loop Join

Для каждой строки из внешней таблицы выполняется поиск во внутренней таблице (обычно по индексу).

```
Для каждой строки T1:
    Поиск в индексе T2 по ключу соединения
```

```
Nested Loop  (cost=0.29..1000.00 rows=100 width=80)
  ->  Seq Scan on users u  (cost=0.00..20.00 rows=100 width=40)
  ->  Index Scan using idx_orders_user_id on orders o  (cost=0.29..9.79 rows=1 width=40)
        Index Cond: (user_id = u.id)
```

**Когда:** одна таблица маленькая, на второй есть индекс по ключу соединения.

#### 6.3.2. Hash Join

Строит хеш-таблицу в памяти из внутренней таблицы (меньшей), затем для каждой строки внешней таблицы проверяет хеш-таблицу.

```
Hash Join  (cost=... rows=... width=...)
  Hash Cond: (o.user_id = u.id)
  ->  Seq Scan on orders o  (cost=... rows=10000 width=...)
  ->  Hash  (cost=... rows=100 width=...)
        ->  Seq Scan on users u  (cost=... rows=100 width=...)
```

**Когда:** нет подходящих индексов, таблицы примерно одного размера, хватает `work_mem`.

#### 6.3.3. Merge Join

Сортирует обе таблицы по ключу соединения, затем сливает как в merge sort.

```
Merge Join  (cost=... rows=... width=...)
  Merge Cond: (o.user_id = u.id)
  ->  Index Scan using idx_orders_user_id on orders o  (cost=...)
  ->  Sort  (cost=...)
        Sort Key: u.id
        ->  Seq Scan on users u  (cost=...)
```

**Когда:** обе таблицы уже отсортированы (по индексу), или очень большие (не влезают в `work_mem` для Hash Join).

#### Сравнение методов соединения

| Метод | Идеален когда | Требования | Плохо когда |
|-------|--------------|------------|-------------|
| **Nested Loop** | Маленькая внешняя + индекс на внутренней | Индекс на внутренней таблице | Нет индекса, большая внешняя |
| **Hash Join** | Нет индексов, таблицы влезают в work_mem | work_mem | Не влезает в work_mem → на диск |
| **Merge Join** | Данные предсортированы (индекс или ORDER BY) | Сортировка | Если нужна доп. сортировка, дороже Hash Join |

### 6.4. Анализ реальных планов выполнения

#### Пример 1: медленный SELECT

```sql
EXPLAIN (ANALYZE, BUFFERS) 
SELECT o.*, u.name 
FROM orders o 
JOIN users u ON o.user_id = u.id 
WHERE o.created_at > '2024-01-01' 
  AND u.status = 'active';
```

```
Nested Loop  (cost=0.42..12500.00 rows=5000 width=120)
             (actual time=0.150..2500.300 rows=4800 loops=1)
  ->  Seq Scan on orders o  (cost=0.00..2500.00 rows=10000 width=80)
                            (actual time=0.050..800.100 rows=10000 loops=1)
        Filter: (created_at > '2024-01-01'::date)
        Rows Removed by Filter: 90000
        Buffers: shared hit=50 read=1500
  ->  Index Scan using users_pkey on users u  (cost=0.42..0.99 rows=1 width=40)
                                               (actual time=0.169..0.169 rows=0 loops=10000)
        Index Cond: (id = o.user_id)
        Filter: (status = 'active')
        Rows Removed by Filter: 1
        Buffers: shared hit=50000 read=10000
```

**Что мы видим:**
- `Rows Removed by Filter: 90000` — Seq Scan прочитал 100000 строк, чтобы оставить 10000. Нужен индекс на `orders.created_at`!
- Loops: 10000 — для каждой строки из orders делается Index Scan на users. Много!
- 0 rows на внутреннем цикле — большинство пользователей не активны.

**Оптимизация:**
1. Создать индекс на `orders(created_at)`.
2. Может быть, поменять порядок JOIN (Hash Join может быть эффективнее).
3. Или создать индекс на `users(status, id)` для фильтрации.

#### Пример 2: медленная сортировка с уходом на диск

```
Sort  (cost=... rows=... width=...)
      (actual time=5000.100..12000.500 rows=1000000 loops=1)
  Sort Key: created_at DESC
  Sort Method: external merge  Disk: 250000kB   ← НА ДИСК!
  Buffers: shared hit=10000 read=5000, temp read=31250 written=31250
```

**Проблема:** сортировка ушла на диск (не хватило `work_mem`).

**Решение:** увеличить `work_mem` (но осторожно — см. раздел 1.6). Или создать индекс на `created_at` (тогда ORDER BY будет использовать индекс и сортировка не нужна).

#### Пример 3: Index Only Scan, но с Heap Fetches

```
Index Only Scan using idx_users_email on users  (cost=... rows=1 width=40)
       (actual time=0.050..0.051 rows=1 loops=1)
  Index Cond: (email = 'alice@example.com')
  Heap Fetches: 1                    ← читает таблицу :(
  Buffers: shared hit=1 read=3
```

**Проблема:** не все страницы all-visible. **Решение:** `VACUUM users;`

### 6.5. EXPLAIN PLAN в Oracle

Oracle использует другую модель:

```sql
-- Создание плана (без выполнения)
EXPLAIN PLAN FOR
SELECT * FROM users WHERE email = 'alice@example.com';

-- Просмотр плана
SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY);

-- Или:
SELECT PLAN_TABLE_OUTPUT FROM TABLE(DBMS_XPLAN.DISPLAY());

-- Расширенный вывод:

    format => 'ALL',          -- все детали
    type   => 'TYPICAL'       -- или 'BASIC', 'ALL', 'ADAPTIVE'

```

**Пример плана Oracle:**

```
------------------------------------------------------------------------------
| Id  | Operation         | Name  | Rows  | Bytes | Cost (%CPU)| Time     |
------------------------------------------------------------------------------
|   0 | SELECT STATEMENT  |       |     1 |    40 |     2   (0)| 00:00:01 |
|   1 |  TABLE ACCESS BY  | USERS |     1 |    40 |     2   (0)| 00:00:01 |
|*  2 |   INDEX RANGE SCAN| U_EM  |     1 |       |     1   (0)| 00:00:01 |
------------------------------------------------------------------------------
```

**Операции Oracle:**
| Операция | Аналог в PostgreSQL |
|----------|---------------------|
| TABLE ACCESS FULL | Seq Scan |
| TABLE ACCESS BY INDEX ROWID | Index Scan |
| INDEX RANGE SCAN | Index Scan / Bitmap Index Scan |
| INDEX UNIQUE SCAN | Index Scan (для уникального ключа) |
| INDEX FAST FULL SCAN | Index Only Scan (без обращения к таблице) |
| NESTED LOOPS | Nested Loop |
| HASH JOIN | Hash Join |
| MERGE JOIN | Merge Join |
| SORT ORDER BY | Sort |
| HASH GROUP BY | HashAggregate |

### 6.6. Типичные проблемы и их решение по плану

| Симптом в EXPLAIN | Проблема | Решение |
|:---|:---|:---|
| `Seq Scan` на большой таблице | Нет индекса | Создать индекс |
| `Rows Removed by Filter` большое | Индекс не фильтрует | Составной индекс, частичный индекс |
| `Heap Fetches` > 0 на Index Only Scan | Страницы не all-visible | `VACUUM таблица` |
| `Sort Method: external merge Disk` | Не хватило work_mem | Увеличить work_mem или создать индекс |
| `Hash` с `Buckets: ...` и `Batches: N` (N>1) | Hash не влез в память | Увеличить work_mem |
| `Nested Loop` с большим числом loops | Неправильный порядок JOIN | Проверить статистику, попробовать другой метод |
| `Filter` вместо `Index Cond` | Индекс есть, но условие не соответствует | Проверить типы данных, collation, function scan |
| `actual time` >> `cost` | Статистика устарела | `ANALYZE таблица` |
| `Workers Planned: 0` на Seq Scan | Параллелизм не включён или не выгоден | Проверить `max_parallel_workers_per_gather` |

### 6.7. Модуль auto_explain

**auto_explain** — расширение, автоматически записывающее план выполнения медленных запросов в лог PostgreSQL. В отличие от ручного `EXPLAIN`, не требует изменения кода приложения.

#### Настройка

```ini
# В postgresql.conf
shared_preload_libraries = 'auto_explain'

# Параметры (можно менять без перезагрузки через ALTER SYSTEM)
auto_explain.log_min_duration = 1000    # запросы дольше 1 сек (0 = все)
auto_explain.log_analyze = on           # включать ANALYZE (реальное выполнение!)
auto_explain.log_buffers = on           # включать BUFFERS
auto_explain.log_timing = on            # включать TIMING
auto_explain.log_triggers = off         # логировать триггеры
auto_explain.log_verbose = off          # VERBOSE режим
auto_explain.log_format = text          # text | json | xml | yaml
auto_explain.log_nested_statements = on # запросы внутри функций и DO-блоков
auto_explain.log_settings = off         # параметры, изменённые для запроса
auto_explain.sample_rate = 1.0          # доля запросов для логирования (0.0–1.0)
```

**Важно:** `auto_explain.log_analyze = on` реально выполняет запрос с `EXPLAIN ANALYZE`, что замедляет его. Используйте осторожно в production — лучше включать через `log_min_duration` с высоким порогом.

## Часть VII. Репликация

### 7.1. Виды репликации: обзор

**Репликация** — процесс копирования данных с одного сервера (мастер/primary) на другой (реплика/standby).

```
                          ┌─────────────┐
          ┌──────────────►│  Реплика 1  │ (read-only)
          │               └─────────────┘
┌─────────┴───┐           ┌─────────────┐
│   Мастер    │──────────►│  Реплика 2  │ (read-only)
│  (read/write)│          └─────────────┘
└─────────────┘           ┌─────────────┐
                          │  Реплика 3  │ (read-only)
                          └─────────────┘
```

**Зачем нужна репликация:**
1. **Высокая доступность (HA)**: если мастер упал, реплику можно повысить.
2. **Чтение с реплик**: распределение read-нагрузки.
3. **Резервное копирование**: бэкапы с реплики без нагрузки на мастер.
4. **Аналитика/отчётность**: тяжёлые запросы на реплике.

### 7.2. Физическая (потоковая) репликация в PostgreSQL

**Физическая репликация** (Streaming Replication) передаёт **WAL-записи** с мастера на реплику. Реплика физически идентична мастеру (блок-в-блок).

```
Мастер                                   Реплика
  │                                        │
  │ WAL sender process                     │ WAL receiver process
  │───────── WAL-записи ─────────────────►│
  │        (через потоковое                │ WAL применяется (recovery)
  │         соединение)                    │ → страницы идентичны мастеру
  │                                        │
  │                                        │ Может быть в режиме
  │                                        │ hot_standby = on (read-only)
```

#### Настройка потоковой репликации

**На мастере (`postgresql.conf`):**
```ini
wal_level = replica          # или logical
max_wal_senders = 5          # количество процессов для отправки WAL
wal_keep_size = 1024         # МБ WAL хранить для отставших реплик
# или слоты репликации (лучше!)
max_replication_slots = 5
```

**На мастере (`pg_hba.conf`):**
```
host replication replicator 192.168.1.0/24 md5
```

**Создание реплики (через pg_basebackup):**
```bash
pg_basebackup -h master_host -D /var/lib/postgresql/data \
    -U replicator -P -v -R -X stream -C -S replica_slot_1
# -R: создать standby.signal + настройки в postgresql.auto.conf
# -X stream: включить WAL сразу при создании
# -C -S: создать слот репликации
```

**На реплике (создаётся автоматически при `-R`):**
```ini
# postgresql.auto.conf (автоматически)
primary_conninfo = 'host=master_host port=5432 user=replicator password=secret'
primary_slot_name = 'replica_slot_1'

# В postgresql.conf вручную:
hot_standby = on             # разрешить read-only запросы
hot_standby_feedback = on    # сообщать мастеру о старых XID'ах (важно!)
```

#### Слоты репликации (Replication Slots)

**Слот репликации** гарантирует, что мастер не удалит WAL, пока все реплики его не прочитали.

```sql
-- На мастере
SELECT * FROM pg_create_physical_replication_slot('replica_slot_1');

-- Посмотреть состояние слотов
SELECT 
    slot_name,
    slot_type,
    active,
    pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) AS lag_bytes,
    ROUND(pg_wal_lsn_diff(pg_current_wal_lsn(), restart_lsn) 
          / 1024.0 / 1024.0, 2) AS lag_mb
FROM pg_replication_slots;
```

**Без слотов:** если реплика отстала, а мастер уже удалил нужный WAL (checkpoint, `wal_keep_size`) — реплика сломана, нужно пересоздавать.

**Со слотами:** WAL не удаляется, пока реплика не подтвердит получение. **Но:** если реплика упала надолго, WAL на мастере будет копиться и может заполнить диск.

#### Утилита pg_rewind

Если мастер упал и реплика была повышена до мастера, старый мастер (после восстановления) отстал и больше не может быть просто репликой. **pg_rewind** синхронизирует бывший мастер с новым мастером без полного pg_basebackup:

```bash
# На старом мастере (теперь он будет репликой):
pg_rewind --source-server='host=new_master port=5432 user=replicator dbname=postgres' \
          --target-pgdata=/var/lib/postgresql/data \
          --progress

# После pg_rewind настраиваем replication, создаём standby.signal и запускаем
```

**Как работает:** pg_rewind копирует только изменившиеся блоки с нового мастера (сравнивая WAL-позиции), а не всю базу. Это быстро для больших баз.

**Требования:**
- `wal_log_hints = on` на мастере (или контрольные суммы данных).
- PG 9.5+.
- `full_page_writes = on` (по умолчанию).

### 7.3. Логическая репликация в PostgreSQL

**Логическая репликация** (PG 10+) передаёт не WAL, а **логические изменения строк** (INSERT, UPDATE, DELETE). Это позволяет:
- Реплицировать только некоторые таблицы.
- Реплицировать между разными мажорными версиями PostgreSQL.
- Частичная двунаправленная репликация (с осторожностью!).
- Репликация в другие системы (Kafka, другие БД через расширения).

```
Мастер                                      Реплика
  │                                           │
  │ WAL (wal_level = logical)                  │
  │   │                                       │
  │   ▼                                       │
  │ Logical Decoding Plugin (pgoutput)         │
  │   │                                       │
  │   ▼                                       │
  │ Публикация (Publication)                   │
  │────────── логические изменения ──────────►│
  │                                           │
  │                                    Подписка (Subscription)
  │                                      │
  │                                      ▼
  │                                    Применение к таблицам
```

#### Настройка логической репликации

```sql
-- На мастере
ALTER SYSTEM SET wal_level = 'logical';
-- Перезапустить PostgreSQL

-- Создать публикацию
CREATE PUBLICATION pub_users FOR TABLE users (id, name, email);
-- Или все таблицы:
CREATE PUBLICATION pub_all FOR ALL TABLES;

-- Добавить таблицы позже
ALTER PUBLICATION pub_users ADD TABLE orders;

-- Посмотреть публикации
SELECT * FROM pg_publication;
\dRp+  -- в psql

-- На реплике
CREATE SUBSCRIPTION sub_users 
CONNECTION 'host=master dbname=mydb user=replicator password=secret'
PUBLICATION pub_users;

-- Мониторинг
SELECT * FROM pg_stat_subscription;
SELECT * FROM pg_stat_replication;
```

#### Ограничения логической репликации

- Не реплицирует DDL (CREATE TABLE, ALTER TABLE). Только DML.
- Не реплицирует последовательности (sequences) по умолчанию.
- Не реплицирует TRUNCATE (до PG 11).
- Не реплицирует большие объекты (large objects).
- Схема таблиц должна существовать на реплике **заранее**.

#### Конфликты при логической репликации

Если на реплике изменили строку, которую обновляет репликация:

```
ERROR: duplicate key value violates unique constraint
ERROR: update/delete on table "users" violates ...
```

Логическая репликация в таких случаях **останавливается**. Нужно ручное разрешение конфликта.

### 7.4. Синхронная и асинхронная репликация

#### Асинхронная (по умолчанию)

Мастер не ждёт подтверждения от реплики перед коммитом.

```
TX1: INSERT ... COMMIT;
       │
       ├──► Запись в WAL на мастере
       ├──► COMMIT возвращает управление клиенту СРАЗУ
       └──► WAL отправляется на реплику ... когда-нибудь
```

**Преимущества:** низкая латентность. **Риск:** потеря данных при падении мастера (RPO > 0).

#### Синхронная

Мастер **ждёт подтверждения** от синхронной реплики.

```
TX1: INSERT ... COMMIT;
       │
       ├──► Запись в WAL на мастере
       ├──► Отправка WAL на реплику
       ├──► ЖДЁМ (sync)
       ├──► Реплика: получила, записала в свой WAL (fsync), ответила
       └──► COMMIT возвращает управление клиенту
```

**Преимущества:** нулевая потеря данных (RPO = 0). **Недостаток:** увеличенная латентность коммита.

#### Настройка в PostgreSQL

```ini
# На мастере
synchronous_commit = on            # уровень синхронности
synchronous_standby_names = 'ANY 1 (replica1, replica2)'  # или 'FIRST 2 (...)'

# На реплике: в primary_conninfo
application_name = 'replica1'      # должно совпадать с именем в synchronous_standby_names
```

**Варианты `synchronous_commit`:**

| Значение | Латентность | Долговечность |
|----------|:-----------:|:-------------:|
| `on` (по умолчанию) | Средняя | WAL синхронизирован на реплике |
| `remote_apply` | Высокая | WAL применён на реплике (не только записан) |
| `remote_write` | Ниже | WAL записан в ОС реплики (не fsync) |
| `local` | Низкая | WAL синхронизирован локально (не ждём реплику) |
| `off` | Минимальная | WAL не синхронизируется даже локально (опасно!) |

### 7.5. Patroni: автоматический failover

**Patroni** — это решение для управления кластером PostgreSQL с автоматическим обнаружением и failover'ом. Использует распределённое хранилище конфигурации (DCS — Distributed Configuration Store): etcd, Consul, ZooKeeper.

```
                     ┌──────────────────┐
                     │   DCS (etcd)     │
                     │  - Кто мастер?    │
                     │  - Конфигурация   │
                     └────┬──────┬──────┘
                          │      │
              ┌───────────┘      └───────────┐
              ▼                              ▼
    ┌──────────────────┐           ┌──────────────────┐
    │ Patroni + PG     │           │ Patroni + PG     │
    │  (Мастер)        │◄─────────►│  (Реплика)       │
    │                  │ streaming │                  │
    └──────────────────┘           └──────────────────┘
```

**Как работает failover:**

1. Patroni на мастере обновляет «lock» в DCS (раз в TTL).
2. Если мастер упал, lock протухает.
3. Patroni на репликах видят это и начинают выборы (leader election).
4. Реплика с наиболее актуальным WAL (или выбранная по приоритету) становится новым мастером.
5. Patroni делает `pg_ctl promote` и обновляет запись в DCS.
6. Остальные реплики перенаправляются на нового мастера.

```yaml
# Пример конфигурации Patroni (patroni.yml)
scope: mycluster
namespace: /db/

restapi:
  listen: 0.0.0.0:8008
  connect_address: 192.168.1.1:8008

etcd:
  hosts: etcd1:2379,etcd2:2379,etcd3:2379

bootstrap:
  dcs:
    ttl: 30
    loop_wait: 10
    retry_timeout: 10
    maximum_lag_on_failover: 1048576  # 1 MB
    synchronous_mode: true
    synchronous_mode_strict: false
  initdb:
    - encoding: UTF8
    - locale: ru_RU.UTF-8

postgresql:
  listen: 0.0.0.0:5432
  connect_address: 192.168.1.1:5432
  data_dir: /var/lib/postgresql/data
  parameters:
    wal_level: replica
    hot_standby: on
    wal_keep_size: 1024
    max_wal_senders: 10
    max_replication_slots: 10
```

### 7.6. Репликация в Oracle: Data Guard и GoldenGate

#### Oracle Data Guard (физическая репликация)

Аналог потоковой репликации PostgreSQL. Redo-записи передаются на standby-базу.

```
Primary Database              Standby Database
  │                             │
  │ Redo Transport               │
  │────────── Redo ────────────►│ Redo Apply (managed recovery)
  │                             │
```

Режимы:
- **Maximum Protection** (SYNC): коммит ждёт записи redo на standby (аналог `remote_write`). RPO = 0.
- **Maximum Availability** (SYNC, но деградирует до ASYNC): как Maximum Protection, но если standby недоступен — продолжает без него.
- **Maximum Performance** (ASYNC): коммит не ждёт standby (по умолчанию).

**Преимущества над PostgreSQL:**
- Automatic block repair (если блок повреждён на standby, он скачивается с primary).
- Real-time query (Active Data Guard) — standby открыт для чтения без задержки, в отличие от hot_standby, который может отставать.
- Snapshot Standby — standby можно открыть для записи (тестирование), потом откатить.

#### Oracle GoldenGate (логическая репликация)

Аналог логической репликации PostgreSQL, но намного мощнее:

- Гетерогенная репликация: Oracle ↔ PostgreSQL, MySQL, Kafka, …
- Фильтрация и трансформация данных на лету.
- Двунаправленная репликация с разрешением конфликтов.
- CDC (Change Data Capture) — только изменения.

### 7.7. Конфликты репликации и их разрешение

#### Виды конфликтов

| Конфликт | Когда возникает |
|----------|----------------|
| **Primary Key violation** | INSERT на реплике (или двух мастерах) с одинаковым PK |
| **Update conflict** | UPDATE на реплике строки, которая была изменена в реплицируемых данных |
| **Delete conflict** | DELETE на реплике строки, которая уже удалена |
| **Unique constraint violation** | Нарушение UNIQUE при двунаправленной репликации |
| **Foreign Key violation** | Нарушение ссылочной целостности |

#### Решения

**PostgreSQL (логическая репликация):**
```sql
-- Конфликт при вставке: пропустить
ALTER SUBSCRIPTION sub_users SET (origin = 'any');

-- Вручную продвинуть репликацию после конфликта
ALTER SUBSCRIPTION sub_users SKIP (lsn = '0/0');
```

**Oracle (GoldenGate):**
- Встроенные процедуры разрешения конфликтов (`DBMS_GOLDENGATE_ADM`).
- CDR (Conflict Detection and Resolution): OVERWRITE, DISCARD, MAX/MIN, EARLIEST/LATEST TIMESTAMP.

**Общий принцип:** избегать конфликтов на уровне дизайна:
- Шардирование ключей (разные диапазоны PK на разных узлах).
- UUID вместо автоинкрементных PK.
- Однонаправленная запись (только в один мастер).

## Часть VIII. Партицирование

### 8.1. Зачем партицировать таблицы

**Партицирование** — разделение большой таблицы на более мелкие физические части (партиции), которые логически представляют собой одну таблицу.

```
Большая таблица (1 ТБ, 10 миллиардов строк)

┌──────────────────────────────────────────────────────────┐
│  ТАБЛИЦА logs                                            │
│                                                          │
│  ┌─────────────────────┐ ┌─────────────────────┐         │
│  │ Партиция 2024-01    │ │ Партиция 2024-02    │  ...    │
│  │ (100 млн строк)     │ │ (95 млн строк)      │         │
│  └─────────────────────┘ └─────────────────────┘         │
│  ┌─────────────────────┐ ┌─────────────────────┐         │
│  │ Партиция 2024-03    │ │ Партиция 2024-04    │  ...    │
│  │ (110 млн строк)     │ │ (105 млн строк)     │         │
│  └─────────────────────┘ └─────────────────────┘         │
└──────────────────────────────────────────────────────────┘
```

#### Преимущества

| Преимущество | Как достигается |
|--------------|-----------------|
| **Partition Pruning** | Запросы сканируют только нужные партиции, а не всю таблицу |
| **Быстрое удаление данных** | `DROP TABLE partition_202001` — мгновенно, без VACUUM |
| **Разные табличные пространства** | «Горячие» партиции на SSD, архивные на HDD |
| **Улучшение VACUUM** | VACUUM работает на партиции меньшего размера |
| **Улучшение управления** | REINDEX, CLUSTER на отдельных партициях |
| **Параллельный Sequential Scan** | Каждая партиция может сканироваться параллельно |

---

### 8.2. Методы партицирования в PostgreSQL

PostgreSQL поддерживает **декларативное партицирование** (начиная с PG 10; значительно улучшено в PG 11–16).

#### 8.2.1. RANGE (диапазонное)

```sql
-- Создание партицированной таблицы
CREATE TABLE logs (
    id SERIAL,
    created_at TIMESTAMP NOT NULL,
    level TEXT,
    message TEXT
) PARTITION BY RANGE (created_at);

-- Создание партиций
CREATE TABLE logs_2024_01 PARTITION OF logs
FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE logs_2024_02 PARTITION OF logs
FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- Партиция по умолчанию (для значений, не попадающих ни в одну партицию)
CREATE TABLE logs_default PARTITION OF logs DEFAULT;
```

**Идеально для:** временных рядов (логи, события, метрики).

#### 8.2.2. LIST (списочное)

```sql
CREATE TABLE orders (
    id SERIAL,
    region_code TEXT,
    order_date DATE,
    amount NUMERIC
) PARTITION BY LIST (region_code);

CREATE TABLE orders_eu PARTITION OF orders
FOR VALUES IN ('DE', 'FR', 'IT', 'ES');

CREATE TABLE orders_asia PARTITION OF orders
FOR VALUES IN ('CN', 'JP', 'KR', 'IN');

CREATE TABLE orders_other PARTITION OF orders DEFAULT;
```

**Идеально для:** категориальных данных (регионы, статусы, типы).

#### 8.2.3. HASH (хешевое)

```sql
CREATE TABLE users (
    id BIGINT,
    name TEXT,
    email TEXT
) PARTITION BY HASH (id);

-- 8 партиций
CREATE TABLE users_p0 PARTITION OF users
FOR VALUES WITH (MODULUS 8, REMAINDER 0);
CREATE TABLE users_p1 PARTITION OF users
FOR VALUES WITH (MODULUS 8, REMAINDER 1);
-- ...
CREATE TABLE users_p7 PARTITION OF users
FOR VALUES WITH (MODULUS 8, REMAINDER 7);
```

**Идеально для:** равномерного распределения данных по заданному числу партиций (распределение хешем).

---

### 8.3. Sub-partitioning (подпартицирование)

Партиция сама может быть партицирована:

```sql
CREATE TABLE logs (
    id SERIAL,
    created_at TIMESTAMP NOT NULL,
    level TEXT,
    region TEXT,
    message TEXT
) PARTITION BY RANGE (created_at);

-- Партиция 2024-01 — сама партицирована по LIST:
CREATE TABLE logs_2024_01 PARTITION OF logs
FOR VALUES FROM ('2024-01-01') TO ('2024-02-01')
PARTITION BY LIST (region);

-- Подпартиции:
CREATE TABLE logs_2024_01_eu PARTITION OF logs_2024_01
FOR VALUES IN ('DE', 'FR', 'IT');
CREATE TABLE logs_2024_01_asia PARTITION OF logs_2024_01
FOR VALUES IN ('CN', 'JP', 'KR');
```

### 8.4. Partition Pruning

**Partition Pruning** — способность планировщика исключать ненужные партиции из плана запроса.

```sql
EXPLAIN SELECT * FROM logs WHERE created_at >= '2024-02-15';
-- Seq Scan on logs_2024_02  ← сканируется ТОЛЬКО партиция февраля!
-- Seq Scan on logs_2024_03
-- ...

EXPLAIN SELECT * FROM logs WHERE created_at = '2024-01-15';
-- Seq Scan on logs_2024_01  ← только одна партиция
```

**Когда pruning работает:**
- ✅ `WHERE partition_key = const`
- ✅ `WHERE partition_key IN (val1, val2)`
- ✅ `WHERE partition_key BETWEEN val1 AND val2`
- ✅ `WHERE partition_key > const`
- ❌ `WHERE partition_key + interval '1 day' = const` (выражение над ключом)
- ❌ `WHERE partition_key = (SELECT ...)` (подзапрос — pruning во время выполнения)

#### Partition Pruning во время выполнения (run-time pruning)

Если значение ключа становится известным только в момент выполнения (например, из подзапроса или параметризованного запроса), PG 11+ может делать pruning во время выполнения:

```sql
-- $1 — параметр, подставленный приложением
SELECT * FROM logs WHERE created_at = $1;
-- Если $1 = '2024-01-15', все партиции кроме 2024-01 исключаются
```

### 8.5. Партицирование в Oracle

Oracle — пионер партицирования (с версии 8, 1997 год). Имеет больше опций:

| Метод | PostgreSQL | Oracle |
|-------|:---:|:---:|
| RANGE | ✅ | ✅ |
| LIST | ✅ | ✅ |
| HASH | ✅ | ✅ |
| RANGE-LIST | ✅ Sub-partition | ✅ Composite Partitioning |
| RANGE-HASH | ✅ Sub-partition | ✅ Composite Partitioning |
| LIST-LIST | ✅ Sub-partition | ✅ |
| **INTERVAL** | ❌ (эмулируется через pg_partman) | ✅ Автоматически создаёт новые партиции |
| **REFERENCE** | ❌ | ✅ Партицирование по внешнему ключу |
| **SYSTEM** | ❌ | ✅ Ручное управление |
| **VIRTUAL COLUMN** | ❌ (можно через expression) | ✅ Партицирование по виртуальному столбцу |

**Oracle INTERVAL:** автоматическое создание партиций при вставке:

```sql
-- Oracle
CREATE TABLE logs (
    id NUMBER,
    created_at DATE,
    message VARCHAR2(1000)
) PARTITION BY RANGE (created_at)
INTERVAL (NUMTODSINTERVAL(1, 'DAY'))  -- Автоматически создавать дневные партиции
(
    PARTITION p0 VALUES LESS THAN (DATE '2024-01-01')
);

-- При вставке строки с created_at = '2024-01-15' — партиция создастся АВТОМАТИЧЕСКИ
```

### 8.6. Управление партициями (attach/detach)

```sql
-- Отсоединить партицию (превращается в обычную таблицу)
ALTER TABLE logs DETACH PARTITION logs_2024_01;
-- logs_2024_01 теперь отдельная таблица

-- Присоединить таблицу как партицию (должна удовлетворять ограничениям)
ALTER TABLE logs ATTACH PARTITION logs_2024_01
FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

-- Удалить партицию и все данные в ней
DROP TABLE logs_2024_01;  -- Мгновенно! Не требует VACUUM!

-- Удалить только партицию (оставить таблицу)
ALTER TABLE logs DETACH PARTITION logs_2024_01 CONCURRENTLY;
```

**Кейс:** архивация данных.
```sql
-- 1. Отсоединяем старую партицию
ALTER TABLE logs DETACH PARTITION logs_2020_01;

-- 2. Архивируем (pg_dump, или перемещаем на медленный диск)
ALTER TABLE logs_2020_01 SET TABLESPACE archive_tablespace;

-- 3. При необходимости — обратно присоединяем
ALTER TABLE logs ATTACH PARTITION logs_2020_01
FOR VALUES FROM ('2020-01-01') TO ('2020-02-01');
```

### 8.7. Лучшие практики партицирования

1. **Выбирайте правильный ключ партицирования.** Запросы должны фильтроваться по этому ключу (Partition Pruning).

2. **Партиция должна быть управляемого размера.** Оптимально — десятки-сотни гигабайт. Слишком маленькие — overhead на управление. Слишком большие — теряется смысл.

3. **Создавайте индексы на КАЖДОЙ партиции отдельно.** Индексы на родительской таблице не создаются автоматически на партициях (в PG 11+ индексы создаются на родительской и автоматически на партициях).

```sql
-- PG 11+: индекс создаётся на всех партициях автоматически
CREATE INDEX idx_logs_created ON logs (created_at);
```

4. **Используйте партицию DEFAULT с осторожностью.** Если в DEFAULT попадает много строк — это узкое горлышко.

5. **Не злоупотребляйте sub-partitioning.** Каждый уровень добавляет overhead. Обычно одного уровня достаточно.

6. **Мониторьте размер партиций:**

```sql
SELECT 
    nmsp_parent.nspname AS parent_schema,
    parent.relname AS parent_table,
    child.relname AS partition_name,
    pg_size_pretty(pg_relation_size(child.oid)) AS size,
    -- Количество строк (если статистика актуальна)
    child.reltuples::bigint AS estimated_rows
FROM pg_inherits
JOIN pg_class parent ON pg_inherits.inhparent = parent.oid
JOIN pg_class child ON pg_inherits.inhrelid = child.oid
JOIN pg_namespace nmsp_parent ON nmsp_parent.oid = parent.relnamespace
WHERE parent.relkind = 'p'  -- партицированная таблица
ORDER BY parent.relname, child.relname;
```

7. **pg_partman** — расширение для автоматического управления партициями (аналог Oracle INTERVAL):

```sql
-- Автоматически создаёт дневные партиции и удаляет старые
SELECT partman.create_parent(
    p_parent_table := 'public.logs',
    p_control := 'created_at',
    p_type := 'native',
    p_interval := '1 day',
    p_premake := 7,        -- создавать на 7 дней вперёд
    p_retention := '90 days' -- удалять старше 90 дней
);
```

## Часть IX. Шардирование и горизонтальное масштабирование

### 9.1. Шардирование vs Партицирование vs Репликация

| Аспект | Репликация | Партицирование | Шардирование |
|--------|-----------|---------------|-------------|
| **Где данные** | Копии на разных узлах | Части на одном узле | Части на разных узлах |
| **Цель** | HA, чтение | Ускорение на одном сервере | Горизонтальное масштабирование |
| **Масштабирование** | Чтения | Записи и чтения (ограничено) | Записи и чтения (линейно) |
| **Сложность** | Низкая-средняя | Средняя | Высокая |
| **Запросы JOIN** | ✅ Полные | ✅ Могут требовать все партиции | ❌ Ограничены (cross-shard join) |
| **Транзакции** | ✅ Полные (на мастере) | ✅ Полные | ❌ Только в пределах шарда |

---

### 9.2. Стратегии шардирования

#### 9.2.1. По диапазону ключа

```
Шард 1: пользователи A–M (user_id 1–500000)
Шард 2: пользователи N–Z (user_id 500001–1000000)
```

✅ Просто. ❌ Неравномерно (пользователи M могут быть активнее).

#### 9.2.2. По хешу ключа

```
shard_id = hash(user_id) % N
```

✅ Равномерное распределение. ❌ Нельзя эффективно делать range-запросы; при добавлении шарда нужен перебаланс.

#### 9.2.3. По бизнес-сущности (Directory-Based)

```
Шард 1: Tenant A, Tenant B
Шард 2: Tenant C
```

✅ Изоляция данных. ❌ Размер шардов разный.

#### 9.2.4. Consistent Hashing

Используется для минимизации перемещения данных при добавлении/удалении шардов.

```
          0
     ┌─────────┐
    /           \
   │   Шард A    │
   │    Шард B   │ N/2
   │   Шард C    │
    \           /
     └─────────┘
          N
```

При добавлении шарда D: только 1/N ключей перемещается, а не все.

### 9.3. Citus: распределённый PostgreSQL

**Citus** — расширение PostgreSQL, превращающее его в распределённую БД. Таблицы делятся на шарды по ключу дистрибуции (distribution column).

```
                  ┌──────────────────┐
                  │   Координатор    │
                  │  (Citus master)  │
                  │                  │
                  │ Принимает запросы│
                  │ Планирует их на  │
                  │ шарды            │
                  └──┬────┬────┬─────┘
                     │    │    │
        ┌────────────┘    │    └────────────┐
        ▼                 ▼                 ▼
  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ Worker 1 │     │ Worker 2 │     │ Worker 3 │
  │ (Шард 1) │     │ (Шард 2) │     │ (Шард 3) │
  │ PG node  │     │ PG node  │     │ PG node  │
  └──────────┘     └──────────┘     └──────────┘
```

#### Дистрибуция таблиц

```sql
-- Распределённая таблица (шардируется по tenant_id)
SELECT create_distributed_table('orders', 'tenant_id');

-- Reference table (полная копия на каждом узле — для небольших справочников)
SELECT create_reference_table('countries');

-- Локальная таблица (только на координаторе)
CREATE TABLE config (key TEXT, value TEXT);
```

#### Типы запросов в Citus

```sql
-- ✅ Быстрый: запрос к одному шарду (tenant_id = конкретный)
SELECT * FROM orders WHERE tenant_id = 42;

-- ✅ Распределённый: агрегация, которая выполняется на каждом worker'е
SELECT tenant_id, SUM(amount) FROM orders GROUP BY tenant_id;

-- ⚠️ Медленный: JOIN между шардированными таблицами по не-ключу дистрибуции
SELECT * FROM orders o JOIN users u ON o.user_id = u.id; -- repartition join

-- ❌ Очень медленный: cross-shard UPDATE/DELETE без ключа дистрибуции
```

#### Когда Citus — правильный выбор

- Multi-tenant SaaS (изоляция по tenant_id).
- Аналитика в реальном времени на больших данных.
- Временные ряды, где данные можно распределить по device_id или user_id.

### 9.4. PostgreSQL FDW (Foreign Data Wrappers)

**FDW** — механизм доступа к данным на **других** серверах PostgreSQL (или других СУБД) как к локальным таблицам. Это не настоящее шардирование, но может использоваться для федеративного доступа.

```sql
-- Расширение
CREATE EXTENSION postgres_fdw;

-- Внешний сервер
CREATE SERVER shard1_server 
FOREIGN DATA WRAPPER postgres_fdw 
OPTIONS (host 'shard1.example.com', port '5432', dbname 'mydb');

-- Mapping пользователя
CREATE USER MAPPING FOR app_user 
SERVER shard1_server 
OPTIONS (user 'app_user', password 'secret');

-- Внешняя таблица (ссылается на таблицу на shard1)
CREATE FOREIGN TABLE orders_shard1 (
    id BIGINT,
    tenant_id INT,
    amount NUMERIC,
    created_at TIMESTAMP
) SERVER shard1_server 
OPTIONS (table_name 'orders');

-- Запрос (postgres_fdw "push-down" отправит WHERE на удалённый сервер)
SELECT * FROM orders_shard1 WHERE tenant_id = 42;
```

**Ограничения FDW:**
- Транзакции только локальные (нет 2PC между разными FDW по умолчанию).
- JOIN между внешними таблицами выполняется локально (может быть медленно).
- Не заменяет шардирование, а дополняет его.

**FDW для других СУБД:**
- `oracle_fdw` — доступ к Oracle.
- `mysql_fdw` — доступ к MySQL.
- `tds_fdw` — доступ к MS SQL Server.
- `mongo_fdw` — доступ к MongoDB.
- `hdfs_fdw` — доступ к Hadoop.

---

### 9.5. Проблемы шардирования

| Проблема | Описание | Решение |
|----------|----------|---------|
| **Cross-shard JOIN** | JOIN таблиц на разных шардах | Денормализация, reference tables (Citus) |
| **Cross-shard транзакции** | ACID между шардами | 2PC, Sagas, eventual consistency |
| **Resharding** | Перебалансировка при добавлении/удалении шарда | Consistent hashing, Citus shard rebalancer |
| **Hot Spot** | Один шард получает непропорционально много нагрузки | Изменить ключ шардирования, разбить горячий шард |
| **Уникальность** | Невозможно глобально уникальное ограничение без координации | UUID, snowflake ID, глобальный сервис ID |
| **Аналитика** | Агрегаты по всем шардам | Распределённые запросы (Citus), Spark, OLAP-решения |

### 9.6. Шардирование в Oracle (Oracle Sharding)

Oracle предоставляет **Oracle Sharding** (с версии 12.2) — нативное шардирование, которое маршрутизирует запросы напрямую к шарду:

```
Клиент (Shard Director)
  │
  │ Определяет шард по sharding key
  │
  ├──► Шард 1 (отдельная Oracle DB)
  ├──► Шард 2 (отдельная Oracle DB)
  └──► Шард 3 (отдельная Oracle DB)
```

- Каждый шард — полноценная Oracle база данных.
- Shard Director маршрутизирует запросы.
- Поддерживает RAC и Data Guard внутри каждого шарда.
- Транзакции ограничены одним шардом.

## Часть X. Кластер и высокая доступность

### 10.1. Что такое кластер базы данных

**Кластер БД** — группа серверов, работающих совместно для обеспечения высокой доступности (HA), отказоустойчивости и балансировки нагрузки.

```
                    ┌─────────────┐
        ┌──────────►│  Сервер 1   │
        │           └─────────────┘
        │
Клиент ─┼───────────►  Сервер 2   │
        │
        │           ┌─────────────┐
        └──────────►│  Сервер N   │
                    └─────────────┘
```

**Основные метрики:**
- **RPO (Recovery Point Objective):** сколько данных можно потерять (в секундах/транзакциях). → определяет тип репликации.
- **RTO (Recovery Time Objective):** сколько времени нужно на восстановление. → определяет механизм failover'а.

### 10.2. Patroni + etcd: отказоустойчивый кластер PostgreSQL

Подробно рассмотрен в разделе [7.5](#75-patroni-автоматический-failover). Архитектура:

```
                    ┌──────────────────┐
                    │   Кластер etcd   │
                    │  (3 или 5 узлов) │
                    └──────────────────┘
                            │
              ┌─────────────┼─────────────┐
              ▼             ▼             ▼
        ┌──────────┐ ┌──────────┐ ┌──────────┐
        │ Patroni  │ │ Patroni  │ │ Patroni  │
        │ PG Node1 │ │ PG Node2 │ │ PG Node3 │
        │ (Master) │ │(Standby) │ │(Standby) │
        └──────────┘ └──────────┘ └──────────┘
              │             ▲
              └─────────────┘
             Streaming Replication
                     │
              ┌──────────────┐
              │   PgBouncer   │  ← Пул соединений
              │ (смотрит DCS) │
              └──────┬───────┘
                     │
              ┌──────┴──────┐
              │  Приложение  │
              └─────────────┘
```

**VIP (Virtual IP) vs HAProxy:**
- VIP: перемещается между узлами. Минус — один IP, сложности с ARP.
- HAProxy: проксирует TCP-трафик. Проверяет, кто мастер, и направляет запись туда.

```haproxy
    server pg2 192.168.1.2:5432 check port 8008
    server pg3 192.168.1.3:5432 check port 8008
```

### 10.3. Pacemaker + Corosync + DRBD

Альтернатива Patroni для более традиционных окружений.

```
┌────────────────────────────────────────┐
│          Corosync (групповая            │
│       коммуникация, кворум)             │
└──────┬─────────────────────┬───────────┘
       │                     │
┌──────┴──────────┐  ┌──────┴──────────┐
│  Узел 1         │  │  Узел 2         │
│  Pacemaker      │  │  Pacemaker      │
│  PostgreSQL     │  │  PostgreSQL     │
│       │         │  │       │         │
│    DRBD         │  │    DRBD         │
│  /dev/drbd0     │◀─┼─ синхронная ───│
│  (Primary)      │  │  репликация     │
│                 │  │  /dev/drbd0     │
│                 │  │  (Secondary)   │
└─────────────────┘  └─────────────────┘
```

- **DRBD (Distributed Replicated Block Device):** синхронная репликация блочного устройства.
- **Corosync:** обеспечивает heartbeat и кворум между узлами.
- **Pacemaker:** ресурсный менеджер; решает, на каком узле запустить PostgreSQL, VIP, DRBD Primary.

**Плюсы:** хорошо отработано, подходит для небольших кластеров (2 узла). **Минусы:** сложно масштабировать больше 2 узлов; нет такой гибкости, как Patroni.

### 10.4. Pgpool-II: пул соединений и балансировка

**Pgpool-II** — многофункциональный посредник между клиентами и PostgreSQL:

```
          ┌──────────┐
          │ Pgpool-II│
          └──────────┘
               │
    ┌──────────┼──────────┐
    ▼          ▼          ▼
┌──────┐  ┌──────┐  ┌──────┐
│ PG 0 │  │ PG 1 │  │ PG 2 │
└──────┘  └──────┘  └──────┘
```

**Функции:**
- Connection Pooling (пул соединений).
- Load Balancing (чтение на реплики, запись на мастер).
- Автоматический failover (watchdog).
- Онлайн-восстановление реплик.

**Сравнение PgBouncer vs Pgpool-II:**

|---|---|---|
| Пул соединений | ✅ Лучший | ✅ |
| Балансировка чтения/записи | ❌ | ✅ |
| Failover | ❌ | ✅ |
| Потребление ресурсов | Минимальное | Среднее |
| Сложность настройки | Минимальная | Высокая |
| Use case | Всегда + Patroni | Когда нужно всё в одном |

### 10.5. RAC (Real Application Clusters) в Oracle

**Oracle RAC** — уникальная технология, позволяющая нескольким экземплярам (instances) Oracle работать с **одной** физической базой данных одновременно. В отличие от PostgreSQL, где мастер только один.

```
┌──────────┐     ┌──────────┐
│ Instance │     │ Instance │
│    1     │     │    2     │
│ (Node 1) │     │ (Node 2) │
└────┬─────┘     └────┬─────┘
     │                │
     └────────┬───────┘
              │
    ┌─────────┴─────────┐
    │  Shared Storage    │
    │  (SAN / ASM)      │
    │  Одна копия данных │
    └───────────────────┘
```

**Как это возможно:**
- **Cache Fusion:** экземпляры обмениваются блоками данных через высокоскоростной interconnect (InfiniBand, 10GbE+).
- Если Instance 1 хочет изменить блок, который держит Instance 2 — блок передаётся напрямую через interconnect, минуя диск.
- Global Resource Directory отслеживает, какой экземпляр каким блоком владеет.

**Цена:** дорогостоящая лицензия Oracle Enterprise Edition + RAC option. Требует специализированного оборудования (SAN, выделенный interconnect).

**Аналогов в PostgreSQL нет.** Это ключевое преимущество Oracle для систем с требованием к горизонтальному масштабированию записи без шардирования.

### 10.6. Расчёт RPO/RTO и выбор архитектуры

| Конфигурация | RPO | RTO | Сложность | Стоимость |
|:---|:---:|:---:|:---:|:---:|
| Один сервер + бэкапы | Часы | Часы-дни | ★☆☆☆☆ | $ |
| Асинхронная репликация | Секунды | 1–5 мин (ручной failover) | ★★☆☆☆ | $$ |
| Синхронная репликация | 0 | 1–5 мин (ручной) | ★★☆☆☆ | $$$ |
| Patroni + etcd | Секунды / 0 | < 30 сек | ★★★☆☆ | $$ |
| Patroni + синхронная репл. | 0 | < 30 сек | ★★★★☆ | $$$ |
| Oracle RAC | 0 | < 5 сек | ★★★★★ | $$$$$ |

## Часть XI. Язык SQL: глубокое погружение

### 11.1. Категории команд SQL

| Категория | Полное название | Команды | Что делает |
|-----------|-----------------|---------|------------|
| **DDL** | Data Definition Language | `CREATE`, `ALTER`, `DROP`, `TRUNCATE`, `RENAME` | Структура БД |
| **DML** | Data Manipulation Language | `SELECT`, `INSERT`, `UPDATE`, `DELETE`, `MERGE` | Данные |
| **DCL** | Data Control Language | `GRANT`, `REVOKE` | Права доступа |
| **TCL** | Transaction Control Language | `BEGIN`, `COMMIT`, `ROLLBACK`, `SAVEPOINT` | Управление транзакциями |
| **DQL** | Data Query Language | `SELECT` | Запросы (часто относят к DML) |

**Важнейшее отличие PostgreSQL:** DDL **транзакционен**! Можно:

```sql
BEGIN;
CREATE TABLE users (id SERIAL PRIMARY KEY, name TEXT);
ALTER TABLE users ADD COLUMN email TEXT;
CREATE INDEX idx_users_email ON users (email);
-- Что-то пошло не так...
ROLLBACK;
-- Все CREATE/ALTER/DROP откатились!
```

В Oracle DDL делает неявный COMMIT до и после команды — откатить нельзя.

### 11.2. JOIN'ы: полный разбор

#### Типы JOIN'ов (наглядная диаграмма)

```
INNER JOIN:          LEFT JOIN:           RIGHT JOIN:
┌─────┬─────┐       ┌─────┬─────┐        ┌─────┬─────┐
│  A  │  B  │       │█████│  B  │        │  A  │█████│
└─────┴─────┘       └─────┴─────┘        └─────┴─────┘
Только совпадающие   Все из A +          Все из B +
строки               совпадающие из B    совпадающие из A

FULL JOIN:           CROSS JOIN:          NATURAL JOIN:
┌─────┬─────┐       ┌─────┬─────┐        Автоматически по
│█████│█████│       │  A  │  B  │        одноимённым столбцам
└─────┴─────┘       │  x  │  y  │        (ОПАСНО! Не использовать
Все из A и B        │  z  │  w  │         в production)
                    └─────┴─────┘
                    Декартово произведение
```

#### Примеры

```sql
-- INNER JOIN: только пользователи, у которых есть заказы
SELECT u.name, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN: все пользователи, даже без заказов (total = NULL)
SELECT u.name, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

-- RIGHT JOIN: все заказы, даже если пользователь удалён
SELECT u.name, o.total
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id;

-- FULL OUTER JOIN: все пользователи и все заказы
SELECT u.name, o.total
FROM users u
FULL OUTER JOIN orders o ON u.id = o.user_id;

-- CROSS JOIN: все комбинации (Декартово произведение)
SELECT * FROM colors CROSS JOIN sizes;
-- цвета: красный, синий; размеры: S, M, L
-- Результат: (красный,S), (красный,M), (красный,L), (синий,S), ...

-- SELF JOIN: таблица соединяется сама с собой
SELECT e1.name AS employee, e2.name AS manager
FROM employees e1
LEFT JOIN employees e2 ON e1.manager_id = e2.id;
```

#### LATERAL JOIN

`LATERAL` позволяет подзапросу в FROM ссылаться на столбцы из предыдущих таблиц:

```sql
-- Для каждого пользователя найти 3 последних заказа
SELECT u.name, o.order_date, o.total
FROM users u
LEFT JOIN LATERAL (
    SELECT order_date, total
    FROM orders
    WHERE user_id = u.id
    ORDER BY order_date DESC
    LIMIT 3
) o ON true;

-- Без LATERAL это невозможно (подзапрос не может видеть u.id)
```

#### SEMI JOIN и ANTI JOIN

**Не путать с синтаксисом!** Это логические операции в плане выполнения, не имеющие прямого синтаксиса в SQL:

```sql
-- SEMI JOIN: EXISTS (есть ли хоть одна строка?)
SELECT * FROM users u
WHERE EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);

-- ANTI JOIN: NOT EXISTS (нет ни одной?)
SELECT * FROM users u
WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.user_id = u.id);
```

PostgreSQL и Oracle эффективно выполняют их через Hash Semi Join / Hash Anti Join в плане выполнения.

### 11.3. Подзапросы и производные таблицы

#### Типы подзапросов

**Скалярный подзапрос** — возвращает одно значение:
```sql
SELECT name, 
       (SELECT COUNT(*) FROM orders WHERE user_id = u.id) AS order_count
FROM users u;
```

**Подзапрос в FROM (производная таблица):**
```sql
SELECT dept, avg_salary
FROM (
    SELECT department AS dept, AVG(salary) AS avg_salary
    FROM employees
    GROUP BY department
) AS dept_stats
WHERE avg_salary > 50000;
```

**Подзапрос в WHERE:**
```sql
SELECT * FROM employees
WHERE department_id IN (SELECT id FROM departments WHERE location = 'NYC');
```

**Коррелированный подзапрос** — ссылается на внешний запрос (медленнее):
```sql
SELECT name, salary
FROM employees e1
WHERE salary > (
    SELECT AVG(salary)
    FROM employees e2
    WHERE e2.department_id = e1.department_id  -- корреляция!
);
```

#### Оптимизация подзапросов

PostgreSQL умеет «разворачивать» (unnest) некоторые подзапросы:
- `IN (SELECT ...)` → Semi Join
- `NOT IN (SELECT ...)` → Anti Join
- `EXISTS` → Semi Join
- `NOT EXISTS` → Anti Join

**Предпочитайте EXISTS вместо IN** (если нет индекса) и **JOIN вместо коррелированных подзапросов**.

### 11.4. Оконные функции (Window Functions)

Оконные функции выполняют вычисления над **набором строк, связанных с текущей строкой** (окном), не группируя их.

```sql
window_function(arg) OVER (
    [PARTITION BY expression]
    [ORDER BY expression [ASC|DESC] [NULLS {FIRST|LAST}]]
    [frame_clause]
)
```

#### Основные оконные функции

| Категория | Функции |
|-----------|---------|
| **Агрегатные** | `SUM()`, `AVG()`, `COUNT()`, `MIN()`, `MAX()` |
| **Ранжирующие** | `ROW_NUMBER()`, `RANK()`, `DENSE_RANK()`, `NTILE(n)` |
| **Смещения** | `LAG()`, `LEAD()`, `FIRST_VALUE()`, `LAST_VALUE()`, `NTH_VALUE()` |
| **Статистические** | `PERCENT_RANK()`, `CUME_DIST()` |

#### Примеры

```sql
-- ROW_NUMBER: уникальный номер строки в окне
SELECT 
    department,
    name,
    salary,
    ROW_NUMBER() OVER (PARTITION BY department ORDER BY salary DESC) AS rank
FROM employees;

-- LAG/LEAD: предыдущее/следующее значение
SELECT 
    date,
    revenue,
    LAG(revenue) OVER (ORDER BY date) AS prev_day_revenue,
    revenue - LAG(revenue) OVER (ORDER BY date) AS change,
    ROUND(100.0 * (revenue - LAG(revenue) OVER (ORDER BY date)) 
          / LAG(revenue) OVER (ORDER BY date), 2) AS change_pct
FROM daily_stats
ORDER BY date;

-- SUM с накоплением (running total)
SELECT 
    date,
    amount,
    SUM(amount) OVER (ORDER BY date) AS running_total
FROM transactions;

-- FIRST_VALUE: первая зарплата в отделе
SELECT 
    department,
    name,
    salary,
    FIRST_VALUE(salary) OVER (PARTITION BY department ORDER BY hire_date) 
        AS first_salary_in_dept
FROM employees;

-- Процентили
SELECT 
    name,
    score,
    NTILE(4) OVER (ORDER BY score) AS quartile,
    PERCENT_RANK() OVER (ORDER BY score) AS percentile
FROM exam_results;
```

#### Рамка окна (Window Frame)

```sql
-- RAMKA: ROWS vs RANGE vs GROUPS
-- По умолчанию: RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW

-- Скользящее среднее за 3 дня (включая текущий)
SELECT 
    date,
    amount,
    AVG(amount) OVER (
        ORDER BY date 
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS moving_avg_3d
FROM daily_sales;

-- Сумма за предыдущий, текущий и следующий день
SUM(amount) OVER (
    ORDER BY date
    ROWS BETWEEN 1 PRECEDING AND 1 FOLLOWING
) AS three_day_window;

-- RANGE: по значению, не по строкам (например, ±7 дней)
SUM(amount) OVER (
    ORDER BY date
    RANGE BETWEEN INTERVAL '7 days' PRECEDING AND CURRENT ROW
) AS weekly_sum;
```

**ROWS vs RANGE:**
- `ROWS`: физические строки (вперёд/назад на N строк).
- `RANGE`: логический диапазон значений (все строки с таким же ORDER BY значением). Для дат можно указать `INTERVAL`.
- `GROUPS` (PG 11+): как ROWS, но включает все строки с одинаковым значением ORDER BY.

### 11.5. CTE (Common Table Expressions) — WITH

CTE — именованный подзапрос, определённый до основного запроса.

#### Обычный CTE

```sql
WITH user_stats AS (
    SELECT user_id, COUNT(*) AS order_count, SUM(total) AS total_spent
    FROM orders
    GROUP BY user_id
)
SELECT u.name, us.order_count, us.total_spent
FROM users u
JOIN user_stats us ON u.id = us.user_id
WHERE us.total_spent > 1000;
```

#### Рекурсивный CTE

Используется для иерархических данных (деревья, графы):

```sql
-- Таблица: employees (id, name, manager_id)
WITH RECURSIVE subordinates AS (
    -- Базовый случай: директор
    SELECT id, name, manager_id, 1 AS level
    FROM employees
    WHERE manager_id IS NULL
    
    UNION ALL
    
    -- Рекурсивный случай: подчинённые
    SELECT e.id, e.name, e.manager_id, s.level + 1
    FROM employees e
    JOIN subordinates s ON e.manager_id = s.id
)
SELECT 
    REPEAT('  ', level - 1) || name AS org_chart,
    level
FROM subordinates
ORDER BY id;
```

**Результат:**
```
CEO                         (level 1)
  VP Engineering            (level 2)
    Senior Developer        (level 3)
      Developer             (level 4)
  VP Sales                  (level 2)
```

#### MATERIALIZED vs NOT MATERIALIZED (PG 12+)

```sql
-- По умолчанию: PG решает сам. Можно указать явно:
WITH cte AS MATERIALIZED (  -- копируется во временную область
    SELECT * FROM expensive_query
)
SELECT * FROM cte WHERE ...;

WITH cte AS NOT MATERIALIZED (  -- инлайнится как подзапрос
    SELECT * FROM simple_query
)
SELECT * FROM cte WHERE ...;
```

#### Множественные CTE

```sql
WITH
monthly_sales AS (
    SELECT date_trunc('month', order_date) AS month, SUM(total) AS revenue
    FROM orders
    GROUP BY date_trunc('month', order_date)
),
sales_with_growth AS (
    SELECT 
        month,
        revenue,
        LAG(revenue) OVER (ORDER BY month) AS prev_revenue,
        ROUND(100.0 * (revenue - LAG(revenue) OVER (ORDER BY month)) 
              / NULLIF(LAG(revenue) OVER (ORDER BY month), 0), 2) AS growth_pct
    FROM monthly_sales
)
SELECT * FROM sales_with_growth WHERE growth_pct > 10;
```

### 11.6. Агрегатные функции и GROUP BY

#### GROUP BY и HAVING

```sql
SELECT 
    department_id,
    COUNT(*) AS emp_count,
    AVG(salary) AS avg_salary,
    MIN(hire_date) AS earliest_hire,
    MAX(salary) AS max_salary
FROM employees
WHERE status = 'active'          -- фильтрует ДО группировки
GROUP BY department_id
HAVING AVG(salary) > 50000      -- фильтрует ПОСЛЕ группировки
ORDER BY avg_salary DESC;
```

#### GROUPING SETS, CUBE, ROLLUP

```sql
-- GROUPING SETS: несколько группировок в одном запросе
SELECT 
    department_id,
    job_title,
    SUM(salary) AS total_salary
FROM employees
GROUP BY GROUPING SETS (
    (department_id),            -- сумма по отделу
    (job_title),                -- сумма по должности
    (department_id, job_title), -- сумма по отделу и должности
    ()                           -- общая сумма
);

-- ROLLUP: иерархические итоги
SELECT 
    COALESCE(region, 'Все регионы') AS region,
    COALESCE(department, 'Все отделы') AS department,
    SUM(salary) AS total_salary
FROM employees
GROUP BY ROLLUP (region, department);
-- Результат: (region, dept), (region, NULL), (NULL, NULL) — итоги по иерархии

-- CUBE: все комбинации
SELECT 
    region,
    department,
    SUM(salary) AS total_salary
FROM employees
GROUP BY CUBE (region, department);
-- Все комбинации: (r,d), (r,NULL), (NULL,d), (NULL,NULL)
```

#### FILTER (агрегаты с условием)

```sql
-- Синтаксис FILTER (PG 9.4+)
SELECT 
    department_id,
    COUNT(*) AS total,
    COUNT(*) FILTER (WHERE salary > 50000) AS high_earners,
    AVG(salary) FILTER (WHERE age > 30) AS avg_salary_30plus,
    SUM(salary) FILTER (WHERE status = 'active') AS active_payroll
FROM employees
GROUP BY department_id;

-- Альтернатива без FILTER (стандартный SQL)
SELECT 
    department_id,
    COUNT(*) AS total,
    COUNT(CASE WHEN salary > 50000 THEN 1 END) AS high_earners,
    AVG(CASE WHEN age > 30 THEN salary END) AS avg_salary_30plus
FROM employees
GROUP BY department_id;
```

#### DISTINCT vs GROUP BY

```sql
-- Эти два запроса эквивалентны:
SELECT DISTINCT department_id FROM employees;
SELECT department_id FROM employees GROUP BY department_id;

-- GROUP BY может то, чего не может DISTINCT (агрегаты):
SELECT department_id, COUNT(*) FROM employees GROUP BY department_id;
```

### 11.7. UNION, INTERSECT, EXCEPT

```sql
-- UNION: объединение (удаляет дубликаты)
SELECT name FROM users WHERE status = 'active'
UNION
SELECT name FROM users WHERE vip = true;

-- UNION ALL: объединение (НЕ удаляет дубликаты — быстрее!)
SELECT name FROM users WHERE status = 'active'
UNION ALL
SELECT name FROM users WHERE vip = true;

-- INTERSECT: пересечение
SELECT user_id FROM orders WHERE order_date >= '2024-01-01'
INTERSECT
SELECT user_id FROM orders WHERE total > 1000;

-- EXCEPT: разность (в PG 9.4+)
SELECT user_id FROM users
EXCEPT
SELECT user_id FROM orders WHERE order_date >= '2024-01-01';
-- Пользователи, не сделавшие заказов в 2024 году
```

### 11.8. NULL: семантика и тонкости

NULL в SQL — это **не значение**, а маркер отсутствия значения. Это трёхзначная логика:

| A | B | A = B | A AND B | A OR B |
|:---|:---|:-----|:-------|:------|
| TRUE | NULL | NULL | NULL | TRUE |
| FALSE | NULL | NULL | FALSE | NULL |
| NULL | NULL | NULL | NULL | NULL |

#### Проверка на NULL

```sql
-- ПРАВИЛЬНО
SELECT * FROM users WHERE email IS NULL;
SELECT * FROM users WHERE email IS NOT NULL;

-- НЕПРАВИЛЬНО (всегда вернёт 0 строк)
SELECT * FROM users WHERE email = NULL;
SELECT * FROM users WHERE email != NULL;

-- IS DISTINCT FROM: безопасное сравнение с NULL
SELECT * FROM users WHERE email IS DISTINCT FROM 'alice@example.com';
-- Работает как != но учитывает NULL: NULL IS DISTINCT FROM 'x' → TRUE
```

#### NULL в агрегатах

```sql
-- COUNT(*) считает строки; COUNT(column) считает NOT NULL значения
SELECT COUNT(*) FROM users;      -- 1000
SELECT COUNT(email) FROM users;  -- 950 (50 NULL)

-- SUM, AVG игнорируют NULL
SELECT AVG(salary) FROM employees; -- NULL не участвуют в среднем

-- COALESCE: первое не-NULL значение
SELECT COALESCE(phone, email, 'нет контакта') AS contact FROM users;

-- NULLIF: вернуть NULL если значения равны
SELECT NULLIF(divisor, 0) FROM calculations; -- защита от деления на 0
```

#### NULL в ORDER BY

```sql
-- По умолчанию NULL считается БОЛЬШЕ любого значения (NULLS LAST в ASC)
SELECT * FROM users ORDER BY email;  -- NULLS LAST по умолчанию в PG
SELECT * FROM users ORDER BY email NULLS FIRST;
SELECT * FROM users ORDER BY email DESC NULLS LAST;  -- NULLS FIRST по умолчанию в DESC
```

### 11.9. Транзакционный DDL (PostgreSQL) vs нетранзакционный DDL (Oracle)

**PostgreSQL:** почти все DDL команды можно откатить.

```sql
BEGIN;
CREATE TABLE test_table (id INT);
INSERT INTO test_table VALUES (1);
DROP TABLE test_table;
ROLLBACK;
-- test_table существовала только внутри транзакции. После ROLLBACK её нет.
```

Исключения в PG: операции с файловой системой (`CREATE TABLESPACE`), некоторые команды управления сервером.

**Oracle:** DDL делает неявные коммиты до и после:

```sql
-- Oracle
INSERT INTO users VALUES (1, 'Alice');
CREATE TABLE orders (...);  -- ← неявный COMMIT перед CREATE!
-- INSERT уже закоммичен, откатить нельзя!
```

Это важно при миграциях: в PostgreSQL можно завернуть все миграции в одну транзакцию (если используются соответствующие lock-режимы).

### 11.10. Регулярные выражения и полнотекстовый поиск

#### Регулярные выражения в PostgreSQL

```sql
-- POSIX regex
SELECT * FROM users WHERE email ~ '^[a-z]+@example\.com$';      -- match
SELECT * FROM users WHERE email ~* 'ALICE@EXAMPLE.COM';        -- case insensitive
SELECT * FROM users WHERE email !~ '@spam\.com$';              -- NOT match

-- Извлечение
SELECT regexp_matches('abc123def', '\d+');        -- {123}
SELECT regexp_replace('abc123def', '\d+', 'XXX'); -- abcXXXdef
SELECT regexp_split_to_table('a,b,c,d', ',');      -- a \n b \n c \n d
```

#### Полнотекстовый поиск (Full-Text Search)

```sql
-- Создание tsvector (документ)
SELECT to_tsvector('russian', 'Привет, мир! Это тестовый документ.');
-- 'документ':5 'мир':2 'привет':1 'тестов':4 'эт':3

-- Создание tsquery (запрос)
SELECT to_tsquery('russian', 'мир & документ');
-- 'мир' & 'документ'

-- Поиск с ранжированием
SELECT 
    title,
    ts_rank(to_tsvector('russian', content), 
            to_tsquery('russian', 'архитектура & база & данных')) AS rank
FROM articles
WHERE to_tsvector('russian', content) @@ to_tsquery('russian', 'архитектура & база & данных')
ORDER BY rank DESC;

-- Выделение совпадений
SELECT 
    ts_headline('russian', content, to_tsquery('russian', 'архитектура'),
                'StartSel=<mark>, StopSel=</mark>, MaxWords=30') AS headline
FROM articles
WHERE ...;
```

### 11.11. INSERT ... ON CONFLICT (UPSERT)

**UPSERT** = UPDATE + INSERT. Если строка с таким ключом существует — обновить, если нет — вставить. В PostgreSQL реализован через `ON CONFLICT` (PG 9.5+):

```sql
-- Базовый синтаксис
INSERT INTO users (id, name, email, login_count)
VALUES (1, 'Alice', 'alice@example.com', 1)
ON CONFLICT (email)           -- конфликт по уникальному столбцу
DO UPDATE SET                 -- обновить
    name = EXCLUDED.name,
    login_count = users.login_count + 1;
-- EXCLUDED — псевдо-таблица со значениями, которые мы ПЫТАЛИСЬ вставить

-- Ничего не делать при конфликте (аналог INSERT IGNORE в MySQL)
INSERT INTO users (id, name, email)
VALUES (1, 'Alice', 'alice@example.com')
ON CONFLICT (email) DO NOTHING;

-- Конфликт по составному ключу или constraint name
INSERT INTO orders (id, user_id, product_id, quantity)
VALUES (100, 1, 42, 5)
ON CONFLICT ON CONSTRAINT orders_pkey
DO UPDATE SET quantity = orders.quantity + EXCLUDED.quantity;

-- Условное обновление (WHERE в DO UPDATE)
INSERT INTO users (id, name, email, updated_at)
VALUES (1, 'Alice', 'alice@example.com', now())
ON CONFLICT (email)
DO UPDATE SET 
    name = EXCLUDED.name,
    updated_at = now()
WHERE users.updated_at < EXCLUDED.updated_at;  -- обновить только если новее
```

**Как работает внутри:**
1. Пробует INSERT.
2. Если нарушен UNIQUE-constraint (или exclusion constraint):
   - `DO NOTHING` → игнорирует строку.
   - `DO UPDATE` → выполняет UPDATE с условием, ограниченным конфликтующей строкой.
3. Это **атомарно** — гонки между параллельными INSERT исключены.

**Важно:** `ON CONFLICT` требует, чтобы конфликт был по уникальному индексу или constraint. Без индекса — ошибка.

### 11.12. MERGE (SQL Standard UPSERT)

**MERGE** (PG 15+) — стандартный SQL-синтаксис для UPSERT, более гибкий, чем `ON CONFLICT`:

```sql
MERGE INTO users AS target
USING (VALUES (1, 'Alice', 'alice@example.com')) AS source(id, name, email)
ON target.email = source.email
WHEN MATCHED THEN
    UPDATE SET name = source.name
WHEN NOT MATCHED THEN
    INSERT (id, name, email) VALUES (source.id, source.name, source.email);

-- MERGE с несколькими условиями
MERGE INTO orders o
USING new_orders n ON o.id = n.id
WHEN MATCHED AND o.status = 'pending' THEN
    UPDATE SET quantity = n.quantity, status = n.status
WHEN MATCHED AND o.status = 'shipped' THEN
    DELETE                                          -- можно удалить!
WHEN NOT MATCHED THEN
    INSERT (id, user_id, product_id, quantity, status)
    VALUES (n.id, n.user_id, n.product_id, n.quantity, n.status);
```

**MERGE vs ON CONFLICT:**
- `ON CONFLICT` — только для INSERT с конфликтом. Проще, быстрее.
- `MERGE` — полноценный синтаксис: можно DELETE при MATCHED, можно несколько WHEN-условий, можно использовать произвольный источник (таблицу, подзапрос).

### 11.13. RETURNING — возврат данных из DML

**RETURNING** (PG-специфичный) позволяет вернуть строки, затронутые INSERT/UPDATE/DELETE, без дополнительного SELECT:

```sql
-- INSERT с возвратом
INSERT INTO users (name, email) 
VALUES ('Alice', 'alice@example.com')
RETURNING id, created_at;

-- UPDATE с возвратом старых и новых значений
UPDATE users 
SET login_count = login_count + 1
WHERE id = 42
RETURNING id, login_count AS new_count;

-- DELETE с возвратом удалённых данных (архивация)
DELETE FROM logs 
WHERE created_at < now() - interval '90 days'
RETURNING *;

-- INSERT ... ON CONFLICT с RETURNING (возвращает и вставленные, и обновлённые)
INSERT INTO users (id, name, email, login_count)
VALUES (1, 'Alice', 'alice@example.com', 1)
ON CONFLICT (email) 
DO UPDATE SET login_count = users.login_count + 1
RETURNING id, login_count, 
          (xmax = 0) AS was_inserted;  -- true для INSERT, false для UPDATE
```

**Применение:**
- Получить сгенерированный ID после INSERT.
- Аудит: вернуть удалённые данные для архивирования.
- Избежать race conditions: INSERT + RETURNING атомарно, в отличие от INSERT + SELECT.

### 11.14. Generated Columns (генерируемые столбцы)

**Generated Columns** (PG 12+) — значения столбца вычисляются автоматически из других столбцов. В PostgreSQL только `STORED` (хранятся на диске):

```sql
CREATE TABLE employees (
    id SERIAL PRIMARY KEY,
    first_name TEXT,
    last_name TEXT,
    full_name TEXT GENERATED ALWAYS AS (first_name || ' ' || last_name) STORED,
    annual_salary NUMERIC,
    monthly_salary NUMERIC GENERATED ALWAYS AS (annual_salary / 12) STORED
);

-- Генерируемые столбцы нельзя обновлять напрямую:
INSERT INTO employees (first_name, last_name, annual_salary)
VALUES ('Bob', 'Smith', 120000);
-- full_name = 'Bob Smith', monthly_salary = 10000
```

**Когда использовать:**
- Денормализация: хранить вычисленное значение, чтобы не вычислять при каждом SELECT.
- Индексация вычисляемого выражения.

```sql
CREATE INDEX idx_emp_monthly ON employees (monthly_salary);
```

### 11.15. Row-Level Security (RLS) — безопасность на уровне строк

**RLS** (PG 9.5+) позволяет ограничить, какие строки видит/изменяет пользователь, на основе политик. Работает на уровне базы данных:

```sql
-- Включить RLS для таблицы
ALTER TABLE projects ENABLE ROW LEVEL SECURITY;

-- Политика: пользователь видит только свои проекты
CREATE POLICY project_owner_policy ON projects
    FOR SELECT
    USING (owner_id = current_setting('app.user_id')::INT);

-- Политика: можно обновлять только свои проекты
CREATE POLICY project_owner_update ON projects
    FOR UPDATE
    USING (owner_id = current_setting('app.user_id')::INT)
    WITH CHECK (owner_id = current_setting('app.user_id')::INT);

-- Политика: только админы могут удалять
CREATE POLICY project_admin_delete ON projects
    FOR DELETE
    USING (current_setting('app.role') = 'admin');
```

**Как это работает с приложением:**
```sql
-- Приложение устанавливает контекст в начале сессии
SET app.user_id = '42';
SET app.role = 'user';

-- Теперь все запросы автоматически фильтруются:
SELECT * FROM projects; -- видны только проекты пользователя 42
```

**Важные нюансы:**
- Владелец таблицы (и суперпользователи) **обходят** RLS по умолчанию. Чтобы включить и для владельца:
  ```sql
  ALTER TABLE projects FORCE ROW LEVEL SECURITY;
  ```
- RLS-политики могут повлиять на производительность — планировщик учитывает их, но не всегда оптимально.
- `BYPASSRLS` атрибут роли позволяет обходить RLS.

## Часть XII. Оптимизация производительности

### 12.1. Статистика и ANALYZE

Планировщик PostgreSQL опирается на статистику распределения данных. Без актуальной статистики — плохие планы.

```sql
-- Ручной сбор статистики
ANALYZE table_name;
ANALYZE table_name (column1, column2); -- только определённые столбцы

-- Статистика хранится в pg_statistic (не читается напрямую)
-- Для чтения есть pg_stats:
SELECT 
    tablename,
    attname,
    n_distinct,       -- количество уникальных значений
    most_common_vals,  -- наиболее частые значения
    most_common_freqs, -- их частоты
    histogram_bounds,  -- гистограмма распределения
    correlation        -- корреляция с физическим порядком
FROM pg_stats
WHERE tablename = 'orders';

-- Расширенная статистика (для коррелированных столбцов!)
CREATE STATISTICS s1 (dependencies, mcv) ON department_id, salary 
FROM employees;
ANALYZE employees;
-- Теперь планировщик знает, что в отделе "IT" зарплаты выше!
```

#### Параметры статистики

```ini
default_statistics_target = 100  # объём выборки для ANALYZE
# Можно увеличить для отдельных столбцов:
ALTER TABLE orders ALTER COLUMN user_id SET STATISTICS 1000;
```

### 12.2. Конфигурация PostgreSQL (postgresql.conf)

**Ключевые параметры производительности:**

| Параметр | Рекомендация | За что отвечает |
|----------|-------------|-----------------|
| `shared_buffers` | 25% RAM | Буферный кэш страниц |
| `effective_cache_size` | 75% RAM | Оценка доступного кэша для планировщика |
| `work_mem` | 64MB–256MB (с осторожностью!) | Память на операцию (сортировка, хеш) |
| `maintenance_work_mem` | 256MB–1GB | Память для VACUUM, CREATE INDEX |
| `wal_buffers` | -1 (авто) или 16–64MB | Буфер WAL |
| `random_page_cost` | 1.0–1.5 для SSD, 4.0 для HDD | Стоимость произвольного доступа |
| `effective_io_concurrency` | 200 для SSD, 2 для HDD | Параллельные I/O операции |
| `max_connections` | Минимально необходимое + PgBouncer | Максимум соединений |
| `checkpoint_timeout` | 5–15 мин | Интервал контрольных точек |
| `max_wal_size` | 2–16 GB | Максимальный размер WAL |
| `min_wal_size` | 1–4 GB | Минимальный размер WAL |
| `autovacuum_max_workers` | 3–10 | Параллельных autovacuum workers |

#### Формулы для начальной конфигурации (выделенный сервер)

```python
# Псевдокод для pg_generate_config()
total_ram_mb = 32768  # 32 GB RAM

shared_buffers = total_ram_mb * 0.25
effective_cache_size = total_ram_mb * 0.75
work_mem = min(total_ram_mb * 0.25 / (max_connections * 2), 256)
maintenance_work_mem = total_ram_mb * 0.05
wal_buffers = -1  # авто
random_page_cost = 1.1 if is_ssd else 4.0
effective_io_concurrency = 200 if is_ssd else 2
```

### 12.3. Мониторинг: ключевые метрики и представления

```sql
-- Количество активных/ожидающих соединений
SELECT state, COUNT(*) 
FROM pg_stat_activity 
GROUP BY state;

-- Соотношение cache hit/read (должно быть > 95% hit)
SELECT 
    SUM(blks_hit) AS cache_hits,
    SUM(blks_read) AS disk_reads,
    ROUND(100.0 * SUM(blks_hit) / NULLIF(SUM(blks_hit) + SUM(blks_read), 0), 2) 
        AS cache_hit_ratio
FROM pg_stat_database;

-- Размеры таблиц и индексов
SELECT 
    schemaname || '.' || tablename AS table_name,
    pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename)) AS total_size,
    pg_size_pretty(pg_relation_size(schemaname || '.' || tablename)) AS table_size,
    pg_size_pretty(pg_total_relation_size(schemaname || '.' || tablename) 
                   - pg_relation_size(schemaname || '.' || tablename)) AS indexes_size
FROM pg_tables
WHERE schemaname NOT IN ('pg_catalog', 'information_schema')
ORDER BY pg_total_relation_size(schemaname || '.' || tablename) DESC
LIMIT 20;

-- Долгие запросы (активные)
SELECT 
    pid,
    now() - query_start AS duration,
    state,
    wait_event_type,
    wait_event,
    LEFT(query, 100) AS query
FROM pg_stat_activity
WHERE state != 'idle'
  AND query_start < now() - interval '1 second'
ORDER BY query_start;

-- Количество dead tuples (важно для VACUUM!)
SELECT 
    schemaname || '.' || relname AS table_name,
    n_live_tup AS live_tuples,
    n_dead_tup AS dead_tuples,
    ROUND(100.0 * n_dead_tup / NULLIF(n_live_tup + n_dead_tup, 0), 2) 
        AS dead_ratio,
    last_autovacuum,
    last_autoanalyze
FROM pg_stat_user_tables
WHERE n_dead_tup > 0
ORDER BY n_dead_tup DESC;

-- Использование индексов
SELECT 
    schemaname || '.' || relname AS table_name,
    indexrelname AS index_name,
    idx_scan AS index_scans,
    idx_tup_read AS tuples_read,
    idx_tup_fetch AS tuples_fetched,
    pg_size_pretty(pg_relation_size(indexrelid)) AS index_size
FROM pg_stat_user_indexes
ORDER BY idx_scan ASC;
```

### 12.4. Оптимизация запросов: методика

#### Алгоритм оптимизации

```
1. Найти медленный запрос
   ├── pg_stat_statements (по total_time)
   ├── pg_stat_activity (текущие долгие запросы)
   └── log_min_duration_statement (логи)

2. Получить план выполнения
   └── EXPLAIN (ANALYZE, BUFFERS) [запрос]

3. Найти «узкое место»
   ├── Seq Scan на большой таблице? → индекс
   ├── Nested Loop с большим loops? → может Hash Join
   ├── Sort на диске? → work_mem или индекс
   ├── Heap Fetches в Index Only Scan? → VACUUM
   ├── Bitmap Heap Scan с потерями? → пересмотреть индекс
   └── Много dead tuples? → autovacuum настройки

4. Применить исправление
   ├── Создать/изменить индекс
   ├── Переписать запрос
   ├── Изменить конфигурацию
   └── VACUUM / REINDEX

5. Проверить результат
   └── EXPLAIN (ANALYZE, BUFFERS) и сравнить
```

#### Пример оптимизации

```sql
-- Медленный запрос (5 секунд)
SELECT o.*, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.created_at BETWEEN '2024-01-01' AND '2024-01-31'
  AND o.status = 'completed';

-- EXPLAIN показывает:
-- Seq Scan on orders → читает 10 млн строк
-- Filter: (status = 'completed') → отбрасывает 90%

-- Решение:
CREATE INDEX idx_orders_created_status ON orders (created_at, status);

-- Новый EXPLAIN:
-- Index Scan using idx_orders_created_status → читает сразу нужные строки
-- Время: 5 сек → 50 мс (в 100 раз быстрее!)
```

### 12.6. JIT-компиляция (Just-In-Time Compilation)

**JIT** (начиная с PG 11) — компиляция частей запроса в нативный машинный код во время выполнения. Особенно эффективна для аналитических запросов с большим количеством вычислений (WHERE-выражения, агрегаты, проекции).

#### Как работает

```
SQL запрос
    │
    ▼
Планировщик → План выполнения
    │
    ▼
Executor: интерпретирует план
    │  Использует JIT?
    ├── Нет → интерпретация (медленнее для CPU-bound запросов)
    │
    └── Да  → LLVM JIT
              ├── Компиляция выражений WHERE в машинный код
              ├── Компиляция агрегатных функций
              ├── Компиляция проекций (SELECT list)
              └── Компиляция операторов соединения
```

PostgreSQL использует **LLVM** для JIT-компиляции (должен быть собран с `--with-llvm`).

#### Параметры

```ini
jit = on                          # включить JIT
jit_above_cost = 100000           # минимальная стоимость плана для JIT
jit_inline_above_cost = 500000    # стоимость для инлайнинга функций
jit_optimize_above_cost = 500000  # стоимость для оптимизаций LLVM
```

**Смысл:** JIT имеет накладные расходы на компиляцию. Для коротких запросов (OLTP) он невыгоден. Для долгих аналитических запросов с миллионами строк — даёт значительный прирост (иногда 20–40%).

#### Когда JIT эффективен

| Сценарий | JIT? |
|----------|:----:|
| OLTP: точечные запросы (< 1 мс) | ❌ Выкл. |
| OLTP: простые запросы с несколькими строками | ❌ Выкл. |
| Аналитика: сканирование миллионов строк с WHERE-выражениями | ✅ Вкл. |
| Аналитика: сложные агрегаты (SUM, AVG по выражениям) | ✅ Вкл. |
| Отчётность: множество вычисляемых столбцов | ✅ Вкл. |

#### Мониторинг JIT

```sql
-- Статистика JIT в плане запроса
EXPLAIN (ANALYZE, BUFFERS) SELECT ...;
-- JIT:
--   Functions: 8
--   Options: Inlining true, Optimization true, Expressions true, Deforming true
--   Timing: Generation 2.5 ms, Inlining 15.3 ms, Optimization 45.2 ms, 
--           Emission 30.1 ms, Total 93.1 ms
```

Если время компиляции (Total) сопоставимо с временем выполнения или превышает его — JIT лучше отключить (`SET jit = off;`).

### 12.7. Пул соединений: PgBouncer

PostgreSQL плохо масштабируется по числу одновременных соединений (процесс на соединение). Пул соединений — обязателен для production.

#### PgBouncer: режимы работы

| Режим | Описание |
|-------|----------|
| **Session pooling** | Соединение закрепляется за клиентом до отключения. Как прямое подключение, но с пулом. |
| **Transaction pooling** (рекомендуется) | Соединение выдаётся только на время транзакции. После COMMIT возвращается в пул. |
| **Statement pooling** | Соединение на один запрос. Самый агрессивный, но ломает транзакции. |

```
Приложение (150 соединений)
  │
  ▼
┌──────────────┐
│  PgBouncer    │  ← 150 клиентских → 20 серверных соединений
│  (пул)        │     Transaction pooling
└──────────────┘
  │
  ▼
PostgreSQL (20 backend-процессов)
```

#### Конфигурация PgBouncer

```ini
# pgbouncer.ini
[databases]
mydb = host=localhost port=5432 dbname=mydb

[pgbouncer]
listen_addr = *
listen_port = 6432
auth_type = md5
auth_file = /etc/pgbouncer/userlist.txt

pool_mode = transaction       # session | transaction | statement
default_pool_size = 20        # соединений на пару user/database
max_client_conn = 200         # максимум клиентских соединений
reserve_pool_size = 5         # резервный пул
reserve_pool_timeout = 3.0    # через сколько сек. задействовать резерв

# Важно!
server_idle_timeout = 600     # разрыв простаивающего серверного соединения
client_idle_timeout = 0       # таймаут простаивающего клиента (0 = отключено)
query_timeout = 0             # таймаут запроса (0 = отключено)
```

**Что нельзя с Transaction pooling:**
- Prepared statements (их нужно подготавливать заново).
- Advisory locks на уровне сессии (используйте `pg_advisory_xact_lock`).
- LISTEN/NOTIFY.
- Курсоры (WITH HOLD).
- Временные таблицы (созданные в одной транзакции не видны в другой).

## Часть XIII. Сравнение PostgreSQL и Oracle

### Сводная таблица сравнения

| Характеристика | PostgreSQL | Oracle |
|:---|:---|:---|
| **Лицензия** | Open Source (PostgreSQL License) | Проприетарная, платная |
| **Модель процессов** | Процесс на соединение | Процесс/поток + Shared Server |
| **MVCC** | In-place (мёртвые кортежи в таблице) | Undo-сегменты |
| **VACUUM** | Да, autovacuum | Нет (undo перезаписывается) |
| **DDL** | Транзакционный | Нетранзакционный |
| **Индексы** | B-Tree, Hash, GiST, SP-GiST, GIN, BRIN, Bloom | B*Tree, Bitmap, IOT, Reverse Key, Domain |
| **Табличные пространства** | ✅ | ✅ |
| **Партицирование** | Декларативное (RANGE, LIST, HASH) | Богаче (INTERVAL, REFERENCE, SYSTEM) |
| **Репликация** | Streaming + Logical | Data Guard + GoldenGate |
| **Шардирование** | Citus, FDW | Oracle Sharding (нативное) |
| **Кластер** | Patroni + etcd | RAC (Real Application Clusters) |
| **Полнотекстовый поиск** | Встроенный | Oracle Text |
| **JSON** | JSONB (бинарный, индексируемый) | JSON (с 12c) |
| **PL/SQL** | PL/pgSQL (есть и другие) | PL/SQL (очень мощный) |
| **Пул соединений** | Внешний (PgBouncer) | Встроенный (Shared Server) |
| **Максимальный размер БД** | ~32 ТБ (практически) | Не ограничен (Exadata — петабайты) |
| **Стоимость владения** | Бесплатно (только железо и поддержка) | Десятки-сотни тысяч $/год |
| **Экосистема** | Огромная (PostGIS, TimescaleDB, Citus, ...) | Enterprise-ориентированная |
| **Сообщество** | Активное open-source | Закрытое, через Oracle Support |
| **Простота освоения** | Средняя | Высокая (требует DBA-экспертизы) |

### Когда что выбирать

**Выбирайте PostgreSQL когда:**
- Бюджет ограничен.
- Нужна гибкость и расширяемость.
- Используются специфичные типы данных (геоданные, JSONB).
- Проект стартап или средний бизнес.
- Нужна облачная совместимость (все облака поддерживают PG).

**Выбирайте Oracle когда:**
- Enterprise с жёсткими требованиями (RPO=0, RTO < 5 сек).
- Нужен RAC для горизонтального масштабирования записи.
- Требуется максимальная поддержка от вендора.
- Уже есть экосистема Oracle (ERP, Siebel, …).
- Бюджет позволяет.

## Заключение

Реляционные базы данных — основа большинства рабочих нагрузок, и эта ситуация сохраняется, несмотря на рост NoSQL-решений. Знание внутреннего устройства PostgreSQL (процессы, память, MVCC, WAL, индексы, планировщик) нужно не для галочки, а чтобы принимать осознанные инженерные решения при проектировании систем.

Основные выводы:

1. **MVCC снимает конфликт читателей и писателей.** Читатели не блокируют писателей, писатели не блокируют читателей. Цена — необходимость VACUUM и риск раздувания таблиц при длинных транзакциях.

2. **Каждый индекс имеет свою цену.** Лишний индекс замедляет запись. Выбирайте тип индекса под задачу (B-Tree, GIN, BRIN — у каждого своя ниша), мониторьте использование и удаляйте неиспользуемые.

3. **EXPLAIN (ANALYZE, BUFFERS) — основной инструмент диагностики.** Прежде чем оптимизировать запрос, посмотрите, что он делает на самом деле: какие узлы плана, сколько буферов, есть ли уход на диск.

4. **Транзакции должны быть короткими.** Длинная транзакция удерживает блокировки и мешает VACUUM очищать мёртвые кортежи. Результат — раздувание таблиц и падение производительности.

5. **Репликация, партицирование и шардирование решают разные задачи.** Репликация — высокая доступность и распределение чтения. Партицирование — управление большими таблицами на одном сервере. Шардирование — горизонтальное масштабирование записи. Не подменяйте одно другим.

6. **Пул соединений обязателен.** Модель «процесс на соединение» плохо переносит сотни прямых подключений. PgBouncer в режиме transaction pooling решает эту проблему.

7. **PostgreSQL — полноценная enterprise-СУБД.** В большинстве случаев она способна работать не хуже Oracle, но требует регулярного обслуживания: настройка autovacuum, мониторинг, перестройка индексов.
