### Введение: что такое Docker и зачем он нужен

**Docker** — это платформа для разработки, доставки и запуска приложений в изолированных окружениях, называемых **контейнерами**. Контейнер упаковывает приложение вместе со всеми его зависимостями (библиотеками, конфигурационными файлами, системными инструментами) в единый, легковесный, переносимый артефакт.

**Ключевая ценность Docker**: гарантия того, что приложение будет работать одинаково в **любом окружении** — на ноутбуке разработчика, на тестовом сервере и в production-облаке.

#### Проблемы, которые решает Docker

| Проблема | Как было до Docker | Как стало с Docker |
|----------|-------------------|-------------------|
| «На моей машине работает» | Разные версии библиотек, ОС, конфигураций на dev/stage/prod | Образ содержит всё необходимое — поведение идентично везде |
| Конфликт зависимостей | Два приложения требуют разные версии одной библиотеки | Каждый контейнер изолирован |
| Долгая настройка окружения | Часы/дни на установку всего стека | `docker compose up` — и окружение готово за минуты |
| Неэффективное использование ресурсов | Каждое приложение — отдельная VM | Десятки контейнеров на одном хосте |
| Сложность масштабирования | Ручное добавление серверов | Оркестрация контейнеров через Kubernetes/Swarm |

#### Docker на разных платформах

Docker работает на Linux, macOS и Windows, но **архитектура принципиально разная**:

| Платформа | Что используется | Ядро контейнеров |
|-----------|-----------------|-------------------|
| **Linux** | Docker Engine (нативно) | Ядро хоста напрямую — namespaces, cgroups, overlay2 на хостовой ФС. Максимальная производительность. |
| **macOS** | Docker Desktop | Linux VM через Apple Hypervisor Framework (ранее VirtualBox). Контейнеры работают внутри VM. |
| **Windows** | Docker Desktop | WSL2 (Windows Subsystem for Linux) — легковесная Linux VM. Также возможны нативные Windows-контейнеры (Windows Server). |

**Ключевое следствие**: на macOS и Windows контейнеры Linux всегда работают внутри виртуальной машины, что даёт небольшой оверхед по производительности (I/O, сеть) и требует выделения памяти/CPU под VM в настройках Docker Desktop.

> **Docker Desktop** — это коммерческий продукт (бесплатен для личного использования, малого бизнеса и open-source), который включает GUI, Kubernetes, Docker Scout и другие утилиты. **Docker Engine** — open-source демон для Linux-серверов.

### История: от виртуальных машин к контейнерам

#### Эпоха Bare Metal

До виртуализации приложения разворачивались напрямую на физических серверах. Проблемы:
- Одно приложение = один сервер (дорого, неэффективно)
- Размещение нескольких приложений на одном сервере вело к конфликтам зависимостей
- Долгое восстановление после сбоев

#### Эпоха виртуальных машин (VM)

VM решают проблему изоляции через **гипервизор**, который эмулирует аппаратное обеспечение. Каждая VM содержит полную гостевую ОС:

```
┌─────────────────────────────────────────────┐
│                  App A                       │
│            ┌───────────────────┐             │
│            │    Bins/Libs      │             │
│            │  ┌─────────────┐  │             │
│            │  │  Guest OS   │  │             │
│            │  │ ┌─────────┐ │  │             │
│            │  │ │Hypervisor│ │  │             │
│            │  │ │┌────────┐│ │  │             │
│            │  │ ││Host OS ││ │  │             │
│            │  │ ││┌──────┐││ │  │             │
│            │  │ │││Server│││ │  │             │
│            │  │ ││└──────┘││ │  │             │
└────────────┴──┴─┴┴────────┴┴─┴─┴─────────────┘
```

**Недостатки VM:**
- Каждая VM содержит ядро ОС (гигабайты дискового пространства)
- Медленный запуск (десятки секунд — минуты)
- Значительные накладные расходы памяти и CPU

#### Эпоха контейнеров

Контейнеры используют ядро хостовой ОС и изолируют только userspace:

```
┌─────────────────────────────────────────────┐
│              App A    App B    App C         │
│            ┌────────┬────────┬────────┐      │
│            │Bins/Lib│Bins/Lib│Bins/Lib│      │
│            │┌────────────────────────┐│      │
│            ││     Docker Engine      ││      │
│            ││  ┌──────────────────┐  ││      │
│            ││  │    Host OS       │  ││      │
│            ││  │  ┌────────────┐  │  ││      │
│            ││  │  │   Server   │  │  ││      │
│            ││  │  └────────────┘  │  ││      │
└────────────┴┴──┴──┴───────────────┴──┴┴──────┘
```

**Ключевые отличия контейнера от VM:**

| Характеристика | Виртуальная машина | Docker-контейнер |
|---------------|-------------------|-----------------|
| Изоляция | Полная (собственное ядро) | На уровне процессов (общее ядро) |
| Запуск | Десятки секунд — минуты | Миллисекунды — секунды |
| Размер образа | Гигабайты | Десятки-сотни мегабайт |
| Плотность на хосте | Единицы-десятки | Десятки-сотни |
| Переносимость | Ограничена гипервизором | Запускается везде, где есть Docker |
| Накладные расходы | 10-20% CPU, память на гостевую ОС | Минимальные (< 2%) |

#### Технологии, лежащие в основе контейнеров (Linux)

Docker не изобретал контейнеризацию — он сделал её удобной. Технологии, на которых основана контейнеризация в Linux:

| Технология | Назначение |
|-----------|-----------|
| **Namespaces** | Изоляция видимости: PID, NET, IPC, MNT, UTS, USER, Cgroup. Каждый контейнер видит только свои процессы, свою сеть, свою файловую систему. |
| **Cgroups (Control Groups)** | Ограничение и учёт ресурсов: CPU, память, дисковый I/O, сеть. Гарантирует, что один контейнер не захватит все ресурсы хоста. |
| **UnionFS (Union Filesystem)** | Слоёная файловая система. Позволяет накладывать слои друг на друга, создавая единое представление. Основа для слоёв образа. |
| **Capabilities** | Гранулярные привилегии. Вместо «root или не root» можно дать конкретные привилегии (например, bind на порт < 1024). |
| **Seccomp** | Фильтрация системных вызовов. Ограничивает, какие syscalls контейнер может выполнять. |

### Архитектура Docker

Docker использует клиент-серверную архитектуру:

```
┌──────────────┐          ┌──────────────────────────────┐       ┌──────────────────┐
│              │  REST    │        Docker Host            │       │                  │
│ Docker CLI   │──────────▶  ┌────────────────────────┐  │       │ Docker Registry  │
│ (docker)     │  UNIX    │  │    Docker Daemon        │  │──────▶│ (Docker Hub,     │
│              │  Socket  │  │    (dockerd)            │  │ pull  │  ECR, GCR, etc.) │
│              │          │  │                         │  │ push  │                  │
│              │          │  │  ┌───────────────────┐  │  │◀──────│                  │
│              │          │  │  │ Container Runtime  │  │  │       │                  │
│              │          │  │  │ (containerd)       │  │  │       └──────────────────┘
│              │          │  │  │  ┌──────────────┐  │  │  │
│              │          │  │  │  │  runc        │  │  │  │
│              │          │  │  │  │  (OCI runtime)│  │  │  │
│              │          │  │  │  └──────────────┘  │  │  │
│              │          │  │  └───────────────────┘  │  │
│              │          │  └────────────────────────┘  │
└──────────────┘          └──────────────────────────────┘
```

#### Docker Daemon (dockerd)

**dockerd** — фоновый процесс, который управляет всеми Docker-объектами: образами, контейнерами, сетями, томами. Он:
- Слушает Docker API-запросы
- Управляет жизненным циклом контейнеров через containerd
- Скачивает образы из registry
- Управляет сетями и томами

С 2016 года Docker использует **containerd** как прослойку между dockerd и OCI-совместимым runtime (runc):

```
dockerd → containerd → containerd-shim → runc → контейнер
```

**containerd-shim** — это процесс-посредник, который остаётся после того, как runc завершил создание контейнера. Он:
- Позволяет обновить/перезапустить dockerd без остановки контейнеров
- Управляет stdin/stdout/stderr контейнера
- Сообщает containerd о завершении контейнера

#### Docker Client (CLI)

Утилита командной строки `docker`. Клиент общается с daemon через:
- **UNIX socket** (`/var/run/docker.sock`) — по умолчанию на Linux
- **TCP socket** (`tcp://host:2375` / `tcp://host:2376` с TLS) — для удалённого управления
- **Named pipe** (`//./pipe/docker_engine`) — на Windows
- **SSH** (`ssh://user@host`) — с Docker 18.09+

Также поддерживается понятие **Docker Context** — переключение между несколькими Docker-хостами одной командой:

```bash
docker context ls                    # список контекстов
docker context create remote --docker "host=ssh://user@server"
docker context use remote            # переключиться на удалённый хост
```

#### Docker Registry

Registry — это хранилище Docker-образов. Бывает:

| Тип | Примеры | Описание |
|-----|---------|----------|
| **Публичный** | Docker Hub, GitHub Container Registry | Бесплатный для публичных образов |
| **Приватный облачный** | Amazon ECR, Google GCR/Artifact Registry, Azure ACR | Интегрирован с облаком |
| **Self-hosted** | Harbor, Docker Registry, GitLab Container Registry | На своей инфраструктуре |

**Формат имени образа**: `registry/repository:tag`
- `registry` — адрес registry (по умолчанию `docker.io`)
- `repository` — имя репозитория (может включать namespace: `library/nginx`)
- `tag` — версия (по умолчанию `latest`)

Примеры:
```
nginx                           → docker.io/library/nginx:latest
mycompany/app:v1.2.3            → docker.io/mycompany/app:v1.2.3
cr.yandex/rp012345/myapp:dev    → Яндекс Cloud Registry
```

#### Docker API

Docker предоставляет REST API для программного управления. Основные версии:
- **v1.41** (текущая) — полный REST API
- Все эндпоинты документированы и версионированы

Примеры эндпоинтов API:
```
GET  /containers/json          — список контейнеров
POST /containers/create        — создать контейнер
POST /containers/{id}/start    — запустить контейнер
GET  /images/json              — список образов
POST /images/create            — скачать образ
```

#### OCI (Open Container Initiative) и стандарты

**OCI** — это проект Linux Foundation (основан в 2015 году), который стандартизирует формат контейнеров и среду их выполнения. Docker — одна из ключевых компаний-основателей OCI. Стандартизация гарантирует, что образы, собранные одним инструментом, могут быть запущены в другом.

**Три ключевых спецификации OCI:**

| Спецификация | Назначение | Что определяет |
|-------------|-----------|----------------|
| **OCI Image Spec** | Формат образа контейнера | Структура манифеста, конфигурации, файловой системы (слои), индекс образов для multi-arch |
| **OCI Runtime Spec** | Запуск контейнера | Как распаковать образ, создать изолированное окружение (namespaces, cgroups) и запустить процесс |
| **OCI Distribution Spec** | API для registry | Стандартные эндпоинты для push/pull образов (Docker Hub API стал основой) |

**Почему это важно:**
- **Совместимость**: образы Docker работают в Podman, containerd, cri-o и наоборот
- **Выбор runtime**: можно использовать runc (по умолчанию), crun (быстрее, на C), gVisor (песочница на уровне syscalls), Kata Containers (VM-уровень изоляции)
- **Инструментарий**: nerdctl (containerd), podman (daemonless), Finch (AWS) — все работают с OCI-образами

```bash
# runc — OCI runtime от Docker (по умолчанию)
# crun — более быстрый OCI runtime на C (используется в Podman по умолчанию)
# gVisor — OCI runtime с песочницей на уровне системных вызовов (Google)
# Kata Containers — OCI runtime с изоляцией на уровне VM (Intel)

# Собрать OCI-совместимый образ
docker build -t myapp .           # Docker
podman build -t myapp .           # Podman — тот же OCI-образ
```

> **Важно**: Docker использует собственный формат образа (Docker Image Manifest v2), который стал основой для OCI Image Spec. Современный Docker полностью совместим с OCI-стандартами.

### Ключевые понятия Docker

#### Образ (Image)

**Образ** — это неизменяемый шаблон, содержащий инструкции для создания контейнера. Это read-only артефакт, состоящий из нескольких слоёв (layers).

Образ — это не один файл, а манифест, ссылающийся на набор слоёв:

```
Образ "myapp:v1"
├── Манифест (JSON)
│   ├── config: { "Cmd": ["/app/start.sh"], "Env": [...], ... }
│   └── layers:
│       ├── sha256:abc123... (базовый слой: Ubuntu)
│       ├── sha256:def456... (установка зависимостей)
│       ├── sha256:ghi789... (копирование кода)
│       └── sha256:jkl012... (конфигурация)
```

#### Слой (Layer)

Каждая инструкция в Dockerfile создаёт запись в истории образа. Слои образа бывают двух типов:

- **Слои файловой системы** — создаются инструкциями `RUN`, `COPY`, `ADD`. Содержат дельту файловой системы и занимают место на диске.
- **Слои метаданных** — создаются инструкциями `CMD`, `ENTRYPOINT`, `ENV`, `EXPOSE`, `WORKDIR`, `USER`, `LABEL`, `HEALTHCHECK`, `ARG`, `SHELL`, `STOPSIGNAL`. Не содержат файлов, только изменяют конфигурацию образа (JSON-метаданные). В `docker history` показываются с размером 0B.

Слои:
- **Иммутабельны** — не изменяются после создания
- **Кэшируются** — повторная сборка переиспользует неизменённые слои
- **Разделяются между образами** — если два образа используют один базовый слой, он хранится один раз

#### Контейнер (Container)

**Контейнер** — это запущенный экземпляр образа. Технически: образ + тонкий read-write слой поверх него + изолированная среда выполнения (namespaces, cgroups).

#### Dockerfile

**Dockerfile** — текстовый файл с инструкциями для сборки образа. Это «рецепт» образа, инфраструктура как код на уровне контейнера.

Для быстрого создания Dockerfile можно использовать `docker init` (доступен с Docker 4.27+):

```bash
docker init                        # интерактивный мастер
# Автоматически определит язык (Node.js, Python, Go, Java и др.)
# и создаст Dockerfile, .dockerignore, compose.yaml
```

#### Том (Volume)

**Volume** — механизм для хранения данных вне жизненного цикла контейнера. Данные в контейнере эфемерны (удаляются вместе с контейнером). Тома позволяют:
- Сохранять данные между перезапусками
- Разделять данные между контейнерами
- Хранить данные на хосте или в облачном хранилище

#### Сеть (Network)

**Сеть** в Docker — это изолированное сетевое пространство, в котором контейнеры могут общаться друг с другом. Docker поддерживает несколько сетевых драйверов.

#### Docker Compose

**Docker Compose** — инструмент для определения и запуска многоконтейнерных приложений. В одном YAML-файле описываются все сервисы, сети и тома приложения.

#### Docker Engine

Термин, объединяющий Docker Daemon, REST API и Docker CLI. Часто используется как синоним всей Docker-платформы.

### Dockerfile: синтаксис и лучшие практики

#### Основные инструкции

| Инструкция | Назначение | Пример |
|-----------|-----------|--------|
| `FROM` | Базовый образ | `FROM ubuntu:22.04` |
| `RUN` | Выполнить команду при сборке | `RUN apt-get update && apt-get install -y curl` |
| `CMD` | Команда по умолчанию при запуске | `CMD ["nginx", "-g", "daemon off;"]` |
| `ENTRYPOINT` | Точка входа (не переопределяется без `--entrypoint`) | `ENTRYPOINT ["/app/start.sh"]` |
| `COPY` | Скопировать файлы из контекста в образ | `COPY . /app` |
| `ADD` | Как COPY, но может распаковывать tar и скачивать URL | `ADD app.tar.gz /app` |
| `WORKDIR` | Рабочая директория | `WORKDIR /app` |
| `ENV` | Переменная окружения | `ENV NODE_ENV=production` |
| `ARG` | Аргумент сборки (только во время build) | `ARG VERSION=1.0` |
| `EXPOSE` | Документировать порт | `EXPOSE 8080` |
| `VOLUME` | Создать точку монтирования тома | `VOLUME /data` |
| `USER` | Пользователь для запуска | `USER 1001` |
| `HEALTHCHECK` | Проверка здоровья контейнера | `HEALTHCHECK --interval=30s CMD curl -f http://localhost:8080/health \|\| exit 1` |
| `LABEL` | Метаданные образа | `LABEL version="1.0.0"` |
| `SHELL` | Оболочка для shell-форм инструкций | `SHELL ["/bin/bash", "-c"]` |
| `ONBUILD` | Инструкция, выполняемая при использовании образа как базового | `ONBUILD COPY . /app` |
| `STOPSIGNAL` | Сигнал для остановки контейнера | `STOPSIGNAL SIGTERM` |

> **Примечание:** `MAINTAINER` — устаревшая инструкция (deprecated), заменена на `LABEL maintainer="..."`. Не используйте её в новых Dockerfile.

#### Формы инструкций: shell vs exec

Многие инструкции (`RUN`, `CMD`, `ENTRYPOINT`) имеют две формы:

**Shell-форма** (запускается через `/bin/sh -c`):
```dockerfile
CMD echo "Hello"
RUN apt-get update
```

**Exec-форма** (прямой запуск без shell, предпочтительная):
```dockerfile
CMD ["echo", "Hello"]
RUN ["apt-get", "update"]
```

| Аспект | Shell-форма | Exec-форма |
|--------|------------|-----------|
| Запуск через shell | Да (`/bin/sh -c`) | Нет (прямой exec) |
| Подстановка переменных | Да (`$VAR`) | Нет (нужен shell) |
| Обработка сигналов | Не передаются процессу (PID 1 = shell) | Сигналы доходят до процесса |
| Рекомендация | Для простых команд | Для production |

#### Взаимодействие CMD и ENTRYPOINT

`CMD` и `ENTRYPOINT` вместе определяют команду запуска контейнера:

```
┌───────────────┬─────────────────┬──────────────────────────────────┐
│               │  Без ENTRYPOINT  │        С ENTRYPOINT              │
├───────────────┼─────────────────┼──────────────────────────────────┤
│ Без CMD       │ Ошибка           │ ENTRYPOINT выполняется один      │
├───────────────┼─────────────────┼──────────────────────────────────┤
│ С CMD         │ CMD выполняется  │ ENTRYPOINT + CMD как аргументы   │
├───────────────┼─────────────────┼──────────────────────────────────┤
│ docker run    │ args заменяют    │ args заменяют CMD,               │
│ с аргументами │ CMD              │ ENTRYPOINT + args                │
└───────────────┴─────────────────┴──────────────────────────────────┘
```

**Рекомендуемый паттерн:**
```dockerfile
ENTRYPOINT ["/app/server"]      # неизменяемая часть
CMD ["--port", "8080"]          # дефолтные аргументы (можно переопределить)
```

#### Multi-stage сборки

Multi-stage сборки позволяют использовать несколько образов в одном Dockerfile и выборочно копировать артефакты между ними. Это решает проблему «build-time vs run-time зависимостей».

```dockerfile
# === Stage 1: Сборка ===
FROM maven:3.9-eclipse-temurin-21 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline    # кэшируем зависимости
COPY src/ ./src/
RUN mvn package -DskipTests

# === Stage 2: Runtime ===
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
USER 1001
ENTRYPOINT ["java", "-jar", "app.jar"]
```

**Преимущества multi-stage:**
- **Маленький финальный образ** — только JRE, не JDK + Maven
- **Меньше поверхности атаки** — нет компиляторов и dev-инструментов
- **Кэширование** — каждый stage кэшируется независимо
- **Несколько целей** — можно собирать разные варианты из одного Dockerfile: `docker build --target builder .`

#### Оптимизация размера образа

1. **Выбирайте минимальный базовый образ:**
   ```
   ubuntu:22.04        ~77 MB
   debian:bookworm-slim ~74 MB
   alpine:3.19          ~7 MB  ← минимальный, но использует musl (не glibc)
   distroless           ~2 MB  ← только приложение и runtime-зависимости
   scratch              ~0 MB  ← пустой образ (для статически скомпилированных бинарников)
   ```

2. **Объединяйте RUN-команды:**
   ```dockerfile
   # Плохо: 3 слоя
   RUN apt-get update
   RUN apt-get install -y curl
   RUN rm -rf /var/lib/apt/lists/*

   # Хорошо: 1 слой + очистка в том же слое
   RUN apt-get update && \
       apt-get install -y --no-install-recommends curl && \
       rm -rf /var/lib/apt/lists/*
   ```

3. **Используйте `.dockerignore`:**
   ```
   node_modules/
   .git/
   *.md
   .env
   dist/
   ```

4. **Удаляйте кэш пакетных менеджеров в том же слое:**
   ```dockerfile
   RUN pip install --no-cache-dir -r requirements.txt
   RUN npm ci --production && npm cache clean --force
   ```

#### Кэширование слоёв

Docker кэширует слои на основе:
1. **Хэша инструкции** (самой строки в Dockerfile)
2. **Хэша файлов**, копируемых через `COPY`/`ADD`

**Правило порядка инструкций**: то, что меняется редко — в начале, что часто — в конце.

```dockerfile
# ❌ Плохо: любое изменение в коде инвалидирует кэш зависимостей
COPY . /app
RUN npm ci

# ✅ Хорошо: копируем package.json отдельно
COPY package.json package-lock.json ./
RUN npm ci                          # кэшируется, пока не меняется package.json
COPY src/ ./src/
```

#### Безопасность Dockerfile

1. **Не запускайте от root:**
   ```dockerfile
   RUN addgroup -S appgroup && adduser -S appuser -G appgroup
   USER appuser
   ```

2. **Используйте фиксированные версии, не latest:**
   ```dockerfile
   FROM node:20.11.1-alpine3.19   # ✅
   FROM node:latest               # ❌ невоспроизводимо
   ```

3. **Проверяйте контрольные суммы (для ADD с URL):**
   ```dockerfile
   ADD --checksum=sha256:abc123... https://example.com/app.tar.gz /app/
   ```

4. **Не храните секреты в образе:**
   ```dockerfile
   # ❌ Секрет остаётся в слое навсегда
   ENV DATABASE_PASSWORD=super_secret

   # ✅ Передавайте во время запуска
   # docker run -e DATABASE_PASSWORD=super_secret myapp
   ```

5. **Используйте `--no-install-recommends` (Debian/Ubuntu):**
   ```dockerfile
   RUN apt-get update && apt-get install -y --no-install-recommends curl
   ```

### Управление процессами в контейнере: PID 1, сигналы и tini

Это одна из самых частых причин проблем в production — непонимание того, как процессы работают внутри контейнера.

#### Проблема PID 1

В Linux процесс с PID 1 (init) имеет особые обязанности:
- **Жатва зомби-процессов** (reaping): все осиротевшие процессы перепривязываются к PID 1, и он должен вызывать `wait()` для их корректного завершения
- **Игнорирование сигналов**: ядро не посылает PID 1 сигналы, для которых не установлен обработчик (в т.ч. SIGTERM)
- **Перезагрузка ядра**: только PID 1 может перезагрузить систему через сигнал SIGINT (неактуально для контейнеров, но важно понимать)

Если ваше приложение запущено как PID 1, но не было спроектировано как init-процесс, вы получаете:
- Зомби-процессы, которые никогда не освобождают ресурсы
- Невозможность остановить контейнер через `docker stop` (SIGTERM игнорируется, через 10 секунд — SIGKILL)

#### Shell-форма и потеря сигналов

Проблема усугубляется shell-формой CMD/ENTRYPOINT:

```dockerfile
# ❌ Сигналы не доходят до приложения
CMD myapp --port 8080
# Реально запускается: /bin/sh -c "myapp --port 8080"
# PID 1 = /bin/sh, который НЕ передаёт сигналы дочернему myapp
```

#### Решение: exec-форма

```dockerfile
# ✅ Сигналы доходят напрямую
CMD ["myapp", "--port", "8080"]
# PID 1 = myapp (сигналы обрабатываются приложением)
```

> Exec-форма использует прямой `exec` вместо `fork + exec`, поэтому процесс приложения становится PID 1 без промежуточного shell.

#### Решение: `--init` и tini

Если приложение не умеет корректно работать как PID 1 (например, bash-скрипт, Java-приложение с дочерними процессами), используйте `--init`:

```bash
docker run --init myapp
```

Флаг `--init` внедряет **tini** (или `docker-init`) как PID 1. tini — минималистичный init-процесс, который:
- Корректно репит зомби-процессы
- Пробрасывает сигналы дочернему процессу
- Завершается, когда дочерний процесс завершается

```dockerfile
# Альтернатива: установить tini в образе
RUN apk add --no-cache tini
ENTRYPOINT ["/sbin/tini", "--", "/app/start.sh"]
```

#### Graceful shutdown (корректное завершение)

При `docker stop` происходит:
1. `docker stop` посылает **SIGTERM** главному процессу (PID 1)
2. Ждёт 10 секунд (настраивается через `--time` или `-t`)
3. Если процесс не завершился — посылает **SIGKILL**

**Рекомендации для корректного завершения:**
- Приложение должно обрабатывать SIGTERM и завершать работу (закрыть соединения, дописать данные)
- Используйте `STOPSIGNAL` если приложению нужен другой сигнал: `STOPSIGNAL SIGQUIT`
- В entrypoint-скриптах используйте `exec` для передачи PID 1:

```bash
#!/bin/bash
# ❌ Плохо: PID 1 = bash, не передаёт сигналы приложению
./myapp

# ✅ Хорошо: PID 1 = myapp (exec заменяет процесс bash на myapp)
exec ./myapp

# ✅ Хорошо: trap для кастомной логики при SIGTERM
trap 'echo "Shutting down..."; kill $PID' SIGTERM
./myapp & PID=$!
wait $PID
```

#### `docker pause` / `docker unpause`

Используют **cgroups freezer** — «замораживают» все процессы в контейнере (состояние процессов сохраняется в памяти, но они не получают процессорное время). В отличие от `docker stop`, не посылает сигналы и не завершает процессы.

```bash
docker pause mycontainer    # заморозить все процессы
docker unpause mycontainer  # возобновить выполнение
```

### BuildKit и buildx: современный движок сборки

#### Что такое BuildKit

**BuildKit** — это современный движок сборки Docker-образов, который пришёл на смену классическому билдеру. Включён по умолчанию в Docker Desktop и Docker Engine с версии 23.0.

**Преимущества BuildKit:**
- **Параллельная сборка** независимых стадий (multi-stage)
- **Умное кэширование** — более гранулярное, кэш можно экспортировать в registry
- **Монтирование секретов** (`--mount=type=secret`) — секреты не попадают в финальный образ
- **Монтирование кэша** (`--mount=type=cache`) — кэш пакетных менеджеров между сборками
- **Монтирование SSH** (`--mount=type=ssh`) — доступ к приватным репозиториям при сборке

Включить BuildKit (если не включён по умолчанию):

```bash
# Для одной команды
DOCKER_BUILDKIT=1 docker build .

# Включить навсегда (в /etc/docker/daemon.json)
{ "features": { "buildkit": true } }

# Или через переменную окружения
export DOCKER_BUILDKIT=1
```

#### buildx и мультиплатформенные сборки

**docker buildx** — расширение CLI для BuildKit, позволяющее:
- Собирать образы для нескольких платформ одновременно (linux/amd64, linux/arm64, etc.)
- Экспортировать кэш в удалённый registry
- Собирать без локального Docker daemon

```bash
# Создание билдера с поддержкой мультиплатформенности
docker buildx create --name multiarch --driver docker-container --use

# Мультиплатформенная сборка
docker buildx build --platform linux/amd64,linux/arm64 \
  -t myapp:latest --push .

# Экспорт кэша в registry
docker buildx build --cache-to type=registry,ref=myregistry.com/cache:myapp \
  --cache-from type=registry,ref=myregistry.com/cache:myapp \
  -t myapp:latest --push .
```

#### Монтирование секретов и кэша

**Секреты (не остаются в образе):**
```dockerfile
# syntax=docker/dockerfile:1
FROM node:20-alpine
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm ci --production
```

```bash
docker build --secret id=npmrc,src=$HOME/.npmrc -t myapp .
```

**Кэш пакетных менеджеров (переиспользуется между сборками):**
```dockerfile
# syntax=docker/dockerfile:1
FROM node:20-alpine
RUN --mount=type=cache,target=/root/.npm \
    npm ci --production
```

```bash
docker build --mount=type=cache,target=/root/.npm -t myapp .
```

### Слои образа и Union Filesystem

#### Copy-on-Write (CoW)

Когда контейнер хочет модифицировать файл из read-only слоя образа:

1. Docker находит файл в read-only слое
2. Копирует его в read-write слой контейнера
3. Модификация применяется к копии
4. Оригинал в read-only слое остаётся неизменным

Это даёт два преимущества:
- **Экономия диска**: 100 контейнеров из одного образа используют общие read-only слои (только один раз на диске)
- **Быстрый старт**: не нужно копировать весь образ для каждого контейнера

```
┌──────────────────────────────────────────────┐
│               Контейнер A                     │
│  ┌────────────────────────────────────┐      │
│  │  Read-Write слой (изменения только A)│      │
│  └────────────────────────────────────┘      │
│  ┌────────────────────────────────────┐      │
│  │  Read-Only слой 3: код приложения   │──────┼─────┐
│  └────────────────────────────────────┘      │     │
│  ┌────────────────────────────────────┐      │     │
│  │  Read-Only слой 2: зависимости      │──────┼──┐  │
│  └────────────────────────────────────┘      │  │  │
│  ┌────────────────────────────────────┐      │  │  │
│  │  Read-Only слой 1: базовый образ    │──────┼──┼──┼─ Общие слои
│  └────────────────────────────────────┘      │  │  │  │ (хранятся один раз)
└──────────────────────────────────────────────┘  │  │  │
                                                  │  │  │
┌──────────────────────────────────────────────┐  │  │  │
│               Контейнер B                     │  │  │  │
│  ┌────────────────────────────────────┐      │  │  │  │
│  │  Read-Write слой (изменения только B)│      │  │  │  │
│  └────────────────────────────────────┘      │  │  │  │
│  ┌────────────────────────────────────┐      │  │  │  │
│  │  Read-Only слой 3: код приложения   │──────┼──┼──┘  │
│  └────────────────────────────────────┘      │  │     │
│  ┌────────────────────────────────────┐      │  │     │
│  │  Read-Only слой 2: зависимости      │──────┼──┘     │
│  └────────────────────────────────────┘      │        │
│  ┌────────────────────────────────────┐      │        │
│  │  Read-Only слой 1: базовый образ    │──────┼────────┘
│  └────────────────────────────────────┘      │
└──────────────────────────────────────────────┘
```

#### Storage Drivers

Storage driver — реализация Union Filesystem. Docker поддерживает несколько:

| Driver | ФС | Статус | Когда использовать |
|--------|----|--------|--------------------|
| **overlay2** | ext4, xfs | ✅ Рекомендуемый | Для всех современных Linux |
| **fuse-overlayfs** | Любая | ✅ Для rootless | Когда Docker работает без root |
| **btrfs** | btrfs | Производственный | Если используется btrfs ФС |
| **devicemapper** | direct-lvm | Устаревший | Устарел, не рекомендуется |
| **aufs** | ext4, xfs | Устаревший | Устарел, не рекомендуется |
| **vfs** | Любая | Для тестов | Не использует CoW, каждый слой — полная копия |

**overlay2** — стандарт де-факто:
- Хорошая производительность
- Эффективное использование inode
- Поддержка в ядре Linux с 4.0+
- Работает с SELinux

### Сети в Docker

#### Bridge Network

**bridge** — default-сеть. Это изолированная (на хосте) виртуальная сеть, в которой контейнеры могут общаться по IP-адресам.

```
Хост (Host)
├── eth0 (физический интерфейс, 192.168.1.100)
│
├── docker0 (bridge, 172.17.0.1)
│   ├── vethA ↔ eth0 контейнера A (172.17.0.2)
│   ├── vethB ↔ eth0 контейнера B (172.17.0.3)
│   └── vethC ↔ eth0 контейнера C (172.17.0.4)
│
└── Правила iptables NAT: port mapping (host:8080 → containerA:80)
```

**User-defined bridge** (пользовательские bridge-сети):
- Автоматическое DNS-разрешение имён контейнеров
- Лучшая изоляция (контейнеры в разных bridge-сетях не видят друг друга)
- Можно подключать/отключать контейнеры на лету

```bash
docker network create --driver bridge mynetwork
docker run --network mynetwork --name app1 myapp
docker run --network mynetwork --name app2 myapp
# app2 может обращаться к app1 по имени: curl app1:8080
```

#### Host Network

**host** — контейнер использует сеть хоста напрямую, без изоляции. Сетевой стек контейнера не изолирован от хоста.

```bash
docker run --network host nginx
# nginx слушает порт 80 на хосте напрямую
```

| Плюсы | Минусы |
|-------|---------|
| Максимальная производительность (нет NAT, bridge) | Нет изоляции |
| Нет необходимости в port mapping | Конфликты портов |
| Подходит для высоконагруженных сетевых приложений | Небезопасно |

#### None Network

**none** — контейнер полностью изолирован от сети. Имеет только loopback-интерфейс. Используется, когда сеть не нужна или настраивается вручную.

#### Overlay Network

**overlay** — сеть, охватывающая несколько Docker-хостов. Использует VXLAN-туннели для соединения контейнеров на разных хостах. Требуется в Docker Swarm или при использовании Docker с внешним key-value store (Consul, etcd).

```
Хост 1                              Хост 2
┌──────────────────┐               ┌──────────────────┐
│  Container A     │               │  Container B     │
│  (10.0.0.2)      │               │  (10.0.0.3)      │
│       │           │               │       │           │
│   vxlan-туннель   │───────────────│   vxlan-туннель   │
│       │           │   физическая  │       │           │
│  eth0 (хост 1)   │     сеть      │  eth0 (хост 2)   │
└──────────────────┘               └──────────────────┘
```

#### Macvlan Network

**macvlan** — каждому контейнеру назначается собственный MAC-адрес, и он выглядит как физическое устройство в локальной сети. Полезно для приложений, которые ожидают прямого подключения к физической сети (legacy-системы, мониторинг).

#### Сравнение сетевых драйверов

| Driver | Изоляция | Производительность | Multi-host | DNS | Типовой сценарий |
|--------|----------|-------------------|------------|-----|-----------------|
| **bridge** | На уровне хоста | Хорошая | Нет | Только user-defined | Разработка, простые приложения |
| **host** | Отсутствует | Максимальная | Нет | Н/Д | Высоконагруженные приложения |
| **overlay** | На уровне сети | Средняя (оверхед VXLAN) | Да | Да | Swarm-кластеры |
| **macvlan** | MAC-уровень | Хорошая | Нет | Нет | Интеграция с физической сетью |
| **none** | Полная | Н/Д | Н/Д | Нет | Пакетная обработка, отладка |

### Тома и хранение данных

#### Volumes (именованные тома)

**Volume** — предпочтительный механизм хранения данных. Управляется Docker, хранится в `/var/lib/docker/volumes/`.

```bash
# Создание тома
docker volume create myvolume

# Монтирование
docker run -v myvolume:/data myapp

# Просмотр
docker volume ls
docker volume inspect myvolume
```

**Преимущества:**
- Управляется Docker (резервное копирование, миграция)
- Работает на Linux и Windows
- Может быть разделён между контейнерами
- Может использовать volume driver для облачного хранилища (AWS EBS, Azure Disk, NFS)

#### Bind Mounts

**Bind mount** — монтирование конкретной файловой системы хоста в контейнер. Контейнер имеет полный доступ к указанному пути на хосте.

```bash
docker run -v /home/user/data:/data myapp
#           ───────┬──────  ──┬──
#           путь на хосте   путь в контейнере
```

| Плюсы | Минусы |
|-------|---------|
| Прямой доступ с хоста | Зависимость от структуры ФС хоста |
| Удобно для разработки (hot reload) | Проблемы с правами (UID/GID mismatch) |
| Можно монтировать конфиги | Непреносимость (разные пути на разных хостах) |

#### tmpfs Mounts

**tmpfs** — временная файловая система в памяти. Данные не сохраняются на диск, теряются при остановке контейнера.

```bash
docker run --tmpfs /tmp:rw,size=128M myapp
docker run --mount type=tmpfs,destination=/tmp,tmpfs-size=128M myapp
```

Используется для:
- Кэшей и временных файлов
- Чувствительных данных (не должны попасть на диск)
- Высокопроизводительного I/O

#### Сравнение и выбор

| Критерий | Volume | Bind Mount | tmpfs |
|----------|--------|------------|-------|
| Управление | Docker | Пользователь | Docker |
| Сохранение после удаления контейнера | Да | Да | Нет |
| Производительность | Хорошая | Хорошая | Отличная (RAM) |
| Облачные драйверы | Да | Нет | Нет |
| Переносимость | Высокая | Низкая | Средняя |
| Безопасность | Высокая | Низкая (доступ к ФС хоста) | Высокая |
| Типовой сценарий | Production-данные, БД | Разработка (hot reload) | Кэши, секреты, IPC |

### Логирование в Docker

Docker поддерживает несколько **logging drivers** — механизмов доставки логов из stdout/stderr контейнера в различные системы сбора логов.

#### Встроенные драйверы логирования

| Драйвер | Назначение | Типовой сценарий |
|---------|-----------|-----------------|
| **json-file** (по умолчанию) | JSON-файлы на хосте: `/var/lib/docker/containers/<id>/<id>-json.log` | Разработка, простые деплои |
| **local** | Бинарный формат с автоматической ротацией | Альтернатива json-file с лучшей производительностью |
| **syslog** | Отправка в syslog-сервер | Традиционные Linux-окружения |
| **journald** | Логи systemd-journald | Хосты с systemd |
| **gelf** | Graylog Extended Log Format (UDP/TCP) | Централизованный сбор в Graylog |
| **fluentd** | Отправка во Fluentd | Централизованный сбор с парсингом |
| **splunk** | Отправка в Splunk (HTTP Event Collector) | Enterprise-окружения |
| **awslogs** | Amazon CloudWatch Logs | AWS ECS, EC2 |
| **gcplogs** | Google Cloud Logging | GCP, GKE |
| **azurelogs** | Azure Log Analytics | Azure Container Instances |

#### Настройка драйвера

```bash
# Для конкретного контейнера
docker run --log-driver fluentd \
  --log-opt fluentd-address=localhost:24224 \
  --log-opt tag="myapp.{{.Name}}" \
  myapp

# Глобально в /etc/docker/daemon.json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}
```

#### Ротация логов для json-file

По умолчанию Docker **не ротирует** json-file логи — файл растёт бесконечно, пока не закончится диск. Это одна из самых частых причин инцидентов в production.

```bash
# Настройка ротации при запуске
docker run --log-opt max-size=10m --log-opt max-file=3 myapp

# Или в Compose
services:
  app:
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
```

#### Важные замечания

- **Логи пишутся в stdout/stderr** — не используйте файловые логи внутри контейнера (их не подхватит драйвер)
- **`docker logs` работает только с json-file и local** — для других драйверов команда `docker logs` не вернёт данные
- **Драйверы с доставкой по сети (fluentd, gelf, splunk)** — если сервер логов недоступен, контейнер может блокироваться на отправке логов. Используйте `mode: non-blocking`

```yaml
logging:
  driver: fluentd
  options:
    fluentd-address: localhost:24224
    mode: non-blocking          # не блокировать контейнер при недоступности Fluentd
    max-buffer-size: 4m         # размер буфера в памяти
```

### Docker Compose

#### Структура docker-compose.yml

> **Важно:** Поле `version` в начале файла является **устаревшим** (deprecated) в Docker Compose v2 (текущий стандарт). Оно игнорируется и может быть безопасно удалено. Оставлено только для обратной совместимости с очень старыми версиями (Compose v1).

```yaml
services:                         # Сервисы = контейнеры
  app:
    build:                        # Сборка из Dockerfile
      context: ./app
      dockerfile: Dockerfile
      args:
        NODE_ENV: production
    image: myapp:latest           # Имя образа
    container_name: myapp-prod    # Фиксированное имя контейнера
    command: ["node", "server.js"] # Переопределение CMD
    entrypoint: ["/entry.sh"]     # Переопределение ENTRYPOINT
    environment:                  # Переменные окружения
      - NODE_ENV=production
      - DB_HOST=postgres
    env_file:                     # Файл с переменными
      - .env.production
    ports:                        # Проброс портов
      - "8080:8080"
    volumes:                      # Монтирование
      - app-data:/data
      - ./config:/app/config:ro
    depends_on:                   # Порядок запуска
      postgres:
        condition: service_healthy
    networks:
      - backend
    restart: unless-stopped
    healthcheck:                  # Проверка здоровья
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 30s
      timeout: 5s
      retries: 3
    deploy:                       # Ограничения ресурсов
      resources:
        limits:
          cpus: "0.5"
          memory: 512M
        reservations:
          cpus: "0.25"
          memory: 256M
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: myapp
      POSTGRES_PASSWORD: ${DB_PASSWORD}  # из .env файла
    volumes:
      - pg-data:/var/lib/postgresql/data
    networks:
      - backend
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U myapp"]
      interval: 5s
      timeout: 3s
      retries: 5

volumes:                          # Именованные тома
  app-data:
  pg-data:

networks:                         # Пользовательские сети
  backend:
    driver: bridge
```

#### Основные директивы

| Директива | Назначение |
|-----------|-----------|
| `build` | Конфигурация сборки образа |
| `image` | Используемый образ (сборка или готовый) |
| `command` | Переопределение команды |
| `ports` | Проброс портов `HOST:CONTAINER` |
| `expose` | Открыть порт внутри сети (без проброса на хост) |
| `volumes` | Монтирование томов и bind mounts |
| `environment` | Переменные окружения |
| `env_file` | Файл с переменными окружения |
| `depends_on` | Порядок запуска (не ждёт готовности без `condition`) |
| `networks` | Подключение к сетям |
| `restart` | Политика перезапуска (`no`, `always`, `on-failure`, `unless-stopped`) |
| `healthcheck` | Проверка здоровья сервиса |
| `deploy` | Настройки для Swarm/ограничения ресурсов |
| `profiles` | Условный запуск сервиса |
| `extends` | Наследование конфигурации из другого файла (устаревает) |
| `extra_hosts` | Добавление записей в /etc/hosts |
| `dns` | Настройка DNS-серверов |
| `secrets` | Секреты (в Swarm) |
| `configs` | Конфигурации (в Swarm) |

#### Профили (Profiles)

Профили позволяют условно запускать сервисы:

```yaml
services:
  app:
    profiles: ["debug", "dev"]  # запускается только с --profile
    # ...
  db:
    # без profiles — запускается всегда
    # ...
```

```bash
docker compose --profile debug up    # app + db
docker compose up                    # только db
```

#### extends и фрагменты

**YAML anchors (рекомендуется):**
```yaml
x-common: &common                    # extension field (определение)
  restart: unless-stopped
  networks:
    - backend

services:
  app:
    <<: *common                      # merge key (слияние)
    image: myapp
  worker:
    <<: *common
    image: myworker
```

**extends (менее предпочтительно, устаревающий):**
```yaml
services:
  app:
    extends:
      file: docker-compose.base.yml
      service: base
    image: myapp
```

#### Переменные окружения и .env

Compose автоматически подставляет переменные из `.env` файла (в той же директории):

```bash
# .env
DB_PASSWORD=super_secret
COMPOSE_PROJECT_NAME=myapp

# docker-compose.yml
environment:
  POSTGRES_PASSWORD: ${DB_PASSWORD}
```

Приоритет переменных:
1. Переменные shell (export)
2. `.env` файл
3. Значения по умолчанию: `${VAR:-default}`

#### Docker Compose vs Docker Swarm

| Характеристика | Docker Compose | Docker Swarm |
|---------------|----------------|-------------|
| Масштаб | Один хост | Кластер из многих хостов |
| Service Discovery | DNS в рамках bridge-сети | Встроенный DNS + VIP |
| Балансировка | Нет | Встроенная (ingress mesh) |
| Секреты | Нет (только .env) | Да (`docker secret`) |
| Rolling Update | Нет | Да |
| Декларативная модель | compose.yml | stack.yml (совместим с compose) |
| Типовой сценарий | Разработка, тестирование | Production (малый/средний) |

### Безопасность Docker

#### Пользователи и привилегии

**Проблема**: Docker daemon работает от root, и пользователи в группе `docker` имеют привилегии, эквивалентные root, через монтирование `/` хоста в контейнер.

```
# Это даёт root на хосте (пользователь docker = root):
docker run -v /:/host -it alpine chroot /host
```

**Решения:**
- **Rootless-режим**: Docker daemon работает от непривилегированного пользователя
- **User namespace remapping**: root в контейнере → непривилегированный пользователь на хосте
- **Не добавлять пользователей в группу docker**: использовать `sudo docker` или podman с аутентификацией

#### Capabilities

Вместо полного root-доступа контейнер получает ограниченный набор capabilities:

```bash
# Посмотреть capabilities контейнера
docker inspect --format '{{.HostConfig.CapAdd}}' CONTAINER

# Запустить с конкретными capabilities
docker run --cap-add NET_BIND_SERVICE --cap-drop ALL myapp

# Основные capabilities:
# CAP_NET_BIND_SERVICE — bind на порт < 1024
# CAP_NET_RAW — raw sockets (ping)
# CAP_SYS_PTRACE — strace, gdb
# CAP_SYS_ADMIN — монтирование ФС, очень мощная
```

#### Seccomp и AppArmor

**Seccomp (Secure Computing Mode):**
- Docker по умолчанию применяет seccomp-профиль, блокирующий ~44 из ~300 системных вызовов
- Можно задать свой профиль: `docker run --security-opt seccomp=/path/to/profile.json`

**AppArmor:**
- Мандатный контроль доступа на уровне процессов
- `docker run --security-opt apparmor=docker-default`

#### Сканирование образов

```bash
docker scan IMAGE                  # Snyk-сканирование (встроенное)
docker scout quickview IMAGE       # Docker Scout
```

Инструменты:
- **Docker Scout**: анализ SBOM, рекомендации по обновлению
- **Trivy** (Aqua Security): сканер уязвимостей
- **Grype** (Anchore): сканирование CVE
- **Clair** (Quay): статический анализ

#### Content Trust (DCT)


### Ограничения ресурсов контейнера

Без ограничений контейнер может использовать все доступные ресурсы хоста. В production это недопустимо — один «взбесившийся» контейнер может положить весь сервер.

#### CPU

```bash
docker run --cpus 1.5 myapp        # максимум 1.5 ядра
docker run --cpuset-cpus 0,2 myapp # использовать только ядра 0 и 2
docker run --cpu-shares 512 myapp  # относительный вес (по умолчанию 1024)
```

| Параметр | Тип | Описание |
|----------|-----|----------|
| `--cpus` | Абсолютный (hard limit) | Максимальное количество ядер (1.5 = полтора ядра) |
| `--cpuset-cpus` | Абсолютный (pin) | На каких конкретно ядрах запускать |
| `--cpu-shares` | Относительный (soft limit) | Вес при конкуренции за CPU. Контейнер с shares=512 получит в 2 раза меньше CPU, чем shares=1024, только когда CPU перегружен |
| `--cpu-period` + `--cpu-quota` | Абсолютный (CFS) | Тонкая настройка через Completely Fair Scheduler (CFS). `--cpu-quota=50000 --cpu-period=100000` = 0.5 ядра |

**CPU shares vs CPU limit:**
- `--cpu-shares` — **soft limit**: если CPU простаивает, контейнер может использовать больше
- `--cpus` — **hard limit**: контейнер никогда не превысит указанный лимит (даже если CPU свободен)

#### Память

```bash
docker run --memory 512M myapp                    # максимум 512 MB
docker run --memory 512M --memory-swap 1G myapp   # 512M RAM + 512M swap = 1G total
docker run --memory 512M --memory-swap 512M myapp # 512M RAM + 0 swap (swap = memory)
docker run --memory 512M --memory-swap -1 myapp   # 512M RAM + unlimited swap
```

| Параметр | Описание |
|----------|----------|
| `--memory` (или `-m`) | Максимальный объём физической памяти |
| `--memory-swap` | RAM + swap вместе. По умолчанию = 2 × memory |
| `--memory-swappiness` | Склонность к свопу (0-100), по умолчанию -1 (наследуется от хоста) |
| `--memory-reservation` | Soft limit — мягкий лимит для планировщика |

#### OOM (Out of Memory)

При превышении лимита памяти ядро вызывает **OOM Killer**, который убивает процесс с наибольшим OOM Score. Счёт определяется используемой памятью процесса.

```bash
# Настроить приоритет при OOM
docker run --oom-score-adj 500 myapp  # выше значение = выше шанс быть убитым (диапазон -1000 до 1000)

# Отключить OOM Killer (опасно!)
docker run --oom-kill-disable --memory 512M myapp
# Контейнер будет висеть/тормозить вместо завершения при нехватке памяти
```

#### Ограничение количества процессов (PIDs)

Без этого ограничения контейнер может исчерпать PID-таблицу хоста через fork-бомбу:

```bash
docker run --pids-limit 100 myapp    # максимум 100 процессов (включая потоки)
```

```yaml
# Compose (deploy поддерживается Docker Swarm и Compose v2)
services:
  app:
    deploy:
      resources:
        limits:
          cpus: "0.5"
          memory: 512M
          pids: 100
```

#### Динамическое изменение ограничений

Для уже запущенного контейнера:

```bash
docker update --cpus 2 --memory 1G mycontainer
docker update --restart unless-stopped mycontainer
```

Ограничения CPU и памяти можно менять на лету, но не все параметры (например, `--pids-limit` нельзя изменить без пересоздания).

### Аутентификация в Docker Registry

```bash
# Вход в registry
docker login                                    # Docker Hub (docker.io)
docker login registry.example.com               # свой registry
docker login -u username -p password registry   # неинтерактивно (небезопасно)
docker login --password-stdin < token.txt       # безопасная передача пароля через stdin

# Выход
docker logout
docker logout registry.example.com
```

**Credential stores** — механизм безопасного хранения учётных данных (вместо plain-text `~/.docker/config.json`):

| Бэкенд | Платформа | Настройка в `config.json` |
|--------|----------|---------------------------|
| `secretservice` | Linux (GNOME Keyring) | `"credsStore": "secretservice"` |
| `osxkeychain` | macOS (Keychain) | `"credsStore": "osxkeychain"` |
| `wincred` | Windows (Credential Manager) | `"credsStore": "wincred"` |
| `pass` | Linux (password-store) | `"credsStore": "pass"` |

**Внешние credential helpers** (для облачных registries):

| Registry | Helper | Установка |
|----------|--------|-----------|
| Amazon ECR | `docker-credential-ecr-login` | Встроен в AWS CLI v2 |
| Google GCR/AR | `docker-credential-gcr` | `gcloud auth configure-docker` |
| Azure ACR | `docker-credential-acr` | `az acr login --name <registry>` |

**Проверка прав доступа:**

```bash
docker pull alpine  # публичный образ — работает без логина
docker push myapp   # push требует авторизации
```

> При использовании CI/CD: не храните пароли в коде. Используйте `--password-stdin`, GitHub Actions secrets, GitLab CI/CD variables, или OIDC-аутентификацию облачных провайдеров (AWS IAM, GCP Workload Identity).

### Альтернативные инструменты и экосистема

Docker — не единственный способ работать с контейнерами. Понимание альтернатив помогает выбрать правильный инструмент под задачу.

#### Сравнение инструментов

| Инструмент | Тип | Демон | Rootless | Совместимость | Ключевое отличие |
|-----------|-----|-------|----------|---------------|-----------------|
| **Docker** | Платформа | dockerd | Да (rootless mode) | OCI-образы | Полная экосистема, де-факто стандарт |
| **Podman** | CLI + libpod | Нет (daemonless) | Да (из коробки) | OCI-образы, Dockerfile | Без демона, systemd-интеграция, Kubernetes-совместимые pod'ы |
| **containerd** | Container runtime | containerd | Нет | OCI-образы | Низкоуровневый runtime (используется в Kubernetes, Docker) |
| **nerdctl** | CLI для containerd | containerd | Да | OCI-образы, Dockerfile, Compose | Docker-совместимый CLI для containerd |
| **Finch** | CLI (AWS) | containerd + Lima VM | Да | OCI-образы, Dockerfile | Open-source альтернатива Docker Desktop для macOS |
| **Rancher Desktop** | GUI + CLI | containerd/ dockerd | Да | OCI-образы, Dockerfile, Compose | Альтернатива Docker Desktop с GUI и Kubernetes |
| **cri-o** | Container runtime | cri-o | Нет | OCI-образы | Специализированный runtime для Kubernetes (замена containerd) |

#### Когда что использовать

| Сценарий | Рекомендация |
|----------|-------------|
| **Локальная разработка** | Docker Desktop (удобство) или Rancher Desktop (open-source) |
| **Linux-сервер, production** | Docker Engine или Podman (если нужен daemonless) |
| **Kubernetes (worker node)** | containerd или cri-o |
| **CI/CD pipeline** | Docker (buildx) или Podman (без привилегий) |
| **Без root-доступа** | Podman или rootless Docker |
| **Systemd-интеграция** | Podman (генерирует systemd unit-файлы: `podman generate systemd`) |

#### Podman: daemonless-альтернатива

```bash
# Podman — полный аналог Docker CLI (те же команды)
podman build -t myapp .
podman run -d --name myapp -p 8080:8080 myapp
podman ps

# Podman может генерировать Kubernetes YAML из запущенных контейнеров
podman generate kube myapp > pod.yaml

# Podman + systemd (автозапуск контейнера как сервиса)
podman generate systemd --new myapp > /etc/systemd/system/myapp.service
systemctl enable --now myapp
```

#### containerd + nerdctl

```bash
# nerdctl повторяет Docker CLI для containerd (без dockerd)
nerdctl build -t myapp .
nerdctl run -d --name myapp -p 8080:8080 myapp
nerdctl compose up                  # поддержка Compose!

# Главное отличие: неймспейсы (аналог контекстов, но встроенные)
nerdctl --namespace production ps
nerdctl --namespace development ps
```

### Docker Compose на практике: локальная разработка


Для локальной разработки Docker Compose — основной инструмент. Вот несколько продвинутых приёмов:

**Watch mode (Compose v2.22+):** Автоматическая синхронизация файлов при изменении исходного кода (замена volumes для hot reload):

```yaml
services:
  app:
    build: .
    develop:
      watch:
        - action: sync
          path: ./src
          target: /app/src
        - action: rebuild
          path: package.json
```

**Include (Compose v2.20+):** Композиция нескольких compose-файлов:

```yaml
include:
  - path: ./infra/docker-compose.yml    # БД, кэш, очереди
  - path: ./services/docker-compose.yml # Микросервисы
```

**Переопределение (override):**
```bash
docker compose -f docker-compose.yml -f docker-compose.override.yml up
# override.yml добавляется/переопределяет базовый
```

Типовой flow Docker в CI/CD:

```
┌─────────┐   Push    ┌──────────┐   Build    ┌──────────────┐
│   Git   │──────────▶│  CI (Jen-│───────────▶│  Docker       │
│  (PR)   │           │ kins/GHA)│            │  Registry     │
└─────────┘           └────┬─────┘            └──────┬────────┘
                           │                        │
                           │                        │ Pull
                           ▼                        ▼
                    ┌──────────────────────────────────┐
                    │         Kubernetes / Сервер       │
                    │  ┌────────────────────────────┐  │
                    │  │  Deploy (ArgoCD / Flux /   │  │
                    │  │  helm upgrade / ansible)   │  │
                    │  └────────────────────────────┘  │
                    └──────────────────────────────────┘
```

**Ключевые шаги Docker в CI:**
1. **Сборка образа** (`docker build` или `docker buildx build`)
2. **Сканирование уязвимостей** (`docker scout` / `trivy image`)
3. **Тегирование** (git SHA, ветка, семантическая версия)
4. **Пуш в registry** (`docker push`)
5. **Пул и деплой** на целевом окружении

### Лучшие практики Docker

1. **Один процесс на контейнер** — контейнер не должен запускать несколько сервисов (для этого используйте Pod в Kubernetes или supervisord в исключительных случаях).

2. **Минимизируйте слои** — объединяйте RUN, очищайте кэш в том же слое.

3. **Минимальный базовый образ** — alpine или distroless для production.

4. **Не храните секреты в образе** — используйте secrets manager, переменные окружения только во время запуска, либо `--mount=type=secret` в BuildKit.

5. **Не запускайте от root** — создайте непривилегированного пользователя.

6. **Используйте .dockerignore** — исключайте node_modules, .git, временные файлы.

7. **Фиксируйте версии** — теги `:latest` непредсказуемы.

8. **Добавляйте HEALTHCHECK** — для обнаружения зависших контейнеров.

9. **Логи в stdout/stderr** — не пишите логи в файлы внутри контейнера.

10. **Тегируйте осмысленно** — используйте семантическое версионирование или git SHA.

11. **Используйте BuildKit** — современный движок сборки с параллелизацией, секретами и умным кэшированием.

12. **Сканируйте образы** — регулярно проверяйте на CVE (Docker Scout, Trivy, Grype).

13. **Используйте Multi-stage сборки** — разделяйте build-зависимости и runtime.

14. **Ограничивайте ресурсы** — задавайте `--cpus`, `--memory` и `--pids-limit` для контейнеров в production. Без этого контейнер может исчерпать ресурсы хоста.

15. **Предпочитайте COPY вместо ADD** — COPY проще и предсказуемее.

16. **Используйте exec-форму для CMD/ENTRYPOINT** — `CMD ["app"]` вместо `CMD app`. Shell-форма теряет сигналы и создаёт лишний процесс `/bin/sh`.

17. **Используйте `--init` или tini** — если приложение не умеет работать как PID 1 (порождает дочерние процессы, не обрабатывает сигналы).

18. **Настраивайте ротацию логов** — по умолчанию Docker не ротирует json-file логи. Добавьте `max-size` и `max-file` в конфигурацию логирования.

19. **Используйте Docker Contexts** — для переключения между несколькими Docker-хостами (локальный, staging, production) без смены переменных окружения.

20. **Избегайте `docker commit`** — образы должны собираться через Dockerfile (воспроизводимость, версионирование, аудит).
