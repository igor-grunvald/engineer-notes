## 1. Архитектура памяти JVM

### 1.1. Общая картина: Process Memory

JVM-процесс потребляет **значительно больше памяти, чем `-Xmx`**. Полная картина выглядит так:

```
┌──────────────────────────────────────────────────────────────────────┐
│                        JVM PROCESS MEMORY                            │
│                                                                      │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │                     MANAGED MEMORY (JVM)                      │   │
│  │                                                                │   │
│  │  ┌──────────────────────────────────────────────────────┐    │   │
│  │  │                    HEAP (-Xmx)                        │    │   │
│  │  │                                                      │    │   │
│  │  │  ┌─────────────────────┐  ┌─────────────────────┐    │    │   │
│  │  │  │   YOUNG GENERATION  │  │   OLD GENERATION    │    │    │   │
│  │  │  │                    │  │   (Tenured)         │    │    │   │
│  │  │  │  ┌───────┐┌──────┐│  │                     │    │    │   │
│  │  │  │  │ Eden  ││ S0   ││  │  Долгоживущие       │    │    │   │
│  │  │  │  │       │├──────┤│  │  объекты            │    │    │   │
│  │  │  │  │       ││ S1   ││  │                     │    │    │   │
│  │  │  │  └───────┘└──────┘│  │                     │    │    │   │
│  │  │  └─────────────────────┘  └─────────────────────┘    │    │   │
│  │  │                                                      │    │   │
│  │  │  ┌──────────────────────────────────────────────┐    │    │   │
│  │  │  │              STRING POOL                     │    │    │   │
│  │  │  │  (часть Heap с Java 7+)                      │    │    │   │
│  │  │  └──────────────────────────────────────────────┘    │    │   │
│  │  └──────────────────────────────────────────────────────┘    │   │
│  │                                                                │   │
│  │  ┌──────────────────────────────────────────────────────┐    │   │
│  │  │                   METASPACE                          │    │   │
│  │  │  (метаданные классов, не в куче!)                     │    │   │
│  │  └──────────────────────────────────────────────────────┘    │   │
│  │                                                                │   │
│  │  ┌──────────────────────────────────────────────────────┐    │   │
│  │  │                  CODE CACHE                           │    │   │
│  │  │  (JIT-скомпилированный машинный код)                   │    │   │
│  │  └──────────────────────────────────────────────────────┘    │   │
│  │                                                                │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  ┌──────────────────────────────────────────────────────────────┐   │
│  │                   NATIVE MEMORY                               │   │
│  │                                                                │   │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────────┐ │   │
│  │  │ Thread   │ │ Direct   │ │ GC       │ │ JNI / Native     │ │   │
│  │  │ Stacks   │ │ Buffers  │ │ Internal │ │ Libraries        │ │   │
│  │  │ (N × Xss)│ │ (NIO)    │ │ (RSet,   │ │ (.so / .dll)     │ │   │
│  │  │          │ │          │ │  Bitmap, │ │                  │ │   │
│  │  │          │ │          │ │  CardTb) │ │                  │ │   │
│  │  └──────────┘ └──────────┘ └──────────┘ └──────────────────┘ │   │
│  └──────────────────────────────────────────────────────────────┘   │
│                                                                      │
│  Total ≈ Heap + Metaspace + CodeCache + (Xss × threads)              │
│           + DirectBuffers + GC_structures + Native_libs               │
└──────────────────────────────────────────────────────────────────────┘
```

**Правило большого пальца:** реальное потребление памяти процессом JVM = `-Xmx` × 1.3–1.5 для типичного серверного приложения. В контейнерах это критически важно учитывать: если задать `-Xmx` близко к лимиту контейнера, процесс будет убит OOM Killer'ом ядра.

### 1.2. Heap (куча) — детально

Heap — центральная область памяти, где живут **все Java-объекты**. Это то, что ограничивается `-Xms` (начальный) и `-Xmx` (максимальный).

#### Структура поколений

```
                          ┌──────────────────────────────────┐
                          │            HEAP                  │
                          │                                  │
  new Object()  ──────→  │  ┌────────────────────────────┐  │
                          │  │     YOUNG GENERATION        │  │
                          │  │                             │  │
                          │  │  ┌──────────┐  ┌────────┐  │  │
                          │  │  │          │  │   S0   │  │  │
                          │  │  │   EDEN   │  ├────────┤  │  │
                          │  │  │          │  │   S1   │  │  │
                          │  │  └──────────┘  └────────┘  │  │
                          │  │       ↑             ↑       │  │
                          │  │   новые        survivor     │  │
                          │  │   объекты      spaces       │  │
                          │  └──────────────┬─────────────┘  │
                          │                 │ promotion       │
                          │                 ▼                 │
                          │  ┌────────────────────────────┐  │
                          │  │     OLD GENERATION          │  │
                          │  │        (Tenured)            │  │
                          │  │                             │  │
                          │  │   объекты, пережившие       │  │
                          │  │   несколько сборок          │  │
                          │  └────────────────────────────┘  │
                          │                                  │
                          │  ┌────────────────────────────┐  │
                          │  │  HUMONGOUS (G1) / LARGE    │  │
                          │  │  объекты > 50% размера     │  │
                          │  │  региона                   │  │
                          │  └────────────────────────────┘  │
                          └──────────────────────────────────┘
```

#### Young Generation — фабрика объектов

**Eden Space:**
- Точка входа для всех новых объектов
- Обычно самый большой регион Young Gen (Eden : S0 : S1 = 8 : 1 : 1 по умолчанию для Serial/Parallel GC; G1 использует динамическое разбиение на регионы)
- Быстро заполняется → частая Minor GC
- Каждый поток аллоцирует в своём TLAB внутри Eden (см. §1.9)

**Survivor Spaces (S0, S1):**
- Два одинаковых по размеру пространства
- В любой момент времени **одно из них пустое** (to-space), другое содержит объекты (from-space)
- Minor GC: живые объекты из Eden + from-Survivor → копируются в to-Survivor
- Роли S0 и S1 меняются местами после каждой сборки
- Каждый объект имеет **age** (возраст), хранящийся в Mark Word заголовка
- `-XX:MaxTenuringThreshold` (по умолчанию 15) — после этого количества выживаний объект переходит в Old Gen
- JVM может динамически корректировать возраст promotion (для Parallel GC управляется `-XX:+UseAdaptiveSizePolicy`; для G1 работает в пределах `MaxTenuringThreshold`)

**Как работает Minor GC (Scavenge):**

```
ДО:                               ПОСЛЕ:
┌──────────┐ ┌────┐ ┌────┐       ┌──────────┐ ┌────┐ ┌────┐
│  Eden    │ │ S0 │ │ S1 │       │  Eden    │ │ S0 │ │ S1 │
│ ████████ │ │ ██ │ │    │  →    │          │ │    │ │ ██ │
│ (полный) │ │жив.│ │пуст│       │ (пустой) │ │пуст│ │жив.│
└──────────┘ └────┘ └────┘       └──────────┘ └────┘ └────┘
                                       ↑            S0↔S1 поменялись ролями
```

1. Eden заполнен → триггер Minor GC → Stop-The-World (все потоки приложения заморожены)
2. Находим все живые объекты в Eden и S0 (from-space), проходя от GC Roots
3. Копируем живые объекты в S1 (to-space), увеличивая их age на 1
4. Если age ≥ `MaxTenuringThreshold` → объект копируется в Old Gen (promotion)
5. Eden и S0 (from) очищаются полностью
6. S1 становится from-space для следующей сборки, S0 — to-space
7. Приложение продолжает работу

#### Old Generation (Tenured) — дом для долгожителей

- Объекты, пережившие порог promotion
- Крупные объекты могут попасть сюда напрямую (pretenuring), минуя Survivor
- Собирается реже, чем Young
- Сборка (Major GC / Full GC) обычно дольше, чем Minor

#### Размеры поколений: ключевые соотношения

| Параметр | Описание | По умолчанию |
|---|---|---|
| `-XX:NewRatio` | Соотношение Old / Young | 2 (Old = 2/3 Heap, Young = 1/3) |
| `-XX:SurvivorRatio` | Соотношение Eden / Survivor | 8 (Eden = 8/10 Young, каждый Survivor = 1/10) |
| `-Xmn` / `-XX:NewSize` / `-XX:MaxNewSize` | Явный размер Young Gen | Не задано |
| `-XX:MaxTenuringThreshold` | Макс. возраст до promotion | 15 (G1: 15, Parallel: 15) |

### 1.3. Stack (стек) — потоковая память

Каждый Java-поток имеет **собственный стек**. Размер задаётся `-Xss` (по умолчанию ~1MB, зависит от платформы).

#### Структура стекового фрейма

```
Thread Stack (LIFO)
│
├── Frame: метод main()
│   ┌──────────────────────────────┐
│   │ Local Variables Array        │  int x = 42;     → сам 42
│   │                               │  Object o = ...;  → ссылка (4/8 байт)
│   │                               │  double d = 3.14; → само значение
│   ├──────────────────────────────┤
│   │ Operand Stack                │  iload_0          → push 42
│   │                               │  iconst_1         → push 1
│   │                               │  iadd             → pop 42, pop 1, push 43
│   ├──────────────────────────────┤
│   │ Frame Data                    │
│   │  - Return Address (куда вернуться)
│   │  - Pointer to Constant Pool  │
│   │  - Previous Frame Pointer    │
│   │  - Exception Table reference │
│   └──────────────────────────────┘
│
├── Frame: метод foo()
│   ┌──────────────────────────────┐
│   │ ...                          │
│   └──────────────────────────────┘
▼
```

#### Что именно хранится в стеке?

| Что | Где |
|---|---|
| Примитивные типы (`int`, `long`, `double`, ...) | Непосредственно в Local Variables Array |
| Ссылки на объекты (reference) | В Local Variables Array (сам объект в куче) |
| Промежуточные результаты вычислений | Operand Stack |
| Return address после вызова метода | Frame Data |

**Важно:** escape analysis может разместить объект на стеке (scalar replacement), если он не «убегает» из метода. Это оптимизация JIT (C2), уменьшающая давление на GC.

#### StackOverflowError vs OutOfMemoryError

- **StackOverflowError:** фреймы заполнили стек (обычно бесконечная/глубокая рекурсия)
- **OutOfMemoryError: unable to create new native thread:** не хватает native-памяти для выделения стека под новый поток (даже если куча пустая!)

### 1.4. Metaspace — метаданные классов

Metaspace появился в Java 8, заменив PermGen. Принципиальное отличие: **Metaspace находится в native-памяти, а не в куче**.

#### Что хранится в Metaspace

```
Metaspace
│
├── Klass metadata (по одному на каждый загруженный класс)
│   ├── Имя класса (fully qualified)
│   ├── Модификаторы (public, abstract, final, ...)
│   ├── Информация о суперклассе и интерфейсах
│   ├── Поля: имя, тип, смещение в объекте, модификаторы
│   ├── Методы: имя, сигнатура, байт-код, таблица исключений
│   ├── Константный пул (Constant Pool) — строки, числа, ссылки
│   ├── Аннотации (для RUNTIME-аннотаций)
│   └── Информация для рефлексии
│
├── String Pool (с Java 7+ — в куче, НЕ в Metaspace!)
│
├── Таблицы виртуальных методов (vtable, itable)
│
└── Данные для Method Handles (Java 7+)
```

#### PermGen vs Metaspace — сравнение

```
┌─────────────────────────────────────────────────────────────────┐
│                      PERMGEN (Java ≤ 7)                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                    HEAP                                  │   │
│  │  ┌──────────────────┐  ┌────────────────────────────┐   │   │
│  │  │   Young + Old    │  │        PermGen             │   │   │
│  │  │                  │  │  -XX:PermSize              │   │   │
│  │  │                  │  │  -XX:MaxPermSize           │   │   │
│  │  │                  │  │  ФИКСИРОВАННЫЙ РАЗМЕР      │   │   │
│  │  └──────────────────┘  └────────────────────────────┘   │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                    METASPACE (Java ≥ 8)                          │
│  ┌────────────────────────────┐  ┌──────────────────────────┐   │
│  │         HEAP               │  │      NATIVE MEMORY       │   │
│  │                            │  │                          │   │
│  │  ┌────────────────────┐    │  │  ┌───────────────────┐   │   │
│  │  │   Young + Old      │    │  │  │    Metaspace      │   │   │
│  │  │                    │    │  │  │                   │   │   │
│  │  │                    │    │  │  │  АВТО-РАСШИРЯЕТСЯ  │   │   │
│  │  │                    │    │  │  │  (до MaxMetaspace) │   │   │
│  │  └────────────────────┘    │  │  └───────────────────┘   │   │
│  └────────────────────────────┘  └──────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

| Характеристика | PermGen | Metaspace |
|---|---|---|
| Расположение | Внутри Heap | Native memory (Off-Heap) |
| Размер по умолчанию | ~64MB (фиксировано) | Без ограничений (авто-рост) |
| Ограничение | `-XX:MaxPermSize` | `-XX:MaxMetaspaceSize` |
| Освобождение памяти | Только при Full GC | При выгрузке классов + GC |
| Ошибка | `OutOfMemoryError: PermGen space` | `OutOfMemoryError: Metaspace` |
| String Pool (interned) | В PermGen (до Java 7) | В куче (Java 7+) |

#### Как Metaspace растёт

```
Metaspace capacity
│
│  ┌───────────────────────────────────────  MaxMetaspaceSize (если задан)
│  │         ▲
│  │         │  Metaspace автоматически расширяется,
│  │         │  когда занято > (100 - MinMetaspaceFreeRatio)%
│  │    ┌────┤
│  │    │    │
│  │    │    │  Расширение происходит блоками (chunks),
│  │    │    │  запрашиваемыми из VirtualSpace
│  │    │    │
│  └────┴────┴─── MetaspaceSize (начальный, ~21MB)
│
└──────────────────────────────────────────────────────→ время
```

**Ключевые флаги Metaspace:**
| Флаг | Описание | По умолчанию |
|---|---|---|
| `-XX:MetaspaceSize` | Начальный размер | ~21MB (платформозависимо) |
| `-XX:MaxMetaspaceSize` | Максимальный размер | Без ограничений |
| `-XX:MinMetaspaceFreeRatio` | Мин. % свободного места до GC | 40 |
| `-XX:MaxMetaspaceFreeRatio` | Макс. % свободного места после GC | 70 |
| `-XX:MinMetaspaceExpansion` | Мин. шаг расширения | ~340KB |
| `-XX:MaxMetaspaceExpansion` | Макс. шаг расширения | Без ограничений |

#### Утечка Metaspace

Если Metaspace переполняется, происходит Full GC. Если и это не помогает — `OutOfMemoryError: Metaspace`.

**Причины:**
- Чрезмерное использование рефлексии / динамических прокси
- Множественные redeploy приложений (каждый ClassLoader со своим набором классов)
- Генерация классов во время выполнения (CGLIB, ByteBuddy, динамические прокси)
- Groovy / JRuby / Nashorn — динамическая компиляция скриптов

### 1.5. Code Cache — JIT-код

Code Cache хранит скомпилированный JIT-компилятором машинный код методов.

```
Code Cache
│
├── Non-profiled code  (C2-скомпилированный, максимально оптимизированный)
├── Profiled code      (C1-скомпилированный со счётчиками для C2)
└── Non-method code    (адаптеры, заглушки, trampolines)
```

**Сегментация (Java 9+):**
- `-XX:+SegmentedCodeCache` — разделяет кэш на три сегмента
- Без сегментации один большой регион может быть фрагментирован (метод скомпилирован C1, перекомпилирован C2 — старый код удалён, осталась дыра)

**Переполнение Code Cache:**
1. JVM не может скомпилировать новый метод → падает в интерпретатор
2. Включается Code Cache Flushing (`-XX:+UseCodeCacheFlushing`) — удаление редко используемого кода
3. Если и это не помогает → JVM перестаёт компилировать новые методы вообще

**Флаги:**
| Флаг | Описание | По умолчанию |
|---|---|---|
| `-XX:ReservedCodeCacheSize` | Размер Code Cache | 240MB (Java 11+) |
| `-XX:+UseCodeCacheFlushing` | Очистка от rarely-used кода | On |
| `-XX:+SegmentedCodeCache` | Разделение на сегменты | On (Java 9+) |

### 1.6. Native Memory — всё, что за пределами кучи

Native-память не управляется GC. Её потребление надо мониторить отдельно.

| Компонент | Описание | Как ограничить |
|---|---|---|
| **Thread Stacks** | `-Xss` × N_threads | `-Xss` + лимит потоков на уровне ОС |
| **Direct Byte Buffers** | `ByteBuffer.allocateDirect()` | `-XX:MaxDirectMemorySize` (по умолчанию = `-Xmx`) |
| **GC Internal Structures** | Remembered Sets, Card Tables, Mark Bitmaps, Mark Stacks | Увеличивается с размером кучи, не настраивается напрямую |
| **JNI / Native Libraries** | Загруженные .so/.dll | Не ограничено JVM |
| **Metaspace** | Метаданные классов | `-XX:MaxMetaspaceSize` |
| **Code Cache** | JIT-код | `-XX:ReservedCodeCacheSize` |
| **Arena / malloc** | Внутренние аллокации JVM | Не ограничено |

**NMT (Native Memory Tracking):**
```
-XX:NativeMemoryTracking=summary   # или detail
jcmd <pid> VM.native_memory summary
```

NMT покажет, сколько native-памяти потребляет каждая подсистема JVM. Особенно полезно при отладке утечек native-памяти (например, незакрытые DirectByteBuffer).

### 1.7. Объект в памяти: Object Header + поля + выравнивание

Каждый объект в куче состоит из трёх частей:

```
┌────────────────────────────────────────────────────────────────┐
│                     Java Object (Heap)                         │
├────────────────────────┬───────────────────┬──────────────────┤
│     Object Header      │   Instance Data   │     Padding      │
│     (12/16 байт)       │   (поля объекта)  │  (до кратности 8)│
├───────────┬────────────┤                   │                  │
│ Mark Word │ Klass Ptr  │  field1: int      │  0000...         │
│ (8 байт)  │ (4/8 байт) │  field2: long     │  (если нужно)    │
│           │            │  field3: ref      │                  │
└───────────┴────────────┴───────────────────┴──────────────────┘
```

#### Mark Word (8 байт на 64-битной JVM)

Mark Word — мультиплексированная структура. Содержимое зависит от состояния объекта:

```
Состояние: Normal (без блокировки)
┌──────────────────────────────────────────────────────────────┐
│       identity_hashcode (25 бит)  │ age(4) │ biased_lock(1) │
│       unused_gap (29 бит)         │        │ lock_bits  (2) │
└──────────────────────────────────────────────────────────────┘

Состояние: Biased Lock (смещённая блокировка)
┌──────────────────────────────────────────────────────────────┐
│   thread_id (54 бита)  │ epoch(2) │ age(4) │ biased(1) │ 01  │
└──────────────────────────────────────────────────────────────┘

Состояние: Lightweight Lock (тонкая блокировка)
┌──────────────────────────────────────────────────────────────┐
│   pointer to lock record on stack (62 бита)            │ 00  │
└──────────────────────────────────────────────────────────────┘

Состояние: Heavyweight Lock (толстая блокировка)
┌──────────────────────────────────────────────────────────────┐
│   pointer to ObjectMonitor (62 бита)                   │ 10  │
└──────────────────────────────────────────────────────────────┘

Состояние: GC (для эвакуации/forwarding)
┌──────────────────────────────────────────────────────────────┐
│   forwarding pointer (62 бита)                         │ 11  │
└──────────────────────────────────────────────────────────────┘
```

GC-флаги в Mark Word:
- **age** (4 бита): количество выживаний в Survivor (max 15)
- **forwarding pointer** (при эвакуации): указывает, куда объект перемещён
- **lock bits**: позволяют понять, заблокирован ли объект, не мешая ли GC

#### Klass Pointer (4/8 байт)

Указатель на метаданные класса в Metaspace:
- С CompressedOops (куча ≤ 32GB): 4 байта
- Без CompressedOops: 8 байт
- Можно отдельно сжать: `-XX:+UseCompressedClassPointers`

#### Instance Data (поля)

Порядок полей в объекте (HotSpot):
1. Поля родительского класса
2. Поля текущего класса
3. Поля сортируются по размеру: `long/double` (8), `int/float/ref` (4), `short/char` (2), `byte/boolean` (1)
4. Ссылки на объекты группируются вместе в конце

Это минимизирует паддинг (gaps) между полями.

#### Выравнивание (Alignment)

Все объекты в HotSpot выравниваются по **8 байтам** (`-XX:ObjectAlignmentInBytes=8`). Если суммарный размер заголовка + полей не кратен 8 — добавляется padding.

**Пример расчёта размера объекта:**
```java
class Person {
    int age;      // 4 байта
    String name;  // 4 байта (Compressed OOPs)
    long id;      // 8 байт
}
```

```
Header:   Mark Word (8) + Klass Ptr (4) = 12 байт
Fields:   id   (8)  → смещение 12 (выровнено)
          age  (4)  → смещение 20
          name (4)  → смещение 24
          Total fields: 16 байт
Итого:    12 + 16 = 28 байт → выравнивается до 32 байт
```

---

### 1.8. Compressed OOPs — сжатие указателей

На 64-битной JVM указатели занимали бы 8 байт, что увеличило бы потребление памяти на ~50%. Compressed OOPs (Ordinary Object Pointers) сжимают указатели до 4 байт.

**Принцип работы:**
```
64-битный указатель  ──compress──→  32-битное смещение  ──decompress──→  64-битный адрес

Формула:  native_address = (compressed_oop × 8) + base
          (объекты выровнены по 8 байт → младшие 3 бита всегда 000)
```

**Ограничения:**
- Куча ≤ 32GB (для 8-байтового выравнивания)
- С `-XX:ObjectAlignmentInBytes=16`: куча ≤ 64GB (но больше паддинга)
- Автоматически включается для `-Xmx < 32GB`

**Типы Compressed указателей:**
| Тип | Флаг | Что сжимает |
|---|---|---|
| Compressed OOPs | `-XX:+UseCompressedOops` | Ссылки на объекты в куче |
| Compressed Class Pointers | `-XX:+UseCompressedClassPointers` | Klass Pointer в заголовке объекта |

**Эффект:** экономия 30-40% памяти кучи для типичного приложения.

---

### 1.9. TLAB — Thread Local Allocation Buffer

Аллокация объекта в Eden из нескольких потоков требует синхронизации. Чтобы избежать блокировок, каждый поток получает эксклюзивный буфер в Eden — **TLAB**.

```
┌──────────────────────────────────────────────────┐
│                    EDEN SPACE                     │
│                                                   │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐             │
│  │ TLAB    │ │ TLAB    │ │ TLAB    │  ...         │
│  │ Thread1 │ │ Thread2 │ │ Thread3 │             │
│  │ ████    │ │ ██████  │ │ ███     │             │
│  └─────────┘ └─────────┘ └─────────┘             │
│                                                   │
│  ┌──────────────────────────────────┐             │
│  │  Shared Eden (для крупных        │             │
│  │  объектов > TLAB размера)        │             │
│  └──────────────────────────────────┘             │
└──────────────────────────────────────────────────┘
```

**Как работает TLAB:**
1. Поток запрашивает TLAB: просто bump-the-pointer (сдвинуть указатель свободной памяти)
2. Поток аллоцирует внутри своего TLAB: тоже bump-the-pointer, **без синхронизации**
3. TLAB заполнен → поток запрашивает новый TLAB (с синхронизацией один раз)
4. Объекты, не влезающие в TLAB, аллоцируются напрямую в Eden с синхронизацией

**Флаги:**
| Флаг | Описание | По умолчанию |
|---|---|---|
| `-XX:+UseTLAB` | Включить TLAB | On |
| `-XX:TLABSize` | Размер TLAB | Автоматически |
| `-XX:-ResizeTLAB` | Отключить динамический размер | Off |
| `-XX:TLABWasteTargetPercent` | % Eden, допустимый для waste | 1% |

## 2. Garbage Collection — фундаментальные концепции

### 2.1. Generational Hypothesis

**Слабая гипотеза поколений (Weak Generational Hypothesis):**
- Большинство объектов умирает молодыми (live fast, die young)
- Мало объектов живёт долго
- Объекты, пережившие несколько сборок, скорее всего проживут ещё дольше

Эта гипотеза — основа разделения кучи на поколения. Young Gen собирается часто и быстро (большинство объектов мёртвые → мало копирования). Old Gen собирается редко.

```
Выживаемость объектов
│
│ ██   ← 95% объектов умирает в первые секунды
│ ██
│ ██
│ █
│ █
│ ▌   ← ~5% доживают до Old Gen
│ ▌
│ ▏   ← <<1% живут всю жизнь приложения
│ ▏
└─────────────────────────→ время жизни
```

---

### 2.2. Виды сборок: Minor, Major, Full GC

```
┌─────────────────────────────────────────────────────────────────┐
│                        ТИПЫ СБОРОК GC                           │
│                                                                 │
│  ┌─────────────────┐   ┌─────────────────┐   ┌───────────────┐ │
│  │   MINOR GC      │   │   MAJOR GC      │   │   FULL GC     │ │
│  │                 │   │                 │   │               │ │
│  │  Область:       │   │  Область:       │   │  Область:     │ │
│  │  Young Gen      │   │  Old Gen        │   │  Heap +       │ │
│  │  (Eden+Survivor)│   │  (Tenured)      │   │  Metaspace    │ │
│  │                 │   │                 │   │               │ │
│  │  Триггер:       │   │  Триггер:       │   │  Триггер:     │ │
│  │  Eden заполнен  │   │  Old заполнен   │   │  Old полон    │ │
│  │                 │   │  или promotion  │   │  Metaspace    │ │
│  │  Длительность:  │   │  некуда         │   │  полон        │ │
│  │  мс             │   │                 │   │  System.gc()  │ │
│  │                 │   │  Длительность:  │   │               │ │
│  │  STW: да        │   │  дольше Minor   │   │  Длительность:│ │
│  │                 │   │                 │   │  секунды      │ │
│  │  Частота: часто │   │  STW: зависит   │   │               │ │
│  │                 │   │  от GC          │   │  STW: да      │ │
│  └─────────────────┘   └─────────────────┘   └───────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

**Minor GC (Young GC, Scavenge):**
- Собирает ТОЛЬКО Young Gen
- Живые объекты копируются в Survivor или Old
- Stop-The-World (почти у всех сборщиков)
- Обычно миллисекунды
- Триггер: Eden заполнен

**Major GC:**
- Собирает Old Gen
- Определение размыто: разные сборщики называют Major GC разные вещи
- У CMS: сборка только Old Gen
- У G1: часть Mixed GC (эвакуация Old-регионов)

**Full GC:**
- Собирает ВСЁ: Heap (Young + Old) + Metaspace
- Самый длительный и тяжёлый
- Всегда Stop-The-World
- Триггеры:
  - Old Gen заполнен → promotion failure
  - Metaspace заполнен
  - Явный вызов `System.gc()` — в HotSpot по умолчанию вызывает Full GC (Stop-The-World). Отключается флагом `-XX:+DisableExplicitGC`; для CMS/G1 может быть направлен в concurrent-цикл через `-XX:+ExplicitGCInvokesConcurrent`
  - Concurrent Mode Failure (CMS)
  - Allocation Failure в Old Gen
  - Humongous Allocation Failure (G1)

---

### 2.3. GC Roots — корневые точки

GC начинает обход графа объектов с **GC Roots** — набора гарантированно живых ссылок.

```
                    GC Roots
                   ┌─────────┐
                   │ Threads │──── Stack frames (локальные переменные)
                   │         │
                   │ Static  │──── static-поля классов
                   │ fields  │
                   │         │
                   │ JNI     │──── JNI-ссылки (GlobalRef)
                   │ Refs    │
                   │         │
                   │ Class   │──── загруженные классы (ClassLoader)
                   │ Loaders │
                   │         │
                   │ Monitors│──── объекты, используемые для synchronized
                   │         │
                   │ JVM     │──── внутренние структуры JVM
                   │ internal│
                   └────┬────┘
                        │
           ┌────────────┼────────────┐
           ▼            ▼            ▼
        [Obj A] ←──── [Obj B] ←──── [Obj C]
           │                         │
           ▼                         ▼
        [Obj D]                  [Obj E] ←──── этот объект ДОСТИЖИМ
                                               (через цепочку от GC Root)

        [Obj F] ←──── НЕТ цепочки от GC Root → МУСОР, будет собран
```

**Полный список GC Roots:**
1. **Локальные переменные** во всех стеках потоков
2. **Активные потоки** (java.lang.Thread)
3. **Статические поля** загруженных классов
4. **JNI References** (глобальные и локальные)
5. **Объекты-мониторы** (используемые в synchronized)
6. **Объекты, удерживаемые JVM** (системный ClassLoader, interned strings, ...)

---

### 2.4. Достижимость и типы ссылок

Объекты классифицируются по достижимости (strength of reachability):

```
  Strongly Reachable  ──── объект достижим через strong-ссылку
       │                    Не удаляется никогда (кроме завершения потока)
       ▼
  Softly Reachable    ──── достижим только через SoftReference
       │                    Удаляется при нехватке памяти перед OOME
       ▼
  Weakly Reachable    ──── достижим только через WeakReference
       │                    Удаляется при ближайшей GC
       ▼
  Phantom Reachable   ──── достижим только через PhantomReference
       │                    Объект финализирован, ждёт освобождения памяти
       ▼
  Unreachable         ──── нет никаких ссылок → будет собран
```

**Типы ссылок в `java.lang.ref`:**

| Тип ссылки | Когда GC удаляет объект | Типичное применение |
|---|---|---|
| **StrongReference** | Никогда (пока достижим) | Обычные ссылки `Object o = ...` |
| **SoftReference** | Только при нехватке памяти (перед OOME) | Кэши, чувствительные к памяти |
| **WeakReference** | При ближайшей сборке мусора | WeakHashMap, канонические отображения |
| **PhantomReference** | После финализации, объект уже мёртв | Управление off-heap памятью (Cleaner) |

**ReferenceQueue:** все не-strong ссылки можно связать с очередью. Когда GC определяет, что referent подлежит удалению, ссылка добавляется в ReferenceQueue — приложение может очистить связанные ресурсы.

### 2.5. Базовые алгоритмы GC

Все сборщики мусора строятся на комбинации трёх фундаментальных алгоритмов.

#### 2.5.1. Mark-Sweep

```
Фаза Mark:                      Фаза Sweep:
┌───┬───┬───┬───┬───┐          ┌───┬───┬───┬───┬───┐
│ A │ B │ C │ D │ E │          │   │ B │   │ D │   │
│ ■ │ ■ │   │ ■ │   │    →     │   │ ■ │   │ ■ │   │
│   │   │   │   │   │          │   │   │   │   │   │
└───┴───┴───┴───┴───┘          └───┴───┴───┴───┴───┘
  A,B,D — живые, C,E — мёртвые    Мёртвые удалены → фрагментация!
```

**Плюсы:**
- Простота реализации
- Не перемещает объекты

**Минусы:**
- **Фрагментация** — после нескольких циклов свободная память разбита на мелкие куски
- Allocation становится сложнее: нужно найти подходящий по размеру свободный блок (free-list)

#### 2.5.2. Mark-Sweep-Compact

```
Фаза Mark:           Фаза Sweep+Compact:
┌───┬───┬───┬───┬───┐     ┌───┬───┬───┬───┬───┐
│ A │ B │ C │ D │ E │     │ A │ B │ D │   │   │
│ ■ │ ■ │   │ ■ │   │  →  │ ■ │ ■ │ ■ │   │   │
│   │   │   │   │   │     │   │   │   │   │   │
└───┴───┴───┴───┴───┘     └───┴───┴───┴───┴───┘
                               A,B,D сдвинуты влево
                               Свободное место — непрерывно!
```

**Плюсы:**
- Нет фрагментации
- Allocation: просто bump-the-pointer

**Минусы:**
- Дороже Sweep (нужно двигать объекты и править ссылки)
- Требует Stop-The-World (нельзя двигать объекты, пока приложение работает)

#### 2.5.3. Copying (копирующий)

```
From-space:          To-space:
┌───┬───┬───┬───┬───┐     ┌───┬───┬───┬───┬───┐
│ A │ B │ C │ D │ E │     │ A │ B │ D │   │   │
│ ■ │ ■ │   │ ■ │   │  →  │ ■ │ ■ │ ■ │   │   │
│   │   │   │   │   │     │   │   │   │   │   │
└───┴───┴───┴───┴───┘     └───┴───┴───┴───┴───┘
                            From-space освобождается целиком
```

**Плюсы:**
- Нет фрагментации
- Allocation: просто bump-the-pointer
- Быстрее Compact (не нужно отдельно чистить удалённые объекты)
- Идеален для Young Gen (мало живых объектов → мало копирования)

**Минусы:**
- Требует 2× памяти (from + to space)
- Дорого, если много живых объектов (каждый нужно скопировать)
- Stop-The-World

### 2.6. Барьеры: Write Barrier, Read Barrier, SATB

Конкурентные сборщики (CMS, G1, ZGC, Shenandoah) выполняют часть работы без остановки приложения. Чтобы отслеживать изменения в графе объектов во время конкурентных фаз, используются **барьеры**.

#### Write Barrier

Внедряется JIT-компилятором в код каждой операции записи ссылки в поле объекта:

```java
// Было:   obj.field = newValue;
// Стало:  barrier(obj, newValue);  // + сохранение информации для GC
//          obj.field = newValue;
```

**Виды Write Barrier:**

| Механизм | Используется | Как работает |
|---|---|---|
| **Card Table** | Parallel, CMS, G1 | Куча разбита на «карты» по 512 байт. При записи ссылки карта помечается «грязной». При GC сканируются только грязные карты. |
| **SATB** (Snapshot-At-The-Beginning) | G1 | Перед перезаписью ссылки старое значение сохраняется в SATB-буфер. Гарантирует, что объекты, живые на начало marking, не будут потеряны. |
| **Remembered Sets (RSet)** | G1 | Каждый регион хранит RSet: список карт других регионов, которые содержат ссылки на объекты внутри. Обновляется через write barrier. |

#### Read Barrier

Внедряется в каждую операцию **чтения** ссылки:

```java
// Было:   Object x = obj.field;
// Стало:  Object x = readBarrier(obj.field);
```

Используется ZGC и Shenandoah для «лечения» ссылок на перемещённые объекты (self-healing).

#### SATB (Snapshot-At-The-Beginning)

Алгоритм, используемый G1 для concurrent marking:
1. В начале marking делается «снимок» графа объектов — все живые объекты считаются живыми на этот момент
2. При перезаписи ссылки **старое значение** сохраняется в SATB-буфер (через write barrier)
3. Маркировщик обрабатывает SATB-буфер, чтобы не пропустить объекты, которые были живы на момент снимка

**Плюс:** гарантирует, что marking не пропустит живые объекты.
**Минус:** некоторые мёртвые объекты могут быть ошибочно признаны живыми (floating garbage) — они будут собраны в следующем цикле.

### 2.7. Фрагментация кучи

```
Без компактификации (Mark-Sweep, CMS):

┌──┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
│A │B │  │C │  │D │  │ E│  │  │
└──┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
     ↑     ↑     ↑        ↑
     свободные блоки разного размера

Проблема: нужно аллоцировать объект размером 3 блока
→ есть три свободных блока по 1, но нет непрерывного куска из 3
→ Allocation Failure → Promotion Failure → Full GC

С компактификацией (Copying, Mark-Compact, G1, ZGC):

┌──┬──┬──┬──┬──┬──────────────────┐
│A │B │C │D │E │                  │
└──┴──┴──┴──┴──┴──────────────────┘
                  ↑ свободное место непрерывно
```

Все современные сборщики (G1, ZGC, Shenandoah) **компактифицируют** кучу. CMS — исключение, именно фрагментация была его главной проблемой.

## 3. Сборщики мусора — полный разбор

### 3.1. Serial GC

```
┌─────────────────────────────────────────────────┐
│                  SERIAL GC                       │
│                                                  │
│  Young GC: Serial Copying (1 поток, STW)        │
│  Old GC:   Serial MSC — Mark-Sweep-Compact      │
│            (1 поток, STW)                        │
│                                                  │
│  Приложение:  [████████████] [██] [█████████]    │
│  GC Thread:   [          ] [██] [         ]      │
│                ← работа →  ←STW→ ← работа →      │
└─────────────────────────────────────────────────┘
```

**Алгоритм:**
- **Young:** Copying. Eden → Survivor / Old. Один поток.
- **Old:** Mark-Sweep-Compact. Один поток.
- **Полностью Stop-The-World.**

**Преимущества:**
- Минимальные накладные расходы (нет синхронизации, нет конкурентных структур)
- Самый маленький footprint по памяти (нет RSet, SATB, marking bitmap сверх минимума)
- Детерминированное поведение (легко предсказать)
- Идеален для 1 CPU

**Недостатки:**
- Неприемлемые паузы на кучах > 512MB
- Не использует multi-core CPU
- Нет конкурентных фаз — приложение полностью останавливается

**Когда использовать:**
- Heap ≤ 100-200MB
- Одноядерные системы / контейнеры с 1 CPU
- Простые CLI-утилиты
- Эмбеддед-системы

**Флаг:** `-XX:+UseSerialGC`

---

### 3.2. Parallel GC (Throughput Collector)

```
┌─────────────────────────────────────────────────┐
│                PARALLEL GC                       │
│                                                  │
│  Young GC: Parallel Scavenge (N потоков, STW)   │
│  Old GC:   Parallel Old (N потоков, STW)        │
│                                                  │
│  Приложение:  [████████████] [██] [█████████]    │
│  GC Threads:  [          ] [██] [         ]      │
│                ← работа →  ←STW→ ← работа →      │
│                (все N CPU делают GC параллельно) │
└─────────────────────────────────────────────────┘
```

**Алгоритм:**
- **Young:** Parallel Scavenge — копирующий сборщик, работающий в N потоков
- **Old:** Parallel Old — Mark-Summary-Compact в N потоков
- **Полностью Stop-The-World** (как Serial, но параллельно)
- Число потоков: `-XX:ParallelGCThreads` (по умолчанию = CPU cores)

**Целевая метрика:** пропускная способность (throughput):
```
Throughput = время_работы_приложения / (время_работы_приложения + время_GC)
```

Целевой показатель задаётся через `-XX:GCTimeRatio` (по умолчанию 99 → 1% времени на GC).

**Особенности:**
- Адаптивный sizing: `-XX:+UseAdaptiveSizePolicy` — JVM автоматически подбирает размеры поколений и пороги promotion для достижения целевой метрики
- Хорошо масштабируется на многопроцессорных системах

**Преимущества:**
- Максимальный throughput среди всех GC
- Хорошо масштабируется (время паузы ~ обратно пропорционально числу ядер)
- Зрелый, стабильный, предсказуемый
- Адаптивное управление размерами поколений

**Недостатки:**
- Полный Stop-The-World на каждой сборке → паузы могут быть секундами на больших кучах
- Не подходит для интерактивных / latency-sensitive приложений
- На кучах > 4GB паузы становятся заметными (> 100ms)

**Когда использовать:**
- Batch-обработка, ETL, научные расчёты
- Приложения, где паузы приемлемы, но важна максимальная скорость обработки
- Heap до 4GB (как практическая граница комфортных пауз)

**Флаг:** `-XX:+UseParallelGC`

**Дополнительные флаги:**
| Флаг | Описание |
|---|---|
| `-XX:ParallelGCThreads=N` | Число потоков GC |
| `-XX:GCTimeRatio=N` | Целевое соотношение: 1/(N+1) времени на GC |
| `-XX:MaxGCPauseMillis=N` | Подсказка JVM о желаемой макс. паузе |
| `-XX:+UseAdaptiveSizePolicy` | Авто-подбор размеров поколений (по умолчанию on) |

### 3.3. CMS — Concurrent Mark Sweep (deprecated)

> ⚠️ **Deprecated в JDK 9 (JEP 291), удалён в JDK 14 (JEP 363).** Не используйте в новых проектах. Описывается для исторического контекста и понимания эволюции GC.

```
┌──────────────────────────────────────────────────────────────┐
│                         CMS GC                               │
│                                                              │
│  Фаза:                                                        │
│  1. Initial Mark       [STW]     ── пометить GC roots        │
│  2. Concurrent Mark    [CONC]    ── обход графа               │
│  3. Remark             [STW]     ── догнать изменения за (2) │
│  4. Concurrent Sweep   [CONC]    ── освободить мёртвые        │
│  5. Concurrent Reset   [CONC]    ── сбросить структуры        │
│                                                              │
│  Приложение:  [██████] [██████████████] [██] [███████████]   │
│  GC:                [██]             [██████] [██]           │
│                  InitMark  ConcMark  Remark   Sweep          │
└──────────────────────────────────────────────────────────────┘
```

**Алгоритм:**

1. **Initial Mark [STW, короткая]:** пометить объекты, напрямую достижимые из GC Roots. Только ссылки из корней (не обход всего графа).

2. **Concurrent Mark [CONC, длинная]:** обход графа объектов, начиная с помеченных на шаге 1. Приложение работает. CMS использует **Write Barrier + Card Table** для отслеживания изменений.

3. **Remark [STW, средняя]:** догнать изменения, которые приложение сделало во время Concurrent Mark. Сканируются карты, помеченные как dirty.

4. **Concurrent Sweep [CONC, длинная]:** освобождение памяти мёртвых объектов. **Объекты не перемещаются** (нет компактификации). Память возвращается в free-list.

5. **Concurrent Reset [CONC]:** сброс внутренних структур CMS для следующего цикла.

**CMS для Young Gen:** использует **ParNew** — параллельный копирующий сборщик для Young Gen (STW, но параллельно в несколько потоков).

**Проблемы CMS:**

- **Фрагментация:** Sweep не компактирует → куча фрагментируется → невозможно аллоцировать крупный объект даже при достаточном свободном месте.

- **Concurrent Mode Failure (CMF):** Old Gen заполняется быстрее, чем CMS успевает собирать. Причина: приложение аллоцирует в Old быстрее, чем работает Concurrent Sweep. Результат → Full GC (полный STW, Serial Old).

- **Floating Garbage:** объекты, умершие во время Concurrent Mark, не видны CMS до следующего цикла → нужен запас памяти в Old Gen для них.

- **CPU-голод:** конкурентные фазы потребляют CPU, конкурируя с приложением. На системах с малым числом ядер это заметно.

**Преимущества (исторические):**
- Первый сборщик с конкурентными фазами (низкие паузы по сравнению с Parallel)
- Подходит для интерактивных приложений (веб-серверы, GUI)
- Хорошая производительность на кучах 2-8GB

**Недостатки:**
- Фрагментация (главная причина deprecation)
- Concurrent Mode Failure → внезапный длинный Full GC
- Не масштабируется на кучи > 8-16GB
- Удалён из JDK 14+ (JEP 363)

**Флаг:** `-XX:+UseConcMarkSweepGC` (недоступен с Java 18)

### 3.4. G1 GC — Garbage First

> **G1 — сборщик по умолчанию с Java 9.** Разработан для замены CMS. Обеспечивает предсказуемые паузы и компактификацию.

#### Архитектура регионов

G1 разбивает кучу на **регионы фиксированного размера** (1MB–32MB, степень двойки). Количество регионов ≈ 2048.

```
┌───────┬───────┬───────┬───────┬───────┬───────┬───────┬───────┐
│  Old  │ Eden  │ Eden  │ Free  │ Surv  │ Hum   │ Old   │ Eden  │
│       │       │       │       │       │       │       │       │
├───────┼───────┼───────┼───────┼───────┼───────┼───────┼───────┤
│ Eden  │ Old   │ Free  │ Eden  │ Old   │ Eden  │ Free  │ Surv  │
│       │       │       │       │       │       │       │       │
├───────┼───────┼───────┼───────┼───────┼───────┼───────┼───────┤
│ Free  │ Eden  │ Old   │ Hum   │ Free  │ Eden  │ Eden  │ Old   │
└───────┴───────┴───────┴───────┴───────┴───────┴───────┴───────┘
                                    ↑
                              Humongous: объекты > 50% размера региона
```

#### Типы регионов

| Тип | Содержимое | Особенности |
|---|---|---|
| **Eden** | Новые объекты | Аналог Eden Space |
| **Survivor** | Объекты после Minor GC | Аналог S0/S1 |
| **Old** | Долгоживущие объекты | Аналог Old Gen |
| **Humongous** | Крупные объекты (> 50% размера региона, строго больше половины) | Занимает целые регионы. Может вызывать проблемы |
| **Free** | Свободная память | Готовы к использованию |

**Размер региона:** `-XX:G1HeapRegionSize=N` (1MB, 2MB, 4MB, 8MB, 16MB, 32MB). Рекомендуется: 2048 регионов → размер региона ≈ Heap / 2048.

#### Цикл G1 — полный разбор

```
 ┌─────────────────────────────────────────────────────────────┐
 │                      G1 GC CYCLE                            │
 │                                                              │
 │  ┌───────────────────────────────────────────────────────┐  │
 │  │  1. YOUNG-ONLY PHASE                                  │  │
 │  │     Серия Young GC (эвакуация Eden → Survivor → Old)  │  │
 │  │     Объекты с age ≥ threshold → Old                   │  │
 │  │     STW, короткие паузы (~10-50ms)                    │  │
 │  └───────────────────────────────────────────────────────┘  │
 │                         │                                    │
 │     Old Gen occupancy ≥ InitiatingHeapOccupancyPercent      │
 │     (по умолчанию 45%)                                      │
 │                         │                                    │
 │                         ▼                                    │
 │  ┌───────────────────────────────────────────────────────┐  │
 │  │  2. CONCURRENT MARKING                                │  │
 │  │                                                       │  │
 │  │  2a. Initial Mark    [STW,   ~2-5ms]                  │  │
 │  │       - Пометить GC roots                             │  │
 │  │       - Совмещается с Young GC (piggyback)            │  │
 │  │                                                       │  │
 │  │  2b. Root Region Scan [CONC]                          │  │
 │  │       - Сканирование Survivor-регионов                │  │
 │  │                                                       │  │
 │  │  2c. Concurrent Mark  [CONC]                          │  │
 │  │       - Обход графа объектов от корней               │  │
 │  │       - SATB-буфер отслеживает изменения              │  │
 │  │       - Попутно: подсчёт живости (liveness)           │  │
 │  │         для каждого региона                           │  │
 │  │                                                       │  │
 │  │  2d. Remark           [STW,   ~5-20ms]                │  │
 │  │       - Догнать изменения за (2c)                     │  │
 │  │       - Обработать SATB-буфер                         │  │
 │  │       - Завершить marking                             │  │
 │  │                                                       │  │
 │  │  2e. Cleanup          [STW + CONC]                    │  │
 │  │       - Подсчёт % мусора в каждом Old-регионе         │  │
 │  │       - Освобождение регионов с 0% живых объектов     │  │
 │  │       - Сортировка Old-регионов по % мусора           │  │
 │  └───────────────────────────────────────────────────────┘  │
 │                         │                                    │
 │                         ▼                                    │
 │  ┌───────────────────────────────────────────────────────┐  │
 │  │  3. SPACE RECLAMATION (MIXED GC)                      │  │
 │  │                                                       │  │
 │  │  Серия Mixed GC:                                      │  │
 │  │  - Эвакуация: все Young-регионы + N Old-регионов      │  │
 │  │    (с наибольшим % мусора)                            │  │
 │  │  - Живые объекты копируются в другие регионы          │  │
 │  │  - Покинутые регионы → Free                           │  │
 │  │  - Количество Old-регионов в CSet подбирается так,    │  │
 │  │    чтобы уложиться в MaxGCPauseMillis                 │  │
 │  │                                                       │  │
 │  │  Заканчивается, когда:                                │  │
 │  │  - Достаточно памяти освобождено                     │  │
 │  │  - Old Gen occupancy упала ниже порога                │  │
 │  └───────────────────────────────────────────────────────┘  │
 └─────────────────────────────────────────────────────────────┘
```

#### Remembered Sets (RSet) — ключевой механизм G1

Проблема: при сборке одного региона нужно знать, какие объекты из **других** регионов на него ссылаются — иначе пришлось бы сканировать всю кучу.

Решение — **Remembered Set (RSet):** каждый регион хранит список внешних ссылок на себя.

```
  Region 1 (Old)              Region 2 (Eden)
┌─────────────────┐         ┌─────────────────┐
│  [Obj A]────────┼────────→│    [Obj B]      │
│       ↑         │         │                 │
│  [Obj C]        │         │    [Obj D]      │
│       │         │         │        ↑        │
│       │         │         │        │        │
│       └─────────┼─────────┼────────┘        │
└─────────────────┘         └─────────────────┘
    RSet Region 1:              RSet Region 2:
    ∅                           {Region 1: card X, card Y}

    Card Table (глобальная):
    Каждая «карта» = 512 байт памяти
    При записи ссылки в поле → карта помечается dirty
```

**Как работает RSet:**
1. Приложение записывает ссылку в поле → Write Barrier → Card Table[dirty] = true
2. Во время GC: для каждого региона в CSet сканируются только dirty-карты из его RSet
3. Refinement-потоки (concurrent refinement threads) асинхронно обновляют RSet, обрабатывая dirty cards из буфера

**Накладные расходы RSet:** 1-5% памяти кучи, зависит от паттерна ссылок между регионами.

#### Collection Set (CSet)

Набор регионов, которые будут эвакуированы в текущей сборке:

```
CSet = { все Young-регионы } + { top-N Old-регионов по % мусора }

Ограничение: время эвакуации CSet ≤ MaxGCPauseMillis
```

G1 сортирует Old-регионы по проценту мусора и включает их в CSet, пока предсказываемое время не превысит целевую паузу. Отсюда название — **Garbage First** (сначала собираем регионы с наибольшим количеством мусора).

#### Humongous Objects — проблема G1

Объект считается humongous, если его размер **строго больше 50%** размера региона G1 (объект ровно в 50% размера региона помещается в обычный регион и не считается humongous).

```
Обычный объект:                 Humongous объект:
┌───────────┐                  ┌───────────┬───────────┐
│  Region   │                  │  Region   │  Region   │
│ ┌───────┐ │                  │ ┌─────────┴─────────┐ │
│ │ Obj A │ │                  │ │     BigObject     │ │
│ └───────┘ │                  │ └───────────────────┘ │
│ ┌───┐     │                  │    + продолжение      │
│ │ B │     │                  │    в соседних         │
│ └───┘     │                  │    регионах           │
└───────────┘                  └───────────┴───────────┘
```

**Проблемы:**
- Humongous-объекты аллоцируются сразу в Old-регионах (не в Eden)
- Для них нужны непрерывные свободные регионы
- Если непрерывных регионов нет → запускается Full GC (даже если памяти в сумме достаточно!)
- До Java 8u40: humongous-объекты не могли быть собраны во время Mixed GC → накопление → Full GC

**Решения:**
- Увеличить размер региона (`-XX:G1HeapRegionSize`)
- Избегать крупных массивов (пересмотреть дизайн данных)
- В Java 8u40+: humongous-объекты могут быть собраны во время cleanup (если стали недостижимы)

#### SATB в G1

G1 использует **Snapshot-At-The-Beginning (SATB)** для concurrent marking. Это гарантирует, что объекты, живые на начало marking, не будут потеряны.

**Алгоритм SATB:**
1. В начале concurrent marking делается snapshot всех ссылок
2. Write Barrier: при перезаписи ссылки **старое значение** помещается в SATB-буфер
3. Маркировщик обрабатывает SATB-буфер: помечает старые значения и их транзитивные достижимые объекты
4. Результат: все объекты, живые на момент снимка, считаются живыми (даже если стали мёртвыми во время marking — floating garbage)

#### Преимущества G1

- **Предсказуемые паузы** — `-XX:MaxGCPauseMillis` (по умолчанию 200ms)
- **Компактификация** — нет фрагментации (в отличие от CMS)
- **Адаптивность** — JVM сама подбирает количество Old-регионов в CSet
- **Масштабируется** до heap ~64GB с приемлемыми паузами
- **Зрелый и стабильный** — в production с Java 7 (полноценно с 8u40)
- **Default с Java 9**

#### Недостатки G1

- **Накладные расходы на RSet** — память (1-5%) и CPU на refinement
- **Write Barrier** на каждую запись ссылки — замедляет приложение (2-5%)
- **На очень больших кучах (>64GB)** — может не укладываться в MaxGCPauseMillis
- **Humongous-объекты** могут вызывать проблемы
- **Full GC всё ещё возможен** — при exhaustion регионов или evacuation failure

#### Ключевые флаги G1

| Флаг | Описание | По умолчанию |
|---|---|---|
| `-XX:+UseG1GC` | Включить G1 | Default с Java 9 |
| `-XX:MaxGCPauseMillis=N` | Целевая максимальная пауза (мягкая цель) | 200 |
| `-XX:G1HeapRegionSize=N` | Размер региона | Авто (Heap / 2048) |
| `-XX:InitiatingHeapOccupancyPercent=N` | Порог Occupancy Old для старта marking | 45 |
| `-XX:G1MixedGCCountTarget=N` | Число Mixed GC в цикле | 8 |
| `-XX:G1MixedGCLiveThresholdPercent=N` | Макс. % живых объектов в регионе для включения в CSet | 85 |
| `-XX:ConcGCThreads=N` | Число конкурентных потоков | ~1/4 от ParallelGCThreads |
| `-XX:ParallelGCThreads=N` | Число потоков для STW-фаз | CPU cores ≤ 8 → cores; > 8 → 5/8 × cores |
| `-XX:G1ReservePercent=N` | Резерв памяти для promotion | 10 |

### 3.5. ZGC — Z Garbage Collector

> **ZGC** — сборщик с паузами <1ms, поддерживает кучи от 8MB до 16TB. Production-ready с Java 15.

#### Ключевые технологии ZGC

##### 1. Colored Pointers (цветные указатели)

ZGC использует 64-битные указатели, резервируя несколько старших битов под метаданные:

```
┌───┬───┬──────────────────────────────────────────────┬───────┐
│   │   │             Адрес объекта (44 бита)           │ Мета  │
│   │   │                                              │ (4 бита)
└───┴───┴──────────────────────────────────────────────┴───────┘
  Неиспользуемые                                         ↑
  биты (16 шт.)                          ┌───────┬───────┬───────┬───────┐
                                         │marked0│marked1│remapped│finaliz│
                                         │       │       │        │ able  │
                                         └───────┴───────┴───────┴───────┘
```

**Цвета и их значение:**
| Биты метаданных | Состояние | Значение |
|---|---|---|
| `marked0` | Marked (чётный цикл) | Объект помечен как живой в чётном цикле GC |
| `marked1` | Marked (нечётный цикл) | Объект помечен как живой в нечётном цикле GC |
| `remapped` | Remapped | Ссылка указывает на текущее (актуальное) положение объекта |
| `finalizable` | Finalizable | Объект имеет непустую очередь финализации |

**Как это работает:**
- Зная цвет указателя, ZGC знает состояние объекта **без обращения к его заголовку**
- При перемещении объекта: old-указатель помечается как не-remapped, new-указатель — как remapped
- На каждый цикл GC чередуются marked0/marked1 (чётный/нечётный цикл)

##### 2. Load Barriers

ZGC встраивает **load barrier** (барьер чтения) в каждое чтение ссылки из кучи. Это «self-healing» механизм:

```java
// Исходный код:
Object obj = someField;

// Что делает JIT со встроенным load barrier:
Object obj = loadBarrier(someField);
// loadBarrier проверяет цвет указателя:
//   - Если remapped: вернуть как есть
//   - Если not-remapped: объект был перемещён
//     → найти forwarding address
//     → обновить ссылку в someField (self-healing)
//     → вернуть новый указатель (remapped)
```

**Почему это работает быстро:**
- Цвет проверяется по старшим битам — одна битовая операция
- При попадании в кэш (обычный случай) — накладные расходы минимальны
- Self-healing: ссылка чинится при первом чтении → следующие чтения не платят штраф

##### 3. Региональная архитектура

Как и G1, ZGC делит кучу на регионы, но трёх размеров:

| Тип региона | Размер | Назначение |
|---|---|---|
| **Small** | 2MB | Объекты < 256KB |
| **Medium** | 32MB | Объекты 256KB–4MB |
| **Large** | N × 2MB | Объекты > 4MB |

#### Multi-Mapping (многоуровневое отображение)

ZGC использует трюк с виртуальной памятью: одни и те же физические страницы отображаются в несколько виртуальных диапазонов — по одному на каждый цвет указателя.

```
Физическая память:     [████████████████████████████]
                              │
                              │ mmap (MAP_SHARED)
                              │
Виртуальные диапазоны:  ┌─────────────────────────────┐ remapped view
                        │ [████████████████████████████]│
                        ├─────────────────────────────┤ marked0 view
                        │ [████████████████████████████]│
                        ├─────────────────────────────┤ marked1 view
                        │ [████████████████████████████]│
                        └─────────────────────────────┘

Цвет указателя = смещение виртуального диапазона.
Чтение через любой цвет попадает в ту же физическую память.
```

Это изящное решение: ZGC может определить «цвет» указателя, просто посмотрев на его старшие биты (они же определяют виртуальный диапазон).

#### Generational ZGC (Java 21+)

Начиная с Java 21 (production-ready: Java 23, по умолчанию), ZGC поддерживает **generational mode**:

```
Generational ZGC:
┌─────────────────────────────────────────┐
│               HEAP                      │
│  ┌─────────────────┐ ┌────────────────┐ │
│  │  Young Regions  │ │  Old Regions   │ │
│  │                 │ │                │ │
│  │  ┌──┐┌──┐┌──┐  │ │  ┌──┐┌──┐┌──┐ │ │
│  │  │E ││E ││S │   │ │  │O ││O ││O │  │ │
│  │  └──┘└──┘└──┘  │ │  └──┘└──┘└──┘ │ │
│  └─────────────────┘ └────────────────┘ │
└─────────────────────────────────────────┘

Преимущества поколенческого режима:
- Меньше объектов в Old → меньше marking
- Меньше накладных расходов на load barriers
- Выше пропускная способность (~10-20%)
- Паузы всё ещё <1ms
```

**Флаги Generational ZGC:**
| Флаг | Описание |
|---|---|
| `-XX:+UseZGC` | Включить ZGC |
| `-XX:+ZGenerational` | Включить поколенческий режим (default с Java 23) |
| `-XX:ZCollectionInterval=N` | Максимальный интервал между GC (секунды) |
| `-XX:ZAllocationSpikeTolerance=N` | Толерантность к всплескам аллокаций |

#### Преимущества ZGC

- **Паузы <1ms** независимо от размера кучи — идеал для latency-critical приложений
- **Компактификация** — нет фрагментации
- **Масштабируется до 16TB** — без деградации времени паузы
- **Self-healing** через load barriers
- **Generational mode** — близкая к G1 пропускная способность при сверхнизких паузах
- **Нет поколений в non-generational** — для приложений где гипотеза поколений не работает

#### Недостатки ZGC

- **Требует 64-битной платформы** (colored pointers не работают на 32-bit)
- **Load barriers снижают throughput** на 5-10% по сравнению с G1
- **Потребляет больше CPU** в конкурентных фазах
- **Не поддерживает Compressed OOPs** (цветные указатели занимают старшие биты)
- **Меньше production-опыта**, чем у G1 (но быстро набирает)
- **Multi-mapping** требует специфичной поддержки ОС (Linux, Windows — ок; macOS — ограничения)

### 3.6. Shenandoah GC

> **Shenandoah** — конкурентный сборщик с компактификацией от Red Hat. Добавлен в JDK 12. Паузы не зависят от размера кучи.

#### Brooks Pointers — альтернатива Colored Pointers

В отличие от ZGC, Shenandoah не использует colored pointers. Каждый объект имеет **дополнительное слово в заголовке — Brooks pointer**:

```
Обычный объект (HotSpot):       Объект с Brooks Pointer (Shenandoah):

┌──────────────────────┐        ┌──────────────────────────────┐
│     Mark Word        │        │         Mark Word            │
├──────────────────────┤        ├──────────────────────────────┤
│     Klass Pointer    │        │         Klass Pointer        │
├──────────────────────┤        ├──────────────────────────────┤
│     Поля объекта     │        │      Brooks Pointer ─────────┼──┐
│                      │        ├──────────────────────────────┤  │
│                      │        │      Поля объекта            │  │
│                      │        │                              │  │
└──────────────────────┘        └──────────────────────────────┘  │
                                                                    │
     Все внешние ссылки указывают на Brooks Pointer ────────────────┘
     Brooks Pointer → указывает на ФАКТИЧЕСКОЕ положение данных объекта
```

**Как работает перемещение:**
1. Объект O копируется из региона A в регион B
2. **Brooks Pointer** объекта O в регионе A обновляется: теперь указывает на данные в регионе B
3. Внешние ссылки всё ещё указывают на Brooks Pointer в регионе A — они продолжают работать!
4. Во время Concurrent Update Refs все внешние ссылки обновляются на Brooks Pointer в регионе B
5. Регион A освобождается

#### Цикл Shenandoah — полный разбор

│                         │                                        │
│                         ▼                                        │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  2. Concurrent Mark         [CONC]                        │  │
│  │     - Обход графа объектов                               │  │
│  │     - Write Barrier: отслеживание изменений              │  │
│  │     - Приложение продолжает работать                      │  │

#### Сравнение Brooks Pointers vs Colored Pointers

| Аспект | Shenandoah (Brooks) | ZGC (Colored) |
|---|---|---|
| **Где хранятся метаданные** | В заголовке каждого объекта (+1 слово) | В старших битах указателя |
| **Дополнительная память** | Да, 8 байт на объект | Нет (но +18 бит неиспользуемых) |
| **Дополнительное разыменование** | Да, всегда через Brooks Pointer | Только при промахе (load barrier) |
| **32-битная поддержка** | Да | Нет |
| **Работает ли без 64-bit** | Да | Нет (нужны биты для цвета) |
| **Простота портирования** | Выше (меньше требований к ОС) | Ниже (multi-mapping) |

#### Generational Shenandoah (JEP 404, Java 21+; default с Java 24)

Режим поколений для Shenandoah был введён в Java 21 (JEP 404). С Java 24 поколенческий режим включён по умолчанию. Принцип аналогичен G1/ZGC: отдельные поколения для молодых и старых объектов.

**Флаги:**
| Флаг | Описание |
|---|---|
| `-XX:+UseShenandoahGC` | Включить Shenandoah |
| `-XX:ShenandoahGCHeuristics` | Режим: `static`, `compact`, `adaptive`, `aggressive` |
| `-XX:ShenandoahPacing` | Управление темпом аллокаций (по умолчанию on) |

#### Преимущества Shenandoah

- **Конкурентная эвакуация** — перемещение объектов без остановки приложения
- **Паузы <10ms** независимо от размера кучи
- **Работает на 32-битных платформах** (в отличие от ZGC)
- **Меньше требований к ОС** (нет multi-mapping)
- **Generational mode** (Java 24+) — улучшенный throughput

#### Недостатки Shenandoah

- **Brooks Pointer** — +8 байт к заголовку каждого объекта
- **Двойное разыменование** (Brooks Pointer → данные) добавляет накладные расходы
- **Concurrent Update Refs** — дополнительная фаза, которой нет в ZGC
- **Меньше распространён**, чем G1 и ZGC — меньше production-опыта
- **Не во всех сборках JDK доступен** (Oracle JDK не включает; OpenJDK, Amazon Corretto, Red Hat — включают)

### 3.7. Epsilon GC — No-Op

> Добавлен в Java 11. Сборщик, который **ничего не собирает**.

**Алгоритм:**
- Объекты аллоцируются (TLAB + Heap)
- Когда куча заполнена → **просто выбрасывает OutOfMemoryError**
- Никакого GC не происходит вообще

**Преимущества:**
- Нулевые накладные расходы на GC
- Детерминированная производительность
- Идеален для:
  - Бенчмарков (измерение чистой производительности кода без GC)
  - Короткоживущих приложений (время жизни < заполнения кучи)
  - Тестирования поведения при OOM
  - Приложений, управляющих памятью вручную (off-heap)

**Недостатки:**
- Не собирает мусор → OOM как только куча заполнена
- Нужно точно знать максимальный объём аллокаций

**Флаг:** `-XX:+UseEpsilonGC` (JDK 11-14 требует также `-XX:+UnlockExperimentalVMOptions`, с JDK 15 — production-фича)

| **Идеальный сценарий** | Маленькие приложения, 1 CPU | Batch, throughput | НЕ ИСПОЛЬЗОВАТЬ | Веб-сервисы, микросервисы | Latency-critical, большие кучи | Latency-critical, 32-bit, кросс-платформа |

## 4. Выбор сборщика мусора

### 4.1. Древо принятия решений

```
  СТАРТ: выбор GC
      │
      ├── Heap < 200MB или 1 CPU?
      │     └── ДА → Serial GC
      │
      ├── Требуется максимальный THROUGHPUT, паузы не критичны?
      │     └── ДА → Parallel GC
      │           │
      │           └── Heap > 4GB → всё равно Parallel, но следи за паузами
      │
      ├── Требуются НИЗКИЕ ПАУЗЫ (< 200ms)?
      │     └── ДА → G1 GC (default)
      │           │
      │           ├── Укладываешься в целевую паузу?
      │           │     └── ДА → Оставайся на G1
      │           │
      │           └── НЕТ → Пробуй ZGC или Shenandoah
      │
      ├── Требуются СВЕРХНИЗКИЕ ПАУЗЫ (< 10ms)?
      │     └── ДА → ZGC или Shenandoah
      │           │
      │           ├── Нужна поддержка 32-bit?
      │           │     └── ДА → Shenandoah
      │           │
      │           ├── Нужна поддержка Oracle JDK?
      │           │     └── ДА → ZGC
      │           │
      │           └── Максимальная производительность?
      │                 └── ZGC (generational)
      │
      └── Heap > 64GB?
            └── ДА → ZGC или Shenandoah
                  (G1 может не укладываться в паузы)
```

### 4.2. Рекомендации по типам приложений

| Тип приложения | Рекомендуемый GC | Почему |
|---|---|---|
| **CLI-утилиты, одноразовые задачи** | Serial | Минимум накладных расходов, быстрое время старта |
| **Batch-обработка (ETL, map-reduce)** | Parallel | Максимальный throughput, паузы не важны |
| **Веб-сервер / REST API** | G1 | Хороший баланс throughput и пауз |
| **Микросервисы (Spring Boot, Quarkus)** | G1 (по умолчанию) | Предсказуемые паузы, стабильность |
| **Платёжные системы, трейдинг** | ZGC | Паузы <1ms критичны для SLA |
| **Игровые серверы, real-time** | ZGC / Shenandoah | Минимальные фризы |
| **Big Data (Apache Spark, Kafka Streams)** | G1 / Parallel | Зависит от latency-требований; часто Parallel для worker-нод |
| **Эмбеддед / IoT** | Serial / Shenandoah | Ограниченные ресурсы; Shenandoah если нужна компактификация |
| **Разработка / тестирование** | G1 или Serial | G1 для интеграционных тестов, Serial для unit |

## Заключение

Выбор GC — это всегда компромисс между **throughput** (пропускная способность), **latency** (паузы) и **memory footprint** (потребление памяти). Ключевые принципы:

1. **Начни с G1** — это default и он покрывает 90% сценариев
2. **Не тюнь без метрик** — сначала собери GC-логи и измерь реальные паузы
3. **Следи за Metaspace** — неограниченный по умолчанию рост может убить процесс
4. **В контейнерах:** `-Xmx` должен быть на 20-30% меньше лимита контейнера (помни про native-память!)
5. **Думай о аллокациях в коде** — лучшая сборка мусора та, которая не запускается. Избегай излишнего создания объектов в горячих путях.
