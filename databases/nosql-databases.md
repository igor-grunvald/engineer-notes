## Введение

**NoSQL** (Not Only SQL) — это класс систем управления базами данных, которые отходят от классической реляционной модели. Термин появился в 1998 году (Карло Строцци, легковесная БД без SQL-интерфейса), но возродился в 2009 году на конференции, посвящённой распределённым нереляционным хранилищам — силами Йохана Оскарссона (Last.fm) и Эрика Эванса (Rackspace).

Ключевая идея NoSQL — **жертвовать одним или несколькими свойствами ACID ради горизонтальной масштабируемости, гибкости схемы и специфической оптимизации под определённый тип нагрузки.**

В отличие от реляционных БД, где данные всегда раскладываются в таблицы со строгой схемой, NoSQL-системы предлагают разнообразие моделей данных, каждая из которых оптимизирована под свой класс задач.

## Часть I. Фундаментальные концепции

### 1.1. Что такое NoSQL и почему он появился

**Исторический контекст.** В начале 2000-х интернет-гиганты (Google, Amazon, Facebook) столкнулись с ограничениями реляционных БД при работе с web-scale нагрузками:

- **Горизонтальное масштабирование** в РСУБД сложно: репликация master-slave упирается в один мастер на запись, шардирование требует ручного управления и ломает SQL-запросы.
- **Гибкость схемы:** в веб-приложениях структура данных меняется часто, ALTER TABLE на таблицах в сотни гигабайт — операция, блокирующая продакшен.
- **Производительность:** JOIN'ы на шардированных данных невозможны, денормализация в реляционной модели неудобна.
- **Специфические нагрузки:** полнотекстовый поиск, работа с графами, временные ряды — всё это требует специализированных структур, которые B-Tree не даёт.

Ответом стали:
- **Google Bigtable** (2006) — колоночное хранение для web-scale.
- **Amazon Dynamo** (2007) — distributed key-value store, пожертвовавший консистентностью ради доступности.
- **MongoDB** (2009) — документная БД с JSON-подобной моделью.
- **Cassandra** (2008, Facebook, позже Apache) — гибрид Bigtable и Dynamo.
- **Redis** (2009, Salvatore Sanfilippo) — in-memory key-value store.

### 1.2. Теорема CAP

Теорема CAP (Эрик Брюэр, 2000; формально доказана Гилбертом и Линчем в 2002) утверждает, что **в распределённой системе при сетевом разделении (Partition) можно гарантировать только одно из двух: Consistency (согласованность) или Availability (доступность).**

```
                        ┌──────────────┐
                        │     CAP      │
                        │              │
              ┌─────────┴───┐  ┌───────┴─────────┐
              │ Consistency │  │  Availability    │
              │ (CP)        │  │  (AP)            │
              └──────┬──────┘  └──────┬──────────┘
                     │                │
              ┌──────┴────────────────┴──────┐
              │   НЕЛЬЗЯ выбрать все три     │
              │   при наличии Partition      │
              └──────────────────────────────┘
```

**Определения:**

| Свойство | Определение |
|----------|-------------|
| **Consistency** | Все узлы видят одни и те же данные в один и тот же момент времени. После записи любой последующий read возвращает записанное значение (линеаризуемость). |
| **Availability** | Каждый запрос к работающему узлу завершается корректным ответом (без ошибки). Система остаётся доступной для операций, даже если часть узлов отказала. |
| **Partition Tolerance** | Система продолжает работать, несмотря на потерю сетевой связности между узлами (произвольное количество сообщений между узлами может быть потеряно или задержано). |

**Критически важный нюанс:** CAP — это не «выбери два из трёх». В распределённой системе Partition Tolerance **нельзя выбрать** — сетевые разделения неизбежны (сеть ненадёжна по определению). Реальный выбор — между CP и AP в момент разделения:
- **CP (Consistency + Partition Tolerance):** при разделении система жертвует доступностью — часть узлов отказывается отвечать, чтобы не нарушить консистентность. Пример: MongoDB (в конфигурации с большинством), HBase, Zookeeper.
- **AP (Availability + Partition Tolerance):** при разделении система продолжает отвечать на всех узлах, но данные могут быть несогласованными (stale reads). Пример: Cassandra (с eventual consistency), DynamoDB (по умолчанию), CouchDB.
- **CA (Consistency + Availability):** возможно только в системе **без** разделения — по сути, классическая монолитная РСУБД.

**Классификация NoSQL по CAP:**

| База данных | Класс CAP | Комментарий |
|-------------|-----------|-------------|
| MongoDB | CP (настраивается) | По умолчанию CP: при потере большинства узлов replica set перестаёт принимать записи. Можно ослабить read concern. |
| Cassandra | AP (настраивается) | По умолчанию AP, но уровень консистентности настраивается на каждую операцию (Tunable Consistency). |
| Redis (standalone) | CA | Вне распределённого контекста. |
| Redis Cluster | CP (условно) | При потере большинства мастеров кластер перестаёт работать. |
| Neo4j (standalone) | CA | Один узел — нет Partition. |
| Neo4j (Aura/Fabric) | CP | Федерация графов жертвует доступностью части данных при разделении. |
| Elasticsearch | AP (по умолчанию) | Можно настроить на CP, но ценой доступности (wait_for_active_shards: all). |
| HBase | CP | Сильно консистентна, при отказе RegionServer данные недоступны до восстановления. |

### 1.3. BASE — антипод ACID

Если ACID — это гарантии реляционного мира (Atomicity, Consistency, Isolation, Durability), то BASE описывает философию NoSQL:

| Буква | Значение | Описание |
|-------|----------|----------|
| **B** | **Basically Available** (базовая доступность) | Система гарантирует доступность в смысле CAP: каждый запрос получает ответ (возможно, не самые свежие данные). Данные доступны всегда, даже ценой несогласованности. |
| **S** | **Soft State** (неустойчивое состояние) | Состояние системы может меняться со временем, даже без поступления новых данных — из-за механизмов eventual consistency. База не обязана быть в непротиворечивом состоянии в каждый момент времени. |
| **E** | **Eventually Consistent** (согласованность в конечном счёте) | Если в систему перестанут поступать новые данные, через некоторое время все узлы сойдутся к одному состоянию. Нет гарантии, что read сразу после write увидит записанное. |

**Глубинная разница ACID vs BASE:**

```
┌─────────────────────────────────────────────────────────────────────┐
│     ACID                          │          BASE                   │
├───────────────────────────────────┼──────────────────────────────────┤
│ Сильная консистентность           │ Eventual consistency            │
│ (read-after-write гарантирован)   │ (stale reads возможны)          │
├───────────────────────────────────┼──────────────────────────────────┤
│ Фокус на отсутствии противоречий  │ Фокус на доступности            │
│ (commit или rollback)             │ (данные можно прочитать всегда) │
├───────────────────────────────────┼──────────────────────────────────┤
│ Вертикальное масштабирование      │ Горизонтальное масштабирование   │
│ (быстрее железо)                  │ (больше узлов)                  │
├───────────────────────────────────┼──────────────────────────────────┤
│ Сложные транзакции — норма        │ Транзакции — дорого и редко      │
├───────────────────────────────────┼──────────────────────────────────┤
│ Схема first, данные second        │ Данные first, схема second       │
├───────────────────────────────────┼──────────────────────────────────┤
│ Join'ы, foreign keys, constraints │ Денормализация, вложенные        │
│                                   │ документы, нет FK               │
├───────────────────────────────────┼──────────────────────────────────┤
│ Примеры: PostgreSQL, Oracle,      │ Примеры: Cassandra, DynamoDB,    │
│ MySQL (InnoDB)                    │ CouchDB, Riak                   │
└─────────────────────────────────────────────────────────────────────┘
```

**Почему eventual consistency — это не баг, а фича:** в высоконагруженных системах (социальные сети, ленты новостей, счётчики лайков) нет бизнес-требования видеть актуальные данные немедленно. Если лайк под постом отображается через 2 секунды — это приемлемо. Но система выдерживает нагрузку в миллионы операций в секунду, которую реляционная БД не потянет.

**Переходные состояния в BASE.** Представим, что пользователь обновил пост в распределённой БД:
1. `t = 0`: write на узел A, узел A подтверждает успех.
2. `t = 0.1`: read с узла B — возвращает старую версию (soft state).
3. `t = 0.5`: механизм anti-entropy или gossip синхронизирует B с A.
4. `t = 0.5`: read с узла B — возвращает новую версию (eventual consistency достигнута).

Этот интервал (0.1–0.5 с) — окно неконсистентности (inconsistency window). Задача инженера — сделать его приемлемым.

### 1.4. PACELC: расширение CAP

Теорема PACELC (Daniel Abadi, 2010) добавляет важный нюанс: **даже при отсутствии Partition (нормальная работа сети), система выбирает между Latency и Consistency.**

```
CAP:  если Partition (P) → выбор между A и C
ELSE: если нет Partition   → выбор между L (Latency) и C (Consistency)

                                             ┌──────────────────┐
                    ┌────────────────────────┤ Latency (скорость)│
                    │                        └──────────────────┘
    ┌─── ЕСТЬ Partition? ───┤
    │                        │                        ┌───────────────────────┐
    │                        └────────────────────────┤ Consistency (мгновен.)│
    │                                                 └───────────────────────┘
    │
    │                        ┌──────────────────┐
    └─── НЕТ Partition? ─────┤ Availability      │  ...тут вопрос не стоит,
                             └──────────────────┘  система доступна по определению
```

**Классификация по PACELC:**

| Тип | Примеры | При разделении | Без разделения |
|-----|---------|---------------|----------------|
| **PC/EC** | Bigtable, HBase, MongoDB (большинство) | Жертвуют A ради C | Жертвуют L ради C |
| **PA/EC** | Cassandra | Жертвуют C ради A | Жертвуют L ради C (но можно настроить) |
| **PA/EL** | DynamoDB (дефолт), Riak | Жертвуют C ради A | Жертвуют C ради L |

Это объясняет, почему MongoDB при нормальной работе всё равно имеет задержки: каждый read должен подтверждаться большинством (read concern: majority), что добавляет network round-trip.

### 1.5. Модели согласованности (Consistency Models)

Модель согласованности определяет, что видит клиент при чтении данных после записи. Это спектр от «вообще никаких гарантий» до «абсолютная согласованность».

```
 СЛАБЫЕ ГАРАНТИИ                             СИЛЬНЫЕ ГАРАНТИИ
 ◄─────────────────────────────────────────────────────────────────►
 │                                                                  │
 Eventual    │  Causal      │  Monotonic   │  Linearizable  │ Strict
 Consistency │ Consistency  │ Read/Writes  │                │ Serial.
             │              │              │                │
```

#### 1.5.1. Eventual Consistency (Согласованность в конечном счёте)

Самая слабая модель. После прекращения записи, через некоторое время все реплики сойдутся к одному состоянию. Гарантий порядка или времени нет.

```
Write(A=1) ──→ Replica1 [A=1]        ...ждём...
               Replica2 [A=0]  ──→   Replica2 [A=1]
               Replica3 [A=0]  ──→   Replica3 [A=1]

Время inconsistencies: от миллисекунд до минут.
```

**Проблема:** если клиент пишет в Replica1, а читает с Replica2 сразу после — он может увидеть старое значение. Это называют **stale read**.

#### 1.5.2. Causal Consistency (Причинно-следственная)

Гарантирует, что операции, связанные причинно-следственной связью, видны в правильном порядке.

```
Клиент A: 1) write(x=1)  2) write(x=2)     ← причинно-связанные операции
Клиент B: read(x)                          ← если видит x=2, то обязан увидеть и x=1

НО: операции, не связанные причинностью, могут быть видны в любом порядке.
```

Реализуется через **векторные часы** (version vectors). Используется в DynamoDB (опционально).

#### 1.5.3. Monotonic Reads

Гарантирует, что клиент не увидит более старые данные после того, как прочитал более новые.

```
Без monotonic reads:          С monotonic reads:
read → x=42                   read → x=42
read → x=17  ← ОТКАТ!         read → x=42 (или новее)
```

#### 1.5.4. Monotonic Writes (Write-Follows-Read)

Запись не завершится, пока все предыдущие чтения не будут учтены. Как если бы запись «ждала» завершения всех видимых клиентом состояний.

#### 1.5.5. Read-Your-Own-Writes (RYOW)

Клиент всегда видит свои собственные записи. Критически важно для UX: пользователь обновил профиль → нажал F5 → должен увидеть обновлённый профиль. Без RYOW он может увидеть старую версию и подумать, что его изменения потеряны.

#### 1.5.6. Linearizability (Линеаризуемость)

Сильнейшая модель: все операции выглядят так, как будто они выполнились атомарно в некоторый момент времени между их началом и завершением. **Каждая операция видна всем клиентам в точке её «мгновенного» выполнения.**

```
Шкала времени: ────────────────────────────────────────────────────
write(x=1):     [.......запись.......]
                        ↑ точка линеаризации
read(x):                                  [..чтение..]
                                           → должен вернуть 1
```

Это то, что в CAP называют Consistency. В реальности дорогая гарантия: требует консенсуса (Paxos/Raft) или записи/чтения с большинства.

#### 1.5.7. Strict Serializability

Линеаризуемость + транзакционная изоляция уровня Serializable. Каждая транзакция выглядит как выполненная в некоторый момент времени, и порядок транзакций соответствует реальному времени. Практически недостижима в распределённых NoSQL без колоссальных накладных расходов.

### 1.6. Таксономия NoSQL: четыре типа хранилищ

Классическая классификация (по модели данных) выделяет четыре основных типа:

## Часть II. Key-Value хранилища

### 2.1. Принцип работы и внутреннее устройство

Key-Value (KV) — простейшая модель NoSQL. База данных ведёт себя как распределённая хеш-таблица (Distributed Hash Table, DHT): каждому ключу соответствует значение, которое трактуется как непрозрачный (opaque) набор байтов. База не знает о структуре значения.

**Анатомия KV-запроса:**

```
Client ──→ SET user:1001 {"name":"Ivan","email":"ivan@example.com"}
Client ──→ GET user:1001
Client ──→ DEL user:1001
```

**Внутреннее устройство (обобщённо):**

```
┌──────────────────────────────────────────────┐
│              Key-Value Store                  │
│                                               │
│  ┌──────────────────────────────────────┐     │
│  │          Hash Function               │     │
│  │   key ──→ hash(key) ──→ bucket       │     │
│  └──────────┬───────────────────────────┘     │
│             │                                  │
│  ┌──────────▼───────────────────────────┐     │
│  │          Storage Engine              │     │
│  │  ┌─────────────┐  ┌───────────────┐  │     │
│  │  │ In-Memory   │  │  On-Disk      │  │     │
│  │  │ Hash Table  │  │  (LSM / B-Tree│  │     │
│  │  │ + SkipList  │  │   + WAL)      │  │     │
│  │  └─────────────┘  └───────────────┘  │     │
│  └──────────────────────────────────────┘     │
└──────────────────────────────────────────────┘
```

- **Хеш-функция:** Равномерно распределяет ключи по «корзинам» (buckets). В распределённых KV (Redis Cluster, DynamoDB) хеш определяет, на каком узле лежит ключ.
- **Storage Engine:** Может быть чисто in-memory (Redis по умолчанию) или с персистентностью на диске через LSM-деревья (RocksDB в TiKV) или B-Tree (LMDB в некоторых KV).
- **WAL (Write-Ahead Log):** Перед записью в основное хранилище данные пишутся в лог для durability.

**Когда KV — правильный выбор:**
- Данные всегда запрашиваются по первичному ключу (нет поиска по полям).
- Нужна экстремальная производительность: миллионы операций в секунду.
- Значения могут быть большими и неструктурированными (blob, картинки).
- Схема данных минимальна или отсутствует.

### 2.2. Redis: архитектура, структуры данных и персистентность

**Redis (Remote Dictionary Server)** — самая популярная KV-база. Написана на C, однопоточная (в основном), работает в оперативной памяти с опциональной персистентностью на диск.

#### 2.2.1. Однопоточность Redis — почему это быстро?

Redis использует **один поток** для обработки команд. Это устраняет накладные расходы на блокировки и переключение контекста. Производительность достигается за счёт:

1. **In-memory:** все данные в RAM — нет случайных дисковых чтений.
2. **I/O Multiplexing:** один поток обслуживает тысячи соединений через epoll/kqueue — неблокирующий ввод-вывод.
3. **Эффективные структуры данных:** SkipList, ZipList, IntSet — каждая реализация оптимизирована под конкретный сценарий.
4. **Минимальный парсинг:** протокол RESP (REdis Serialization Protocol) текстовый, но предельно простой.

```
┌──────────────────────────────────────────────┐
│                 Redis Server                  │
│                                               │
│  ┌──────────────────────────────────────┐     │
│  │          Event Loop (epoll)          │     │
│  │  ┌──────────────────────────────┐    │     │
│  │  │  Command Queue (FIFO)        │    │     │
│  │  │  SET key val ← GET key ← ... │    │     │
│  │  └──────────┬───────────────────┘    │     │
│  │             ▼                        │     │
│  │  ┌──────────────────────────────┐    │     │
│  │  │  Single Thread Execution     │    │     │
│  │  │  processCommand() → result   │    │     │
│  │  └──────────┬───────────────────┘    │     │
│  │             ▼                        │     │
│  │  ┌──────────────────────────────┐    │     │
│  │  │  Response Queue              │    │     │
│  │  └──────────────────────────────┘    │     │
│  └──────────────────────────────────────┘     │
│                                               │
│  ┌─────────────┐  ┌──────────────┐            │
│  │  Main Dict  │  │   Expires    │            │
│  │  (hashtable)│  │   Dict       │            │
│  └─────────────┘  └──────────────┘            │
│                                               │
│  + Background Threads (Redis 6+):              │
│  - BIO (Background I/O): close fd, fsync      │
│  - Lazy free: async delete large objects       │
└──────────────────────────────────────────────┘
```

#### 2.2.2. Структуры данных Redis

Redis — это не просто `GET`/`SET`. Это **data structure server**: значения могут быть разными структурами, и для каждой есть атомарные операции.

| Структура | Назначение | Примеры команд | Внутреннее представление |
|-----------|-----------|---------------|--------------------------|
| **String** | Простое значение, счётчик, битовый массив | `SET`, `GET`, `INCR`, `SETBIT` | SDS (Simple Dynamic String) |
| **List** | Очередь, стек, хронология | `LPUSH`, `RPOP`, `LRANGE` | QuickList (гибрид linked list + ziplist) |
| **Set** | Множество уникальных элементов | `SADD`, `SINTER`, `SUNION` | IntSet или Hash Table |
| **Sorted Set** | Рейтинг, очередь с приоритетом | `ZADD`, `ZRANGEBYSCORE` | SkipList + Hash Table |
| **Hash** | Объект с полями | `HSET`, `HGET`, `HGETALL` | ZipList или Hash Table |
| **HyperLogLog** | Приблизительный подсчёт уникальных элементов | `PFADD`, `PFCOUNT` | HyperLogLog (12 КБ на ключ, ошибка 0.81%) |
| **Bitmap** | Битовые операции, bloom-фильтры | `SETBIT`, `BITCOUNT`, `BITOP` | Строка + битовые операции |
| **Geospatial** | Гео-индексы | `GEOADD`, `GEORADIUS` | Sorted Set (Z-order curve / Geohash) |
| **Stream** | Event sourcing, очереди сообщений | `XADD`, `XRANGE`, `XREADGROUP` | Radix Tree (Rax) |

**Как Redis хранит структуры внутри:**

```
Глобальная хеш-таблица сервера:

┌─────────────────────────────────────────────────────┐
│  dict[0] → основной словарь                         │
│  ┌─────────────────────────────────────────────────┐│
│  │ bucket[0] → [key1 → redisObject]                ││
│  │ bucket[1] → NULL                                ││
│  │ bucket[2] → [key2 → redisObject] → [key3 → ...] ││
│  │ ...                                             ││
│  │ bucket[N] → ...                                 ││
│  └─────────────────────────────────────────────────┘│
│                                                     │
│  redisObject:                                       │
│  ┌─────────────────────────────────────────┐        │
│  │ type: STRING | LIST | SET | ZSET | HASH │        │
│  │ encoding: INT | EMBSTR | RAW | ZIPLIST  │        │
│  │            | SKIPLIST | INTSET | ...    │        │
│  │ lru: 12345    (LRU clock)              │        │
│  │ refcount: 1                             │        │
│  │ *ptr → внутреннее представление          │        │
│  └─────────────────────────────────────────┘        │
└─────────────────────────────────────────────────────┘
```

#### 2.2.3. Персистентность: RDB и AOF

Redis — in-memory БД, но без персистентности данные потеряются при перезапуске. Два основных механизма:

**RDB (Redis Database Backup)** — снапшот состояния памяти на диск.

```
Алгоритм:
1. Redis форкается (fork()).
2. Дочерний процесс обходит все данные (используя Copy-On-Write память родителя).
3. Пишет бинарный дамп на диск.
4. Родитель продолжает обслуживать запросы.

Преимущества:
- Минимальное влияние на производительность (только fork).
- Компактный файл (сжатие LZF опционально).
- Быстрое восстановление.

Недостатки:
- Данные между снапшотами теряются.
- fork() при большом объёме памяти может быть медленным (COW page tables).
```

**AOF (Append-Only File)** — журнал всех операций записи.

```
Алгоритм:
1. Каждая команда, модифицирующая данные, дописывается в AOF-файл.
2. При перезапуске Redis проигрывает AOF для восстановления состояния.

Режимы fsync:
- always: fsync после каждой команды → самый надёжный, медленный.
- everysec: fsync раз в секунду → баланс (потеря до 1 секунды данных).
- no: отдаём fsync на откуп ОС → быстро, но ненадёжно.

AOF rewrite:
- AOF растёт бесконечно → периодически перезаписывается (компактизация).
- BGREWRITEAOF форкает дочерний процесс, который пишет компактную версию.

RDB + AOF гибрид (Redis 4+):
- AOF файл начинается с RDB-снапшота, затем команды — ускорение восстановления.
```

### 2.3. Redis Cluster и шардирование

Redis Cluster — распределённая версия Redis с автоматическим шардированием и failover'ом. **Не использует consistent hashing** — использует **hash slots**.

```
Hash Slots: 16384 слота (0..16383).

Хеширование ключа:
  slot = CRC16(key) % 16384

Каждый мастер-узел владеет диапазоном слотов:
  Node A: слоты 0..5460
  Node B: слоты 5461..10922
  Node C: слоты 10923..16383
```

**Архитектура Redis Cluster:**

```
┌────────────────────────────────────────────────────────────────┐
│                     Redis Cluster                               │
│                                                                 │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐                    │
│  │ Master A │   │ Master B │   │ Master C │                    │
│  │ slots:   │   │ slots:   │   │ slots:   │                    │
│  │ 0..5460  │   │ 5461..   │   │ 10923..  │                    │
│  └────┬─────┘   └────┬─────┘   └────┬─────┘                    │
│       │              │              │                           │
│  ┌────▼─────┐   ┌────▼─────┐   ┌────▼─────┐                    │
│  │ Slave A1 │   │ Slave B1 │   │ Slave C1 │                    │
│  │ (replica)│   │ (replica)│   │ (replica)│                    │
│  └──────────┘   └──────────┘   └──────────┘                    │
│                                                                 │
│  Все узлы общаются через gossip-подобный протокол              │
│  (Redis Cluster Bus, порт 17000, бинарный протокол)            │
└────────────────────────────────────────────────────────────────┘
```

**Ключевые особенности Redis Cluster:**
- **Hash tags:** если ключ содержит `{...}`, хешируется только содержимое скобок — для colocation связанных ключей. `user:{1001}:name` и `user:{1001}:email` попадут в один слот.
- **MOVED redirect:** клиент спрашивает узел A → узел А знает, что ключ лежит на B → возвращает `MOVED <slot> <IP:port>` → клиент переподключается.
- **ASK redirect:** при миграции слотов: `ASK <slot> <IP:port>` — временный редирект на один запрос.
- **Умные клиенты:** клиенты (Jedis, Lettuce, redis-py-cluster) кэшируют карту слотов и сами направляют запросы к нужному мастеру.

**Проблемы Redis Cluster:**
- Multi-key операции требуют, чтобы все ключи были в одном слоте (`hash tag`).
- Транзакции (`MULTI/EXEC`) не работают кросс-шардово (надо использовать Lua в пределах одного слота).
- Потоковые структуры (Streams) ограничены одним слотом.

### 2.4. Redis Sentinel: высокая доступность

Для standalone Redis (не Cluster) используется Sentinel — система мониторинга и автоматического failover'а.

```
┌──────────────────────────────────────────────────┐
│                Redis + Sentinel                   │
│                                                   │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐         │
│  │Sentinel 1│  │Sentinel 2│  │Sentinel 3│        │
│  │(quorum)  │  │(quorum)  │  │(quorum)  │        │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘        │
│       │             │             │               │
│       └─────────────┼─────────────┘               │
│                     │ Мониторят                   │
│              ┌──────▼──────┐                      │
│              │   Master    │                      │
│              └──────┬──────┘                      │
│                     │ Репликация                  │
│              ┌──────▼──────┐                      │
│              │   Slave 1   │                      │
│              └─────────────┘                      │
│              ┌─────────────┐                      │
│              │   Slave 2   │                      │
│              └─────────────┘                      │
└──────────────────────────────────────────────────┘
```

**Процесс failover'а:**

1. Sentinel'ы (не менее 3) мониторят мастер через `PING`.
2. Если `quorum` Sentinel'ов считают мастер недоступным (SDOWN → ODOWN):
   - Проводят выборы лидера среди Sentinel'ов (Raft-подобный алгоритм).
   - Лидер выбирает нового мастера среди slave'ов (по приоритету и replication offset).
   - Переконфигурирует всех, рассылая `SLAVEOF new-master`.
3. Старый мастер при возвращении становится slave'ом.

**Отличие Sentinel от Cluster:**
- Sentinel: один мастер на все данные, только репликация на чтение.
- Cluster: шардирование данных + репликация каждого шарда. Масштабируется на запись линейно.

### 2.5. Индексы в Key-Value: когда ключ — единственный индекс

В чистом KV-хранилище **ключ — это единственный индекс**. Нет возможности `WHERE age > 25` внутри значения — база просто не видит структуру значения.

**Паттерны построения вторичных индексов вручную (на примере Redis):**

```python
# Основные данные
SET user:1001 '{"name":"Ivan","age":30,"city":"Moscow"}'

# Вторичный индекс "пользователи по городу"
SADD city:Moscow user:1001
SADD city:SPb user:1002

# Запрос: все пользователи из Москвы
SMEMBERS city:Moscow  # → [1001, ...]
# Затем для каждого: GET user:<id>

# Вторичный индекс "пользователи по возрасту" через Sorted Set
ZADD age_index 30 user:1001
ZADD age_index 25 user:1002

# Запрос: пользователи старше 28
ZRANGEBYSCORE age_index 28 +inf
```

**Плюсы:** полный контроль, атомарность (при использовании Lua или MULTI/EXEC).
**Минусы:** ручное управление, дублирование данных, риск рассинхронизации индекса с данными.

**Redis Stack / RediSearch** предоставляет встроенные вторичные индексы через модули — инвертированные индексы над JSON-документами (см. Elasticsearch, Часть VI).

### 2.6. Amazon DynamoDB: внутреннее устройство и Dynamo-подобные системы

DynamoDB — полностью управляемый сервис AWS, но его архитектура (основанная на статье Dynamo, 2007) повлияла на множество open-source NoSQL: **Cassandra**, **Riak**, **Voldemort**.

#### 2.6.1. Архитектура Dynamo (2007)

```
┌──────────────────────────────────────────────────────────┐
│                   Dynamo Ring                            │
│                                                          │
│         Node A (0..49)          Node D (150..199)        │
│             ● ───────────────────── ●                    │
│            /                          \                  │
│           /                            \                 │
│     Node B ●──────────RING──────────● Node E             │
│    (50..99)                        (200..249)            │
│           \                            /                 │
│            \                          /                  │
│             ● ───────────────────── ●                    │
│         Node C (100..149)       Node F (250..299)        │
│                                                          │
│  Ключевые принципы:                                      │
│  - Consistent hashing (кольцо)                           │
│  - Virtual nodes (каждый физический узел = N виртуальных)│
│  - Replication factor = 3 (данные на coordinator + 2)    │
│  - Hinted handoff (временная передача данных)            │
│  - Gossip-протокол для membership                        │
│  - Merkle trees для anti-entropy                         │
│  - Vector clocks для версионирования                     │
└──────────────────────────────────────────────────────────┘
```

#### 2.6.2. Consistent Hashing в Dynamo

Вместо `hash(key) % N` (где при изменении N все ключи переезжают), используется кольцо:

```
  0 ─────────────────────────────────────────── 2^160
     │      │      │      │      │      │
     ●A    ●B     ●C     ●D     ●E     ●F         ← позиции узлов на кольце

Ключ key = "user:1234":
  hash("user:1234") = 95 → попадает между B и C
  → отвечает узел C (coordinator)

Репликация (RF=3): C копирует на D и E
```

При добавлении/удалении узла перераспределяется только часть данных (примерно 1/N от всех ключей), а не все, как при `hash % N`.

#### 2.6.3. DynamoDB сегодня

Современная DynamoDB — это не просто KV, а **документно-ориентированная БД с KV-корнями**:
- Поддержка вторичных индексов (GSI — Global Secondary Index, LSI — Local Secondary Index).
- Транзакции (TransactWriteItems / TransactGetItems).
- Streams (DynamoDB Streams — событийная модель).
- Адаптивная ёмкость (Adaptive Capacity).
- On-demand scaling.

## Часть III. Документные хранилища (Document Stores)

### 3.1. Модель данных: документы, коллекции, вложенные структуры

Документные БД хранят данные в виде документов — самодостаточных иерархических структур в JSON-подобном формате (BSON в MongoDB). Каждый документ может иметь произвольный набор полей, вложенные объекты и массивы.

```
Реляционная модель:                    Документная модель:

┌──────────┐  ┌──────────────┐         ┌────────────────────────────────┐
│  users   │  │   orders     │         │  {                             │
├──────────┤  ├──────────────┤         │    "_id": ObjectId("..."),     │
│ id (PK)  │  │ id (PK)      │         │    "name": "Ivan",             │
│ name     │  │ user_id (FK) │   →     │    "email": "ivan@test.com",   │
│ email    │  │ product      │         │    "orders": [                 │
└──────────┘  │ price        │         │      { "product": "Книга",     │
              └──────────────┘         │        "price": 500 },         │
                                       │      { "product": "Кофе",      │
                                       │        "price": 300 }          │
                                       │    ]                           │
                                       │  }                             │
                                       └────────────────────────────────┘
```

**Преимущества вложенной модели:**
- **Локальность данных:** все данные пользователя (профиль + заказы) лежат в одном документе → один disk seek вместо JOIN'а.
- **Отсутствие impedance mismatch:** JSON-документ напрямую сериализуется в объект языка программирования.
- **Гибкость схемы:** можно добавить поле `phone` только для одного пользователя, не меняя другие документы.

**Недостатки:**
- **Дублирование данных:** заказы внутри пользователя дублируются, если нужны отчёты по всем заказам (надо денормализовывать).
- **Размер документа:** MongoDB ограничивает один документ 16 МБ (проблема для очень больших вложенных массивов).

### 3.2. MongoDB: архитектура и внутреннее устройство

MongoDB — самая популярная документная БД. Архитектурно состоит из:

```
┌─────────────────────────────────────────────────────────────┐
│                    MongoDB Deployment                        │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │                  Replica Set                          │   │
│  │                                                       │   │
│  │  ┌──────────┐   ┌──────────┐   ┌──────────┐          │   │
│  │  │ PRIMARY  │──▶│SECONDARY │──▶│SECONDARY │          │   │
│  │  │ (writes) │   │ (reads)  │   │ (reads)  │          │   │
│  │  └──────────┘   └──────────┘   └──────────┘          │   │
│  │       │                                              │   │
│  │       │ Oplog репликация (асинхронная)               │   │
│  │       ▼                                              │   │
│  │  ┌──────────┐                                        │   │
│  │  │ ARBITER  │  (только голосует, без данных)         │   │
│  │  └──────────┘                                        │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │                 Sharded Cluster                       │   │
│  │                                                       │   │
│  │  ┌────────────┐  ┌────────────┐  ┌────────────┐       │   │
│  │  │ Config     │  │ Config     │  │ Config     │       │   │
│  │  │ Server RS  │  │ Server RS  │  │ Server RS  │       │   │
│  │  └────────────┘  └────────────┘  └────────────┘       │   │
│  │       (хранят метаданные: какие шарды, где данные)     │   │
│  │                                                       │   │
│  │  ┌────────────┐  ┌────────────┐  ┌────────────┐       │   │
│  │  │ mongos     │  │ mongos     │  │ mongos     │       │   │
│  │  │ (router)   │  │ (router)   │  │ (router)   │       │   │
│  │  └──────┬─────┘  └──────┬─────┘  └──────┬─────┘       │   │
│  │         └────────────────┼──────────────┘              │   │
│  │                          │                             │   │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐               │   │
│  │  │ Shard 1  │ │ Shard 2  │ │ Shard 3  │               │   │
│  │  │(Repl.Set)│ │(Repl.Set)│ │(Repl.Set)│               │   │
│  │  └──────────┘ └──────────┘ └──────────┘               │   │
│  └──────────────────────────────────────────────────────┘   │

```

**Процесс обработки запроса (write):**

```
1. Клиент → mongos (router). mongos парсит запрос, определяет целевой шард(ы)
   по shard key.

2. mongos → Primary нужного шарда.

3. Primary:
   a. Записывает операцию в oplog (специальная capped collection).
   b. Применяет изменение к дисковому storage engine (WiredTiger).
   c. Возвращает подтверждение клиенту.

4. Secondary'и реплицируют oplog → применяют изменения асинхронно.
```

### 3.3. WiredTiger: движок хранения MongoDB

С версии 3.2 WiredTiger — стандартный storage engine MongoDB (сменил MMAPv1). Это полноценный storage engine с поддержкой сжатия, MVCC и document-level locking.

**Архитектура WiredTiger:**

```
┌─────────────────────────────────────────────────────────┐
│                    WiredTiger                            │
│                                                          │
│  ┌──────────────────────────────────────────────────┐   │
│  │                 Cache (RAM)                       │   │
│  │  ┌──────────────────────────────────────────┐    │   │
│  │  │ B-Tree pages (leaf + internal)            │    │   │
│  │  │ - Чистые страницы (read from disk)       │    │   │
│  │  │ - Грязные страницы (modified in memory)  │    │   │
│  │  └──────────────────────────────────────────┘    │   │
│  │  Размер: wiredTigerCacheSizeGB (по умолчанию    │   │
│  │  50% RAM - 1 GB)                                  │   │
│  └──────────────────┬───────────────────────────────┘   │
│                     │ Eviction (когда кэш > 80%)        │
│                     ▼                                   │
│  ┌──────────────────────────────────────────────────┐   │
│  │              Disk (B-Tree)                        │   │
│  │                                                   │   │
│  │  /data/db/                                        │   │
│  │  ├── collection-0-*.wt   (данные коллекции)       │   │
│  │  ├── index-*.wt          (индексы)                │   │
│  │  ├── _mdb_catalog.wt     (метаданные)             │   │
│  │  ├── WiredTiger.wt       (системная таблица)      │   │
│  │  ├── WiredTiger.turtle   (последний checkpoint)   │   │
│  │  └── journal/            (WAL журнал)             │   │
│  └──────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
```

**MVCC в WiredTiger:**

WiredTiger использует MVCC (Multiversion Concurrency Control), как PostgreSQL:

```
Каждый документ имеет:
  - transaction_id: ID транзакции, записавшей версию
  - start_ts / commit_ts: временные метки

При чтении:
  - Берётся версия с commit_ts ≤ времени начала транзакции
  - Если есть незакоммиченные версии с меньшим ts — игнорируются

При записи:
  - Создаётся новая версия (не перезаписывается существующая)
  - Старая версия остаётся доступной для конкурентных читателей

Очистка:
  - Checkpoint фиксирует «самый старый активный transaction_id»
  - Всё, что старше — может быть удалено при следующем checkpoint
```

**Checkpoint в WiredTiger:**

Checkpoint — это точка согласованности на диске. WiredTiger делает checkpoint каждые 60 секунд (или по объёму журнала).

```
Процесс checkpoint'а:
1. Фиксирует «снапшот» всех грязных страниц в B-Tree на диск.
2. Создаёт новый корень B-Tree, атомарно переключает (turtle-файл).
3. Старые версии данных, не нужные для активных транзакций, освобождаются.

Журнал (WAL) между checkpoint'ами:
  - Все изменения пишутся в journal/ (WAL) синхронно (или каждые 50 мс).
  - При крахе: после восстановления checkpoint'а проигрывается журнал.
```

**Сжатие в WiredTiger:**

| Алгоритм | Назначение |
|----------|-----------|
| **snappy** | По умолчанию. Хороший баланс скорость/сжатие. |
| **zlib** | Более сильное сжатие, медленнее. |
| **zstd** | Лучшее соотношение скорость/сжатие (MongoDB 4.2+). |
| **none** | Без сжатия. |

### 3.4. Индексы в MongoDB: полный разбор

MongoDB поддерживает богатый набор индексов. **Без индексов запросы выполняются полным сканированием коллекции (COLLSCAN)**, что недопустимо на production.

#### 3.4.1. Индекс по умолчанию: `_id`

Каждый документ имеет поле `_id`, которое автоматически индексируется. По умолчанию `_id` — это ObjectId:

```
ObjectId (12 байт):
┌──────────┬─────────────────────┬──────────────┐
│ 4 байта  │       5 байт        │   3 байта    │
│ timestamp│    random value     │   counter    │
│ (epoch)  │ (3 байта machine id │ (increment)  │
│          │  + 2 байта process  │              │
│          │  id / 5 случ. байт) │              │
└──────────┴─────────────────────┴──────────────┘

Гарантирует монотонное возрастание → хорошо для B-Tree вставки.
```

#### 3.4.2. Single Field Index

```javascript
// Создание индекса по одному полю
db.users.createIndex({ name: 1 })  // 1 = ascending, -1 = descending

// Как работает внутри:
// B-Tree над значениями поля name, в листьях — указатели на документы (RecordId).
```

#### 3.4.3. Compound Index (Составной индекс)

```javascript
// Порядок полей ВАЖЕН!
db.orders.createIndex({ user_id: 1, status: 1, created_at: -1 })

// Поддерживает запросы:
// - { user_id: X }
// - { user_id: X, status: Y }
// - { user_id: X, status: Y, created_at: Z }  ← prefix rule
// - { user_id: X, created_at: Z }              ← пропуск поля (неэффективно)

// НЕ поддерживает:
// - { status: Y }                              ← не префикс!
```

**Правило префиксов (ESR rule — Equality, Sort, Range):**

```
При построении compound index порядок должен быть:
  1. Equality поля (точные совпадения) — ПЕРВЫМИ
  2. Sort поля (поля сортировки) — ВТОРЫМИ
  3. Range поля (поля диапазона $gt/$lt) — ПОСЛЕДНИМИ

Пример:
  Запрос: db.orders.find({ user_id: 42, date: { $gt: ISODate(...) } })
                            .sort({ total: -1 })

  Индекс: { user_id: 1, total: -1, date: 1 }
           └─ equality ─┘└─ sort ─┘└─ range ─┘
```

#### 3.4.4. Multikey Index

Автоматически создаётся, когда индексируется поле-массив. MongoDB создаёт отдельную запись в индексе для КАЖДОГО элемента массива.

```javascript
// Документ: { tags: ["mongodb", "database", "nosql"] }

db.articles.createIndex({ tags: 1 })

// Внутри B-Tree:
// "database" → [doc1, doc37, ...]
// "mongodb"  → [doc1, doc42, ...]
// "nosql"    → [doc1, ...]
```

**Ограничения multikey:**
- Нельзя создать compound multikey, где оба поля — массивы.
- Размер индексной записи пропорционален размеру массива.

#### 3.4.5. Text Index

Специальный индекс для полнотекстового поиска (основан на инвертированном индексе, как Elasticsearch).

```javascript
// Один text index на коллекцию (можно несколько полей)
db.articles.createIndex({ title: "text", body: "text" })

// Запрос:
db.articles.find({ $text: { $search: "mongodb indexing" } })

// Внутри:
// - Токенизация (разбиение на слова)
// - Удаление стоп-слов
// - Стемминг (для поддерживаемых языков)
// - Инвертированный индекс: слово → [документы + позиции]
// - TF-IDF для релевантности
```

#### 3.4.6. Geospatial Indexes

Для работы с геоданными (координаты, полигоны, etc.):

| Тип | Структура | Для |
|-----|-----------|-----|
| **2d** | Geohash на плоскости | Устаревший, плоские координаты |
| **2dsphere** | Сферическая геометрия (S2-ячейки) | GeoJSON, Земля, расстояния в метрах |

```javascript
db.places.createIndex({ location: "2dsphere" })

db.places.find({
  location: {
    $near: {
      $geometry: { type: "Point", coordinates: [37.6, 55.7] },
      $maxDistance: 1000  // метры
    }
  }
})
```

#### 3.4.7. Hashed Index

Хеширует значение поля и индексирует хеш. Используется для **shard key на случайных значениях** (равномерное распределение по шардам).

```javascript
db.users.createIndex({ user_id: "hashed" })
// user_id: 12345 → хеш: 0xA3F2... → B-Tree по хешам

// Поддерживает только equality-запросы (не range!)
db.users.find({ user_id: 12345 })  // OK
db.users.find({ user_id: { $gt: 12345 } })  // НЕ использует hashed index!
```

#### 3.4.8. Wildcard Index

Индексирует все поля документа (или поддерево) — для schemaless данных.

```javascript
// Индексировать ВСЕ скалярные поля документа
db.users.createIndex({ "$**": 1 })

// Индексировать все поля внутри product_attributes
db.products.createIndex({ "product_attributes.$**": 1 })
```

#### 3.4.9. TTL Index (Time-To-Live)

Автоматически удаляет документы через заданное время.

```javascript
// Документы с createdAt старше 1 часа автоматически удаляются
db.sessions.createIndex({ createdAt: 1 }, { expireAfterSeconds: 3600 })

// Механизм:
// Фоновый поток (TTL Monitor) каждые 60 секунд сканирует индекс,
// находит документы, где createdAt + expireAfterSeconds < now() и удаляет.
```

#### 3.4.10. Как MongoDB выбирает индекс?

При поступлении запроса планировщик запросов (Query Planner):

1. **Генерирует план-кандидаты** для каждого подходящего индекса (включая COLLSCAN).
2. **Выполняет "гонку" (trial run):** каждый план выполняется на небольшом числе документов — замеряется количество прочитанных документов и время.
3. **Выбирает победителя**, который затем кэшируется (plan cache) для аналогичных запросов.
4. При изменении данных (индексы перестраиваются, данные меняются) — план переоценивается.

```
Просмотр плана выполнения:

db.users.find({ name: "Ivan" }).explain("executionStats")

// Ключевые метрики:
// - stage: IXSCAN (индекс) или COLLSCAN (полный скан) — COLLSCAN = ПЛОХО
// - totalDocsExamined: сколько документов просмотрено
// - nReturned: сколько возвращено
// - executionTimeMillis: время выполнения
```

### 3.5. Шардирование в MongoDB

MongoDB поддерживает горизонтальное шардирование на уровне коллекций. Данные распределяются по шардам на основе **shard key**.

#### 3.5.1. Архитектура шардированного кластера

```
┌────────────────────────────────────────────────────────┐
│              MongoDB Sharded Cluster                    │
│                                                        │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │  mongos  │  │  mongos  │  │  mongos  │   (роутеры)  │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘              │
│       │             │             │                     │
│       └─────────────┼─────────────┘                     │
│                     │                                   │
│          ┌──────────▼──────────┐                        │
│          │  Config Servers     │                        │
│          │  (Replica Set)      │                        │
│          │                     │                        │
│          │  Хранят:             │                        │
│          │  - databases/        │                        │
│          │    collections       │                        │
│          │  - chunks            │                        │
│          │  - shard mapping     │                        │
│          └──────────┬──────────┘                        │
│                     │                                   │
│     ┌───────────────┼───────────────┐                   │
│     │               │               │                   │
│  ┌──▼──────┐   ┌────▼────┐   ┌─────▼─────┐             │
│  │ Shard 1 │   │ Shard 2 │   │ Shard 3   │             │
│  │  (RS)   │   │  (RS)   │   │  (RS)     │             │
│  └─────────┘   └─────────┘   └───────────┘             │
└────────────────────────────────────────────────────────┘
```

#### 3.5.2. Chunks и Balancer

Данные делятся на **чанки (chunks)** — непрерывные диапазоны shard key значений.

```
Shard Key: user_id (hashed)

Chunk 1: hash(user_id) ∈ [minKey, -2000000000)     → Shard 1
Chunk 2: hash(user_id) ∈ [-2000000000, 0)          → Shard 1
Chunk 3: hash(user_id) ∈ [0, 2000000000)           → Shard 2
Chunk 4: hash(user_id) ∈ [2000000000, maxKey)      → Shard 3

Размер чанка по умолчанию: 128 МБ (настраивается через `--chunkSize` в МБ, допустимый диапазон: 1–1024 МБ. Изменение размера чанка не приводит к автоматическому перераспределению существующих чанков — только новые split/merge учитывают новый размер).
```

**Balancer** — фоновый процесс на Config Servers, который мигрирует чанки между шардами для равномерного распределения:

```
Если на шарде X чанков > (avg + threshold) → Balancer мигрирует чанк на шард Y:

1. Начинает миграцию (донор → получатель).
2. Все операции идут на донора.
3. Данные чанка копируются на получателя.
4. Изменения за время копирования (delta) применяются на получателя.
5. Метаданные в Config Server обновляются атомарно (чанк теперь на получателе).
6. Донор удаляет старые данные асинхронно.
```

#### 3.5.3. Выбор Shard Key — КРИТИЧЕСКИ ВАЖНО

Shard key нельзя изменить после шардирования коллекции. Это решение навсегда.

**Хороший shard key:**
1. **Высокая кардинальность (cardinality):** много уникальных значений → чанки можно равномерно распределить.
2. **Равномерное распределение запросов:** запросы не должны идти на один шард (hot shard).
3. **Поддержка целевых запросов:** `mongos` может направить запрос на один шард (targeted query), а не на все (scatter-gather).

```javascript
// ПЛОХОЙ shard key: { country: 1 }
// - Низкая кардинальность (всего 200 стран)
// - Неравномерное распределение (Китай vs Ватикан)
// - Размер чанка ограничен → чанк страны может быть больше 128 МБ (jumbo chunk)

// ПЛОХОЙ shard key: { _id: 1 } (monotonically increasing)
// - Все новые записи идут на один шард (самый правый чанк) → hot shard

// ХОРОШИЙ shard key: { user_id: "hashed" }
// - Высокая кардинальность
// - Равномерное распределение (хеш)
// - Targeted queries по user_id

// ХОРОШИЙ shard key: { organization_id: 1, timestamp: 1 }
// - Запросы в пределах организации идут на один шард
// - Достаточная кардинальность
```

### 3.6. Репликация в MongoDB: Replica Sets

Replica Set — это группа процессов `mongod`, поддерживающих один и тот же набор данных.

```
┌──────────────────────────────────────────────────┐
│                 Replica Set                       │
│                                                   │
│  ┌──────────────────────────────┐                 │
│  │         PRIMARY              │                 │
│  │  - Принимает все записи      │                 │
│  │  - Записывает в oplog        │                 │
│  │  - Может обслуживать чтения   │                 │
│  └──────────┬───────────────────┘                 │
│             │                                     │
│    Oplog (асинхронная репликация)                 │
│             │                                     │
│  ┌──────────▼───────────┐ ┌──────────────────┐    │
│  │    SECONDARY 1       │ │   SECONDARY 2    │    │
│  │ - Копирует oplog     │ │ - Копирует oplog │    │
│  │ - Применяет операции │ │ - Применяет опер.│    │
│  │ - Обслуживает чтения │ │ - Обслуживает чт.│    │
│  └──────────────────────┘ └──────────────────┘    │
│                                                   │
│  ┌──────────────────────┐                         │
│  │      ARBITER         │  (опционально)          │
│  │ - Не хранит данные   │                         │
│  │ - Только голосует    │                         │
│  │ - Нужен для нечётного│                         │
│  │   числа голосующих   │                         │
│  └──────────────────────┘                         │
└──────────────────────────────────────────────────┘
```

**Oplog (Operation Log):**

Oplog — это специальная **capped collection** (коллекция фиксированного размера, работает как кольцевой буфер). Хранит все операции, изменяющие данные:

```javascript
// Пример записи в oplog:
{
  "ts": Timestamp(1689000000, 1),   // время операции
  "t": NumberLong(1),               // терм (эпоха выборов)
  "h": NumberLong("..."),           // хеш операции
  "v": 2,                           // версия протокола
  "op": "i",                        // i=insert, u=update, d=delete, c=command
  "ns": "mydb.users",               // namespace
  "ui": UUID("..."),                // UUID коллекции
  "o": { _id: 1, name: "Ivan" },   // сам документ (для insert/delete)
  "o2": { _id: 1 }                 // условие (для update/delete)
}
```

**Выборы лидера (Raft-подобный протокол):**

Когда Primary становится недоступным:
1. Secondary с самым свежим oplog (наибольший `ts`) инициирует выборы.
2. Голосование: нужно большинство голосов (nodes/2 + 1).
3. Новый Primary избирается, остальные Secondary'и переключаются на репликацию с него.
4. Старый Primary при возврате становится Secondary (и откатывает незакоммиченные операции — rollback).

**Read Preference и Write Concern:**

```javascript
// Write Concern: сколько узлов должны подтвердить запись
db.collection.insertOne(doc, { writeConcern: { w: "majority", j: true } })
// w: 1      — только primary подтверждает
// w: "majority" — большинство голосующих узлов
// j: true   — запись в журнал (journal) обязательна

// Read Preference: с какого узла читать
db.collection.find().readPref("secondaryPreferred")
// "primary"            — только с primary (строгая консистентность)
// "primaryPreferred"   — с primary, если доступен, иначе secondary
// "secondary"          — только с secondary (может быть stale)
// "secondaryPreferred" — с secondary, если доступен, иначе primary
// "nearest"            — с узла с наименьшей сетевой задержкой
```

**Read Concern — уровень изоляции при чтении:**

| Read Concern | Гарантия |
|-------------|----------|
| `local` | Возвращает данные с узла без гарантии, что они закоммичены на большинстве (могут быть откачены при rollback). |
| `available` | То же, что local (для нешардированных). |
| `majority` | Возвращает данные, закоммиченные на большинстве узлов (не могут быть откачены). Высокая durability. |
| `linearizable` | Гарантирует, что возвращаются все записи, успешно подтверждённые до начала чтения (однако медленнее). |
| `snapshot` | Для транзакций: снапшот-изоляция на момент начала транзакции. |

### 3.7. Транзакции в MongoDB: от single-document к multi-document ACID

**Исторически** MongoDB гарантировала атомарность только на уровне одного документа. Начиная с версии 4.0 (2018) — multi-document ACID транзакции в пределах одного replica set. C версии 4.2 — распределённые транзакции на шардированном кластере.

```javascript
// Multi-document ACID транзакция
const session = client.startSession();

session.startTransaction({
  readConcern: { level: "snapshot" },
  writeConcern: { w: "majority" }
});

try {
  const orders = session.client.db("mydb").collection("orders");
  const inventory = session.client.db("mydb").collection("inventory");

  await orders.insertOne({ product: "book", qty: 1 }, { session });
  await inventory.updateOne(
    { product: "book" },
    { $inc: { qty: -1 } },
    { session }
  );

  await session.commitTransaction();
} catch (error) {
  await session.abortTransaction();
} finally {
  session.endSession();
}
```

**Как это работает внутри:**

1. **MVCC (Multiversion Concurrency Control):** WiredTiger хранит несколько версий каждого документа с timestamp'ами.
2. **Snapshot Isolation:** транзакция видит согласованный снапшот данных на момент `start_transaction_ts`.
3. **WiredTiger Transactions:** каждая MongoDB-транзакция отображается на WiredTiger-транзакцию, которая даёт ACID на уровне storage engine.
4. **Двухфазный коммит (2PC) для шардированных транзакций:** координатор собирает готовность от всех участвующих шардов, затем коммитит.

**Ограничения транзакций MongoDB:**

- Максимальная длительность: 60 секунд по умолчанию (настраивается `transactionLifetimeLimitSeconds`).
- Производительность: транзакции медленнее, чем обычные операции (дополнительные round-trips, удержание блокировок).
- Влияние на кэш WiredTiger: старые версии документов не удаляются, пока идёт транзакция (может расти кэш).
- Не для всех операций: некоторые административные команды не поддерживаются в транзакциях.

### 3.8. Change Streams: реактивные потоки изменений

Change Streams — механизм MongoDB (начиная с 3.6), позволяющий приложениям подписываться на изменения данных в реальном времени без поллинга (tail-чтения) oplog напрямую. Основаны на oplog, но предоставляют безопасный, фильтруемый и возобновляемый API.

**Принцип работы:**

```
┌─────────────────────────────────────────────────────────────────┐
│                    Change Streams                                │
│                                                                  │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐                 │
│  │ PRIMARY  │────▶│SECONDARY │────▶│SECONDARY │                 │
│  │  oplog   │     │  oplog   │     │  oplog   │                 │
│  └────┬─────┘     └──────────┘     └──────────┘                 │
│       │                                                          │
│       │ Change Stream cursor (читает oplog как tailable cursor)  │
│       ▼                                                          │
│  ┌─────────────────────────────────────────┐                     │
│  │           Change Stream                  │                     │
│  │  ┌───────────────────────────────────┐  │                     │
│  │  │ События: insert, update, delete,  │  │                     │
│  │  │ replace, invalidate, drop         │  │                     │
│  │  └───────────────────────────────────┘  │                     │
│  └───────────────────┬─────────────────────┘                     │
│                      │                                           │
│                      ▼                                           │
│  ┌──────────────────────────────────────────┐                    │
│  │            Приложение                     │                    │
│  │  - Реактивные обновления UI               │                    │
│  │  - Синхронизация с поисковым индексом     │                    │
│  │  - Отправка в Kafka / Event Bus           │                    │
│  │  - Триггеры бизнес-логики                 │                    │
│  └──────────────────────────────────────────┘                    │
└─────────────────────────────────────────────────────────────────┘
```

**Пример использования (Node.js):**

```javascript
const collection = db.collection('orders');

// Подписка на все изменения в коллекции orders
const changeStream = collection.watch([
  { $match: { operationType: { $in: ['insert', 'update'] } } }
]);

changeStream.on('change', (change) => {
  console.log('Изменение:', change.operationType, change.fullDocument);
  // Например: синхронизация с Elasticsearch
  // Или: публикация события в Kafka
});

// Возобновляемость: если соединение разорвано,
// Change Stream продолжит с последнего обработанного токена
// (resume token = временная метка oplog)
```

**Ключевые свойства Change Streams:**

| Свойство | Описание |
|----------|----------|
| **Возобновляемость (Resume)** | Каждое событие содержит `_id` (resume token) — указатель на позицию в oplog. После разрыва соединения можно продолжить с того же места. |
| **Фильтрация** | Можно фильтровать по типу операции (`insert`, `update`, `delete`), по полям документа (`$match`), по `fullDocument` (состояние «до» или «после» изменения). |
| **Total ordering** | В пределах одного шарда события упорядочены глобально. На шардированном кластере — глобальный порядок не гарантирован (события из разных шардов могут перемешиваться). |
| **Durability** | События читаются из oplog, который реплицирован на большинство узлов (majority write concern). Change Stream возвращает только закоммиченные изменения. |

**Типичные сценарии использования:**

- **Синхронизация полиглотных хранилищ:** MongoDB → Elasticsearch, MongoDB → Kafka, MongoDB → кэш Redis. Change Streams — стандартный способ реализации CDC (Change Data Capture) в экосистеме MongoDB.
- **Реактивные UI:** обновление интерфейса при изменении данных другим пользователем (WebSocket + Change Stream).
- **Event Sourcing:** сохранение всех изменений как потока событий.
- **Триггеры и оповещения:** уведомление при изменении критичных документов.

**Ограничения:**
- Change Streams доступны только для replica set и sharded cluster (не standalone).
- На шардированном кластере события могут приходить в разном порядке из разных шардов.
- Изменения в коллекции могут быть пропущены, если коллекция была удалена и пересоздана (событие `invalidate`).

### 3.9. Aggregation Pipeline

Aggregation Pipeline — это аналог SQL GROUP BY / JOIN / WINDOW для MongoDB, реализованный как конвейер операторов.

```
Pipeline: документы проходят через последовательность стадий
          ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
collection→│$match│→───│$group│→───│$sort │→───│$limit│→ результат
          └──────┘    └──────┘    └──────┘    └──────┘
```

**Ключевые стадии (stages):**

| Стадия | Аналог SQL | Описание |
|--------|-----------|----------|
| `$match` | `WHERE` | Фильтрация документов. |
| `$group` | `GROUP BY` | Агрегация с `_id` для группировки. |
| `$sort` | `ORDER BY` | Сортировка. |
| `$project` | `SELECT` | Выбор/трансформация полей. |
| `$lookup` | `LEFT JOIN` | Объединение с другой коллекцией. |
| `$unwind` | `UNNEST` | Разворачивание массива в отдельные документы. |
| `$bucket` / `$bucketAuto` | Распределение по корзинам (гистограммы). |
| `$facet` | Параллельные pipeline'ы (multi-faceted search). |
| `$merge` / `$out` | Запись результата в коллекцию. |

```javascript
// Аналог SQL: SELECT status, COUNT(*), AVG(amount)
//             FROM orders
//             WHERE created_at > '2024-01-01'
//             GROUP BY status
//             HAVING COUNT(*) > 5

db.orders.aggregate([
  { $match: { created_at: { $gt: ISODate("2024-01-01") } } },
  { $group: {
      _id: "$status",
      count: { $sum: 1 },
      avgAmount: { $avg: "$amount" }
  }},
  { $match: { count: { $gt: 5 } } },
  { $sort: { count: -1 } }
])
```

**Оптимизация Aggregation Pipeline:**

MongoDB может «сливать» (coalesce) последовательные стадии:
- `$match` + `$sort` на первом месте использует индекс (как в SQL: WHERE + ORDER BY на одном индексе).
- `$match` + `$project` → проецирование только нужных полей снижает объём передаваемых данных.
- `$lookup` на индексированном поле выполняется как indexed lookup, а не COLLSCAN.

### 3.10. Couchbase и CouchDB: сравнение с MongoDB

| Характеристика | MongoDB | Couchbase | CouchDB |
|---------------|---------|-----------|---------|
| **Модель данных** | BSON документы | JSON документы | JSON документы |
| **Запросы** | MongoDB Query Language (MQL) | N1QL (SQL для JSON) | MapReduce + Mango Query |
| **Транзакции** | Multi-document ACID (4.0+) | ACID (с 6.5+) | Нет (только single-doc) |
| **Репликация** | Oplog (асинхронная) | DCP (Database Change Protocol) | Multi-Version (MVCC) |
| **Шардирование** | Автоматическое (mongos, config servers) | Автоматическое (vBuckets) | Встроенное (shared nothing) |
| **Индексы** | B-Tree, compound, text, geo, hashed | GSI (Global Secondary Index) | B-Tree (Mango) |
| **Мобильная синхронизация** | Atlas Device Sync | Couchbase Mobile (Sync Gateway) | PouchDB + CouchDB (протокол репликации) |
| **CAP-классификация** | CP (настраивается) | CP | AP |
| **Типичное применение** | Общего назначения, веб-приложения | Интерактивные приложения, mobile | Offline-first, edge computing |

## Часть IV. Колоночные (Column-Family) хранилища

### 4.1. Модель данных: строки, колонки, семейства колонок

**Важно:** Column-family БД — это НЕ колоночное хранение в смысле columnar storage (как ClickHouse или Vertica). Терминология путает.

- **Columnar storage (колоночное хранение):** данные таблицы хранятся по колонкам на диске для аналитических запросов (сжатие, векторизация). ClickHouse, Vertica, Parquet.
- **Column-family (семейства колонок):** разреженная, многомерная, распределённая хеш-таблица. Данные группируются по строкам, затем по семействам колонок. Cassandra, HBase.

**Модель данных Cassandra/Wide-Column:**

```
Таблица: sensor_data

┌────────────────┬──────────────────────┬──────────────────────┐
│  Partition Key │   Column Family:     │   Column Family:     │
│  sensor_id     │   readings           │   metadata           │
├────────────────┼──────────────────────┼──────────────────────┤
│  sensor_1      │ temp: 23.5 @ 10:00  │ location: "floor_1"  │
│  (partition)   │ temp: 23.8 @ 10:01  │ model: "TH-200"      │
│                │ humidity: 60 @ 10:00│                      │
│                │ humidity: 61 @ 10:01│                      │
├────────────────┼──────────────────────┼──────────────────────┤
│  sensor_2      │ temp: 19.0 @ 10:00  │ location: "floor_2"  │
│                │ temp: 19.1 @ 10:01  │ model: "TH-200"      │
└────────────────┴──────────────────────┴──────────────────────┘
```

**В отличие от реляционной модели:**
- Нет жёсткой схемы: строки в одной таблице могут иметь разный набор колонок.
- Нет JOIN'ов: денормализация — основной подход.
- Запросы оптимизируются под конкретные access patterns (как данные будут читаться).

### 4.2. Apache Cassandra: архитектура и внутреннее устройство

Cassandra — гибрид Amazon Dynamo (распределённое кольцо, eventual consistency) и Google Bigtable (модель данных column-family + LSM-деревья).

#### 4.2.1. Peer-to-Peer архитектура (НЕТ Master/Slave!)

```
┌─────────────────────────────────────────────────────────────┐
│                 Cassandra Cluster (P2P)                      │
│                                                              │
│    ┌─────────┐          ┌─────────┐          ┌─────────┐     │
│    │ Node A  │◄────────►│ Node B  │◄────────►│ Node C  │     │
│    │ (rack1) │  gossip  │ (rack1) │  gossip  │ (rack2) │     │
│    └────┬─────┘          └────┬─────┘          └────┬─────┘     │
│         │                    │                    │           │
│         │    ┌─────────┐     │                    │           │
│         └───►│ Node D  │◄────┴────────────────────┘           │
│              │ (rack2) │                                      │
│              └─────────┘                                      │
│                                                              │
│  ВСЕ узлы равны — любой узел может принять запрос на чтение  │
│  или запись. Узел-координатор перенаправляет запрос к        │
│  ответственным узлам (replicas) на основе хеша ключа.        │
└─────────────────────────────────────────────────────────────┘
```

**Gossip-протокол (подробно в 7.6):** узлы обмениваются информацией о состоянии кластера через gossip-сообщения. Каждую секунду каждый узел отправляет gossip digest трём случайным узлам.

**Snitch:** компонент, который определяет топологию датацентра и стоек (rack). Используется для размещения реплик с учётом отказоустойчивости.

**Как Cassandra обрабатывает запись:**

```
1. Клиент отправляет write на ЛЮБОЙ узел (становится координатором).
2. Координатор хеширует partition key → определяет token → определяет replica set.
3. Координатор отправляет write на все реплики одновременно.
4. Ждёт подтверждения от N узлов (consistency level, напр. QUORUM = RF/2 + 1).
5. Возвращает подтверждение клиенту.

Если реплика недоступна:
  - Координатор сохраняет hint в системной таблице (hinted handoff).
  - Когда реплика возвращается, hint доставляется.
```

**Как Cassandra обрабатывает чтение:**

```
1. Клиент → Координатор.
2. Координатор определяет реплики (RF реплик).
3. Отправляет read request на RF реплик (или меньше, зависит от consistency level).
4. Ждёт ответы от N реплик (consistency level).
5. Сравнивает версии (используя timestamp'ы).
6. Если есть расхождения — запускает read repair: самая свежая версия записывается на stale реплики.
7. Возвращает клиенту последнюю версию.
```

### 4.3. Storage Engine: LSM-деревья и SSTables

Cassandra использует LSM-Tree (Log-Structured Merge Tree) — оптимизацию для записи, а не чтения.

```
Процесс записи в Cassandra (путь данных):

   Write
     │
     ▼
┌─────────┐   ┌──────────────┐
│  Commit │   │  Memtable    │  (память)
│  Log    │   │  (Sorted     │
│  (WAL)  │   │   SkipList   │
└────┬────┘   └──────┬───────┘
     │               │ (при заполнении)
     │               ▼
     │        ┌──────────────┐
     │        │  SSTable     │  (диск, неизменяемый
     │        │  (Sorted     │   файл, отсортированный
     │        │   Strings    │   по ключу)
     │        │   Table)     │
     │        └──────────────┘
     │
     ▼
   Долговременное хранение на диске (для восстановления после краха)
```

**Детальный путь записи:**

1. Данные приходят на узел.
2. Записываются в **Commit Log** (append-only лог на диске) — для durability.
3. Данные попадают в **Memtable** — in-memory структура (SkipList), отсортированная по ключу. Memtable партицирован по таблицам.
4. Когда Memtable заполняется (по размеру или времени) → flush на диск в виде **SSTable** (Sorted Strings Table).
5. SSTable — неизменяемый (immutable) файл. Содержит данные, отсортированные по partition key + clustering key.
6. Commit Log очищается: данные теперь в SSTable.

**SSTable структура:**

```
┌─────────────────────────────────────────────────┐
│                  SSTable                        │
│                                                  │
│  ┌────────────────────────────────────────┐     │
│  │  Data file (*-Data.db)                 │     │
│  │  ┌──────────────────────────────────┐  │     │
│  │  │ Partition 1 (row)                │  │     │
│  │  │  ├─ Clustering row 1             │  │     │
│  │  │  │   ├─ column: value            │  │     │
│  │  │  │   └─ column: value            │  │     │
│  │  │  └─ Clustering row 2             │  │     │
│  │  ├──────────────────────────────────┤  │     │
│  │  │ Partition 2                      │  │     │
│  │  │  └─ ...                          │  │     │
│  │  └──────────────────────────────────┘  │     │
│  └────────────────────────────────────────┘     │
│  ┌────────────────────────────────────────┐     │
│  │  Index file (*-Index.db)               │     │
│  │  ┌──────────────────────────────────┐  │     │
│  │  │ Partition Key → offset в Data    │  │     │
│  │  │ (разреженный индекс)             │  │     │
│  │  └──────────────────────────────────┘  │     │
│  └────────────────────────────────────────┘     │
│  ┌────────────────────────────────────────┐     │
│  │  Bloom filter (*-Filter.db)            │     │
│  │  ┌──────────────────────────────────┐  │     │
│  │  │ Вероятностная структура:         │  │     │
│  │  │ "Есть ли ключ X в этом SSTable?" │  │     │
│  │  │ - false: точно нет (99%)         │  │     │
│  │  │ - true: возможно есть (1%)       │  │     │
│  │  └──────────────────────────────────┘  │     │
│  └────────────────────────────────────────┘     │
│  ┌────────────────────────────────────────┐     │
│  │  Statistics file (*-Stats.db)          │     │
│  │  Метрики: размер, количество ключей,   │     │
│  │  min/max timestamp и т.д.             │     │
│  └────────────────────────────────────────┘     │
└─────────────────────────────────────────────────┘
```

**Чтение из Cassandra:**

```
Read path (упрощённо):

1. Поиск в Memtable (память, быстро).
2. Поиск в Row Cache (если включён, хранит часто читаемые строки).
3. Поиск в Bloom-фильтрах всех SSTable'ов:
   - Если bloom filter говорит "нет" → пропускаем SSTable.
   - Если "возможно" → проверяем Index file → находим offset партиции.
4. Читаем данные из Data file.
5. Объединяем результаты из разных SSTable'ов (merge sort).
6. Возвращаем самую свежую версию (по timestamp/writetime).

Проблема: чем больше SSTable'ов, тем медленнее чтение.
Решение: Compaction (см. 4.7).
```

### 4.4. Индексы в Cassandra: первичный ключ, secondary indexes, SSTable-attached indexes

**Первичный ключ в Cassandra:**

Первичный ключ состоит из двух частей:

```
PRIMARY KEY ((partition_key), clustering_key_1, clustering_key_2, ...)

- Partition Key: определяет, на каком узле хранятся данные (хешируется).
- Clustering Key: определяет порядок строк внутри партиции.
```

```sql
-- Пример: показания датчиков
CREATE TABLE sensor_data (
    sensor_id UUID,
    date DATE,
    time TIMESTAMP,
    temperature DOUBLE,
    humidity DOUBLE,
    PRIMARY KEY ((sensor_id), date, time)
) WITH CLUSTERING ORDER BY (date DESC, time ASC);

-- Данные внутри партиции упорядочены по date DESC, time ASC:
-- sensor_1: [2024-01-03, ...], [2024-01-02, ...], [2024-01-01, ...]
```

**Запросы в Cassandra ОБЯЗАНЫ следовать структуре PRIMARY KEY:**

```sql
-- OK: указан полный partition key
SELECT * FROM sensor_data WHERE sensor_id = ? AND date = ?;

-- OK: указан partition key + префикс clustering key
SELECT * FROM sensor_data WHERE sensor_id = ? AND date >= ?;

-- OK: можно использовать IN (но осторожно — медленно, scatter-gather)
SELECT * FROM sensor_data WHERE sensor_id IN (?, ?, ?);

-- НЕЛЬЗЯ: нет partition key → scatter на все узлы (катастрофа)
SELECT * FROM sensor_data WHERE date = ?;  -- ❌ ALLOW FILTERING
```

#### 4.4.1. Secondary Indexes

Cassandra поддерживает вторичные индексы, но они сильно отличаются от B-Tree индексов в РСУБД:

```sql
-- Создание вторичного индекса
CREATE INDEX ON sensor_data (temperature);

-- Запрос с использованием индекса
SELECT * FROM sensor_data WHERE temperature > 100;
```

**Как работает вторичный индекс в Cassandra:**

```
Вторичный индекс — это скрытая таблица:
  indexed_value → partition_key_of_original_row

Проблемы:
1. Индекс не знает partition key → запрос идёт на ВСЕ узлы кластера!
2. Каждый узел ищет в своей локальной индексной таблице.
3. Результаты собираются координатором.

Это эффективно только когда:
- Запрос содержит partition key (тогда поиск идёт на одном узле).
- Количество уникальных значений индекса невелико.
```

**SASI (SSTable-Attached Secondary Index) — улучшенный вторичный индекс:**

```sql
-- Более эффективный индекс (Cassandra 3.4+)
CREATE CUSTOM INDEX ON sensor_data (temperature)
USING 'org.apache.cassandra.index.sasi.SASIIndex';
```

SASI хранит индекс внутри SSTable (а не как отдельную таблицу), используя B+Tree или префиксное дерево. Поддерживает range-запросы и LIKE.

**Storage-Attached Indexing (SAI) — Cassandra 4.0+:**

SAI — новый подход к индексации с глобальным индексированием, оптимизированным для распределённой природы Cassandra. SAI-индексы хранятся вместе с данными в SSTable и поддерживают:
- Числовые range-запросы
- Поиск по строкам (включая regex-like операции)
- Коллекции (set, list, map)
- Лучшая производительность, чем у SASI

```sql
CREATE INDEX ON sensor_data (temperature) USING 'sai';
```

#### Materialized Views (материализованные представления)

Cassandra 3.0+ поддерживает автоматическую денормализацию через **Materialized Views (MV)** — таблицы, которые автоматически поддерживаются сервером при записи в базовую таблицу, но с другим partition key. Это избавляет от ручного дублирования данных в несколько таблиц.

```sql
-- Базовая таблица: пользователи
CREATE TABLE users (
    id UUID PRIMARY KEY,
    name TEXT,
    email TEXT,
    city TEXT
);

-- Материализованное представление: поиск пользователей по email
CREATE MATERIALIZED VIEW users_by_email AS
    SELECT * FROM users
    WHERE email IS NOT NULL AND id IS NOT NULL
    PRIMARY KEY (email, id);
-- Теперь запрос SELECT * FROM users_by_email WHERE email = ? работает
-- без ALLOW FILTERING. Данные автоматически синхронизируются сервером.
```

**Как работает внутри:**

```
1. При записи в базовую таблицу users:
   - Данные пишутся в основную партицию (по id).
   - Сервер автоматически вычисляет партицию для MV (по email) и пишет копию.
   - Обе записи атомарны в пределах одного узла (batch log).

2. При чтении из MV:
   - Запрос работает как обычная таблица (partition key → узел → SSTable).
   - Никаких дополнительных накладных расходов на чтение.
```

**Критические ограничения Materialized Views (Cassandra 3.x/4.x):**

| Ограничение | Описание |
|-------------|----------|
| **Проблема восстановления после сбоя** | Если узел был недоступен во время записи в базовую таблицу, после восстановления MV может не получить обновление (нет механизма hinted handoff для MV). Требуется ручной `nodetool rebuild_view`. |
| **Производительность записи** | Каждая запись в базовую таблицу удваивает/утраивает нагрузку (основная + N представлений). Запись не считается выполненной, пока все MV не обновлены локально. |
| **Только локальные чтения** | Запрос к MV читает только локальные реплики — не гарантирует актуальность при низком consistency level на базовой таблице. |
| **Не для production с частыми сбоями** | Из-за проблем с восстановлением после сбоя многие команды избегают MV в production. Альтернатива: ручная денормализация в несколько таблиц на уровне приложения. |

**Альтернатива MV:** денормализация на уровне приложения (запись в несколько таблиц через BATCH с одним partition key) — более контролируема и надёжна, хоть и требует больше кода.

**Статус в Cassandra 4.0+:** MV получили значительные исправления, но проблемы с hinted handoff для MV остаются. Сообщество рекомендует оценивать риски перед использованием в production.

### 4.5. Шардирование и партицирование в Cassandra

Cassandra **изначально спроектирована для шардирования** — это не дополнение, а фундамент.

#### 4.5.1. Кольцо токенов

```
Hash Ring (MD5 или Murmur3):

  Token Range: 0 ──── 2^64-1 ──── 0

  Node A: token 0
  Node B: token 2^63
  Node C: token 2^63/2
  Node D: token 2^63/2 + 2^63

  Key "sensor_42" → hash → token 123456789
  → Принадлежит диапазону между Node A и Node C
  → Первая реплика: Node C (первый узел по часовой стрелке от токена)
  → Вторая реплика: Node A (следующий)
  → Третья реплика: Node D (следующий)
```

#### 4.5.2. Murmur3Partitioner (по умолчанию)

- Хеширует partition key через Murmur3Hash → 64-битный токен.
- Равномерное распределение: соседние ключи в пространстве ключей дают далеко разнесённые токены.

#### 4.5.3. Virtual Nodes (Vnodes)

Каждый физический узел владеет не одним диапазоном токенов, а несколькими (по умолчанию 16, но настраивается — `num_tokens`):

```
Без Vnodes:
  Node A: [0, 2^63)          ← огромный диапазон
  Node B: [2^63, 2^64)

С Vnodes (num_tokens=4):
  Node A: [T1..T2), [T5..T6), [T9..T10), [T13..T14)
  Node B: [T3..T4), [T7..T8), [T11..T12), [T15..T16)
  ...
  Каждый узел владеет несколькими «виртуальными узлами» на кольце.

Преимущества Vnodes:
  - Автоматическая балансировка (при добавлении узла данные перетекают равномерно).
  - Более быстрое восстановление (при падении узла нагрузка распределяется по всем).
  - Гетерогенные кластеры (можно дать больше vnodes мощному узлу).
```

### 4.6. Согласованность в Cassandra: Tunable Consistency

Уникальная фича Cassandra — **настраиваемая согласованность на уровне каждого запроса**.

```
Consistency Level для ЗАПИСИ:

┌──────────────┬──────────────────────────────────────────────────┐
│     Уровень  │  Сколько реплик должны подтвердить запись         │
├──────────────┼──────────────────────────────────────────────────┤
│ ANY          │ Хотя бы одна (включая hinted handoff). Быстро,   │
│              │ но данные могут быть потеряны.                   │
│ ONE          │ Одна реплика.                                    │
│ TWO          │ Две реплики.                                     │
│ THREE        │ Три реплики.                                     │
│ QUORUM       │ Большинство (RF/2 + 1). Баланс.                  │
│ LOCAL_QUORUM │ Большинство в локальном датацентре.              │
│ EACH_QUORUM  │ Большинство в КАЖДОМ датацентре.                 │
│ ALL          │ ВСЕ реплики. Сильнейшая гарантия, но любая       │
│              │ недоступная реплика → ошибка.                    │
└──────────────┴──────────────────────────────────────────────────┘

Consistency Level для ЧТЕНИЯ определяется аналогично.
```

**Формула сильной согласованности:**

```
R + W > N

Где:
  R = read consistency level (сколько узлов читаем)
  W = write consistency level (сколько узлов подтверждают запись)
  N = replication factor (сколько всего реплик)

Пример: RF=3, W=QUORUM(2), R=QUORUM(2)
  → 2 + 2 > 3 → сильная согласованность

Пример: RF=3, W=ONE(1), R=ONE(1)
  → 1 + 1 < 3 → eventual consistency (stale reads возможны)
```

**Это позволяет тонко настраивать компромисс между консистентностью и доступностью на уровне отдельных операций:**

```sql
-- Критичная операция (списание денег): сильная согласованность
INSERT INTO transactions (...) ... ;
-- выполняем с CONSISTENCY LOCAL_QUORUM (write) + LOCAL_QUORUM (read)

-- Некритичная операция (счётчик просмотров): eventual
UPDATE page_views SET count = count + 1 WHERE page_id = ?;
-- выполняем с CONSISTENCY ONE
```

### 4.7. Compaction и ремонт данных

#### 4.7.1. Compaction

Поскольку каждая запись создаёт новый SSTable (flush memtable), со временем накапливаются десятки/сотни файлов. Чтение замедляется. **Compaction** — процесс слияния нескольких SSTable'ов в один, с удалением старых версий данных (tombstones).

```
До compaction:

  SSTable 1: key=A, timestamp=10, value=v1
  SSTable 2: key=A, timestamp=15, value=v2  ← более новая
  SSTable 3: key=B, timestamp=20, value=v3

После compaction (Size-Tiered):

  SSTable new: key=A, timestamp=15, value=v2  (старая версия t=10 удалена)
               key=B, timestamp=20, value=v3
```

**Стратегии Compaction:**

| Стратегия | Принцип | Для чего |
|-----------|---------|----------|
| **Size-Tiered (STCS)** | Сливает SSTable'ы примерно одинакового размера. Дефолт. | Общие нагрузки с преобладанием записи. |
| **Leveled (LCS)** | Организует SSTable'ы в уровни фиксированного размера (~160 МБ каждый). Уровни не пересекаются по диапазонам ключей. | Нагрузки с преобладанием чтения (read-heavy). |
| **Time-Window (TWCS)** | SSTable'ы делятся по временным окнам. Внутри окна — STCS. | Временные ряды (TTL-данные, логи). |

```
Size-Tiered Compaction (STCS):

  Уровни:
  4 MB:  [■][■][■][■] → при накоплении 4+ сливаются в 16 MB
 16 MB:  [■■][■■] → при накоплении 4+ сливаются в 64 MB
 64 MB:  [■■■■] → ...

  + Эффективно по дисковому I/O
  - Может требовать вдвое больше дискового пространства на время compaction
```

```
Leveled Compaction (LCS):

  Уровень 0: [SSTable 160MB] [SSTable 160MB] ← новые данные
  Уровень 1: [SST 160MB][SST 160MB][SST 160MB][...] ← в 10× больше L0
  Уровень 2: [SST 160MB][SST 160MB][SST 160MB][......] ← в 10× больше L1

  Ключи в пределах уровня не пересекаются → для чтения достаточно
  посмотреть по одному SSTable на уровне → быстрое чтение.

  + Отличная производительность чтения (90% запросов → 1 SSTable)
  - Больше I/O на запись (чаще компактим)
```

#### 4.7.2. Tombstones и проблема зомби-данных

При удалении данных Cassandra не удаляет их физически сразу (SSTable'ы неизменяемы), а записывает **tombstone** — маркер удаления.

```
DELETE FROM sensor_data WHERE sensor_id = ? AND date = ?;

→ Запись: key=sensor_id:date, value=☠TOMBSTONE☠, timestamp=now()

Tombstone живёт gc_grace_seconds (по умолчанию 10 дней).
После этого при compaction удаляется физически.
```

**Проблема:** если узел был недоступен дольше `gc_grace_seconds`, после восстановления он может «воскресить» удалённые данные (зомби-данные), потому что tombstone истёк, а старые данные на восстановленном узле остались.

**Решение:** регулярный **repair** (см. ниже) с интервалом меньше `gc_grace_seconds`.

#### 4.7.3. Anti-Entropy Repair

Механизм синхронизации данных между репликами:

```
1. Запускается `nodetool repair` (или автоматически).
2. Узлы обмениваются Merkle Trees (хэш-деревьями) данных.
3. Если хеши ветвей расходятся → данные в этих ветвях неконсистентны.
4. Узлы обмениваются недостающими/устаревшими данными и исправляются.
```

**Read Repair (ремонт при чтении):**

При чтении, если координатор замечает, что реплики вернули разные версии данных, он записывает самую свежую версию на stale реплики немедленно. Это **оппортунистический ремонт**.

### 4.8. HBase vs Cassandra: сравнение

| Характеристика | Cassandra | HBase |
|---------------|-----------|-------|
| **Архитектура** | Peer-to-peer (все узлы равны) | Master-Slave (HMaster + RegionServers) |
| **Координация** | Gossip-протокол | Zookeeper |
| **Хранение** | LSM-Tree (Cassandra SSTables) | LSM-Tree (HDFS HFiles) |
| **CAP** | AP (настраиваемая) | CP (сильная согласованность) |
| **Согласованность** | Tunable (на каждый запрос) | Strong (serializable) |
| **Зависимости** | Нет (автономная) | HDFS, Zookeeper |
| **Запросы** | CQL (SQL-подобный) | HBase API (get/put/scan) + Apache Phoenix (SQL) |
| **Транзакции** | LWT (Lightweight Transactions) | ACID на уровне строки, checkAndPut |
| **Когда использовать** | Высокая доступность, много датацентров, write-heavy | Сильная согласованность, интеграция с Hadoop экосистемой |

#### ScyllaDB: эволюция Cassandra на C++

ScyllaDB — это переработка Cassandra на C++ с шардированной архитектурой (shard-per-core), созданная для устранения проблем с производительностью и предсказуемостью оригинальной Cassandra на Java.

**Ключевые архитектурные отличия:**

```
│  │  Все потоки     │              │  │ - RAM pin │  │            │
│  │  разделяют      │              │  │ - disk I/O│  │            │

│                                     долей данных, памяти и I/O  │
│                                   - Предсказуемая latency        │

└─────────────────────────────────────────────────────────────────┘
```

**Сравнение ScyllaDB vs Cassandra:**

| Характеристика | Cassandra | ScyllaDB |
|---------------|-----------|----------|
| **Язык** | Java | C++ (Seastar framework) |
| **Архитектура** | Один JVM-процесс на узел | Shard-per-core: данные и запросы изолированы по ядрам |
| **GC-паузы** | Есть (G1GC, ZGC в новых версиях) | Нет (ручное управление памятью в стиле C++) |
| **Планировщик** | Потоки ОС (много контекстных переключений) | Userspace scheduling (Seastar, cooperative) |
| **I/O** | Стандартный Java NIO | Асинхронный I/O (AIO/DIO) в обход page cache |
| **Производительность** | ~10-50K записей/сек/ядро | ~100-500K записей/сек/ядро |
| **Latency P99** | 100-1000 мс (зависит от GC) | 1-10 мс (предсказуемо) |
| **Совместимость** | — | CQL-совместима, SSTable-совместима |
| **Compaction** | STCS, LCS, TWCS | Incremental Compaction Strategy (ICS) — без write amplification, как у LCS |
| **Ремонт** | Полный (full repair) или инкрементальный | RBNO (Repair Based on Node Operations) — без Merkle Trees, на основе журнала операций |
| **Лицензия** | Apache 2.0 | AGPL (ScyllaDB Open Source) / Commercial (Enterprise) |
| **Когда выбирать** | Устоявшаяся экосистема, много датацентров, консервативный стек (Java) | Требования к низкой latency (P99 < 10ms), high throughput, чувствительность к GC-паузам |

**Когда миграция на ScyllaDB оправдана:**
- Cassandra не укладывается в latency SLO из-за GC-пауз.
- Узлы используются на 30-40% CPU из-за накладных расходов JVM, а нужно 80%+.
- Эксплуатационные затраты на тюнинг GC становятся неприемлемыми.

**Когда лучше остаться на Cassandra:**
- Недавно появившаяся ScyllaDB может иметь меньшее комьюнити и меньше драйверов/инструментов.
- Сложные CQL-запросы с пользовательскими функциями (UDF/UDA) — ScyllaDB поддерживает не всё.
- Организационная экспертиза в Java/JVM (операционная команда знает Cassandra глубоко).

## Часть V. Графовые базы данных

### 5.1. Модель данных: вершины, рёбра, свойства, метки

Графовые БД моделируют данные как граф: **вершины** (nodes/vertices) соединяются **рёбрами** (edges/relationships). Каждая вершина и ребро могут иметь **свойства** (properties) и **метки** (labels).

```
Графовая модель:

    ┌─────────────────────┐       [:KNOWS {since: 2015}]
    │ Node: Person        │──────────────────────────────┐
    │ name: "Иван"        │                              │
    │ age: 30             │                              ▼
    │ id: 1               │              ┌──────────────────────────┐
    └──────────┬──────────┘              │ Node: Person             │
               │                         │ name: "Мария"            │
               │ [:BOUGHT {date: ...}]   │ age: 28                  │
               │                         │ id: 2                    │
               ▼                         └────────────┬─────────────┘
    ┌──────────────────────────┐                       │
    │ Node: Product            │       [:LIKES]        │
    │ name: "Книга SQL"        │◄──────────────────────┘
    │ price: 1500              │
    │ id: 100                  │
    └──────────────────────────┘

  Запрос: "Найди друзей Ивана, которые купили ту же книгу, что и Мария"
  В графовой БД: нативный обход графа, а не рекурсивный CTE или множественные JOIN'ы.
```

**Почему графы неудобны в РСУБД:**

```sql
-- SQL: найти друзей друзей, не являющихся моими друзьями (уровень 2 связи)
WITH RECURSIVE friends AS (
    SELECT friend_id FROM friendships WHERE user_id = 1
    UNION
    SELECT f.friend_id
    FROM friendships f
    JOIN friends fr ON f.user_id = fr.friend_id
    WHERE depth < 2
)
SELECT ... -- JOIN'ы растут экспоненциально с глубиной обхода
```

В графовой БД каждый обход — это O(количество рёбер), а не O(размер таблицы × глубина рекурсии).

### 5.2. Neo4j: архитектура и внутреннее устройство

Neo4j — самая популярная графовая БД, написанная на Java. Использует **native graph storage** — граф хранится в виде связанных записей на диске, оптимизированных для обхода.

#### 5.2.1. Native Graph Storage

```
Файлы хранения Neo4j (в директории data/databases/<db>/):

┌─────────────────────────────────────────────────────────┐
│  neostore.nodestore.db      — вершины (nodes)           │
│  neostore.relationshipstore.db — рёбра (relationships)  │
│  neostore.propertystore.db  — свойства (properties)     │
│  neostore.relationshipgroupstore.db — группы рёбер      │
│  neostore.labeltokenstore.db — метки                     │
│  neostore.relationshiptypestore.db — типы рёбер          │
│  neostore.schema.db         — схема (индексы, constraints)│
└─────────────────────────────────────────────────────────┘
```

**Структура записи вершины (Node Record, ~15 байт):**

```
┌──────────────────────────────────────────────────────┐
│  Node Record (neostore.nodestore.db)                 │
│                                                      │
│  [inUse] [nextRelId] [nextPropId] [labels]           │
│                                                      │
│  inUse:      флаг (1=активна, 0=удалена)             │
│  nextRelId:  указатель на первое ребро вершины        │
│  nextPropId: указатель на первое свойство вершины     │
│  labels:     битовая маска или указатель на хранилище │
│              меток (label store)                      │
└──────────────────────────────────────────────────────┘
```

**Структура записи ребра (Relationship Record, ~34 байта):**

```
┌──────────────────────────────────────────────────────┐
│  Relationship Record (neostore.relationshipstore.db) │
│                                                      │
│  [inUse] [firstNode] [secondNode] [relType]          │
│  [firstPrevRelId] [firstNextRelId]                   │
│  [secondPrevRelId] [secondNextRelId]                 │
│  [nextPropId]                                        │
│                                                      │
│  Важно: двусвязные списки рёбер для КАЖДОЙ вершины. │
│  Это даёт O(1) добавление ребра и O(degree) обход    │
│  соседей вершины.                                    │
└──────────────────────────────────────────────────────┘
```

**Обход графа (Graph Traversal) — почему быстро:**

```
Чтобы найти всех друзей узла 1 ([:KNOWS] → узел):

1. Читаем Node Record для узла 1 → получаем nextRelId.
2. Читаем Relationship Record → получаем secondNode (ID друга).
3. Переходим по secondNextRelId → следующее ребро [:KNOWS].
4. Повторяем, пока не пройдём все рёбра.

Каждый шаг — это один disk seek (или чтение из page cache), а не сканирование
всей таблицы friendships с WHERE user_id = 1.

На глубине N:
  SQL: O(N × размер_таблицы) в худшем случае (N join'ов)
  Neo4j: O(суммарное_число_пройденных_рёбер)
```

#### 5.2.2. Page Cache и файлы хранения

Neo4j использует page cache для минимизации дисковых операций:

```
┌────────────────────────────────────────────┐
│              Neo4j Page Cache              │
│                                            │
│  ┌──────────────────────────────────────┐  │
│  │  Страницы файла nodestore.db         │  │
│  │  Страницы файла relationshipstore.db │  │
│  │  Страницы файла propertystore.db     │  │
│  └──────────────────────────────────────┘  │
│                                            │
│  Размер: dbms.memory.pagecache.size       │
│  (рекомендация: чем больше, тем лучше     │
│   для графовых обходов)                   │
└────────────────────────────────────────────┘
```

**Плотность хранения (Pointer Density):**

В отличие от РСУБД, где JOIN вычисляется через поиск в B-Tree по foreign key, Neo4j хранит прямые указатели (record IDs). Обход графа не требует индексного поиска — только переход по указателю.

### 5.3. Индексы в Neo4j: native indexes и fulltext

#### 5.3.1. Индекс по умолчанию (NONE!)

**Важно:** Neo4j не создаёт индексов автоматически. Без индексов поиск узла по свойству требует полного сканирования всех узлов с заданной меткой (Node Label Scan).

```cypher
-- Без индекса: медленно (O(N) где N = все Person)
MATCH (p:Person {name: "Иван"}) RETURN p

-- Создаём индекс:
CREATE INDEX person_name_idx FOR (p:Person) ON (p.name);

-- Теперь: O(log N) через поиск по B-Tree индексу
```

#### 5.3.2. Типы индексов

| Индекс | Назначение | Синтаксис |
|--------|-----------|-----------|
| **BTREE / RANGE** | Стандартный B-Tree индекс по свойству. Поддерживает equality и range. | `CREATE INDEX FOR (p:Person) ON (p.name)` |
| **LOOKUP** | Индекс для поиска узлов только по метке (без свойств). | `CREATE LOOKUP INDEX node_label_lookup FOR (n) ON EACH labels(n)` |
| **TEXT** | Полнотекстовый индекс (Lucene-based). | `CREATE FULLTEXT INDEX txt_idx FOR (p:Person) ON EACH [p.name, p.bio]` |
| **POINT** | Пространственный индекс для геоданных. | `CREATE POINT INDEX point_idx FOR (p:Place) ON (p.location)` |
| **Composite** | Составной индекс по нескольким свойствам. | `CREATE INDEX composite_idx FOR (p:Person) ON (p.age, p.name)` |

#### 5.3.3. Token Index (Neo4j 5.x+)

Новый тип индекса — поиск по полному значению свойства (а не range scan):

```cypher
-- Token index: оптимизирован для equality-поиска
CREATE INDEX token_idx FOR (p:Person) ON (p.email);

-- Эффективен для точных значений (email, username и т.д.)
```

#### 5.3.4. Как индексы помогают в графовых запросах

```cypher
-- Запрос: найди друзей Ивана, которым больше 30 лет

-- План выполнения (PROFILE):
-- 1. NodeIndexSeek: Person(name='Иван') → использует индекс → находит узел 1
-- 2. Expand(All): (p)─[:KNOWS]→(friend) — обход графа (указатели)
-- 3. Filter: friend.age > 30 — фильтрация свойства

-- Без индекса на Person.name:
-- NodeByLabelScan: Person → перебор ВСЕХ Person → ищем name='Иван' → катастрофа на миллионах узлов
```

### 5.4. Язык Cypher: обзор

Cypher — декларативный язык запросов Neo4j, вдохновлённый SQL, но оптимизированный для графов. Описывает **паттерны** в графе с помощью ASCII-арт синтаксиса.

```cypher
-- Базовые паттерны:
()              -- вершина (анонимная)
(p:Person)      -- вершина с меткой Person
(p:Person {name: "Иван"})  -- вершина с условием
-->             -- направленное ребро
--              -- ненаправленное ребро
[:KNOWS]        -- ребро с типом KNOWS
[:KNOWS*2..4]   -- путь длиной от 2 до 4 рёбер (variable-length)

-- Примеры:
-- 1. Найти всех, кого знает Иван
MATCH (ivan:Person {name: "Иван"})-[:KNOWS]->(friend)
RETURN friend.name

-- 2. Друзья друзей (глубина 2)
MATCH (ivan:Person {name: "Иван"})-[:KNOWS*2]->(foaf)
WHERE foaf <> ivan
RETURN DISTINCT foaf.name

-- 3. Кратчайший путь между двумя людьми
MATCH p = shortestPath((a:Person {name: "Иван"})-[*]-(b:Person {name: "Пётр"}))
RETURN p

-- 4. Рекомендация: книги, которые купили друзья, но не купил Иван
MATCH (ivan:Person {name: "Иван"})-[:KNOWS]->(friend)-[:BOUGHT]->(book:Product)
WHERE NOT (ivan)-[:BOUGHT]->(book)
RETURN book.name, COUNT(*) AS frequency
ORDER BY frequency DESC
```

**Ключевые операции, уникальные для графовых БД:**

| Операция | Описание | Аналог в SQL |
|----------|----------|-------------|
| `shortestPath` | Кратчайший путь (алгоритм Дейкстры, встроен в движок) | Рекурсивный CTE + оконные функции (очень медленно) |
| `[*N..M]` | Variable-length path | Рекурсивный CTE |
| `allShortestPaths` | Все кратчайшие пути | Почти невозможно |
| Graph algorithms (GDS) | PageRank, Community Detection, Betweenness | Почти невозможно в SQL без расширений |

### 5.5. Транзакции в Neo4j: ACID на графах

Neo4j — одна из немногих NoSQL БД, которая предоставляет **полные ACID-транзакции** (как в РСУБД).

```java
// Java API: типичная транзакция
try (Transaction tx = db.beginTx()) {
    Node ivan = tx.createNode(Label.label("Person"));
    ivan.setProperty("name", "Иван");

    Node maria = tx.createNode(Label.label("Person"));
    maria.setProperty("name", "Мария");

    Relationship knows = ivan.createRelationshipTo(maria, RelationshipType.withName("KNOWS"));
    knows.setProperty("since", 2020);

    tx.commit();  // атомарно: либо все узлы + ребро, либо ничего
}
```

**Как Neo4j обеспечивает ACID:**

- **Atomicity:** изменения буферизуются в памяти транзакции. При commit всё сбрасывается в Transaction Log (WAL) синхронно, затем применяется к store files.
- **Consistency:** Neo4j проверяет constraints (unique, node key) на этапе commit.
- **Isolation:** Read Committed — видит только закоммиченные данные. Можно повысить до Repeatable Read через блокировки (Java API: `tx.executeRead()` / `tx.executeWrite()`).
- **Durability:** Transaction Log записывается на диск синхронно (или асинхронно, если `dbms.tx_log.logging_level=loggable`).

```
Процесс commit'а:

  1. Запись изменений в логическую Transaction Log (append-only).
  2. fsync() лог-файла на диск.
  3. Применение изменений к store files (иногда асинхронно, если
     включён lazy flushing).
  4. Запись checkpoint record в лог.

  При восстановлении:
  1. Найти последний checkpoint.
  2. Проиграть лог от checkpoint до конца.
```

### 5.6. Шардирование и масштабирование графов: Fabric и федерация

Графы плохо шардируются. Причина: обход графа не знает границ шарда — сосед узла может быть на другом шарде, что требует network round-trip для каждого шага обхода.

**Neo4j Fabric (Enterprise):**

Fabric не шардирует один граф, а **федерирует** несколько графов (каждый на своём сервере/кластере) и позволяет выполнять запросы, объединяющие данные из них.

```cypher
-- Fabric: запрос к нескольким графам (шардам)
USE fabric.graph1
MATCH (a:Person {name: "Иван"})-[:KNOWS]->(b)
RETURN b.name AS friend

UNION

USE fabric.graph2
MATCH (a:Person {name: "Иван"})-[:KNOWS]->(b)
RETURN b.name AS friend
```

**Альтернативы масштабирования графов:**

| Подход | Система | Как работает |
|--------|---------|-------------|
| **Read Replicas** | Neo4j Cluster | Полный граф реплицируется на чтение (scale-out reads). Запись — только на leader. |
| **Federation** | Neo4j Fabric | Разные графы на разных серверах, UNION результатов. |
| **Распределённый нативный граф** | Dgraph, JanusGraph | Граф шардируется по вершинам, рёбра хранятся на том же шарде, что и вершина. Обход требует сетевых вызовов. |
| **Graph Compute** | Apache Giraph, GraphX | Граф обрабатывается пакетно (BSP — Bulk Synchronous Parallel), а не интерактивно. |

### 5.7. Другие графовые БД: JanusGraph, Amazon Neptune, ArangoDB

| База | Архитектура | Модель данных | Особенности |
|------|------------|---------------|-------------|
| **JanusGraph** | Распределённая, над Cassandra/HBase/Bigtable | Property Graph (как Neo4j) | Масштабируется горизонтально, хранилище pluggable, поддержка Gremlin, но сложная настройка. |
| **Amazon Neptune** | Управляемый сервис AWS | Property Graph + RDF (W3C) | Поддерживает Gremlin, SPARQL, openCypher. Высокая доступность (Multi-AZ), но дорого. |
| **ArangoDB** | Multi-model (документы + графы + KV) | JSON документы + графы поверх | Можно JOIN'ить документы и графы. AQL (ArangoDB Query Language) — универсальный. |
| **TigerGraph** | Распределённый нативный граф | Property Graph | Deep-link analytics (обходы на глубину 10+ шагов за секунды). GSQL — свой язык. |
| **OrientDB** | Multi-model (документы + графы) | Гибридная | Поддерживает SQL, Gremlin, гибридные запросы. |

## Часть VI. Поисковые движки (Search Engines)

### 6.1. Elasticsearch: архитектура и внутреннее устройство

Elasticsearch — распределённый поисковый движок на базе Apache Lucene. Хотя он не является БД общего назначения, его часто используют как NoSQL-хранилище для поиска и аналитики.

```
┌────────────────────────────────────────────────────────────────┐
│                   Elasticsearch Cluster                        │
│                                                                │
│  ┌──────────────────────────────────────────────────────┐      │
│  │                     Node 1                           │      │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐             │      │
│  │  │ Shard P0 │ │ Shard R1 │ │ Shard R2 │ (мастер)    │      │
│  │  │ (primary)│ │ (replica)│ │ (replica)│             │      │
│  │  └──────────┘ └──────────┘ └──────────┘             │      │
│  └──────────────────────────────────────────────────────┘      │
│                                                                │
│  ┌──────────────────────────────────────────────────────┐      │
│  │                     Node 2                           │      │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐             │      │
│  │  │ Shard P1 │ │ Shard R0 │ │ Shard R2 │             │      │
│  │  │ (primary)│ │ (replica)│ │ (replica)│             │      │
│  │  └──────────┘ └──────────┘ └──────────┘             │      │
│  └──────────────────────────────────────────────────────┘      │
│                                                                │
│  ┌──────────────────────────────────────────────────────┐      │
│  │                     Node 3                           │      │
│  │  ┌──────────┐ ┌──────────┐ ┌──────────┐             │      │
│  │  │ Shard P2 │ │ Shard R0 │ │ Shard R1 │             │      │
│  │  │ (primary)│ │ (replica)│ │ (replica)│             │      │
│  │  └──────────┘ └──────────┘ └──────────┘             │      │
│  └──────────────────────────────────────────────────────┘      │
└────────────────────────────────────────────────────────────────┘
```

**Иерархия Elasticsearch:**

```
Cluster (кластер)
  └─ Node (узел, один процесс Elasticsearch)
      └─ Index (индекс, логическая группа документов)
          └─ Shard (шард, Lucene instance)
              └─ Segment (сегмент, неизменяемый мини-индекс Lucene)
                  └─ Inverted Index + Doc Values + Stored Fields
```

**Каждый Shard — это полноценный экземпляр Lucene.** Он хранит:
- **Инвертированный индекс:** для полнотекстового поиска.
- **Doc Values:** колоночное хранение для сортировки и агрегаций.
- **Stored Fields:** оригинальные значения полей для возврата в ответе.
- **Point Values:** для числовых, date и geo-point range-запросов (BKD-деревья).

### 6.2. Инвертированный индекс: сердце полнотекстового поиска

Инвертированный индекс (Inverted Index) — это структура данных, которая отображает **терм → список документов, где этот терм встречается**.

```
Документы:
  Doc1: "MongoDB is a document database"
  Doc2: "Cassandra is a column-family database"
  Doc3: "MongoDB vs Cassandra comparison"

Инвертированный индекс:

  Терм          → Postings List (список документов + позиции)
  ─────────────────────────────────────────────────────────
  mongodb       → [Doc1: pos0] [Doc3: pos0]
  cassandra     → [Doc2: pos0] [Doc3: pos2]
  database      → [Doc1: pos4] [Doc2: pos5]
  document      → [Doc1: pos3]
  column-family → [Doc2: pos3]
  comparison    → [Doc3: pos3]
  vs            → [Doc3: pos1]
```

**Структура инвертированного индекса в Lucene (упрощённо):**

```
Term Dictionary (словарь терминов, отсортированный):
  Отсортированный список всех уникальных терминов в индексе.
  Хранится в Term Index (FST — Finite State Transducer, минимизированный
  конечный автомат, очень эффективный по памяти).

Term Index (индекс терминов):
  Позволяет найти термин в Term Dictionary за O(длина_термина),
  а не O(log N).

Postings List:
  Для каждого терма:
    - Список ID документов (сжатый: delta encoding + битовые упаковки)
    - Частота терма в каждом документе (Term Frequency, TF)
    - Позиции терма в документе (для phrase-запросов)
    - Смещения (offsets) для сниппетов (highlighting)
    - Payloads (опциональные метаданные)
```

**Как выполняется поиск:**

```
Запрос: "mongodb database"

1. Analysis: "mongodb database" → ["mongodb", "database"]
2. Поиск в Term Dictionary: "mongodb" → Postings List [Doc1, Doc3]
                            "database" → Postings List [Doc1, Doc2]
3. Пересечение списков (conjunction): [Doc1, Doc3] ∩ [Doc1, Doc2] = [Doc1]
4. Scoring: вычисление BM25 релевантности (TF-IDF заменён на BM25 в Lucene 6+).
5. Возврат результатов, отсортированных по score.
```

### 6.3. Шардирование и репликация в Elasticsearch

#### 6.3.1. Шардирование

Индекс делится на **primary shards**. Количество primary shards фиксируется при создании индекса и не может быть изменено (только reindex).

```
Распределение документов по шардам:

shard = hash(_routing) % num_primary_shards

По умолчанию _routing = _id документа.
Если указан явный routing: shard = hash(routing_value) % num_primary_shards
Это позволяет размещать связанные документы на одном шарде (colocation).
```

**Важно:** количество шардов определяет максимальный объём данных индекса. Каждый шард — это Lucene instance, который имеет накладные расходы и не может быть бесконечно большим.

```
Рекомендации:
- Размер шарда: 10–50 ГБ (не более)
- Количество шардов: зависит от объёма данных и пропускной способности
- Не больше 20 шардов на 1 ГБ heap (правило большого пальца)
```

#### 6.3.2. Репликация

Каждый primary shard имеет 0+ replica shards. Реплики:
- Обеспечивают высокую доступность (при потере primary, replica становится primary).
- Увеличивают пропускную способность чтения (запросы могут читать с реплик).

```
Процесс записи:

1. Клиент → coordinating node.
2. Coordinating node определяет primary shard (по routing).
3. Coordinating node → primary shard.
4. Primary shard:
   a. Валидирует документ.
   b. Выполняет запись локально (Lucene).
   c. Отправляет запись на все replica shards параллельно.
5. Когда все in-sync replicas подтвердят → success клиенту.
   (wait_for_active_shards: 1/all/half — настраивается)
```

#### 6.3.3. Сегменты и обновление (Refresh)

Lucene пишет данные в **сегменты** (segments) — неизменяемые мини-индексы. Данные не видны для поиска, пока не произойдёт **refresh**.

```
Indexing → In-memory buffer → Refresh (1 сек по умолчанию) → Segment (на диске)
                                                                ↓
                                                          Доступен для поиска

Commit (flush):
  Segment → fsync на диск → Transaction Log очищается.
  Происходит автоматически каждые 30 минут или при заполнении transaction log.

Transaction Log (Translog):
  Аналог WAL. Все операции записываются в translog до применения к Lucene.
  При крахе: восстанавливаем последний commit + проигрываем translog.
```

**Near Real-Time (NRT) Search:**

Данные становятся доступны для поиска через ~1 секунду после индексации (refresh interval). Это не ACID-транзакция в реальном времени, а компромисс между производительностью индексации и скоростью видимости.

### 6.4. Index Lifecycle Management (ILM)

ILM — это встроенный в Elasticsearch механизм автоматического управления жизненным циклом индексов. Он решает проблему: индексы со временем растут, и данные меняют свою «температуру» (частота доступа снижается).

**Фазы жизненного цикла индекса:**

```
                  ┌──────────┐
                  │   HOT    │  Активная запись и частый поиск.
                  │          │  Требует быстрых дисков (SSD), максимум реплик.
                  └────┬─────┘
                       │ (rollover: по размеру / времени / числу документов)
                       ▼
                  ┌──────────┐
                  │   WARM   │  Индексация прекращена, поиск реже.
                  │          │  Можно перевести на менее быстрые диски,
                  └────┬─────┘  уменьшить число реплик, принудительно слить сегменты.
                       │
                       ▼
                  ┌──────────┐
                  │   COLD   │  Данные почти не запрашиваются.
                  │          │  Можно использовать поисковые снапшоты (searchable
                  └────┬─────┘  snapshots) — данные хранятся в дешёвом S3/HDFS.
                       │
                       ▼
                  ┌──────────┐
                  │  DELETE  │  Данные старше retention policy — удаляются.
                  │          │
                  └──────────┘
```

**Настройка ILM-политики (пример):**

```json
PUT _ilm/policy/logs_policy
{
  "phases": {
    "hot": {
      "min_age": "0ms",
      "actions": {
        "rollover": {
          "max_primary_shard_size": "50GB",
          "max_age": "7d"
        },
        "set_priority": { "priority": 100 }
      }
    },
    "warm": {
      "min_age": "7d",
      "actions": {
        "forcemerge": { "max_num_segments": 1 },
        "shrink": { "number_of_shards": 1 },
        "allocate": { "number_of_replicas": 1 }
      }
    },
    "cold": {
      "min_age": "30d",
      "actions": {
        "searchable_snapshot": { "snapshot_repository": "my-repo" }
      }
    },
    "delete": {
      "min_age": "90d",
      "actions": { "delete": {} }
    }
  }
}
```

**Ключевые действия ILM:**

| Действие | Фаза | Описание |
|----------|------|----------|
| **Rollover** | Hot | Создаёт новый индекс для записи при достижении порога (размер, возраст, число документов). Текущий индекс переходит в следующую фазу. |
| **Force Merge** | Warm | Принудительно сливает сегменты Lucene (освобождает место от удалённых документов). Дорого по I/O, но ускоряет поиск. |
| **Shrink** | Warm | Уменьшает число primary shards (например, с 6 до 1), когда индекс уже не пишется активно. |
| **Allocate** | Warm/Cold | Перемещает шарды на узлы с другими характеристиками (HDD вместо SSD, меньше реплик). |
| **Searchable Snapshot** | Cold | Монтирует индекс как поисковый снапшот — данные хранятся в объектном хранилище (S3), локально кэшируются часто запрашиваемые сегменты. |
| **Delete** | Delete | Удаляет индекс безвозвратно. |
| **Readonly** | Warm | Блокирует запись в индекс и переводит его в режим «только чтение». |

**Зачем нужен ILM:**

- **Экономия ресурсов:** данные на hot-узлах (дорогие SSD, много реплик) автоматически перемещаются на cold-узлы (HDD, снапшоты в S3). Стоимость хранения падает в 5–10×.
- **Стабильная производительность записи:** rollover предотвращает ситуацию, когда один индекс вырастает до петабайта и запись в него деградирует.
- **Соответствие retention-политикам:** автоматическое удаление данных старше N дней (GDPR, аудит).

### 6.5. Анализ текста: анализаторы, токенизаторы, фильтры

Анализ текста — это процесс преобразования сырого текста в термы для индексации и поиска.

```
Pipeline анализа:

  "The Quick Brown Foxes Jumped Over the Lazy Dog!"
  
  1. Character Filters (опционально):
     - HTML Strip: удалить HTML-теги
     - Mapping: заменить символы (например, ö → o)

  2. Tokenizer (токенизатор):
     - Standard Tokenizer: ["The", "Quick", "Brown", "Foxes", "Jumped", ...]
     - Whitespace: по пробелам
     - Pattern: по регулярному выражению

  3. Token Filters (фильтры токенов):
     - Lowercase: ["the", "quick", "brown", "foxes", "jumped", ...]
     - Stemming: ["the", "quick", "brown", "fox", "jump", ...]  (Foxes → Fox)
     - Stop Words: ["quick", "brown", "fox", "jump", ...]        (the, over, ...)

  → Итоговые термы для индексации: ["quick", "brown", "fox", "jump", "lazy", "dog"]
```

**Тот же pipeline применяется и к поисковому запросу при поиске!** Это гарантирует, что `"JUMPED"` в документе найдётся по запросу `"jump"`.

**Встроенные анализаторы:**

| Анализатор | Состав |
|-----------|--------|
| **standard** | Standard Tokenizer + Lower Case + Stop (по умолчанию) |
| **simple** | Lowercase Tokenizer (не-буквы → границы) |
| **whitespace** | Whitespace Tokenizer (только пробелы — границы) |
| **keyword** | Keyword Tokenizer (весь текст = один терм) |
| **english** | Standard Tokenizer + English Stemmer + English Stop Words |

### 6.6. Релевантность: TF-IDF, BM25 и scoring

**TF-IDF (исторический, Lucene ≤ 5.x):**

```
TF (Term Frequency) = sqrt(частота_терма_в_документе)
   ↓ Чем чаще терм в документе → тем выше (но с насыщением).

IDF (Inverse Document Frequency) = log((N - n + 0.5) / (n + 0.5) + 1)
   ↓ Чем в большем числе документов встречается терм → тем менее он значим.

Field Length Norm: 1 / sqrt(количество_термов_в_поле)
   ↓ Чем короче документ → тем выше вес совпадения.
```

**BM25 (Lucene 6+, Elasticsearch 5+):**

BM25 (Best Match 25) — вероятностная модель, заменившая TF-IDF. Улучшения:

```
BM25(q, d) = Σ IDF(qi) × (TF(qi, d) × (k1 + 1)) / (TF(qi, d) + k1 × (1 - b + b × |d| / avgDL))

Где:
  - k1: насыщение TF (по умолчанию 1.2) — как быстро рост TF перестаёт влиять на score
  - b: нормализация длины документа (по умолчанию 0.75)
       0 = без нормализации, 1 = полная нормализация
  - |d| / avgDL: отношение длины документа к средней длине по индексу

Особенности BM25:
  - TF влияет на score с насыщением (не бесконечно).
  - Длинные документы штрафуются меньше, чем в TF-IDF (параметр b).
  - Естественным образом обрабатывает крайние случаи.
```

**Практический scoring в Elasticsearch:**

```json
GET /articles/_search
{
  "query": {
    "bool": {
      "should": [
        { "match": { "title": "mongodb indexing" } },
        { "match": { "body": "mongodb indexing" } }
      ]
    }
  },
  "explain": true  // показывает, как считается score для каждого документа
}
```

## Часть VII. Внутренние механизмы NoSQL: общий взгляд

### 7.1. Storage Engines: B-Tree vs LSM-Tree

Это фундаментальный выбор, определяющий, для какой нагрузки оптимизирована БД.

```
┌───────────────────────────────────────────────────────────────────┐
│                    B-TREE (чтение)                                 │
│                                                                   │
│   Структура:                                                       │
│                    [30 | 60]          ← внутренний узел             │
│                   /    |    \                                     │
│            [10|20]  [40|50]  [70|80]  ← листья (данные)           │
│                                                                   │
│   Запись: найти место в листе → вставить. Если лист полон →       │
│           split (разделение узла) → может каскадно дойти до корня.│
│           Запись «на месте» (in-place update) с WAL.              │
│                                                                   │
│   Чтение: O(log_B N) — всегда, предсказуемо.                      │
│                                                                   │
│   Фрагментация: со временем растёт (in-place updates), нужен REINDEX.│
│                                                                   │
│   Примеры: WiredTiger (MongoDB), InnoDB (MySQL), PostgreSQL.      │
└───────────────────────────────────────────────────────────────────┘

┌───────────────────────────────────────────────────────────────────┐
│                    LSM-TREE (запись)                               │
│                                                                   │
│   Структура:                                                       │
│   Память:  Memtable (Sorted SkipList / RB-Tree)                   │
│   Диск:    SSTable 1 (level 0), SSTable 2 (level 0), ...          │
│            → периодически сливаются (compaction) в level 1, 2... │
│                                                                   │
│   Запись: append в Memtable (быстро) + WAL.                       │
│           При заполнении Memtable → flush в SSTable (сортированный)│
│           SSTable НЕИЗМЕНЯЕМ — нет in-place update.               │
│                                                                   │
│   Чтение: поиск в Memtable → поиск в SSTable'ах (Bloom-фильтры).  │
│           Медленнее B-Tree из-за поиска по нескольким файлам.      │
│                                                                   │
│   Compaction: сливает SSTable'ы, удаляет старые версии.            │
│           Write Amplification: один и тот же байт может быть       │
│           записан несколько раз при последовательных compaction.   │
│                                                                   │
│   Примеры: Cassandra, LevelDB/RocksDB, HBase, InfluxDB.           │
└───────────────────────────────────────────────────────────────────┘
```

**Сравнение:**

| Характеристика | B-Tree | LSM-Tree |
|---------------|--------|----------|
| **Запись** | Медленнее (in-place update + split) | Быстрее (append-only) |
| **Чтение** | Быстрее (один путь от корня к листу) | Медленнее (поиск по нескольким SSTable) |
| **Фрагментация** | Есть (in-place updates) | Нет (immutable файлы) |
| **Сжатие** | Посредственное | Отличное (блоки) |
| **Write Amplification** | Низкая (1×) | Высокая (10–30× для LCS) |
| **Read Amplification** | Низкая (1×) | Высокая (несколько SSTable на чтение) |
| **Space Amplification** | Средняя (fragmentation) | Низкая (сжатие) |
| **Когда использовать** | Read-heavy, OLTP | Write-heavy, временные ряды, логи |

**Современные тенденции:** WiredTiger (MongoDB) использует B-Tree с MVCC. RocksDB (Facebook) — LSM-Tree, оптимизированный для SSD. Некоторые БД (TiKV/TiDB) используют RocksDB как storage engine с Raft-репликацией поверх.

### 7.2. Шардирование: стратегии и проблемы

**Шардирование** — это горизонтальное разделение данных по независимым узлам. Сложнейшая проблема в распределённых системах.

#### 7.2.1. Стратегии шардирования

```
┌─────────────────────────────────────────────────────────────────┐
│  1. Range-based Sharding (диапазонное)                          │
│                                                                 │
│  Shard A: user_id 1..1000000                                   │
│  Shard B: user_id 1000001..2000000                             │
│  Shard C: user_id 2000001..3000000                             │
│                                                                 │
│  ✓ Простые range-запросы (BETWEEN)                             │
│  ✓ Можно направить запрос на конкретный шард                    │
│  ✗ Неравномерное распределение (hot spots на "активных"        │
│    пользователях с большими ID)                                 │
│                                                                 │
│  Пример: HBase (range partitions), MongoDB (range shard key)    │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  2. Hash-based Sharding (хеш-шардирование)                      │
│                                                                 │
│  shard_id = hash(key) % N                                      │
│                                                                 │
│  ✓ Равномерное распределение данных                             │
│  ✗ Range-запросы требуют scatter-gather на все шарды           │
│  ✗ При изменении N все данные переезжают (если hash % N)       │
│                                                                 │
│  Пример: Redis Cluster (hash slots), Cassandra (Murmur3)        │
│                                                                 │
│  Решение проблемы изменения N: Consistent Hashing (7.4)         │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  3. Directory-based Sharding (шардирование по справочнику)      │
│                                                                 │
│  Отдельный сервис (или Config Servers) хранит маппинг:          │
│    user_1 → Shard A                                            │
│    user_2 → Shard A                                            │
│    user_3 → Shard B                                            │
│                                                                 │
│  ✓ Гибкое управление (можно двигать данные между шардами)       │
│  ✓ Можно реализовать свои стратегии балансировки                │
│  ✗ Дополнительный hop к справочнику (latency)                   │
│  ✗ Единая точка отказа (нужно реплицировать справочник)         │
│                                                                 │
│  Пример: MongoDB Config Servers (для chunk mapping)             │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  4. Entity/Projection Sharding (сущностное)                     │
│                                                                 │
│  Разные коллекции/таблицы живут на разных шардах:               │
│  Shard A: users, profiles                                      │
│  Shard B: orders, payments                                     │
│                                                                 │
│  ✓ Простота (нет логики маршрутизации внутри коллекции)         │
│  ✗ Не решает проблему масштабирования одной большой коллекции   │
│                                                                 │
│  Пример: MongoDB (sharding на уровне коллекции)                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 7.2.2. Проблемы шардирования

**1. Перебалансировка (Rebalancing):** при добавлении/удалении узлов данные нужно перемещать. Это дорого по I/O и времени. Consistent Hashing минимизирует объём перемещаемых данных.

**2. Горячие шарды (Hot Shards):** неравномерная нагрузка на один шард. Например, если shard key = `country`, и 80% пользователей из США → один шард получает 80% трафика.

**3. Кросс-шардовые запросы (Scatter-Gather):** запрос без shard key должен быть отправлен на все шарды, затем результаты объединены (merge-sort). Это медленно.

```sql
-- Без shard key: запрос уходит на ВСЕ шарды
SELECT * FROM orders WHERE status = 'PENDING';  -- scatter-gather!

-- С shard key: запрос уходит на ОДИН шард
SELECT * FROM orders WHERE user_id = 42;  -- targeted query
```

**4. Кросс-шардовые JOIN'ы:** в NoSQL JOIN'ы либо отсутствуют, либо очень дорогие (требуют пересылки данных между шардами).

**5. Кросс-шардовые транзакции:** требуют 2PC (Two-Phase Commit) — дорого, медленно, блокировки.

**6. Упорядоченность (Ordering):** глобальная сортировка (`ORDER BY`, `LIMIT`) требует собрать данные со всех шардов, отсортировать глобально, затем взять top-N. Не масштабируется.

### 7.3. Партицирование: вертикальное vs горизонтальное

**Важное различие:** в NoSQL-мире термины «партицирование» и «шардирование» часто используются как синонимы. Но есть нюансы:

```
┌─────────────────────────────────────────────────────────────────┐
│  Вертикальное партицирование (Vertical Partitioning)            │
│                                                                 │
│  Разделение таблицы ПО КОЛОНКАМ:                                │
│                                                                 │
│  Исходная таблица:                                              │
│  users (id, name, email, bio, avatar_binary, settings_json, ...) │
│                                                                 │
│  После вертикального партицирования:                            │
│  users_main (id, name, email)         ← часто используемые      │
│  users_profile (id, bio, avatar_binary) ← редко используемые    │
│  users_settings (id, settings_json)     ← специфичные           │
│                                                                 │
│  Преимущества:                                                  │
│  - Часто используемые колонки влезают в кэш (higher hit rate)   │
│  - Разные профили доступа к разным группам колонок              │
│  - Можно хранить на разных дисках (SSD vs HDD)                  │
│                                                                 │
│  В NoSQL:                                                       │
│  - Cassandra: column families — это форма вертикального         │
│    партицирования (разные семейства колонок)                     │
│  - MongoDB: можно проектировать коллекции по доступу            │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Горизонтальное партицирование (Horizontal Partitioning)        │
│  = Шардирование                                                 │
│                                                                 │
│  Разделение таблицы ПО СТРОКАМ (см. 7.2)                        │
│                                                                 │
│  Внутри одного узла: партицирование (partitions)                │
│  Между узлами: шардирование (sharding)                          │
│                                                                 │
│  Пример (PostgreSQL):                                           │
│  Таблица measurements партицирована по диапазонам date внутри    │
│  одного сервера PostgreSQL — это партицирование, не шардирование.│
│                                                                 │
│  Пример (Cassandra):                                            │
│  Данные партицированы по partition key и распределены по узлам   │
│  кластера — это одновременно и партицирование, и шардирование.   │
│  В терминологии Cassandra: partition = token range на узле.     │
└─────────────────────────────────────────────────────────────────┘
```

**Партицирование vs Шардирование в терминологии разных систем:**

| Система | Термин | Значение |
|---------|--------|----------|
| PostgreSQL | Partitioning | Деление таблицы на партиции в пределах одного инстанса (декларативное партицирование) |
| PostgreSQL + Citus | Sharding + Partitioning | Шардирование через Citus (распределённые шарды), партицирование внутри шарда |
| MongoDB | Sharding | Горизонтальное распределение чанков по шардам |
| Cassandra | Partitioning | Данные всегда партицированы по partition key; «шардирование» как термин не используется |
| Elasticsearch | Sharding | Индекс делится на shards; шард — экземпляр Lucene |
| Redis Cluster | Sharding | Hash slots распределены по мастерам |

### 7.4. Consistent Hashing: математика распределённых систем

Consistent Hashing (Каргер, Лехман и др., 1997) решает проблему перераспределения ключей при изменении числа узлов.

**Проблема модулярного хеширования:**

```
hash(key) % N → shard_id

При N=3:
  user_1 → hash % 3 = 0 → Shard A
  user_2 → hash % 3 = 1 → Shard B
  user_3 → hash % 3 = 2 → Shard C

При добавлении Shard D (N=4):
  user_1 → hash % 4 = 0 → Shard A (OK, не изменился)
  user_2 → hash % 4 = 1 → Shard B (OK)
  user_3 → hash % 4 = 3 → Shard D (ПЕРЕЕХАЛ!)
  ...
  ~75% ключей переезжают при добавлении одного узла!
```

**Consistent Hashing решает это:**

```
1. Хешируем и ключи, и узлы в одно кольцевое пространство (0..2^160).

2. Для ключа: находим ближайший узел по часовой стрелке (successor).

3. При добавлении узла:
   - Новый узел вставляется в кольцо.
   - Забирает ключи ТОЛЬКО у своего successor'а (соседа по часовой стрелке).
   - Остальные узлы не трогаются.
   Переезжает ~K/N ключей (K = количество ключей, N = количество узлов).

4. При удалении узла:
   - Ключи передаются successor'у.
   - Остальные узлы не трогаются.

Пример с virtual nodes (Vnodes):
  Каждый физический узел представлен V точками на кольце.
  При V=150 на узел: неравномерность распределения становится
  пренебрежимо малой (стандартное отклонение ~0.5%).
```

**Математика consistent hashing:**

```
Для N узлов и K ключей:

- Среднее количество ключей на узле: K/N
- При добавлении узла переезжает: ~K/N ключей (вместо K·(1 - 1/(N+1)) для mod N)
- Максимальная нагрузка на узел при случайном распределении: O((K/N) · log N)
- При использовании Vnodes (V виртуальных узлов на физический):
  - Стандартное отклонение: ~1/√V от среднего
  - V=150: stddev ≈ 8.2%, V=1000: stddev ≈ 3.2%
```

### 7.5. Репликация: leader-follower, multi-leader, leaderless

**Три топологии репликации в распределённых системах:**

```
┌─────────────────────────────────────────────────────────────────┐
│  1. Leader-Follower (Master-Slave, Primary-Secondary)           │
│                                                                 │
│                ┌──────────┐                                     │
│        ┌───────│  LEADER  │───────┐                             │
│        │ writes└──────────┘writes │                             │
│        ▼                          ▼                             │
│  ┌──────────┐              ┌──────────┐                         │
│  │ Follower │              │ Follower │                         │
│  │ (ro)     │              │ (ro)     │                         │
│  └──────────┘              └──────────┘                         │
│                                                                 │
│  Все записи → Leader. Leader реплицирует на Follower'ов.        │
│  Чтения: с Follower'ов (или с Leader).                          │
│                                                                 │
│  ✓ Простота реализации                                          │
│  ✓ Отсутствие конфликтов записи (пишет только лидер)            │
│  ✗ Бутылочное горлышко на запись (только лидер)                 │
│  ✗ Failover требует выбора нового лидера                        │
│                                                                 │
│  Примеры: MongoDB (Replica Set), Redis (Sentinel),              │
│           стандартная репликация PostgreSQL, MySQL               │

┌─────────────────────────────────────────────────────────────────┐
│  2. Multi-Leader (Master-Master)                                │
│                                                                 │
│  ┌──────────┐              ┌──────────┐                         │
│  │ LEADER A │◄──conflicts──►│ LEADER B │                         │
│  │ (writes) │              │ (writes) │                         │
│  └──────────┘              └──────────┘                         │
│                                                                 │
│  Каждый лидер принимает записи и реплицирует на других.         │
│                                                                 │
│  ✓ Пишем в любой датацентр (latency для пользователей)          │
│  ✓ Нет единой точки отказа на запись                            │
│  ✗ Конфликты записи (write conflicts) — нужна стратегия          │
│    разрешения: LWW (Last Write Wins), merge, application logic  │
│  ✗ Сложнее реализация                                           │
│                                                                 │
│  Примеры: CouchDB (протокол репликации),                        │
│           Cassandra (с несколькими датацентрами),               │
│           DynamoDB (глобальные таблицы)                         │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  3. Leaderless (Dynamo-style)                                   │
│                                                                 │
│       ┌──────────┐                                             │
│       │  Node A  │◄──── coordination ────┐                     │
│       └──────────┘                      │                     │
│       ┌──────────┐              ┌───────┴──────┐               │
│       │  Node B  │◄─────────────┤   CLIENT     │               │
│       └──────────┘              └───────┬──────┘               │
│       ┌──────────┐                      │                     │
│       │  Node C  │◄─────────────────────┘                     │
│       └──────────┘                                             │
│                                                                 │
│  Клиент пишет/читает сразу на несколько реплик.                  │
│  Координация через Quorum (R + W > N).                          │
│                                                                 │
│  ✓ Нет лидера → нет failover'а                                  │
│  ✓ Можно настраивать баланс консистентность/доступность          │
│  ✗ Сложнее логика клиента                                       │
│  ✗ Конфликты (разрешаются при чтении через version vectors)      │
│                                                                 │
│  Примеры: Cassandra (по сути), DynamoDB, Riak                  │
└─────────────────────────────────────────────────────────────────┘
```

### 7.6. Gossip-протокол: как узлы узнают друг о друге

Gossip-протокол — это децентрализованный механизм распространения информации в кластере. Используется в Cassandra, DynamoDB, Consul.

```
Алгоритм Gossip (на примере Cassandra):

1. Каждую секунду каждый узел выбирает случайный узел из своего списка.
2. Отправляет ему «digest» (краткую информацию о состоянии кластера).
3. Получатель сравнивает полученный digest со своим состоянием:
   - Если данные получателя свежее → отправляет обновления отправителю.
   - Если данные отправителя свежее → запрашивает обновления у отправителя.
4. Оба узла обновляют своё состояние.

Свойства gossip:
  - Информация распространяется за O(log N) раундов.
  - Устойчив к отказам узлов (информация идёт разными путями).
  - Рано или поздно все узлы узнают о всех изменениях (eventual consistency
    на уровне membership).
```

**Gossip в Cassandra (GossipDigestSyn):**

```
Структура Gossip Digest:

┌────────────────────────────────────────────────────────┐
│  GossipDigestSynMessage                                │
│  ┌──────────────────────────────────────────────────┐  │
│  │ cluster_name: "MyCluster"                        │  │
│  │ digests: [                                       │  │
│  │   { endpoint: NodeA, generation: 5, version: 23 },│  │
│  │   { endpoint: NodeB, generation: 3, version: 42 },│  │
│  │   ...                                            │  │
│  │ ]                                                │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────┘

Generation: увеличивается при перезапуске узла.
Version: увеличивается при каждом изменении состояния.
```

**Phi Accrual Failure Detection (Cassandra):**

Вместо «узел мёртв, если не ответил на 3 ping'а» используется вероятностный подход:

```
φ (phi) = -log10(вероятность, что узел жив, учитывая историю задержек)

Если φ > threshold (по умолчанию 8) → узел считается мёртвым (suspect).

Преимущества:
  - Не зависит от абсолютных таймаутов (адаптивно).
  - Учитывает «дрожание» сети (jitter).
  - Редкие ложные срабатывания.
```

### 7.7. Векторные часы (Version Vectors): разрешение конфликтов

В распределённых системах с несколькими писателями возникает проблема: **как определить порядок операций, если часы на разных узлах рассинхронизированы?**

**Timestamp-based подход (Last Write Wins — LWW):**

```
Write A: key=X, value=V1, timestamp=1000 (node A)
Write B: key=X, value=V2, timestamp=1002 (node B)

Результат: V2 (побеждает более поздний timestamp).

Проблема: если часы на node B спешат на 5 секунд, write B всегда побеждает,
даже если фактически был раньше. Данные теряются.
```

**Векторные часы (Version Vectors):**

Каждый узел хранит [version_on_node_A, version_on_node_B, version_on_node_C, ...].

```
Начальное состояние:
  Node A: [A:0, B:0, C:0]
  Node B: [A:0, B:0, C:0]
  Node C: [A:0, B:0, C:0]

Write на Node A:
  Node A: [A:1, B:0, C:0]  ← инкрементировали A

Синхронизация A → B:
  Node B: [A:1, B:0, C:0]  ← получили вектор от A

Одновременные записи (concurrent writes):
  Write на Node A: [A:2, B:0, C:0]
  Write на Node B: [A:1, B:1, C:0]  (ещё не видел A:2)

Сравнение векторов:
  [2,0,0] и [1,1,0] — НЕСРАВНИМЫ (ни один не является потомком другого).
  → КОНФЛИКТ! Обе версии сохраняются, клиент разрешает.

Разрешение:
  Клиент читает обе версии → merge → пишет новую версию с вектором [A:2, B:1, C:0].
```

**В DynamoDB:**

DynamoDB использует векторные часы для отслеживания версий элементов (items). Когда возникает конфликт, DynamoDB может либо:
- Применить LWW (Last Write Wins) — последняя запись побеждает.
- Вернуть все версии клиенту для разрешения.

**В Cassandra:**

Cassandra использует упрощённый подход — **timestamp на запись** (client-provided timestamp или server timestamp). При конфликте побеждает запись с большим timestamp. Если timestamp'ы равны — побеждает лексикографически большее значение.

### 7.8. Quorum Reads/Writes: R + W > N

Формула кворума — ключевая концепция Dynamo-подобных систем.

```
Дано:
  N = replication factor (сколько всего реплик данных)
  R = read consistency level (со скольких реплик читаем)
  W = write consistency level (сколько реплик подтверждают запись)

Условие сильной согласованности:
  R + W > N

Доказательство (на пальцах):
  Множество реплик, участвовавших в записи: размер ≥ W
  Множество реплик, участвовавших в чтении:  размер ≥ R
  
  Если R + W > N, то по принципу Дирихле эти множества ПЕРЕСЕКАЮТСЯ.
  → Хотя бы одна реплика участвовала и в записи, и в чтении.
  → Читатель гарантированно увидит записанное значение.
```

**Практические конфигурации (при N=3):**

| W | R | R+W | Гарантия | Use case |
|---|---|------|----------|----------|
| 1 | 1 | 2 < 3 | Eventual consistency (stale reads возможны) | Счётчики просмотров, кэш |
| 2 | 2 | 4 > 3 | Сильная согласованность | Финансовые операции, запись профиля |
| 3 | 1 | 4 > 3 | Сильная согласованность при чтении с 1 реплики | Write-heavy, редкое чтение |
| 1 | 3 | 4 > 3 | Сильная согласованность при записи на 1 реплику | Read-heavy, редкая запись |

**Виды кворумов в Cassandra:**

```
LOCAL_QUORUM: большинство реплик в локальном датацентре
  → R + W > RF_local
  → согласованность в пределах одного ДЦ, низкая latency

EACH_QUORUM: большинство реплик в КАЖДОМ датацентре
  → согласованность глобально, но медленно (ждём все ДЦ)

QUORUM: большинство ВСЕХ реплик во всех датацентрах
  → глобальная согласованность, быстрее чем EACH_QUORUM
```

### 7.9. Anti-Entropy и Merkle Trees

**Проблема:** в распределённой системе реплики могут рассинхронизироваться из-за сетевых сбоев, пропущенных hinted handoff, ошибок.

**Anti-Entropy** — процесс активной синхронизации реплик (в отличие от пассивной репликации). Cassandra использует **Merkle Trees** для эффективного сравнения данных на репликах.

```
Merkle Tree (хэш-дерево):

                     ┌───────┐
                     │ ROOT  │  hash(h(h(1,2)) + h(h(3,4)))
                     │ HASH  │
                     └───┬───┘
              ┌──────────┴──────────┐
        ┌─────┴────┐          ┌─────┴────┐
        │ HASH 1-2 │          │ HASH 3-4 │
        │ h(1)⊕h(2)│          │ h(3)⊕h(4)│
        └──┬────┬──┘          └──┬────┬──┘
      ┌────┘    └────┐      ┌────┘    └────┐
   ┌──┴──┐      ┌───┴──┐ ┌──┴──┐     ┌───┴──┐
   │Hash 1│     │Hash 2│ │Hash 3│    │Hash 4│
   │h(leaf│     │h(leaf│ │h(leaf│    │h(leaf│
   │  1)  │     │  2)  │ │  3)  │    │  4)  │
   └──────┘     └──────┘ └──────┘    └──────┘
     DATA          DATA     DATA         DATA

Как работает сравнение:

1. Узлы A и B строят Merkle Tree для общего диапазона токенов (Cassandra строит деревья для диапазонов токенов, а не для индивидуальных ключей — каждый лист дерева соответствует диапазону хеш-токенов).
2. Сравнивают ROOT HASH.
   - Если равны → данные идентичны, конец.
   - Если различаются → спускаемся на уровень ниже.
3. Сравнивают HASH 1-2 и HASH 3-4.
   - Какие различаются → идём в те листья.
4. Находят конкретные различающиеся диапазоны токенов.
5. Пересылают только различающиеся данные (а не всю реплику).

Сложность: O(log N) шагов сравнения + O(D) пересылка, где D = объём различий.
```

## Часть VIII. Транзакции в NoSQL

### 8.1. Почему транзакции в NoSQL — это сложно

Транзакции в распределённых NoSQL-системах сложны по нескольким причинам:

1. **Данные распределены по узлам:** ACID-транзакция, затрагивающая несколько шардов, требует координации (2PC).
2. **Нет глобальных блокировок:** в P2P-системах (Cassandra) нет центрального координатора, который мог бы выдать блокировки.
3. **Сетевые разделения:** в момент Partition часть узлов недоступна — транзакция не может быть завершена корректно.
4. **Производительность:** блокировки и 2PC радикально снижают пропускную способность в распределённой среде (latency растёт).

Именно поэтому ранние NoSQL жертвовали транзакциями ради масштабируемости. Но последние годы принесли компромиссы.

### 8.2. Single-Document Transactions

**Простейшая форма транзакции в NoSQL** — атомарность на уровне одного документа/строки.

```
MongoDB (изначально):
  db.users.updateOne(
    { _id: 1 },
    { $set: { name: "Ivan" }, $inc: { loginCount: 1 } }
  )
  // ОДНОМОМЕНТНО: либо оба поля обновляются, либо ни одно.
  // Но: нельзя атомарно обновить users и orders.

Redis:
  // MULTI/EXEC на одном ключе — избыточно, команды и так атомарны.
  INCR user:1:visits  // атомарно, даже в кластере (для одного ключа).

Cassandra:
  // Операции в пределах одной партиции атомарны (batch с одним partition key).
  BEGIN BATCH
    INSERT INTO users (...) VALUES (...);
    INSERT INTO user_events (...) VALUES (...);
  APPLY BATCH;
  // Атомарно, если все в одной партиции (один partition key).
```

Это покрывает ~80% случаев использования, где связанные данные могут быть сгруппированы в один документ/партицию.

### 8.3. Multi-Document ACID в MongoDB

Начиная с MongoDB 4.0 (2018), появились **multi-document ACID транзакции** в пределах replica set. С версии 4.2 — на шардированных кластерах.

**Детали реализации:**

```
1. Snapshot Isolation:
   - Транзакция видит снапшот данных на момент start_transaction.
   - Все чтения в транзакции идут из этого снапшота (не видят чужих незакоммиченных изменений).

2. WiredTiger Transactions:
   - Каждая MongoDB-транзакция использует WiredTiger-транзакцию для MVCC.
   - WiredTiger хранит несколько версий каждого документа с timestamp'ами.
   - Старые версии не удаляются, пока существует активная транзакция, которой они нужны.

3. Двухфазный коммит (2PC) для шардированных транзакций:
   Phase 1 (Prepare):
     - Координатор отправляет prepare на все участвующие шарды.
     - Каждый шард проверяет, может ли закоммитить (проверка constraints, блокировок).
     - Отвечает "ready" или "abort".
   Phase 2 (Commit):
     - Если все ответили "ready" → координатор отправляет commit всем.
     - Если хотя бы один ответил "abort" → координатор отправляет abort всем.

4. Ограничения:
   - Максимальная длительность транзакции: 60 секунд (по умолчанию).
   - Влияет на кэш WiredTiger (старые версии не удаляются).
   - Производительность ниже, чем у single-document операций.
```

**Когда использовать multi-document транзакции в MongoDB:**

- Перевод денег между счетами (списание + зачисление).
- Обновление денормализованных данных в нескольких коллекциях.
- Атомарное создание заказа + резервирование товара + запись в историю.

**Когда НЕ использовать:**

- Высоконагруженные операции (миллионы в секунду) — транзакции замедляют пропускную способность.
- Операции, которые можно выразить через single-document update (вложенные документы).
- Операции, которые можно сделать идемпотентными (повторная обработка безопасна).

### 8.4. Lightweight Transactions (LWT) в Cassandra

Cassandra не поддерживает полноценные ACID-транзакции, но предоставляет **Lightweight Transactions (LWT)** — ограниченную форму compare-and-set (CAS).

```sql
-- LWT: вставить, только если такого пользователя ещё нет
INSERT INTO users (id, name, email)
VALUES (uuid(), 'Ivan', 'ivan@test.com')
IF NOT EXISTS;

-- LWT: обновить, только если текущий email совпадает
UPDATE users SET email = 'new@test.com'
WHERE id = 12345
IF email = 'old@test.com';
```

**Как работают LWT внутри (основано на Paxos):**

```
1. Координатор получает запрос на LWT.
2. Запускает раунд консенсуса (Paxos-подобный протокол):
   a. Prepare/Promise: координатор предлагает proposal и получает обещания
      от большинства реплик.
   b. Propose/Accept: если большинство обещало, координатор предлагает значение.
   c. Commit: если большинство приняло → операция выполнена.

3. Результат возвращается клиенту.

Примечание: Cassandra использует упрощённую версию Paxos без
долгоживущего лидера — каждый LWT-запрос начинает новый раунд.
Это не полный Multi-Paxos.

Производительность:
  LWT в 5–10× медленнее обычных операций (из-за раундов консенсуса).
  Не для высоконагруженных сценариев.
```

**Сравнение LWT с ACID-транзакциями в MongoDB:**

| | MongoDB Multi-Doc | Cassandra LWT |
|---|---|---|
| **Тип операций** | Любые (CRUD в нескольких коллекциях) | Только IF EXISTS / IF condition |
| **Атомарность** | Полная (несколько документов/коллекций) | Одна операция |
| **Изоляция** | Snapshot Isolation | Линеаризуемость (Paxos) |
| **Производительность** | Замедление на 20–50% | Замедление на 5–10× |
| **Когда использовать** | Сложные бизнес-транзакции | Простые CAS-операции |

### 8.5. Транзакции в Redis: MULTI/EXEC и Lua-скрипты

**MULTI/EXEC (Pipeline-транзакции):**

```
Redis транзакция:

MULTI           ← начало транзакции
  SET key1 val1
  INCR counter
  LPUSH list item1
EXEC            ← выполнить все команды атомарно

Особенности:
  - Команды между MULTI и EXEC НЕ ВЫПОЛНЯЮТСЯ, а ставятся в очередь.
  - При EXEC выполняются последовательно, без прерывания другими клиентами.
  - Отката при ошибке НЕТ (если команда синтаксически верна, но семантически
    ошибочна — она выполняется, остальные команды тоже выполняются).
  - Нельзя использовать результат одной команды в другой (нет read-modify-write).
```

**Lua-скрипты (EVAL):**

```lua
-- Атомарный скрипт: проверить баланс и списать
EVAL "
  local balance = redis.call('GET', KEYS[1])
  if tonumber(balance) >= tonumber(ARGV[1]) then
    redis.call('DECRBY', KEYS[1], ARGV[1])
    return 'OK'
  else
    return 'INSUFFICIENT_FUNDS'
  end
" 1 user:1:balance 100
```

**Как Redis выполняет Lua-скрипты:**

```
1. Весь скрипт выполняется как единая атомарная операция.
2. Никакие другие команды не выполняются, пока скрипт работает.
3. Redis загружает скрипт в кэш (SCRIPT LOAD), затем вызывает по SHA (EVALSHA).

Ограничения:
  - Должен выполняться быстро (блокирует event loop).
  - Детерминированность: на репликах скрипт должен дать тот же результат.
  - В Redis Cluster: все ключи, используемые скриптом, должны быть в одном слоте
    (hash tags).
```

**Сравнение MULTI/EXEC и Lua:**

| | MULTI/EXEC | Lua Script |
|---|---|---|
| **Атомарность** | Да | Да |
| **Условная логика** | Нет (команды фиксированы) | Да (if/else, циклы) |
| **Read-modify-write** | Нет | Да |
| **Блокировка event loop** | На время EXEC | На время выполнения скрипта |
| **Кросс-шардово (Cluster)** | Не поддерживается | Только через hash tags |

#### Redis Functions (Redis 7+): эволюция Lua-скриптов

Начиная с Redis 7, появились **Redis Functions** — серверные функции, которые загружаются в Redis как библиотеки и управляются аналогично хранимым процедурам. В отличие от Lua-скриптов (EVAL/EVALSHA), Functions существуют как долгоживущие объекты на сервере: они переживают перезапуск (при AOF/RDB), реплицируются и не требуют повторной загрузки скрипта каждым клиентом.

**Ключевые отличия Functions от EVAL-скриптов:**

```
┌─────────────────────────────────────────────────────────────────┐
│  EVAL-скрипты (старый способ)                                    │
│                                                                  │
│  1. Клиент отправляет SCRIPT LOAD <lua код>                     │
│  2. Сервер возвращает SHA1 хеш.                                  │
│  3. Клиент вызывает EVALSHA <sha> ...                            │
│  4. При рестарте Redis: все скрипты ТЕРЯЮТСЯ (кэш очищен).      │
│  5. Клиент должен заново загрузить скрипты.                      │
│                                                                  │
│  Проблемы:                                                       │
│  - Клиент должен отслеживать NOSCRIPT ошибку и перезагружать.   │
│  - Скрипты не реплицируются как код (только как эффект).        │
│  - Нет пространства имён — все скрипты в одной куче.            │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Redis Functions (Redis 7+)                                      │
│                                                                  │
│  1. Разработчик пишет библиотеку (.lua файл) с функциями:       │
│     #!lua name=mylib                                            │
│     redis.register_function('transfer', function(keys, args)     │
│       ... логика ...                                            │
│     end)                                                         │
│                                                                  │
│  2. Администратор загружает: FUNCTION LOAD <код>                │
│  3. Клиент вызывает: FCALL transfer 2 key1 key2 amount          │
│                                                                  │
│  4. При рестарте: Functions СОХРАНЯЮТСЯ в RDB/AOF.              │
│  5. Реплицируются как код на все реплики и в Cluster.           │
│  6. FUNCTION LIST — показывает все загруженные библиотеки.      │
│  7. FUNCTION DELETE — удаляет библиотеку.                       │
└─────────────────────────────────────────────────────────────────┘
```

**Пример: библиотека для банковских переводов:**

```lua
#!lua name=banking

-- Функция перевода между счетами
redis.register_function('transfer', function(keys, args)
    local from_key = keys[1]
    local to_key = keys[2]
    local amount = tonumber(args[1])

    local balance = redis.call('GET', from_key)
    if not balance or tonumber(balance) < amount then
        return redis.error_reply('INSUFFICIENT_FUNDS')
    end

    redis.call('DECRBY', from_key, amount)
    redis.call('INCRBY', to_key, amount)
    return 'OK'
end)
```

```bash
# Загрузка библиотеки (один раз, администратором)
CAT banking.lua | redis-cli -x FUNCTION LOAD REPLACE

# Вызов функции из приложения
redis-cli FCALL transfer 2 account:A account:B 1000
```

**Сравнение EVAL-скриптов и Redis Functions:**

| | EVAL / EVALSHA | Redis Functions |
|---|---|---|
| **Загрузка** | Клиент отправляет скрипт | Администратор загружает библиотеку через FUNCTION LOAD |
| **Живучесть** | Теряются при рестарте (volatile кэш) | Сохраняются в RDB/AOF, переживают рестарт |
| **Репликация** | Реплицируется только результат выполнения | Реплицируется код библиотеки |
| **Управление** | SCRIPT LOAD / SCRIPT FLUSH / SCRIPT KILL | FUNCTION LOAD / DELETE / LIST / DUMP / RESTORE |
| **Пространство имён** | Плоское (все скрипты в одном кэше) | Библиотеки с именами, изоляция |
| **Вызов** | EVALSHA <sha> | FCALL <function_name> <num_keys> <keys...> <args...> |
| **Атомарность** | Да | Да |
| **Когда использовать** | Простые одноразовые скрипты, ad-hoc операции | Production-логика, часто используемые процедуры, библиотеки |

### 8.6. Distributed Transactions: 2PC и Saga в мире NoSQL

**Two-Phase Commit (2PC):**

Используется MongoDB для шардированных транзакций (см. 8.3). Классический алгоритм:

```
Phase 1: VOTING
  Coordinator: "Можете ли вы закоммитить транзакцию?"
  Participant 1: "YES" (подготовил данные, ждёт)
  Participant 2: "YES"
  Participant 3: "YES"

Phase 2: DECISION
  Coordinator: "COMMIT" (все сказали YES)
  Participants: коммитят.

НО если в Phase 1 кто-то сказал "NO":
  Coordinator: "ABORT"
  Participants: откатывают.

Проблемы 2PC:
  - Блокирующий: если координатор упал после Phase 1 (после "YES"),
    участники БЛОКИРУЮТ ресурсы в ожидании решения координатора.
  - Медленный: минимум 2 network round-trips.
  - Единая точка отказа: координатор.
```

**Saga Pattern (распределённая транзакция без 2PC):**

Альтернатива 2PC для длительных бизнес-транзакций. Saga разбивает транзакцию на последовательность локальных транзакций (шагов), каждая из которых имеет компенсирующую операцию.

```
Пример Saga: путешествие (бронь отеля + авиа + такси)

Шаг 1: Забронировать отель
  Успех → Шаг 2
  Неудача → конец (ничего не делаем)

Шаг 2: Купить авиабилет
  Успех → Шаг 3
  Неудача → Компенсация шага 1: ОТМЕНИТЬ бронь отеля

Шаг 3: Заказать такси
  Успех → конец
  Неудача → Компенсация шага 2: ВЕРНУТЬ авиабилет
           → Компенсация шага 1: ОТМЕНИТЬ бронь отеля

Реализация:
  - Choreography (хореография): каждый шаг публикует событие,
    следующий шаг подписан на событие, компенсации тоже по событиям.
  - Orchestration (оркестрация): центральный Saga Orchestrator
    управляет последовательностью шагов и компенсаций.
```

**Saga vs 2PC:**

| | Saga | 2PC |
|---|---|---|
| **Атомарность** | Нет (промежуточные состояния видны) | Да (всё или ничего) |
| **Изоляция** | Нет (dirty reads возможны) | Да (блокировки) |
| **Длительность** | Минуты/часы/дни | Секунды |
| **Блокировки** | Нет | Да |
| **Компенсация** | Бизнес-логика (semantic rollback) | Технический откат |
| **Производительность** | Высокая (нет блокировок) | Низкая (блокировки + раунды) |

## Часть IX. SQL vs NoSQL: детальное сравнение

### 9.1. Модель данных: схема vs schemaless

```
┌─────────────────────────────────────────────────────────────────┐
│  Реляционная модель (Schema-first)                              │
│                                                                 │
│  1. Сначала проектируем схему (DDL).                            │
│  2. Данные должны соответствовать схеме.                         │
│  3. Изменение схемы — ALTER TABLE (блокирующая операция).        │
│                                                                 │
│  Преимущества:                                                  │
│  - Данные всегда валидны (constraints).                         │
│  - Оптимизатор запросов знает структуру данных.                  │
│  - Явные связи между таблицами (foreign keys).                   │
│                                                                 │
│  Недостатки:                                                    │
│  - Сложно с быстро меняющейся структурой данных.                 │
│  - Impedance mismatch (ORM).                                    │
│  - JOIN'ы на шардированных данных — боль.                        │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  Документная модель (Schemaless)                                │
│                                                                 │
│  1. Данные first, структура — неявная.                           │
│  2. Разные документы могут иметь разные поля.                    │
│  3. "Миграция" = приложение начинает писать новые поля.          │
│                                                                 │
│  Преимущества:                                                  │
│  - Быстрая итерация (не нужны ALTER TABLE).                     │
│  - Данные в том виде, в котором их использует приложение.        │
│  - Нет JOIN'ов — все связанные данные в одном документе.         │
│                                                                 │
│  Недостатки:                                                    │
│  - Нет гарантий структуры (можно записать мусор).                │
│  - Дублирование данных (денормализация).                         │
│  - Сложные аналитические запросы на агрегацию.                   │
└─────────────────────────────────────────────────────────────────┘
```

### 9.2. Масштабирование: вертикальное vs горизонтальное

```
Вертикальное (Vertical Scaling / Scale-Up):
  Больше CPU, RAM, SSD на одном сервере.
  ✓ Простота: ничего не меняется в коде.
  ✗ Предел: максимальная конфигурация железа.
  ✗ Дорого: big-iron hardware.
  ✗ Единая точка отказа.

  → Типично для РСУБД (PostgreSQL, Oracle).

Горизонтальное (Horizontal Scaling / Scale-Out):
  Больше серверов (узлов) в кластере.
  ✓ Практически неограниченный рост.
  ✓ Дешёвое commodity hardware.
  ✓ Отказоустойчивость (репликация).
  ✗ Сложность: шардирование, консистентность, сетевые задержки.

  → Типично для NoSQL (Cassandra, MongoDB, Elasticsearch).
```

**Миф: «РСУБД не масштабируются горизонтально».**

Реальность: PostgreSQL + Citus, CockroachDB, Google Spanner — распределённые SQL-базы существуют. Но:
- Они сложнее в настройке.
- Не все SQL-запросы эффективны в распределённой среде (JOIN'ы, подзапросы).
- Задержки выше, чем у NoSQL для простых операций.

### 9.3. Производительность: join'ы vs денормализация

**Реляционный подход (нормализация + JOIN'ы):**

```sql
SELECT u.name, o.product, o.price
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE u.id = 42;
```

Преимущества:
- Данные не дублируются (одна копия каждого факта).
- Обновление в одном месте.
- JOIN — это просто поиск по B-Tree индексу (при правильной схеме).

Недостатки:
- JOIN на шардированных данных требует выборки со всех шардов.
- Много JOIN'ов (5+) в одном запросе — медленно.

**Документный подход (денормализация):**

```javascript
db.users.findOne({ _id: 42 })
// Возвращает:
// {
//   _id: 42,
//   name: "Ivan",
//   orders: [
//     { product: "Книга", price: 500 },
//     { product: "Кофе", price: 300 }
//   ]
// }
```

Преимущества:
- Один disk seek вместо N (для JOIN'ов).
- Локальность данных (все данные пользователя в одном документе).
- Отлично для OLTP: «показать профиль пользователя со всеми заказами».

Недостатки:
- Дублирование: заказы внутри каждого пользователя — обновление цены продукта требует обновления ВСЕХ документов.
- Размер документа ограничен (16 МБ в MongoDB).
- Аналитические запросы («сумма всех продаж продукта X») требуют сканирования всех документов.

**Бенчмарк (приблизительный, для понимания порядка величин):**

| Операция | PostgreSQL (1 сервер) | MongoDB (1 сервер) | Cassandra (3 узла) |
|----------|----------------------|--------------------|--------------------|
| Простой lookup по PK | 0.1–0.5 мс | 0.1–0.5 мс | 0.5–2 мс |
| JOIN 2 таблиц | 0.5–2 мс | Aggregation $lookup: 1–5 мс | Нет поддержки |
| Полнотекстовый поиск | 5–50 мс (GIN) | 5–50 мс (Text Index) | Нет |
| Аналитика (агрегация 1M строк) | 100–500 мс | 200–2000 мс | 50–200 мс (широкое пар-ние) |
| Запись (простая, 1 запись) | 0.2–1 мс | 0.1–0.5 мс | 0.1–0.5 мс |

> **Примечание:** цифры сугубо ориентировочные и зависят от конкретного железа (NVMe vs SSD vs HDD), объёма данных (помещается ли в RAM/page cache), конфигурации индексов и конкурентной нагрузки. Используйте как точку отсчёта для порядка величин, а не как эталонные значения.

### 9.4. Консистентность: сильная (ACID) vs eventual (BASE)

```
┌─────────────────────────────────────────────────────────────────┐
│  СИЛЬНАЯ КОНСИСТЕНТНОСТЬ (ACID, SQL)                            │
│                                                                 │
│  После commit: все читатели видят записанные данные.             │
│                                                                 │
│  Write(x=5) → commit → Read(x) = 5 (гарантировано)               │
│                                                                 │
│  Цена:                                                          │
│  - Блокировки (locks, latches).                                 │
│  - Меньшая доступность при сетевых сбоях (CP).                   │
│  - Хуже горизонтальное масштабирование.                           │
│                                                                 │
│  Когда нужно:                                                    │
│  - Финансовые транзакции (списание/зачисление).                  │
│  - Инвентаризация (не продать один товар дважды).                │
│  - Аудит (каждое изменение должно быть учтено).                  │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  EVENTUAL CONSISTENCY (BASE, NoSQL)                             │
│                                                                 │
│  После write: читатели МОГУТ видеть старые данные некоторое время.│
│                                                                 │
│  Write(x=5) → Read(x) может быть 0 (stale read)                 │
│              → ... через N мс ...                                │
│              → Read(x) = 5                                      │
│                                                                 │
│  Цена:                                                          │
│  - Старые данные (stale reads).                                 │
│  - Конфликты (надо разрешать).                                   │
│  - Сложнее логика приложения (надо учитывать возможность          │
│    неконсистентности).                                           │
│                                                                 │
│  Когда приемлемо:                                                │
│  - Социальные сети (лайки, комментарии).                         │
│  - Аналитика (данные за прошлый час, а не за последнюю секунду). │
│  - Кэши (можно перестроить).                                    │
│  - IoT/метрики (потеря одной точки данных не критична).          │
└─────────────────────────────────────────────────────────────────┘
```

### 9.5. Гибкость схемы: миграции vs schemaless

**Реляционный подход (миграции):**

```sql
-- Миграция 001: добавляем колонку phone
ALTER TABLE users ADD COLUMN phone VARCHAR(20);

-- Миграция 002: добавляем таблицу user_roles
CREATE TABLE user_roles (
    user_id INTEGER REFERENCES users(id),
    role VARCHAR(50),
    PRIMARY KEY (user_id, role)
);

-- Миграция 003: заполняем дефолтные значения для phone
UPDATE users SET phone = 'N/A' WHERE phone IS NULL;
```

Проблемы:
- На большой таблице (миллиарды строк) ALTER TABLE блокирует таблицу на часы.
- Миграции нужно версионировать, тестировать, откатывать.
- Разные окружения (dev/staging/prod) могут быть на разных версиях схемы.

**Schemaless подход (MongoDB):**

```javascript
// Старые документы: без поля phone
{ _id: 1, name: "Ivan" }

// Новые документы: с полем phone
{ _id: 2, name: "Maria", phone: "+79161234567" }

// Код приложения:
const phone = user.phone || "N/A";  // обрабатываем отсутствие поля
```

Преимущества:
- Нет блокирующих миграций.
- Можно добавлять поля для части документов (A/B тестирование).
- Структура может различаться для разных типов документов (polymorphic collections).

Недостатки:
- В коде нужно обрабатывать все возможные формы документа (нет гарантии схемы).
- Аналитические запросы сложнее (поле может отсутствовать или иметь разный тип).
- MongoDB 3.2+ ввели **JSON Schema Validation** — компромисс между schemaless и контролем:

```javascript
db.createCollection("users", {
  validator: {
    $jsonSchema: {
      bsonType: "object",
      required: ["name", "email"],
      properties: {
        name: { bsonType: "string" },
        email: { bsonType: "string", pattern: "^.+@.+$" },
        age: { bsonType: "int", minimum: 0, maximum: 150 }
      }
    }
  }
});
```

### 9.6. Сложность запросов: SQL vs ограниченные API

**SQL — выразительный, стандартизированный, мощный:**

```sql
-- Сложный аналитический запрос
WITH monthly_sales AS (
    SELECT
        date_trunc('month', order_date) AS month,
        product_id,
        SUM(quantity * price) AS revenue,
        RANK() OVER (PARTITION BY date_trunc('month', order_date)
                     ORDER BY SUM(quantity * price) DESC) AS rank
    FROM orders
    WHERE order_date >= '2024-01-01'
    GROUP BY month, product_id
)
SELECT m.month, p.name, m.revenue
FROM monthly_sales m
JOIN products p ON m.product_id = p.id
WHERE m.rank <= 3
ORDER BY m.month, m.rank;
```

**NoSQL-запросы — специфичны для каждой БД, ограничены:**

```javascript
// MongoDB Aggregation Pipeline: тот же запрос
db.orders.aggregate([
  { $match: { order_date: { $gte: ISODate("2024-01-01") } } },
  { $group: {
      _id: {
        month: { $dateTrunc: { date: "$order_date", unit: "month" } },
        product_id: "$product_id"
      },
      revenue: { $sum: { $multiply: ["$quantity", "$price"] } }
  }},
  { $sort: { "_id.month": 1, revenue: -1 } },
  { $group: {
      _id: "$_id.month",
      top_products: { $push: { product_id: "$_id.product_id", revenue: "$revenue" } }
  }},
  { $project: {
      top_products: { $slice: ["$top_products", 3] }
  }},
  { $unwind: "$top_products" },
  { $lookup: {
      from: "products",
      localField: "top_products.product_id",
      foreignField: "_id",
      as: "product"
  }},
  // ... ещё несколько стадий
]);
// Многословно, неоптимально, сложно отлаживать.
```

**Вывод:** для сложной аналитики (отчёты, BI, ETL) SQL остаётся вне конкуренции. NoSQL хорош для простых CRUD-операций и специфичных паттернов (графовые обходы, полнотекстовый поиск, timeseries).

### 9.7. Полиглотное хранение (Polyglot Persistence)

**Идея:** использовать разные типы БД для разных частей приложения, а не пытаться всё запихнуть в одну «универсальную» базу.

```
Архитектура e-commerce платформы:

┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│  ┌───────────────┐    ┌──────────────────┐                      │
│  │ PostgreSQL    │    │ MongoDB          │                      │
│  │ - Заказы      │    │ - Каталог товаров│                      │
│  │ - Платежи     │    │ - Корзина        │                      │
│  │ - Финансы     │    │ - Профили        │                      │
│  └───────┬───────┘    └────────┬─────────┘                      │
│          │                     │                                │
│  ┌───────┴───────┐    ┌───────┴──────────┐                      │
│  │ Redis         │    │ Elasticsearch    │                      │
│  │ - Сессии      │    │ - Поиск товаров  │                      │
│  │ - Кэш         │    │ - Автодополнение │                      │
│  │ - Rate limit  │    │ - Фасеты/фильтры │                      │
│  └───────┬───────┘    └────────┬─────────┘                      │
│          │                     │                                │
│  ┌───────┴───────┐    ┌───────┴──────────┐                      │
│  │ Neo4j         │    │ Cassandra/       │                      │
│  │ - Рекомендации│    │ TimescaleDB      │                      │
│  │ - Соц. граф   │    │ - Логи кликов    │                      │
│  └───────────────┘    │ - Аналитика      │                      │
│                       └──────────────────┘                      │
└─────────────────────────────────────────────────────────────────┘
```

**Правила полиглотного хранения:**
1. Для каждой подсистемы — своя БД, оптимальная под её нагрузку.
2. Данные ОБЯЗАТЕЛЬНО дублируются (и это нормально!). Например, товар есть в PostgreSQL и в Elasticsearch, и в MongoDB.
3. Для синхронизации использовать Change Data Capture (Debezium, Kafka Connect), а не двойную запись в коде.
4. Не переусердствовать: 2–3 типа БД — норма, 5+ — сигнал пересмотреть архитектуру.

### 10.1. Дерево принятия решений

```
│   ├─ Нужна мобильная синхронизация (offline-first)?
│   │   └─ ДА → CouchDB + PouchDB / Couchbase Mobile
│   └─ Нужна максимальная производительность на простых операциях?

│   ├─ Нужна SQL-совместимость?

│   ├─ Нужна максимальная скорость записи (миллионы точек/сек)?

└─ Данные — широкие колонки, нужна гибкая схема и настраиваемая
```

### 10.3. Типичные сценарии использования

| Сценарий | Рекомендуемая БД | Почему |
|----------|-----------------|--------|
| **Веб-приложение (CRUD, пользователи, контент)** | MongoDB | Гибкая схема, документная модель близка к объектам в коде, Aggregation Pipeline для отчётов. |
| **Кэш / сессии / rate limiting** | Redis | In-memory, атомарные операции, TTL, pub/sub для инвалидации. |
| **Поиск по документам/товарам** | Elasticsearch | Полнотекстовый поиск, BM25, фасеты, suggest, русская морфология (плагины). |
| **Социальная сеть (граф друзей)** | Neo4j | Обходы графа за O(degree), рекомендации (collaborative filtering), кратчайшие пути. |
| **IoT / логи / временные ряды** | Cassandra / ScyllaDB | Оптимизирована для записи, временные ряды (TWCS), горизонтальное масштабирование. |
| **Аналитика / BI / отчёты** | PostgreSQL + ClickHouse | SQL, оконные функции, CTE, columnar storage для аналитики. |
| **Финансовые транзакции** | PostgreSQL (основные) + MongoDB (каталог) | ACID, строгая согласованность, аудит. MongoDB — для гибких структур продуктов. |
| **Корзина интернет-магазина** | Redis + MongoDB | Redis для быстрой записи/чтения корзины, MongoDB для персистентного хранения. |
| **Offline-first мобильное приложение** | CouchDB + PouchDB | Протокол репликации CouchDB решает конфликты на клиенте и сервере. |
| **Рекомендательная система** | Neo4j + Elasticsearch | Neo4j для коллаборативной фильтрации на графе, Elasticsearch для поиска похожих товаров. |

### 10.4. Антипаттерны: когда NoSQL не нужен

1. **«Используем MongoDB, потому что это модно»** — если данные реляционные (заказы, платежи, пользователи, товары с чёткими связями), SQL-база сделает это лучше.

2. **«MongoDB вместо PostgreSQL для всего»** — когда нужны JOIN'ы, транзакции и отчёты, MongoDB потребует гору Aggregation Pipeline кода, который будет медленным и сложным.

3. **«Elasticsearch как основная БД»** — ES не предназначена быть source of truth: нет транзакций, данные могут теряться при сбоях (translog не даёт гарантий ACID), индексация асинхронная (near real-time, а не real-time).

4. **«Cassandra для сложных отчётов»** — CQL не поддерживает JOIN, GROUP BY ограничен, ORDER BY только по clustering key. Для аналитики — Spark SQL поверх Cassandra.

5. **«Redis как персистентное хранилище всего»** — Redis в первую очередь in-memory, объём данных ограничен RAM. RDB/AOF не дают гарантий ACID. Для персистентного KV — лучше DynamoDB или RocksDB.

6. **«Одна база данных на все случаи жизни»** — полиглотное хранение не враг, а инструмент. Не надо бояться использовать PostgreSQL + Redis + Elasticsearch вместе.

```javascript
db.users.find({ name: "Ivan" }).explain("queryPlanner")  // только план
db.users.find({ name: "Ivan" }).explain("executionStats") // + статистика
db.users.find({ name: "Ivan" }).explain("allPlansExecution") // все планы-кандидаты

// Ключевые метрики:
{
  "executionStats": {
    "executionSuccess": true,
    "nReturned": 1,               // сколько документов возвращено
    "executionTimeMillis": 0,      // время выполнения
    "totalKeysExamined": 1,        // ключей индекса просмотрено
    "totalDocsExamined": 1,        // документов просмотрено
    "executionStages": {
      "stage": "IXSCAN",           // COLLSCAN = плохо, IXSCAN = хорошо
      "indexName": "name_1",
      ...
    }
  }
}
```

**Оптимизация индексов MongoDB:**

```javascript
// 1. Удалить неиспользуемые индексы (занимают RAM и замедляют запись)
db.collection.aggregate([{ $indexStats: {} }])
// → смотрит, какие индексы не используются (accesses.ops = 0)

// 2. Покрывающие индексы (Covered Queries) — все поля запроса в индексе
db.users.find(
  { name: "Ivan" },
  { _id: 0, name: 1, email: 1 }
).explain("executionStats")
// Если stage = "PROJECTION_COVERED" — данные взяты из индекса, без обращения к документам.

// 3. Индекс для сортировки (избежать SORT stage)
db.orders.find({ user_id: 42 }).sort({ created_at: -1 })
// Индекс: { user_id: 1, created_at: -1 } — и фильтрация, и сортировка из индекса.
```

### 11.2. Оптимизация Cassandra: ключевые параметры

**1. Правильный выбор Partition Key:**

```sql
-- Плохо: все данные в одной партиции → hot partition
-- sensor_id = 'single' для всех записей
CREATE TABLE events (
    sensor_id TEXT,
    time TIMESTAMP,
    value DOUBLE,
    PRIMARY KEY (sensor_id, time)
);

-- Хорошо: распределение по bucket'ам
CREATE TABLE events (
    sensor_id TEXT,
    day TEXT,               -- '2024-01-15'
    time TIMESTAMP,
    value DOUBLE,
    PRIMARY KEY ((sensor_id, day), time)
);
```

**2. Compaction Strategy:**

```sql
-- Для временных рядов (TTL, данные по дням)
CREATE TABLE events (...) WITH
    compaction = {
        'class': 'TimeWindowCompactionStrategy',
        'compaction_window_unit': 'DAYS',
        'compaction_window_size': 1
    };
```

**3. Кэширование в Cassandra:**

- **Key Cache:** кэширует местоположение partition key → SSTable (включён по умолчанию, весь в памяти).
- **Row Cache:** кэширует часто читаемые строки (отключён по умолчанию, лучше не включать — Row Cache инвалидируется при каждой записи).
- **File Cache (OS Page Cache):** Cassandra полагается на page cache ОС для кэширования SSTable'ов.

**4. Мониторинг метрик:**

```
nodetool tpstats   — статистика thread pool'ов (dropped mutations = плохо)
nodetool cfstats   — статистика таблиц (SSTable count, read latency)
nodetool compactionstats — прогресс compaction
nodetool netstats   — статистика сети
nodetool tablestats — детальная статистика по таблицам
```

### 11.3. Redis: пайплайны и Lua для производительности

**Pipelining (конвейер команд):**

```
Без пайплайна:
  Client → SET key1 val1
  Client ← OK
  Client → SET key2 val2
  Client ← OK
  Client → SET key3 val3
  Client ← OK
  Время: 3 RTT (Round Trip Time)

С пайплайном:
  Client → SET key1 val1
           SET key2 val2
           SET key3 val3
  Client ← OK
           OK
           OK
  Время: 1 RTT → ускорение в 3 раза
```

```python
import redis
r = redis.Redis()

# Без пайплайна: медленно
for i in range(1000):
    r.set(f"key:{i}", f"value:{i}")

# С пайплайном: быстро
pipe = r.pipeline()
for i in range(1000):
    pipe.set(f"key:{i}", f"value:{i}")
pipe.execute()
```

**Lua-скрипты для производительности:**

```lua
-- Атомарный скрипт, который делает несколько операций на серверной стороне
-- без передачи данных туда-обратно
EVAL "
  local user_key = KEYS[1]
  local session_key = KEYS[2]
  local user_data = ARGV[1]

  redis.call('SET', user_key, user_data)
  redis.call('EXPIRE', user_key, 3600)
  redis.call('SADD', session_key, user_key)
  return 'OK'
" 2 user:123 session:abc "{json data}"
```

**Советы по производительности Redis:**
- Использовать `HSCAN` / `SSCAN` / `ZSCAN` вместо `HGETALL` / `SMEMBERS` / `ZRANGE` для больших коллекций (избежать блокировки).
- `KEYS *` — **запрещено на production** (блокирует event loop на время выполнения). Использовать `SCAN`.
- Мониторить `slowlog` для выявления медленных команд.
- `RENAME` на больших ключах может быть долгим — лучше `COPY` + `DEL`.

### 11.4. Общие практики оптимизации NoSQL

**1. Денормализация:**

Хранить данные так, как они запрашиваются, а не так, как они логически выглядят.

```
Принцип: 1 read = 1 disk seek.

Пример:
  Вместо:
    Collection users: { _id: 1, name: "Ivan" }
    Collection orders: { _id: 100, user_id: 1, ... }  ← отдельный запрос

  Лучше:
    Collection users: {
      _id: 1,
      name: "Ivan",
      recent_orders: [...]  ← встроенные заказы (первые 10)
    }
```

**2. Предварительное вычисление (Pre-aggregation):**

Хранить агрегированные значения вместо вычисления на лету.

```
Пример:
  Вместо: COUNT(*) WHERE user_id = ? (каждый раз)
  Лучше: countOrders в документе пользователя, обновлять атомарно ($inc).
```

**3. Атомарные операции для счётчиков:**

```javascript
// MongoDB: $inc, $push, $addToSet, $pull
db.users.updateOne({ _id: 1 }, { $inc: { loginCount: 1 } })

// Redis: INCR / DECR / HINCRBY
INCR user:1:logins

// Cassandra: counter columns (специальный тип данных)
UPDATE page_views SET count = count + 1 WHERE page_id = ?;
```

**4. TTL (Time-To-Live) для автоочистки:**

```javascript
// MongoDB: TTL Index
db.sessions.createIndex({ createdAt: 1 }, { expireAfterSeconds: 3600 })

// Redis: EXPIRE
SET session:abc123 "data" EX 3600

// Cassandra: TTL на уровне колонки
INSERT INTO events (id, data) VALUES (?, ?) USING TTL 86400;
```

**5. Избегать дорогих операций:**
- Full collection scan (COLLSCAN в MongoDB, ALLOW FILTERING в Cassandra).
- Неиндексированные JOIN'ы/$lookup.
- Сортировка без индекса (in-memory sort на больших выборках).
- DISTINCT на неиндексированных полях.

## Часть XII. Бэкап и восстановление

Бэкап в NoSQL сложнее, чем в РСУБД, из-за распределённой природы данных, eventual consistency и больших объёмов. Стратегии бэкапа различаются для каждой системы.

### 12.1. MongoDB: mongodump и файловые снапшоты

**1. Логический бэкап (`mongodump` / `mongorestore`):**

```bash
# Создание бэкапа
mongodump --uri="mongodb://localhost:27017" --out=/backup/$(date +%Y%m%d)

# Восстановление
mongorestore --uri="mongodb://localhost:27017" /backup/20260714/
```

| Плюсы | Минусы |
|-------|--------|
| Можно фильтровать коллекции и запросы | Медленно: читает все документы через сеть, сериализует в BSON |
| Бэкап переносим между версиями | Не консистентно на лету (нет снапшота на момент бэкапа) |
| Частичное восстановление | Нагружает primary |

**2. Файловый снапшот (filesystem snapshot) — production-стандарт:**

```
Принцип:
  1. Создать снапшот файловой системы (EBS snapshot в AWS, LVM snapshot в Linux).
  2. Файлы данных: /data/db/ (WiredTiger файлы + journal).
  3. Снапшот + журнал = точка согласованности (point-in-time recovery).

Требования:
  - Снапшот ДОЛЖЕН захватывать journal/ директорию.
  - WiredTiger автоматически восстанавливается при старте,
    проигрывая journal после снапшота.
  - Рекомендуется делать на secondary (не нагружает primary, но
    данные могут отставать от primary на несколько секунд).

Альтернатива:
  - MongoDB Atlas: Continuous Cloud Backup (point-in-time restore).
  - mongodump + oplog: для инкрементального бэкапа
    (mongodump --oplog фиксирует позицию в oplog).
```

**3. Стратегия бэкапа для production:**

| Частота | Метод | Для чего |
|---------|-------|----------|
| Ежедневно | Файловый снапшот (EBS/LVM) | Полное восстановление |
| Ежечасно | Инкрементальный (oplog-based) | Point-in-time recovery |
| Еженедельно | `mongodump` | Аудит, миграция, тестирование |

**4. Проверка бэкапа:** обязательно регулярно разворачивать бэкап в тестовом окружении и проверять целостность данных. Непроверенный бэкап = отсутствие бэкапа.

### 12.2. Cassandra: nodetool snapshot и Medusa

**1. `nodetool snapshot` — встроенный механизм:**

```bash
# Создание снапшота на всех узлах (hard links на SSTable)
nodetool snapshot -t backup_20260714

# Снапшоты хранятся в: /var/lib/cassandra/data/<keyspace>/<table>/snapshots/
# Каждый снапшот — это hard link на существующие SSTable-файлы
# → занимает почти 0 места (только новые записи после снапшота создают новые файлы)

# Удаление снапшота
nodetool clearsnapshot -t backup_20260714

# Восстановление:
# 1. Скопировать SSTable из snapshot/ обратно в директорию таблицы
# 2. nodetool refresh <keyspace> <table>
```

**Особенности Cassandra-бэкапа:**

- Снапшот — это **hard links** на SSTable, а не копии. Мгновенно, не занимает места на диске (до тех пор, пока compaction не удалит оригинальные SSTable).
- Снапшот нужно делать **на каждом узле кластера** (данные распределены).
- Требуется **синхронизация по времени** — снапшоты на всех узлах должны быть сделаны примерно одновременно (в пределах нескольких секунд), иначе восстановленные данные будут неконсистентны.

**2. Medusa — production-инструмент для Cassandra:**

Medusa (от Spotify, open-source) автоматизирует бэкап Cassandra:

```
Возможности:
  - Создаёт снапшоты на всех узлах параллельно.
  - Загружает SSTable напрямую в S3/GCS/Azure Blob (без промежуточного копирования).
  - Версионирование: хранит N последних бэкапов.
  - Инкрементальные бэкапы: только изменившиеся SSTable.
  - Восстановление: скачивает SSTable из S3 на узел, делает refresh.
  - Интеграция с Kubernetes (Cassandra Operator).
```

**3. Стратегия бэкапа для production:**

```
Уровень 1: снапшоты (nodetool + Medusa в S3)
  - Частота: каждые 1-4 часа (зависит от RPO).
  - Хранение: 7-30 дней в S3.

Уровень 2: инкрементальные бэкапы (Medusa)
  - Частота: каждые 15-60 минут.
  - Хранение: 3-7 дней.

Уровень 3: repair (профилактический)
  - nodetool repair каждые gc_grace_seconds/2 дней.
  - Защищает от зомби-данных при восстановлении.
```

### 12.3. Elasticsearch: Snapshot & Restore API

**1. Регистрация репозитория снапшотов:**

```json
// Регистрация S3-репозитория
PUT _snapshot/my_backup_repo
{
  "type": "s3",
  "settings": {
    "bucket": "es-backups",
    "region": "us-east-1",
    "compress": true
  }
}

// Поддерживаемые типы: fs (локальная ФС), s3, gcs, azure, hdfs
```

**2. Создание снапшота:**

```json
// Снапшот конкретных индексов
PUT _snapshot/my_backup_repo/snapshot_20260714
{
  "indices": "logs-2026-*,metrics-*",
  "ignore_unavailable": true,
  "include_global_state": true
}

// Проверка статуса
GET _snapshot/my_backup_repo/snapshot_20260714/_status

// Восстановление
POST _snapshot/my_backup_repo/snapshot_20260714/_restore
{
  "indices": "logs-2026-07-*",
  "rename_pattern": "logs-(.+)",
  "rename_replacement": "restored_logs-$1"
}
```

**3. Особенности:**

- **Инкрементальный по умолчанию:** Elasticsearch копирует только сегменты, которых ещё нет в репозитории. Первый снапшот — полный, последующие — только дельта.
- **Не блокирует запись:** индекс продолжает принимать записи во время снапшота.
- **SLM (Snapshot Lifecycle Management):** автоматические снапшоты по расписанию (аналог cron для бэкапов).

```json
// SLM-политика: снапшот каждый час, хранить 30 дней
PUT _slm/policy/hourly_snapshots
{
  "schedule": "0 0 * * * ?",
  "name": "<hourly-{now/d}>",
  "repository": "my_backup_repo",
  "config": {
    "indices": ["*"],
    "include_global_state": false
  },
  "retention": {
    "expire_after": "30d",
    "min_count": 5,
    "max_count": 720
  }
}
```

**4. Стратегия бэкапа для production:**

| Частота | Метод | Хранение |
|---------|-------|----------|
| Каждый час | SLM инкрементальный снапшот в S3 | 30 дней |
| Ежедневно | Полный снапшот (force merge перед снапшотом) | 90 дней |
| Ежемесячно | Архивирование в Glacier (через S3 lifecycle policy) | 1 год |

### 12.4. Redis: RDB/AOF и BGSAVE

**1. RDB-снапшот (BGSAVE):**

```bash
# Создание снапшота в фоне (не блокирует)
redis-cli BGSAVE
# Файл: dump.rdb

# Автоматический снапшот по условию (redis.conf):
save 900 1      # через 900 сек, если изменился ≥ 1 ключ
save 300 10     # через 300 сек, если изменилось ≥ 10 ключей
save 60 10000   # через 60 сек, если изменилось ≥ 10000 ключей
```

**2. AOF (Append-Only File) как бэкап:**

AOF — это журнал всех команд записи. Может использоваться как непрерывный бэкап:

```bash
# Копирование AOF-файла
cp /var/lib/redis/appendonly.aof /backup/
# При восстановлении: Redis проигрывает AOF и восстанавливает состояние.
```

**3. Стратегия бэкапа для production:**

```
Вариант A (простой):
  - BGSAVE каждые 15 минут.
  - Копировать dump.rdb в S3/NFS.
  - Хранить 7 последних RDB-файлов.

Вариант B (минимальные потери):
  - AOF с fsync everysec.
  - RDB каждые 1-6 часов.
  - Копировать RDB + инкремент AOF после последнего RDB.
  - Point-in-time recovery: проиграть AOF после RDB.

Вариант C (облачный):
  - Redis Cloud / ElastiCache: автоматические снапшоты.
  - Redis Cluster: BGSAVE на каждом мастере + копирование в S3.
```

**Важно:** Redis-бэкап из RAM на диск требует `fork()` — процесс копирует все данные (Copy-on-Write). На больших инстансах (50+ ГБ RAM) `BGSAVE` может занять минуты и потреблять дополнительную память на COW-страницы.

**4. Восстановление:**

```bash
# Остановить Redis
# Скопировать dump.rdb или appendonly.aof в /var/lib/redis/
# Запустить Redis — он автоматически загрузит данные.
```

### 12.5. Neo4j: neo4j-admin backup/dump

**1. `neo4j-admin database dump` (логический бэкап):**

```bash
# Создание дампа
neo4j-admin database dump --to-path=/backup/ neo4j

# Восстановление
neo4j-admin database load --from-path=/backup/neo4j.dump neo4j --force
```

- Работает на **остановленной** базе (offline) в Community Edition.
- Enterprise Edition: **online backup** через `neo4j-admin database backup` (не блокирует запись).

**2. Neo4j Enterprise Online Backup:**

```bash
# Инкрементальный бэкап (копирует только изменения с последнего бэкапа)
neo4j-admin database backup --from=<server_address> --to-path=/backup/ neo4j

# Может работать по расписанию как cron-задача:
# 0 */4 * * * neo4j-admin database backup --from=neo4j://primary:7687 --to-path=/backup/ neo4j
```

**3. Файловый снапшот:**

```
Аналогично MongoDB: снапшот ФС директорий с данными Neo4j:
  data/databases/neo4j/    (основные store files)
  data/transactions/neo4j/  (transaction logs)

При восстановлении Neo4j проигрывает transaction logs и
восстанавливает консистентное состояние.
```

**4. Стратегия бэкапа для production:**

| Частота | Метод | Для чего |
|---------|-------|----------|
| Каждые 4-6 часов | Online backup (Enterprise) / `neo4j-admin dump` | Защита от сбоев |
| Ежедневно | Полный файловый снапшот (store files + tx logs) | Point-in-time recovery |
| Еженедельно | Тестовое восстановление | Верификация бэкапа |

### 12.6. Общие принципы бэкапа в NoSQL

1. **Бэкап без верификации — не бэкап.** Регулярно разворачивайте бэкап в тестовом окружении и проверяйте целостность данных.

2. **Распределённый бэкап — распределённая ответственность.** В Cassandra, MongoDB (шардированный), Elasticsearch — бэкап нужно делать со всех шардов/узлов, а не с одного.

3. **RPO и RTO:**
   - **RPO (Recovery Point Objective):** сколько данных вы готовы потерять (определяет частоту бэкапа).
   - **RTO (Recovery Time Objective):** сколько времени вы готовы ждать восстановления (определяет метод бэкапа: файловый снапшот восстанавливается быстрее логического).

4. **Храните бэкапы вне кластера:** S3/GCS/Azure Blob/NFS — не на тех же дисках, что и сама БД.

5. **Шифруйте бэкапы:** особенно если храните в облаке. MongoDB Enterprise, Elasticsearch (плагин), Cassandra (Medusa) поддерживают шифрование бэкапов.

6. **TTL на бэкапы:** автоматически удаляйте старые бэкапы, чтобы не платить за хранение бесконечно.

## Заключение

NoSQL — это не замена SQL, а расширение арсенала инструментов разработчика. Понимание того, как устроены разные типы NoSQL-систем изнутри, позволяет принимать осознанные архитектурные решения.

**Ключевые выводы:**

1. **CAP — это реальность.** В распределённой системе вы всегда выбираете между консистентностью и доступностью в момент сетевого разделения. Не существует системы, которая даёт и то, и другое при Partition.

2. **BASE не хуже ACID — он про другое.** Eventual consistency — инженерный компромисс, позволяющий масштабироваться горизонтально. Для многих бизнес-задач задержка в 100 мс между записью и видимостью данных абсолютно приемлема.

3. **Транзакции в NoSQL возможны, но с оговорками.** MongoDB даёт multi-document ACID, но с ограничением по времени и влиянием на производительность. Cassandra даёт LWT через Paxos, но это медленно. Графовые БД (Neo4j) дают полноценный ACID, но плохо шардируются.

4. **Нет «лучшей» NoSQL-базы — есть правильная под задачу.** Key-Value для кэша, Document для каталога, Wide-Column для IoT, Graph для соцсети, Search для поиска. Полиглотное хранение — это норма.

5. **Shard Key — решение навсегда.** Выбор ключа шардирования определяет масштабируемость системы на годы вперёд. Range для удобных запросов, Hash для равномерности, Compound для целевых запросов.

6. **Индексы в NoSQL — отдельная наука.** От B-Tree в MongoDB до инвертированных индексов в Elasticsearch, от primary key в Cassandra до нативных графовых указателей в Neo4j. Без понимания индексов NoSQL работает медленно.

7. **Денормализация — друг, а не враг.** В мире NoSQL дублирование данных осознанно и контролируемо. Плата за дублирование — более быстрые чтения, плата за нормализацию — JOIN'ы, которые не масштабируются.

8. **Мониторинг и наблюдаемость критичны.** Без понимания того, что происходит внутри распределённого NoSQL-кластера (compaction в Cassandra, chunk migration в MongoDB, shard balance в Elasticsearch), вы будете реагировать на инциденты, а не предотвращать их.

9. **Будущее — за multi-model и конвергенцией.** PostgreSQL с JSONB даёт документную модель, MongoDB с ACID даёт транзакции, Redis с модулями даёт поиск и графы. Границы между типами БД размываются.

10. **Выбирайте базу данных под access patterns, а не под моду.** Проанализируйте, как ваше приложение читает и пишет данные, какие гарантии вам реально нужны, какой объём данных ожидается — и только потом выбирайте инструмент.
