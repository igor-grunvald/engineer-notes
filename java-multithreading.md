## 1. Основы потоков

### 1.1. Что такое поток (thread) в Java?

Поток — это «рабочий» внутри вашей программы. У каждого рабочего свой список задач (стек вызовов), но все они работают в одном цеху (куча/память процесса) и имеют доступ к общим материалам. JVM позволяет запустить множество таких рабочих одновременно.

**Что есть у каждого потока:**
- **Собственный стек (stack)** — изолированная область памяти для локальных переменных и стека вызовов. По умолчанию размер стека ~1 МБ (platform thread), настраивается через `-Xss`.
- **Program Counter (PC)** — указатель на текущую выполняемую инструкцию.
- **Собственный набор регистров** (на уровне ОС).
- **ThreadLocal-переменные** — изолированные копии данных.

**Что разделяют потоки одного процесса:**
- **Кучу (heap)** — все объекты, созданные через `new`.
- **Статические поля классов**.
- **Файловые дескрипторы** (открытые файлы, сокеты).
- **Metaspace** (метаданные классов) — начиная с Java 8.

**Как thread mapping работает на уровне ОС:**
В HotSpot JVM каждый `java.lang.Thread` отображается 1-в-1 на нативный поток ОС (pthread в Linux, CreateThread в Windows). Именно поэтому создание потока — дорогая операция: нужно сделать системный вызов, выделить память под стек, зарегистрировать поток в планировщике ОС. Virtual Threads (Java 21+) меняют эту модель — тысячи виртуальных потоков разделяют небольшой пул платформенных.

### 1.2. Какие два основных способа создания потока в Java?

1. **Наследование от класса `Thread`** и переопределение метода `run()`.
2. **Реализация интерфейса `Runnable`** и передача экземпляра в конструктор `Thread`.

Также с Java 8+ можно использовать лямбда-выражения: `new Thread(() -> { ... }).start()`.

```java
// Способ 1: наследование от Thread
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("Поток " + getName() + " работает");
    }
}
new MyThread().start();

// Способ 2: реализация Runnable
class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Задача выполняется в " + Thread.currentThread().getName());
    }
}
new Thread(new MyTask()).start();

// Способ 2 (Java 8+): лямбда
new Thread(() -> System.out.println("Лямбда в потоке")).start();
```

> **Важно**: `Callable` и `CompletableFuture` не являются способами создания потока напрямую — они используются с `ExecutorService` для выполнения задач в пуле потоков.

### 1.3. Почему предпочтительнее реализовывать Runnable, а не наследовать Thread?

- **Избегание ограничения наследования**: Java не поддерживает множественное наследование. Если класс уже наследует другой класс, он не может наследовать `Thread`, но может реализовать `Runnable`.
- **Разделение логики**: `Runnable` отделяет задачу (логику `run()`) от механизма выполнения (поток `Thread`), что соответствует принципу единственной ответственности (SRP).
- **Совместимость**: Многие API в Java, такие как `ExecutorService`, предназначены для работы с `Runnable` и `Callable`.

### 1.4. В чем разница между методами start() и run()?

**`start()`** — метод класса `Thread`. Запускает новый поток: JVM создаёт нативный поток ОС, внутри которого вызывает метод `run()`. Вызывать `start()` следует только один раз для одного экземпляра `Thread` — повторный вызов приведёт к `IllegalThreadStateException`.

**`run()`** — метод интерфейса `Runnable` (переопределяется в `Thread`), содержит код, который должен выполняться в потоке. Если вызвать `run()` напрямую (без `start()`), код выполнится в том же самом потоке (например, в `main`), новый поток создан не будет — это обычный синхронный вызов метода.

```java
Thread thread = new Thread(() -> {
    System.out.println("Выполняется в: " + Thread.currentThread().getName());
});

thread.run();  // Выведет: "Выполняется в: main" — всё в main-потоке
thread.start(); // Выведет: "Выполняется в: Thread-0" — новый поток
```

**Что происходит внутри `start()` (упрощённо):**
1. Проверка `threadStatus == 0` (NEW). Если нет → `IllegalThreadStateException`.
2. Переводит статус в `RUNNABLE`.
3. Вызывает нативный метод `start0()` — JNI-вызов, который создаёт поток ОС.
4. Поток ОС вызывает `Thread.run()` уже в своём контексте.

### 1.5. В каких состояниях может находиться поток?

Согласно `Thread.State` (enum):

| Состояние | Описание | Как попадает |
|-----------|----------|-------------|
| `NEW` | Поток создан, но еще не запущен (`start()` не вызывался) | `new Thread()` |
| `RUNNABLE` | Поток готов к выполнению или выполняется (зависит от планировщика ОС) | `start()` |
| `BLOCKED` | Поток заблокирован и ждет освобождения монитора | Попытка войти в `synchronized`, уже занятый другим потоком |
| `WAITING` | Поток ждет сигнала от другого потока (`wait()`, `join()`, `park()`) | `Object.wait()`, `Thread.join()`, `LockSupport.park()` |
| `TIMED_WAITING` | Поток ждет в течение определенного времени (`sleep()`, `wait(timeout)`, `join(timeout)`, `parkNanos()`) | `Thread.sleep()`, `wait(ms)`, `join(ms)` |
| `TERMINATED` | Поток завершил выполнение | Выход из `run()` или необработанное исключение |

```java
// Демонстрация переходов между состояниями
Object lock = new Object();
Thread thread = new Thread(() -> {
    System.out.println("2. RUNNABLE — поток работает");
    
    synchronized (lock) {
        try {
            System.out.println("4. TIMED_WAITING — sleep(1000)");
            Thread.sleep(1000);
            System.out.println("5. WAITING — wait()");
            lock.wait(); // освобождает монитор, ждёт notify
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }
    System.out.println("7. TERMINATED — run() завершён");
});

System.out.println("1. NEW — поток создан");
thread.start();

Thread.sleep(100); // даём потоку войти в synchronized
System.out.println("3. BLOCKED — main ждёт монитор");
synchronized (lock) {
    lock.notify(); // будим thread из wait()
}
thread.join(); // ждём завершения
System.out.println("6. main дождался через join() — теперь точно TERMINATED");
System.out.println("Состояние: " + thread.getState()); // TERMINATED
```

> **Важно**: В спецификации JVM нет отдельного состояния `RUNNING`. Состояние `RUNNABLE` охватывает как готовность к выполнению, так и непосредственное выполнение. Поток может быть `RUNNABLE` даже когда ждёт I/O (например, чтение из сокета) — с точки зрения JVM он «готов к выполнению», просто ОС его приостановила до готовности данных. Это частая путаница: поток, читающий из сокета, в дампе потоков будет показан как `RUNNABLE`, а не `WAITING` или `BLOCKED`.

### 1.6. Что такое демон-поток (daemon thread)?

Это поток, который выполняет задачи в фоновом режиме. JVM завершает работу, когда все **пользовательские (user) потоки** завершены, не дожидаясь завершения демон-потоков. Демон-потоки автоматически завершаются при остановке JVM.

Примеры: сборщик мусора (GC), потоки для фоновой обработки.

### 1.7. Как создать демон-поток?

Вызвать метод `setDaemon(true)` для объекта потока **перед** его запуском:

```java
Thread daemon = new Thread(() -> { /* задача */ });
daemon.setDaemon(true);
daemon.start();
```

> Если вызвать `setDaemon(true)` после `start()`, будет выброшено `IllegalThreadStateException`.

### 1.8. Как получить текущий поток?

Вызвать статический метод `Thread.currentThread()`, который возвращает ссылку на текущий поток.

### 1.9. Можно ли запустить поток более одного раза?

Нет. Если вызвать `start()` на объекте `Thread`, который уже был запущен, будет выброшено исключение `IllegalThreadStateException`.

### 1.10. Что делает метод Thread.join()?

Это механизм синхронизации, который заставляет **текущий поток** (вызывающий `join()`) ждать, пока поток, на котором вызван `join()`, не завершит свое выполнение.

```java
Thread worker = new Thread(() -> {
    System.out.println("Worker начал работу");
    Thread.sleep(2000);
    System.out.println("Worker завершил работу");
});
worker.start();

System.out.println("Main ждёт worker...");
worker.join(); // main блокируется здесь до завершения worker
System.out.println("Main продолжает после worker");
```

**Как join() устроен внутри:**
`join()` реализован через `wait()` на самом объекте `Thread`. Когда поток завершается, JVM вызывает `notifyAll()` на объекте потока, что пробуждает все ожидающие `join()`. Именно поэтому не стоит использовать `wait()`/`notify()` на объектах `Thread` вручную — можно сломать внутренний механизм `join()`.

```java
// Упрощённая реализация join():
public final synchronized void join(long millis) throws InterruptedException {
    if (millis == 0) {
        while (isAlive()) {
            wait(0); // ждёт, пока поток не завершится (notifyAll при смерти)
        }
    }
}
```

Перегруженная версия `join(long millis)` позволяет задать таймаут ожидания.

### 1.11. Как корректно остановить или завершить работу потока (кооперативная отмена)?

В Java нет принудительного механизма остановки потока (методы `stop()`, `suspend()`, `resume()` объявлены устаревшими и опасными). Используется **кооперативная отмена** через флаг прерывания:

1. **Проверка флага прерывания** внутри потока:
   ```java
   while (!Thread.currentThread().isInterrupted()) {
       // работа
   }
   ```
2. **Вызов `interrupt()`** из другого потока для установки флага.
3. **Обработка `InterruptedException`** — при его получении флаг прерывания сбрасывается, и нужно восстановить статус при необходимости:
   ```java
   try {
       Thread.sleep(1000);
   } catch (InterruptedException e) {
       Thread.currentThread().interrupt(); // восстановить флаг
       // завершить работу
   }
   ```

### 1.12. Что происходит, когда interrupt() вызывается у потока, который заблокирован в sleep() или wait()?

Поток немедленно выходит из состояния ожидания и выбрасывает `InterruptedException`, а **флаг прерывания сбрасывается** (устанавливается в `false`).

> **Важно**: Флаг прерывания сбрасывается только при выходе из блокирующих методов, которые бросают `InterruptedException` (`sleep()`, `wait()`, `join()`). Если `interrupt()` вызывается на потоке, который **не находится** в состоянии ожидания, флаг прерывания просто устанавливается в `true`, и поток может проверить его позже через `isInterrupted()` или `interrupted()`.

### 1.13. Какой результат вернет метод Thread.interrupted()?

`Thread.interrupted()` — статический метод, который:
- Возвращает `true`, если текущий поток был прерван.
- **Сбрасывает** (очищает) статус прерывания в `false`.

Это позволяет проверить сигнал и сразу его "погасить". В отличие от `isInterrupted()`, который не сбрасывает флаг.

### 1.14. Каково влияние установки высокого приоритета потока (setPriority())?

Это **предпочтение (hint)** для планировщика ОС, которое может увеличить шанс потока на получение процессорного времени, но **не гарантирует** этого. Реальное поведение зависит от реализации JVM и планировщика ОС. Диапазон приоритетов: от `Thread.MIN_PRIORITY` (1) до `Thread.MAX_PRIORITY` (10), значение по умолчанию — `Thread.NORM_PRIORITY` (5).

**Платформенные различия:**
- **Windows**: использует 7 уровней приоритета — Java-приоритеты 1-10 маппятся на эти 7 уровней (несколько Java-приоритетов могут соответствовать одному уровню ОС).
- **Linux**: планировщик CFS (Completely Fair Scheduler) практически **игнорирует** приоритеты Java-потоков (nice-значения имеют минимальный эффект для non-root процессов). Установка приоритета через `setPriority()` часто не даёт заметного эффекта.
- В большинстве случаев **полагаться на приоритеты для корректности программы нельзя** — это приводит к хрупкому, платформозависимому коду.

**Риски:**
- **Инверсия приоритетов**: высокоприоритетный поток ждёт низкоприоритетный, удерживающий блокировку.
- **Starvation (голодание)**: низкоприоритетные потоки могут никогда не получить CPU, если всегда есть готовые высокоприоритетные.

### 1.15. Что такое Virtual Threads (Java 21+)? Чем отличаются от платформенных потоков?

**Virtual Threads** (виртуальные потоки, Project Loom) — легковесные потоки, управляемые JVM, а не ОС. Введены в Java 21 как финальная фича (JEP 444).

**Ключевые отличия от платформенных (обычных) потоков:**

| Характеристика | Platform Thread | Virtual Thread |
|---|---|---|
| Управление | Планировщик ОС | Планировщик JVM (ForkJoinPool) |
| Память (стек) | ~1 МБ (фиксированный) | Начинается с ~200 байт, растёт динамически в heap |
| Создание | Дорогое (системный вызов) | Дешёвое (~1 мкс, объект в куче) |
| Количество | Ограничено (~10 000) | Миллионы |
| Блокировка | Блокирует поток ОС | Размонтирует (unmount) виртуальный поток с platform-потока |
| Переключение | Контекст ОС (~1-10 мкс) | Внутри JVM (~наносекунды) |

**Как работают:**
- JVM создаёт небольшой пул **platform-потоков** (carrier threads)
- Виртуальные потоки **монтируются (mount)** на platform-потоки только на время выполнения CPU-кода
- Когда виртуальный поток выполняет блокирующую операцию (I/O, sleep, `lock()`), он **размонтируется (unmount)** с platform-потока, освобождая его для других виртуальных потоков
- Планировщик JVM (work-stealing `ForkJoinPool`) управляет монтированием/размонтированием

**Когда использовать:**
- ✅ I/O-bound задачи с высокой конкурентностью (серверы, HTTP-запросы, БД)
- ✅ Когда нужно держать миллион одновременных соединений
- ❌ CPU-bound задачи (не дают преимущества, виртуальный поток не размонтируется)
- ⚠️ **До JDK 24**: код в `synchronized` блоке/методе **pin-ил** виртуальный поток к platform-потоку (не размонтировался). **Начиная с JDK 24 (JEP 491)**: `synchronized` больше не pin-ит виртуальные потоки — JVM может размонтировать виртуальный поток даже внутри synchronized-блока.

**Создание:**
```java
// Способ 1: через Thread.ofVirtual()
Thread vThread = Thread.ofVirtual().start(() -> { /* задача */ });

// Способ 2: Thread.startVirtualThread() — короче (JDK 21+)
Thread vThread2 = Thread.startVirtualThread(() -> { /* задача */ });

// Способ 3: через Executors (рекомендуется для продакшена)
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> { /* задача */ });
}

// Создание platform-потока (явно, для симметрии)
Thread pThread = Thread.ofPlatform().start(() -> { /* задача */ });
```

> **Важно**: `Executors.newVirtualThreadPerTaskExecutor()` создаёт новый виртуальный поток **для каждой задачи** — это не пул! Виртуальные потоки настолько дёшёвые, что их создание per-task является нормой.

## 2. Java Memory Model (JMM)

### 2.1. Что такое JMM? Какие проблемы решает?

**Java Memory Model (JMM)** — это формальная спецификация, которая определяет, как потоки взаимодействуют через общую память. JMM гарантирует, что многопоточные программы ведут себя предсказуемо на разных платформах.

JMM решает три основные проблемы:

#### 1. Видимость (Visibility)

Представьте: два человека (потока) смотрят на общую доску объявлений (память). Один стирает и пишет новое число. Но у каждого на столе лежит стикер (кэш процессора) с «копией» этого числа. Если первый поменял доску, а второй продолжает смотреть на свой стикер — он видит устаревшие данные. Это проблема **видимости** (visibility).

**Конкретный пример проблемы видимости — этот код может работать вечно:**

```java
public class VisibilityProblem {
    private static boolean ready = false;  // Не volatile!
    private static int number = 0;

    public static void main(String[] args) throws InterruptedException {
        Thread reader = new Thread(() -> {
            while (!ready) {  // Может НИКОГДА не увидеть ready = true
                Thread.yield();
            }
            System.out.println("number = " + number); // Может увидеть 0, а не 42!
        });
        reader.start();

        number = 42;
        ready = true;  // Без volatile/синхронизации — нет гарантии видимости

        reader.join();
    }
}
```

Почему так происходит: JIT-компилятор может закэшировать `ready` в регистре или переупорядочить запись `number = 42` и `ready = true`. Поток-читатель может увидеть `ready == true`, но `number == 0`, или вообще никогда не увидеть `ready == true`.

JMM решает её через отношение **Happens-Before** и механизмы `synchronized`/`volatile`: после выхода из `synchronized`-блока или записи в `volatile` все «стикеры» обновляются с доски.

#### 2. Переупорядочивание (Reordering)

Компилятор и процессор могут изменять порядок выполнения инструкций для оптимизации производительности, пока это не меняет результат в однопоточной программе. В многопоточной среде такое переупорядочивание может нарушить логику.

Отношение **Happens-Before** накладывает **барьеры памяти (Memory Barriers)**, которые запрещают переупорядочивание операций, пересекающих границы синхронизированных блоков или операций с `volatile` переменными.

#### 3. Атомарность 64-битных типов (Atomicity)

Чтение и запись 64-битных примитивов (`long` и `double`) **не гарантируется атомарной** без `volatile`. Запись может быть разбита на две 32-битные операции, и другой поток может прочитать "половину" старого и "половину" нового значения.

JMM требует, чтобы при использовании `volatile` чтение и запись `long` и `double` становились атомарными. Для 32-битных примитивов и ссылок атомарность гарантируется по умолчанию.

#### Правило Happens-Before

Если действие A находится в отношении Happens-Before с действием B, то результаты A будут видны в B, и A будет выполнено до B в порядке программы. Основные источники Happens-Before:

- Правило монитора: разблокировка монитора happens-before последующая блокировка того же монитора
- Правило volatile: запись volatile-переменной happens-before последующее чтение той же переменной
- Правило потока: `start()` happens-before первое действие запущенного потока; последнее действие потока happens-before возврат из `join()`
- Транзитивность: если A happens-before B и B happens-before C, то A happens-before C

### 2.2. Какое влияние имеет final на потокобезопасность и видимость?

Модификатор `final` предоставляет важные гарантии JMM:

#### 1. Гарантия видимости при публикации (Safe Publication)

**Без `final`**: Если инициализировать обычное поле в конструкторе, JMM не гарантирует, что другой поток увидит присвоенное значение сразу после того, как ссылка на объект станет доступна. Из-за переупорядочивания ссылка может быть опубликована раньше, чем запишутся значения полей.

**С `final`**: JMM гарантирует, что значение `final`-поля будет зафиксировано в памяти **до того**, как ссылка на объект станет доступна другим потокам.

#### 2. Неизменяемость (Immutability)

Объекты, у которых все поля помечены как `final` (и сами поля не являются ссылками на изменяемые объекты), называются **неизменяемыми (immutable)**. Они потокобезопасны без блокировок, так как их состояние не может измениться после создания.

Пример: класс `String` потокобезопасен благодаря тому, что его внутренний массив байтов и другие поля являются `final`.

#### 3. Ограничения

- **This Escape**: Гарантия `final` работает, только если ссылка на объект не "убегает" из конструктора до его завершения (например, передача `this` в слушатели или запуск потока внутри конструктора).
- **Ссылки на изменяемые объекты**: Если `final` поле хранит ссылку на `ArrayList`, сама ссылка не изменится, но содержимое списка можно модифицировать из разных потоков.
- **Reflection**: Через `setAccessible(true)` можно изменить `final`-поле, но это нарушает гарантии JMM.

## 3. Синхронизация (synchronized, volatile, мониторы)

### 3.1. Что такое synchronized?

`Synchronized` — ключевое слово, используемое для синхронизации потоков при доступе к общим ресурсам. Оно обеспечивает **взаимное исключение (mutual exclusion)**, гарантируя, что только один поток может выполнять синхронизированный блок или метод в данный момент времени.

Также `synchronized` гарантирует **видимость** изменений: после выхода из синхронизированного блока все изменения становятся видны другим потокам, входящим в синхронизированный блок на том же мониторе.

### 3.2. Что такое монитор (monitor) в Java?

Монитор — это абстрактный механизм, который инкапсулирует концепцию блокировки (Lock) и условий ожидания (Wait/Notify). Это ключевой элемент, обеспечивающий потокобезопасность и координацию потоков.

Монитор **не является отдельным классом** — каждый объект в Java имеет свой собственный **неявный, встроенный монитор** (intrinsic lock).

### 3.3. Как работает synchronized? Что такое инфляция (inflation) блокировок?

**На высоком уровне:**
1. Когда поток пытается войти в синхронизированный блок/метод, он должен захватить монитор объекта.
2. Если монитор свободен, поток захватывает его и входит в критическую секцию.
3. Если монитор уже захвачен другим потоком, текущий поток блокируется (переходит в состояние `BLOCKED`) и ждёт освобождения.
4. После выполнения кода блокировка автоматически освобождается (даже при возникновении исключения).

**А теперь — как это устроено внутри JVM: инфляция блокировок**

JVM не использует тяжёлые мьютексы ОС с самого начала — это было бы слишком дорого. Вместо этого блокировка проходит через **три стадии «инфляции»**:

| Стадия | Где хранится | Когда используется | Стоимость |
|---|---|---|---|
| **Biased Locking** (смещённая) | Флаги в заголовке объекта (mark word) | Только один поток repeatedly захватывает блокировку | ~0 (почти бесплатно) |
| **Lightweight Locking** (лёгкая) | CAS в mark word + стек вызовов | Несколько потоков, но без реальной конкуренции | ~несколько CAS-операций |
| **Heavyweight Locking** (тяжёлая) | Монитор ОС (pthread_mutex) | Реальная конкуренция — потоки реально ждут | Переключение контекста ОС |

**Как это выглядит на практике:**

```
Поток T1 100 раз захватывает synchronized(obj):
  → Biased Locking. Mark word указывает на T1. Захват "бесплатный".

Пришёл поток T2, тоже хочет synchronized(obj):
  → JVM отзывает смещение (bias revocation). Блокировка становится Lightweight.
  → T2 пытается захватить через CAS. Если T1 не в临界ической секции — успех.
  → Если T1 внутри — T2 крутится в spin-loop (короткое время), потом инфляция.

Конкуренция высокая, несколько потоков ждут:
  → JVM инфлирует блокировку до Heavyweight.
  → Потоки паркуются в очередь к монитору ОС. Переключение контекста.
```

**Важные изменения в JDK:**
- **JDK 15**: Biased Locking **отключён по умолчанию** (JEP 374). Причина: biased locking давал выигрыш только для старых однопоточных коллекций (Vector, Hashtable), а в современном коде overhead от bias revocation перевешивал выгоду.
- **JDK 21**: Biased Locking **полностью удалён** из кодовой базы. Современные synchronized сразу начинают с lightweight.

> **На собеседовании**: Если расскажете про инфляцию блокировок и почему biased locking удалили — это производит впечатление глубокого понимания JVM.

### 3.4. Как называется встроенный в Java механизм блокировки, связанный с synchronized?

**Intrinsic Lock** (Внутренняя блокировка) или **Monitor Lock**.

### 3.5. Какие существуют способы использования synchronized?

#### На уровне метода (Synchronized Method)

```java
public synchronized void method() { // блокировка на this
    // критическая секция
}
```

Блокировка захватывается при входе в метод и автоматически освобождается при его завершении. Пока один поток держит блокировку, никакой другой поток не может войти ни в этот, ни в любой другой `synchronized` метод того же объекта.

#### На уровне блока (Synchronized Block)

```java
public void method() {
    // некритичный код
    synchronized (lockObject) {
        // критическая секция
    }
}
```

Обеспечивает более тонкий контроль (гранулярность), позволяя синхронизировать только критическую секцию, а не весь метод. Это улучшает производительность.

### 3.6. Какой объект используется в качестве монитора при синхронизации метода экземпляра?

`this` — текущий экземпляр класса.

### 3.7. Какой объект используется в качестве монитора при синхронизации статического метода?

Объект `Class` данного класса (`ClassName.class`).

### 3.8. Что происходит, когда поток входит в synchronized метод, который уже занят другим потоком?

Поток немедленно блокируется (переходит в состояние `BLOCKED`). Он не может войти в синхронизированный метод, так как монитор уже удерживается другим потоком. Блокировка длится до освобождения монитора. После освобождения все ожидающие потоки соревнуются за захват монитора (зависит от планировщика JVM).

### 3.9. Для чего нужен volatile?

`Volatile` исключает кэширование значений переменной в локальных кэшах потоков и гарантирует, что:

- Запись в `volatile`-переменную происходит сразу в основную память.
- Чтение `volatile`-переменной происходит всегда из основной памяти.
- Устанавливается отношение **Happens-Before**: запись happens-before последующее чтение.

**Ограничение**: `volatile` не решает проблему неатомарных операций (например, инкремент `i++`).

### 3.10. Почему volatile недостаточно для обеспечения потокобезопасности при i++?

Операция `i++` **не является атомарной** — она состоит из трех шагов:
1. Чтение значения из памяти
2. Модификация (увеличение на 1)
3. Запись результата обратно в память

`Volatile` гарантирует видимость каждого отдельного чтения и записи, но не защищает от **Race Condition** между этими шагами. Два потока могут одновременно прочитать одно и то же значение, увеличить его и записать — результат будет потерян.

Для атомарного инкремента нужно использовать `AtomicInteger`, `synchronized` или `Lock`.

### 3.11. Какая основная проблема volatile с точки зрения производительности?

`Volatile` отключает оптимизации кэширования и переупорядочивания, что приводит к задержкам:

1. **Memory Barrier**: При записи в `volatile` вставляется барьер памяти, процессор сбрасывает кэши в RAM. Чтение/запись в RAM в десятки раз медленнее работы с кэшем L1.
2. **Запрет переупорядочивания**: Процессор не может выполнять инструкции параллельно вокруг `volatile`-поля.
3. **Инвалидация кэшей**: При записи в `volatile` кэш-линии во всех остальных ядрах помечаются как невалидные (протокол MESI), что создает дополнительный трафик на шине данных.

## 4. Ожидание и уведомление (wait/notify, LockSupport)

### 4.1. Что такое wait(), notify() и notifyAll()?

Эти методы определены в классе `Object` и используются для координации потоков:

- **`wait()`**: Заставляет текущий поток освободить блокировку объекта и перейти в состояние `WAITING` до вызова `notify()`/`notifyAll()` на том же объекте или до истечения таймаута (`wait(long timeout)`).
- **`notify()`**: Пробуждает **один** (случайный) поток, ожидающий на мониторе данного объекта. Пробужденный поток не продолжит выполнение, пока вызывающий поток не освободит блокировку.
- **`notifyAll()`**: Пробуждает **все** потоки, ожидающие на мониторе данного объекта. Все они будут конкурировать за блокировку.

### 4.2. В чем разница между wait() и sleep()?

| Характеристика | `wait()` | `sleep()` |
|---|---|---|
| Освобождает блокировки | **Да**, немедленно освобождает монитор | **Нет**, удерживает все блокировки |
| Откуда вызывается | Только из `synchronized` блока | Из любого места |
| Пробуждение | По `notify()`/`notifyAll()` или таймауту | По истечении таймаута или `interrupt()` |
| Назначение | Координация потоков | Пауза выполнения |

### 4.3. Почему wait(), notify() и notifyAll() должны вызываться в synchronized блоке?

1. **Техническое требование**: Если вызвать любой из этих методов вне синхронизированного контекста, будет выброшено `IllegalMonitorStateException`.
2. **Предотвращение Race Condition и потери уведомлений (Lost Notification)**: Это обеспечивает атомарность проверки условия и ожидания. Без `synchronized` возможна ситуация, когда поток проверит условие (например, "очередь пуста"), затем другой поток добавит элемент и вызовет `notify()`, но первый поток еще не начал ожидать — уведомление будет потеряно.

### 4.4. Что за класс LockSupport и для чего он?

`LockSupport` — служебный класс из пакета `java.util.concurrent.locks`, который является фундаментом для создания современных механизмов синхронизации (`ReentrantLock`, `Semaphore`, `CountDownLatch`). Это инструмент для адресной блокировки и разблокировки потоков без использования классических мониторов (`synchronized`).

**Основные методы: `park()` и `unpark()`**

Работа строится вокруг концепции **"бинарного разрешения" (permit)**. У каждого потока есть ровно один такой "виртуальный билет", который может быть либо в наличии, либо отсутствовать.

- **`park()`**: Приостанавливает текущий поток, если у него нет "билета". Если билет уже был выдан ранее (методом `unpark()`), поток поглощает его и продолжает работу без блокировки.
- **`unpark(Thread thread)`**: Выдает "билет" конкретному потоку. Если поток был заблокирован на `park()`, он просыпается. Если поток не был в состоянии ожидания, его следующий вызов `park()` не заблокирует его.

```java
// Демонстрация: unpark ДО park — поток не заблокируется
Thread worker = new Thread(() -> {
    System.out.println("Worker: засыпаю на 1 сек...");
    try { Thread.sleep(1000); } catch (InterruptedException e) {}

    System.out.println("Worker: вызываю park()");
    LockSupport.park();  // Не заблокируется, т.к. permit уже выдан!
    System.out.println("Worker: park() прошёл без блокировки!");
});

worker.start();
System.out.println("Main: выдаю permit заранее");
LockSupport.unpark(worker); // Выдаём permit ДО park() — он сохранится
```

**Преимущества перед wait/notify:**
- Не требует захвата монитора (можно вызывать вне `synchronized`)
- Адресная блокировка: `unpark(thread)` будит конкретный поток, а не случайный
- Не может быть потеряно уведомление (permit накапливается, но не более одного)
- `park()` также реагирует на `interrupt()` — возвращает управление без `InterruptedException` (но флаг прерывания устанавливается)

> **Важно**: `unpark()` накапливает **только один permit**. Повторный вызов `unpark()` до `park()` даёт лишь один permit — следующий `park()` не заблокирует поток, но второй `park()` подряд уже заблокирует.

## 5. java.util.concurrent.locks (Lock, Condition, ReadWriteLock)

### 5.1. Что такое java.util.concurrent.locks.Lock? Какие есть реализации?

`Lock` — это интерфейс, представляющий высокоуровневый, гибкий механизм блокировки, альтернативу `synchronized`.

**Реализации:**

- **`ReentrantLock`** — самая популярная реализация с опцией честной блокировки (fairness). Поддерживает повторный вход (reentrancy).
- **`ReentrantReadWriteLock`** — предоставляет отдельную блокировку для чтения (разрешает множеству читателей одновременно) и эксклюзивную блокировку для записи. Содержит внутренние классы `ReadLock` и `WriteLock`, реализующие `Lock`.
- **`StampedLock`** (Java 8+) — более современная высокопроизводительная блокировка, поддерживающая оптимистичное чтение (optimistic read). Не является реентерабельной.

### 5.2. Чем Lock лучше synchronized?

| Характеристика | `Lock` | `synchronized` |
|---|---|---|
| Гибкость | Явный захват и освобождение (`lock()`/`unlock()`) | Автоматический захват и освобождение |
| Попытка захвата | `tryLock()` — неблокирующая попытка | Нет аналога |
| Прерываемость | `lockInterruptibly()` — можно прервать ожидание | Нельзя прервать ожидание блокировки |
| Condition | Несколько `Condition` на один `Lock` | Только один неявный набор `wait/notify` |
| Справедливость | Можно создать fair-блокировку | Всегда несправедливая |
| Таймаут | `tryLock(timeout, unit)` | Нет аналога |

**Важно**: `unlock()` всегда должен вызываться в блоке `finally` для гарантии освобождения блокировки.

### 5.3. Что такое Condition и как он связан с Lock?

`Condition` — интерфейс, который факторизует механизм ожидания/уведомления из неявного монитора (`synchronized`) в отдельный объект, привязанный к явной блокировке (`Lock`).

**Соответствие методов:**

| Монитор (synchronized) | Lock + Condition |
|---|---|
| `wait()` | `condition.await()` |
| `notify()` | `condition.signal()` |
| `notifyAll()` | `condition.signalAll()` |

**Главное преимущество**: можно создать несколько `Condition` для одной блокировки и адресовать уведомления только потокам, ждущим конкретного условия. Например, в блокирующей очереди можно иметь `notEmpty` (для потребителей) и `notFull` (для производителей).

```java
// Ручная реализация блокирующей очереди на ReentrantLock + Condition
class BoundedBuffer<T> {
    private final T[] items;
    private int putIndex, takeIndex, count;
    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull  = lock.newCondition();  // Для производителей
    private final Condition notEmpty = lock.newCondition();  // Для потребителей

    public BoundedBuffer(int capacity) {
        items = (T[]) new Object[capacity];
    }

    public void put(T item) throws InterruptedException {
        lock.lock();
        try {
            while (count == items.length) {
                notFull.await();  // Ждём: "буфер полон"
            }
            items[putIndex] = item;
            if (++putIndex == items.length) putIndex = 0;
            count++;
            notEmpty.signal();   // Будим одного потребителя: "буфер не пуст"
        } finally {
            lock.unlock();
        }
    }

    public T take() throws InterruptedException {
        lock.lock();
        try {
            while (count == 0) {
                notEmpty.await();  // Ждём: "буфер пуст"
            }
            T item = items[takeIndex];
            if (++takeIndex == items.length) takeIndex = 0;
            count--;
            notFull.signal();   // Будим одного производителя: "буфер не полон"
            return item;
        } finally {
            lock.unlock();
        }
    }
}
```

Этот пример демонстрирует главное преимущество `Condition` — **разделение ожидающих по разным условиям**. `notFull.signal()` будит именно производителя, а не потребителя, который всё равно не сможет работать. С `wait()/notify()` пришлось бы использовать `notifyAll()` и будить всех, создавая лишнюю конкуренцию.

### 5.4. Что такое ReentrantLock? Что означает "reentrant"?

`ReentrantLock` — это класс из пакета `java.util.concurrent.locks`, реализующий интерфейс `Lock` и предоставляющий более гибкий и мощный механизм синхронизации по сравнению со встроенным `synchronized`.

**Reentrant** (рус. "повторно входимый" или "реентрантный") означает, что один и тот же поток может неоднократно захватывать (или "повторно входить") в блокировку, которую он уже удерживает, без возникновения самоблокировки (deadlock).

**Как работает реентрантность:**
1. Когда поток впервые вызывает `lock()`, он захватывает блокировку. Счетчик владения (hold count) увеличивается до 1.
2. Если тот же поток вызывает `lock()` повторно до освобождения, блокировка не блокируется — счетчик владения просто увеличивается (до 2, 3 и т.д.).
3. Поток должен вызвать `unlock()` столько же раз, сколько он вызвал `lock()`.
4. Только когда счетчик владения снова достигнет нуля, блокировка будет полностью освобождена, и ее сможет захватить другой поток.

```java
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    // критическая секция
    lock.lock(); // повторный захват — разрешен
    try {
        // вложенная критическая секция
    } finally {
        lock.unlock();
    }
} finally {
    lock.unlock();
}
```

### 5.5. Расскажите про ReentrantReadWriteLock?

`ReentrantReadWriteLock` — это реализация интерфейса `ReadWriteLock`, которая предоставляет более высокий уровень параллелизма по сравнению с обычным `ReentrantLock` или `synchronized`. Используется для синхронизации доступа к общим данным, когда операции чтения происходят гораздо чаще, чем операции записи (read-heavy scenarios).

Поддерживает два типа блокировок:

- **Блокировка Чтения (ReadLock)**: Может быть получена одновременно несколькими потоками (Shared Lock), если в данный момент нет активного потока, удерживающего блокировку записи. Потоки-читатели не блокируют друг друга.
- **Блокировка Записи (WriteLock)**: Является исключительной (Exclusive Lock). В каждый момент времени может быть получена только одним потоком. Блокирует как других писателей, так и всех читателей.

**Реентрантность**: поток, удерживающий блокировку записи, может запросить ее снова, а поток-читатель может запросить блокировку чтения снова. Поток, удерживающий блокировку записи, может также получить и блокировку чтения (понижение блокировки, Lock Downgrading).

**Fairness (честность)**: Поддерживает два режима:
- **Нечестный (non-fair, по умолчанию)**: Поток может "вклиниться" перед ожидающими — выше производительность, но возможна несправедливость.
- **Честный (fair)**: Потоки получают блокировку в порядке очереди (FIFO) — ниже производительность, но предсказуемое поведение.

```java
ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
Lock readLock = rwLock.readLock();
Lock writeLock = rwLock.writeLock();

// Чтение
readLock.lock();
try {
    // множество читателей одновременно
} finally {
    readLock.unlock();
}

// Запись
writeLock.lock();
try {
    // эксклюзивный доступ
} finally {
    writeLock.unlock();
}
```

### 5.6. Какие недостатки у ReentrantReadWriteLock?

1. **Сложность использования**: требует явного управления двумя разными блокировками (`readLock()` и `writeLock()`) и правильной работы с ними через блоки `try-finally`.
2. **Риск голодания (Starvation)**: Если поток записи ждет, а постоянно приходят новые потоки чтения, то поток записи может "голодать" (никогда не получить доступ). В Java это смягчается: `ReentrantReadWriteLock` обычно отдает приоритет ожидающим писателям.
3. **Накладные расходы**: В сценариях с низкой конкуренцией или когда запись происходит так же часто, как чтение, накладные расходы на управление двумя внутренними блокировками могут сделать его медленнее, чем простой `ReentrantLock`.
4. **Lock downgrading разрешён, upgrade — нет**: Поток может понизить блокировку с writeLock до readLock (lock downgrading), но попытка повысить readLock до writeLock (lock upgrading) приведёт к deadlock'у, так как поток будет ждать сам себя.

### 5.7. Что вы знаете про StampedLock?

`StampedLock` (Java 8+) — это высокопроизводительный механизм синхронизации, предоставляющий более гибкий подход к блокировкам чтения/записи, чем `ReentrantReadWriteLock`. Используется для значительного увеличения параллелизма в read-heavy сценариях.

Вместо возврата объекта блокировки, `StampedLock` возвращает числовое значение — **штамп (stamp)** (`long`). Этот штамп используется для снятия блокировки и проверки достоверности данных. Если штамп равен 0, это означает, что попытка получения блокировки не удалась.

**Три режима блокировки:**

1. **Блокировка Записи (Exclusive/Pessimistic Write)** — `writeLock()`. Исключительный доступ, блокирует всех читателей и других писателей.
2. **Блокировка Чтения (Shared/Pessimistic Read)** — `readLock()`. Общий доступ, позволяет нескольким потокам читать одновременно, блокируя писателей.
3. **Оптимистичное Чтение (Optimistic Read)** — `tryOptimisticRead()`. Самый быстрый и "неблокирующий" режим. Поток не получает явной блокировки; он просто читает данные, получает штамп, а затем проверяет его через `validate(stamp)`. Если `validate()` возвращает `true` — данные считаны корректно. Если `false` — нужно повторить чтение с блокировкой чтения.

> **Важно**: `StampedLock` **не поддерживает реентерабельность**. Повторный захват блокировки тем же потоком может привести к deadlock'у. Это сделано сознательно для повышения производительности.

```java
StampedLock lock = new StampedLock();

// Оптимистичное чтение
long stamp = lock.tryOptimisticRead();
int data = readData();
if (!lock.validate(stamp)) {
    // конфликт — переходим к пессимистичному чтению
    stamp = lock.readLock();
    try {
        data = readData();
    } finally {
        lock.unlockRead(stamp);
    }
}

// Запись
long stamp = lock.writeLock();
try {
    writeData();
} finally {
    lock.unlockWrite(stamp);
}
```

### 5.8. Какие различия между StampedLock и ReentrantReadWriteLock?

| Характеристика | StampedLock | ReentrantReadWriteLock |
|---|---|---|
| Производительность (чтение) | Выше. `tryOptimisticRead` не изменяет внутреннее состояние, дружественен к кэшу CPU. | Ниже. Каждое `readLock()` увеличивает счетчик читателей, вызывая инвалидацию кэша. |
| Режимы доступа | Три (Запись, Чтение, Оптимистичное Чтение) | Два (Запись, Чтение) |
| Реентрантность | Не реентрантная для писателей и читателей. Повторный захват может привести к дедлоку. | Реентрантная для обоих типов блокировок. |
| API | Более сложный, требует работы со штампами (`long`) и ручной валидации. | Простой (возвращает объекты `Lock`). |
| Конверсия | Поддерживает атомарную конверсию (например, из оптимистичного чтения в чтение или из чтения в запись). | Поддерживает понижение (downgrading) из записи в чтение. |

### 5.9. Что такое AbstractQueuedSynchronizer (AQS)? Как он работает?

**AQS** — это сердце почти всех синхронизаторов в `java.util.concurrent`. `ReentrantLock`, `Semaphore`, `CountDownLatch`, `ReentrantReadWriteLock`, `CyclicBarrier` — все они построены на AQS.

Представьте себе **очередь в поликлинике**: люди (потоки) ждут своей очереди к врачу (ресурсу). AQS — это администратор, который ведёт очередь и решает, кто следующий.

**Два основных понятия AQS:**

1. **State** — 32-битное целое число (`volatile int state`), которое значит разное в разных синхронизаторах:
   - `ReentrantLock`: 0 = свободен, >0 = захвачен (счётчик повторных входов)
   - `Semaphore`: количество оставшихся разрешений
   - `CountDownLatch`: текущее значение счётчика

2. **Очередь потоков** — двунаправленный список (CLH-очередь), где каждый узел представляет поток, ожидающий блокировку. Потоки в очереди **не busy-wait-ят** — они паркуются через `LockSupport.park()`.

**Как работает захват блокировки (на примере ReentrantLock):**

```
1. Поток вызывает lock()
2. CAS-ом пытается установить state из 0 в 1
3. Если получилось — блокировка наша, выходим
4. Если нет — проверяем: может, мы уже владелец? (reentrant)
   → Да: state++, выходим
   → Нет: поток добавляется в хвост очереди и паркуется
5. При unlock(): state--, если стало 0 — будим голову очереди
```

**Два режима AQS:**
- **Exclusive** (эксклюзивный): только один поток может владеть. `ReentrantLock`, `ReentrantReadWriteLock.WriteLock`
- **Shared** (разделяемый): несколько потоков могут владеть одновременно. `Semaphore`, `CountDownLatch`, `ReentrantReadWriteLock.ReadLock`

**Какие методы нужно переопределить:**
- `tryAcquire(int arg)` — попытка эксклюзивного захвата
- `tryRelease(int arg)` — попытка эксклюзивного освобождения
- `tryAcquireShared(int arg)` — попытка разделяемого захвата
- `tryReleaseShared(int arg)` — попытка разделяемого освобождения
- `isHeldExclusively()` — удерживается ли блокировка текущим потоком

Сам AQS уже реализует очередь, парковку/распарковку потоков, пробуждение следующего в очереди. Вам нужно только определить, что означает `state` и при каких условиях захват успешен.

> **На собеседовании**: если вас спросят «как устроен ReentrantLock изнутри?» — расскажите про AQS, state, CLH-очередь и CAS. Это показывает глубокое понимание платформы.

**Минимальный пример своего синхронизатора на AQS — BinaryLatch (одноразовый затвор):**

```java
class BinaryLatch extends AbstractQueuedSynchronizer {
    // 0 = закрыт, 1 = открыт
    public BinaryLatch() { setState(0); }

    public void await() throws InterruptedException {
        acquireSharedInterruptibly(1);
    }

    public void release() {
        releaseShared(1);
    }

    @Override
    protected int tryAcquireShared(int arg) {
        // 1 = успех (открыт), -1 = неудача (идём в очередь)
        return getState() == 1 ? 1 : -1;
    }

    @Override
    protected boolean tryReleaseShared(int arg) {
        setState(1);  // Открываем
        return true;  // true = надо разбудить ожидающих
    }
}

// Использование:
BinaryLatch latch = new BinaryLatch();
new Thread(() -> {
    doWork();
    latch.release(); // Открываем затвор
}).start();
latch.await(); // Ждём открытия
```

Всего ~15 строк кода, а за ними — CLH-очередь, парковка через `LockSupport`, пробуждение головы очереди — всё это AQS даёт бесплатно.

## 6. Атомарные классы (java.util.concurrent.atomic)

### 6.1. Какой класс использовать для атомарного инкремента целочисленной переменной?

`AtomicInteger` — обеспечивает атомарные операции над `int` без использования `synchronized` или `Lock`, используя механизм **CAS (Compare-And-Swap)** на уровне процессора.

```java
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet(); // атомарный инкремент
```

### 6.2. Какие группы атомарных классов существуют в Java?

1. **Атомарные скалярные типы**: `AtomicInteger`, `AtomicLong`, `AtomicBoolean`, `AtomicReference<V>`
2. **Атомарные массивы**: `AtomicIntegerArray`, `AtomicLongArray`, `AtomicReferenceArray<E>`
3. **Атомарные обновлятели полей (Field Updaters)**: `AtomicIntegerFieldUpdater<T>`, `AtomicLongFieldUpdater<T>`, `AtomicReferenceFieldUpdater<T, V>` — позволяют атомарно обновлять `volatile` поля объектов через отражение
4. **Атомарные накопители (Accumulators и Adders)**:
   - `LongAdder` / `DoubleAdder` — более производительные, чем `AtomicLong`, при большом количестве потоков (используют striped cells)
   - `LongAccumulator` / `DoubleAccumulator` — обобщенная версия с пользовательской функцией накопления

### 6.3. Как работает compareAndSet (CAS) операция?

**CAS (Compare-And-Swap)** — атомарная инструкция процессора, основа всех неблокирующих (lock-free) алгоритмов в Java. **CAS (Compare-And-Swap)** — это атомарная инструкция, поддерживаемая большинством современных процессоров. Она является основой для создания неблокирующих (lock-free) алгоритмов в Java, в частности, для реализации классов из пакета `java.util.concurrent.atomic`.

Суть простая: «Процессор, запиши в ячейку M значение B, **но только если** сейчас там лежит A. И скажи мне, получилось или нет». Вся операция выполняется аппаратно, одним неделимым действием.

Три операнда:
- **M** — адрес в памяти (какую переменную меняем)
- **A** — ожидаемое значение (что мы думаем, там лежит)
- **B** — новое значение (что хотим записать)

**Как это выглядит в коде и под капотом:**

```java
AtomicInteger counter = new AtomicInteger(0);
counter.compareAndSet(0, 1); // «если там 0, запиши 1»
```

Процессор делает:
1. Атомарно сравнивает `[M] == A`
2. Если да — записывает B в M, возвращает `true`
3. Если нет (другой поток уже поменял) — возвращает `false`, запись не происходит

**Что делать при неудаче:** CAS не блокирует поток. Если `false` — вы сами решаете, что делать. Обычно — повторяете попытку в цикле (spin-loop):

```java
while (!counter.compareAndSet(expected, expected + 1)) {
    expected = counter.get(); // перечитали, другой поток опередил
}
```

Это называется **CAS-loop** или «оптимистичная блокировка»: мы оптимистично предполагаем, что конфликтов не будет, а если будут — просто повторяем. В этом вся суть lock-free программирования.

### 6.4. Какие есть проблемы у CAS операций?

**1. Проблема ABA (The ABA Problem)**

CAS работает, сравнивая текущее значение в памяти (M) с ожидаемым старым значением (A). Проблема возникает, когда:
- Поток 1 читает значение A.
- Поток 2 изменяет значение с A на B, а затем снова возвращает его к A.
- Поток 1 выполняет CAS, видит, что текущее значение все еще A (как и ожидалось), и успешно выполняет запись.

CAS завершается успешно, но промежуточное изменение (A → B → A) было пропущено. В большинстве случаев с простыми счетчиками (например, `AtomicInteger`) это не критично. Однако это может привести к ошибкам в сложных алгоритмах, таких как неблокирующие стеки или очереди.

**Решение**: `AtomicStampedReference` — использует дополнительное "клеймо" или номер версии (stamp). CAS будет успешным, только если совпадают как значение, так и номер версии.

**2. Цикл CAS и Высокая Конкуренция (Spinning & High Contention)**

При высокой конкуренции (когда много потоков пытаются изменить одну и ту же переменную) большинство потоков будут постоянно терпеть неудачу, многократно повторяя цикл CAS. Это приводит к пустой трате ресурсов ЦП (CPU Waste), так как потоки тратят циклы ЦП на бесполезные повторные попытки (spinning) вместо выполнения полезной работы. В условиях высокой конкуренции CAS может оказаться медленнее, чем традиционная блокировка.

**3. Ограниченная Атомарность (Limited Scope)**

CAS может гарантировать атомарность только для **одной переменной** (или одного адресного слова в памяти). Если нужно атомарно изменить два или более несвязанных поля или выполнить последовательность операций, одной инструкции CAS недостаточно. В таких случаях нужно использовать механизм блокировки (`Lock`) или специальные классы (`AtomicMarkableReference`, `AtomicReferenceFieldUpdater`).

### 6.5. Какой метод в Java 9+ позволяет эффективно работать с памятью на низком уровне (аналог CAS)?

**`VarHandle`** (Java 9+) — это класс, предоставляющий атомарный доступ к переменным с различным уровнем барьеров памяти. Он является более гибкой и производительной альтернативой `AtomicIntegerFieldUpdater` и подобным классам.

`VarHandle` позволяет:
- Атомарно читать и записывать поля объектов и массивов
- Выполнять CAS-операции (`compareAndSet()`)
- Выполнять операции get-and-set, get-and-add
- Контролировать уровень барьеров памяти (acquire/release/opaque)

**Режимы доступа к памяти (Memory Ordering Modes):**

| Режим | Гарантии |
|---|---|
| **Plain** (обычный) | Без гарантий — как обычное чтение/запись (компилятор/CPU могут переупорядочить) |
| **Opaque** | Атомарность без гарантий порядка — операция видна другим потокам атомарно, но не накладывает ограничений на порядок других операций |
| **Acquire** (чтение) | Последующие чтения/записи не могут быть переупорядочены ДО этого чтения (load barrier) |
| **Release** (запись) | Предыдущие чтения/записи не могут быть переупорядочены ПОСЛЕ этой записи (store barrier) |
| **Volatile** | Acquire + Release — полный барьер памяти (как `volatile`-переменная) |

```java
VarHandle handle = MethodHandles.lookup()
    .findVarHandle(MyClass.class, "field", int.class);
// Атомарное чтение с acquire-семантикой
int val = (int) handle.getAcquire(obj);
// Атомарная запись с release-семантикой  
handle.setRelease(obj, 42);
```

## 7. ThreadLocal

### 7.1. Какова цель использования ThreadLocal?

`ThreadLocal` позволяет хранить значения, **уникальные для каждого потока**. Каждый поток имеет свою собственную копию переменной, что устраняет необходимость в синхронизации.

**Примеры использования:**
- Хранение ID транзакции или пользователя для всего контекста запроса
- `SimpleDateFormat` (не потокобезопасный) — можно хранить свой экземпляр для каждого потока
- Сохранение состояния соединения с БД

```java
private static final ThreadLocal<SimpleDateFormat> dateFormat =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));
```

**Как устроен внутри — ThreadLocalMap:**

У каждого потока (`Thread`) есть поле `threadLocals` — это `ThreadLocalMap`. Это **не** обычный `HashMap`, а специализированная структура:

- **Ключ** — слабая ссылка (`WeakReference`) на сам объект `ThreadLocal`
- **Значение** — сильная ссылка на хранимый объект
- **Разрешение коллизий** — открытая адресация (линейное пробирование), а не цепочки
- Причина: ожидается, что ThreadLocal-переменных на поток мало, и открытая адресация работает быстрее

**Почему возникают утечки памяти:**

```
Thread → ThreadLocalMap → Entry[] → [WeakRef(ThreadLocal key), StrongRef(value)]
                                               ↑                       ↑
                                          Собирается GC          НЕ собирается GC,
                                          если нет внешней      пока жив поток!
                                          ссылки на ThreadLocal
```

Сценарий утечки:
1. Вы используете `ThreadLocal` в пуле потоков (например, Tomcat)
2. Поток никогда не умирает — возвращается в пул
3. `ThreadLocal`-переменная вышла из скоупа → ключ собран GC
4. Но **значение** осталось в `ThreadLocalMap`, потому что поток жив
5. Каждый запрос добавляет новое значение → память растёт

**Решение**: всегда вызывать `remove()` в `finally`:
```java
try {
    threadLocal.set(value);
    // ... работа ...
} finally {
    threadLocal.remove(); // обязательно!
}
```

**InheritableThreadLocal** — позволяет дочерним потокам наследовать значение от родителя при создании потока. Значение **копируется** (а не разделяется), поэтому изменения в дочернем потоке не видны родителю. **Важно**: не работает с виртуальными потоками — используйте `ScopedValue` вместо него.

> **Важно**: `ThreadLocal` может вызвать утечку памяти в средах с пулом потоков (например, в веб-серверах), если не вызывать `remove()` после использования.

### 7.2. Что такое ScopedValue (Java 21+)? Чем отличается от ThreadLocal?

**ScopedValue** (JEP 446, Java 21) — иммутабельный (неизменяемый) аналог `ThreadLocal`, спроектированный для эффективной работы с виртуальными потоками. Позволяет передавать данные «вниз» по цепочке вызовов без передачи через параметры методов.

**Ключевые отличия от ThreadLocal:**

| Характеристика | ThreadLocal | ScopedValue |
|---|---|---|
| Изменяемость | Изменяемый (mutable) — можно `set()` | Неизменяемый (immutable) — значение задаётся один раз при входе в scope |
| Жизненный цикл | Ручное управление (нужен `remove()`) | Автоматическое — значение живёт, пока активен scope |
| Утечки памяти | Возможны при использовании с пулами потоков | Невозможны (structured concurrency гарантирует очистку) |
| Наследование | `InheritableThreadLocal` (копирует значение) | Автоматически передаётся в дочерние виртуальные потоки |
| Производительность | Каждый `get()` идёт через `ThreadLocalMap` | Оптимизирован JIT-компилятором (может инлайниться) |

**Пример использования:**
```java
// Объявление
private static final ScopedValue<User> CURRENT_USER = ScopedValue.newInstance();

// Использование
ScopedValue.where(CURRENT_USER, loggedInUser)
    .run(() -> {
        // Здесь CURRENT_USER.get() вернёт loggedInUser
        handleRequest();
    });
// Здесь CURRENT_USER уже недоступен
```

**Когда использовать ScopedValue вместо ThreadLocal:**
- Передача контекста запроса (пользователь, транзакция, локаль) «вниз» по стеку
- Работа с виртуальными потоками (ScopedValue — «родная» фича для них)
- Когда нужна гарантия очистки без `try-finally`

**Когда всё ещё нужен ThreadLocal:**
- Когда значение должно изменяться внутри потока (кеш, накопление данных)
- Когда жизненный цикл значения не привязан к лексическому scope

## 8. ThreadGroup

### 8.1. Что такое ThreadGroup?

`ThreadGroup` — класс для организации потоков в группы и управления ими как единым целым. Позволяет:
- Прервать все потоки в группе (`interrupt()`)
- Установить общий максимальный приоритет (`setMaxPriority()`)
- Создавать иерархическую структуру групп

Если поток не привязан к группе, он принадлежит корневой группе, создаваемой автоматически при запуске JVM.

### 8.2. Почему методы suspend() и resume() в ThreadGroup объявлены устаревшими?

**`suspend()`**:
1. Приостанавливает поток, **не освобождая мониторы и блокировки**, что может привести к deadlock'у.
2. Невозможно гарантировать, что поток будет приостановлен в безопасном месте.

**`resume()`**:
1. Устарел вместе с `suspend()`, так как используется только для его отмены.
2. Небезопасен: если `resume()` вызвать до `suspend()`, поток останется приостановленным навсегда.

> **Примечание**: Метод `stop()` также deprecated, но по другим причинам — он принудительно завершает поток в любой момент, даже во время критической операции, оставляя данные в несогласованном состоянии (например, поток остановлен посередине перевода денег между счетами).

### 8.3. Какие проблемы безопасности у ThreadGroup?

- **Ненадежная граница доверия**: ThreadGroup контролирует только взаимодействие между потоками, но не доступ к системным ресурсам.
- **Проблема наследования**: Новый поток наследует ThreadGroup от создателя, что создает неявные связи доверия.
- **Несовершенный контроль привилегий**: Только грубые проверки (может ли поток A прервать поток B).
- **Уязвимость к DoS**: `setMaxPriority()` может использоваться для "замаривания голодом" других потоков.

### 8.4. Почему ThreadGroup устарел?

1. **Проблемы безопасности** — модель безопасности неэффективна.
2. **Нереализованный функционал** — `suspend()`, `resume()`, `stop()` опасны и deprecated.
3. **Не потокобезопасна** — многие методы небезопасны в высококонкурентной среде.
4. **Устаревшая модель** — современные приложения используют `ExecutorService`, `ThreadPoolExecutor`, которые предоставляют более мощный и безопасный способ управления потоками.

## 9. Callable, Future, FutureTask

### 9.1. Что такое Callable?

`Callable<V>` — функциональный интерфейс, аналог `Runnable`, но:
- Возвращает результат типа `V` (метод `call()`)
- Может бросать проверяемые исключения (`throws Exception`)

```java
Callable<Integer> task = () -> {
    Thread.sleep(1000);
    return 42;
};
```

### 9.2. Выбрасывает ли Callable проверяемые исключения? Выбрасывает ли Runnable?

- **`Callable.call()`**: сигнатура включает `throws Exception` — может выбрасывать проверяемые исключения.
- **`Runnable.run()`**: не позволяет выбрасывать проверяемые исключения (их нужно обрабатывать внутри).

### 9.3. Что такое Future?

`Future<V>` — интерфейс, представляющий результат асинхронной операции, который может быть недоступен немедленно. Это "расписка" на получение результата в будущем.

**Основные методы:**

| Метод | Описание |
|---|---|
| `get()` | Блокирующий, ожидает завершения задачи и возвращает результат |
| `get(long timeout, TimeUnit unit)` | Блокирующий с таймаутом, бросает `TimeoutException` |
| `isDone()` | Неблокирующий, проверяет завершение задачи |
| `isCancelled()` | Проверяет, была ли задача отменена |
| `cancel(boolean mayInterruptIfRunning)` | Пытается отменить выполнение задачи |

### 9.4. Что такое FutureTask?

`FutureTask<V>` — конкретный класс, реализующий одновременно `Runnable` и `Future<V>`. Служит мостом между синхронным и асинхронным кодом:
- Как `Runnable` — можно передать в `Thread` или `ExecutorService`
- Как `Future` — позволяет управлять состоянием и получать результат

**Дополнительные методы (protected):**
- `set(V v)` — ручное завершение с результатом
- `setException(Throwable t)` — ручное завершение с ошибкой
- `done()` — обратный вызов при завершении (можно переопределить)

## 10. ExecutorService и пулы потоков

### 10.1. Опишите иерархию исполнителей (Executor, ExecutorService, ScheduledExecutorService).

```
Executor (интерфейс)
  └── execute(Runnable command) — базовый запуск задачи
      │
      └── ExecutorService (интерфейс)
            ├── submit(Callable/Runnable) → Future
            ├── shutdown() / shutdownNow()
            ├── invokeAll() / invokeAny()
            │
            └── ScheduledExecutorService (интерфейс)
                  ├── schedule() — однократная задержка
                  ├── scheduleAtFixedRate() — периодически с фикс. интервалом
                  └── scheduleWithFixedDelay() — периодически с фикс. задержкой
```

### 10.2. Какие основные реализации ExecutorService вы знаете?

1. **`ThreadPoolExecutor`** — стандартная гибкая реализация. Содержит пул потоков, очередь задач, настройки (corePoolSize, maximumPoolSize, keepAliveTime).
2. **`ScheduledThreadPoolExecutor`** — расширяет `ThreadPoolExecutor`, добавляет возможность планирования задач.
3. **`ForkJoinPool`** — специализированный пул для рекурсивных задач (разделяй и властвуй) с work-stealing.

### 10.3. Как создать пул потоков с помощью Executors?

| Метод | Описание |
|---|---|
| `newFixedThreadPool(int n)` | Пул с фиксированным числом потоков. Если все заняты, задачи ждут в очереди. |
| `newCachedThreadPool()` | Создает потоки по мере необходимости, переиспользует свободные. Удаляет простаивающие 60 сек. |
| `newSingleThreadExecutor()` | Пул из одного потока. Гарантирует последовательное выполнение задач. |
| `newScheduledThreadPool(int n)` | Пул для планирования задач с задержкой/периодически. |
| `newWorkStealingPool()` (Java 8+) | Пул с work-stealing (на основе `ForkJoinPool`). parallelism по умолчанию = `Runtime.getRuntime().availableProcessors()`. |

> **Рекомендация**: Используйте конструктор `ThreadPoolExecutor` напрямую или `Executors.newFixedThreadPool()` для явного контроля параметров. `newCachedThreadPool()` может создать неограниченное количество потоков.

### 10.4. В чем разница между shutdown(), shutdownNow() и awaitTermination()?

- **`shutdown()`**: "Мягкая" остановка. Перестает принимать новые задачи, но завершает уже отправленные (включая ожидающие в очереди).
- **`shutdownNow()`**: "Жесткая" остановка. Пытается прервать активные задачи (`interrupt()`) и возвращает список задач, которые не были запущены.
- **`awaitTermination(long timeout, TimeUnit unit)`**: Блокирующий метод. Ждет указанное время, пока все задачи не завершатся после вызова `shutdown()`. Возвращает `true`, если все задачи завершились.

**Правильный паттерн остановки:**
```java
executor.shutdown();
if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
    executor.shutdownNow();
}
```

### 10.5. В чем фундаментальное различие между ThreadGroup и ExecutorService?

- **`ThreadGroup`**: Старый механизм (Java 1.0). Фокусируется на **потоках** как объектах управления. Не предоставляет пулов и управления очередями.
- **`ExecutorService`**: Современный подход. Фокусируется на **задачах**. Отделяет логику выполнения от создания ресурсов, обеспечивает переиспользование потоков и гибкое управление результатами.

### 10.6. Почему рекомендуется использовать ExecutorService вместо new Thread()?

1. **Накладные расходы**: Создание потока ОС — дорогая операция. Пулы переиспользуют потоки.
2. **Управляемость**: Легко ограничить количество потоков, чтобы не уронить систему.
3. **Удобство**: Инструменты для получения результатов (`Future`, `CompletableFuture`) и обработки исключений.
4. **Управление жизненным циклом**: `shutdown()`, `awaitTermination()`.

### 10.7. Какие существуют политики обработки отклонённых задач (RejectedExecutionHandler)?

Когда очередь задач `ThreadPoolExecutor` переполнена (bounded queue + все потоки заняты + достигнут `maximumPoolSize`), новые задачи **отклоняются**. Стратегия отклонения задаётся через `RejectedExecutionHandler`:

| Политика | Поведение |
|---|---|
| **`AbortPolicy`** (по умолчанию) | Бросает `RejectedExecutionException` |
| **`CallerRunsPolicy`** | Выполняет задачу **в вызывающем потоке** (том, который вызвал `submit()`/`execute()`). Это замедляет производителя, создавая естественный backpressure. |
| **`DiscardPolicy`** | Молча отбрасывает задачу **без уведомления**. Опасно — задача теряется бесследно. |
| **`DiscardOldestPolicy`** | Удаляет **самую старую** задачу из очереди и пытается вставить новую. Полезна, когда актуальность данных важнее полноты (например, последнее значение датчика). |

**Пример:**
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4, 8, 60, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(100),
    new ThreadPoolExecutor.CallerRunsPolicy() // backpressure
);
```

> **Рекомендация**: `CallerRunsPolicy` — хороший выбор для большинства сценариев, так как он естественно ограничивает скорость поступления задач, предотвращая `OutOfMemoryError`.

### 10.8. Что такое ForkJoinPool и для каких задач он предназначен?

`ForkJoinPool` — специализированный пул для задач типа **«Разделяй и властвуй» (Divide and Conquer)**. Его главная фишка — **work-stealing (кража работы)**.

**Как устроен изнутри:**

У каждого worker-потока есть **своя двусторонняя очередь (deque)** задач. Это не общая очередь на всех — именно это и даёт прирост производительности.

```
Worker-1: [задача A1, задача A2, задача A3] ← голова ... хвост →
Worker-2: [задача B1, задача B2]           ← голова ... хвост →
Worker-3: [пусто]                           ← нечего делать →
```

**Дисциплина доступа — почему LIFO для своих и FIFO для чужих:**

- **Свои задачи** — поток берёт с **головы** (LIFO, как стек). Почему? Последняя добавленная задача — самая «горячая», её данные ещё в кэше CPU. Это максимизирует cache locality.
- **Кража чужих задач** — поток-вор берёт с **хвоста** чужой очереди (FIFO). Почему? В хвосте лежит самая старая, самая крупная задача — её украсть выгоднее всего (больше работы = меньше краж = меньше накладных расходов).

**Почему Work-Stealing эффективен:**
1. Нет единой очереди → нет contention за одну блокировку
2. LIFO для своих задач → cache-friendly
3. Кража с хвоста → крупные задачи перераспределяются
4. Потоки без работы не блокируются на общей очереди, а «воруют»

**ForkJoinTask** — базовый класс для задач, имеет две специализации:
- `RecursiveAction` — без возврата результата
- `RecursiveTask<V>` — с возвратом результата

**Где используется:**
- `CompletableFuture` по умолчанию
- Параллельные стримы (`parallelStream()`)
- Виртуальные потоки (планировщик — тоже `ForkJoinPool`)

**Пример RecursiveTask — параллельное суммирование массива:**

```java
class SumTask extends RecursiveTask<Long> {
    private static final int THRESHOLD = 10_000;
    private final int[] array;
    private final int low, high;

    SumTask(int[] array, int low, int high) {
        this.array = array;
        this.low = low;
        this.high = high;
    }

    @Override
    protected Long compute() {
        if (high - low <= THRESHOLD) {
            // Задача достаточно мала — решаем последовательно
            long sum = 0;
            for (int i = low; i < high; i++) sum += array[i];
            return sum;
        }
        // Делим пополам
        int mid = (low + high) / 2;
        SumTask left = new SumTask(array, low, mid);
        SumTask right = new SumTask(array, mid, high);

        left.fork();  // Запускаем левую часть асинхронно (в своей deque)
        long rightResult = right.compute(); // Правую считаем в текущем потоке
        long leftResult = left.join();      // Ждём левую

        return leftResult + rightResult;
    }
}

// Использование:
ForkJoinPool pool = new ForkJoinPool();
SumTask task = new SumTask(array, 0, array.length);
long result = pool.invoke(task);
```

**Почему важно `left.fork()` + `right.compute()`, а не `left.fork()` + `right.fork()`:**
- Вызывая `right.compute()` в текущем потоке, мы продолжаем работать, а не блокируемся
- `left.fork()` кладёт задачу в свою deque — её может украсть другой поток (work-stealing)
- Если бы мы сделали `left.fork()` + `right.fork()` + `left.join()` + `right.join()`, текущий поток бы простаивал на `join()`, пока другой поток выполняет обе подзадачи

## 11. Синхронизаторы (CountDownLatch, CyclicBarrier, Semaphore, Exchanger, Phaser)

### 11.1. Что такое CountDownLatch?

`CountDownLatch` — синхронизатор, работающий как счетчик, который уменьшается вызовами `countDown()`. Потоки ждут (`await()`), пока счетчик не станет равен 0.

**Особенности:**
- **Одноразовый**: после достижения 0 не может быть сброшен.
- Используется, когда один или несколько потоков ждут завершения N операций.

```java
CountDownLatch latch = new CountDownLatch(3);
// В трех потоках:
latch.countDown();
// В ожидающем потоке:
latch.await(); // блокируется, пока счетчик не станет 0
```

### 11.2. Что такое CyclicBarrier? Чем отличается от CountDownLatch?

`CyclicBarrier` — точка сбора ("барьер"). Потоки вызывают `await()` и ждут, пока у барьера не соберется фиксированное количество участников. После этого все одновременно продолжают работу.

**Отличия от CountDownLatch:**

| Характеристика | CountDownLatch | CyclicBarrier |
|---|---|---|
| Многоразовость | Одноразовый | Многоразовый (сбрасывается автоматически) |
| Механизм | Счетчик уменьшается | Потоки собираются в точке |
| Действие при срабатывании | Потоки просто продолжают | Можно указать `Runnable` (barrier action) |
| Типичное применение | Ожидание завершения N задач | Синхронизация фаз параллельной работы |

### 11.3. Что такое Semaphore?

`Semaphore` ограничивает количество потоков, имеющих одновременный доступ к ресурсу. Работает через "разрешения" (permits).

```java
Semaphore semaphore = new Semaphore(5); // 5 одновременных доступов
semaphore.acquire(); // захватить разрешение (блокируется, если нет свободных)
// использование ресурса
semaphore.release(); // освободить разрешение
```

**Применение**: ограничение подключений к БД, rate-limiting, пулы объектов.

### 11.4. Что такое Exchanger?

`Exchanger<V>` — синхронизатор, позволяющий **двум потокам** обмениваться объектами в точке синхронизации. Один поток ждет другой, они передают друг другу данные и продолжают работу.

```java
Exchanger<String> exchanger = new Exchanger<>();
// Поток A:
String dataFromB = exchanger.exchange("data from A");
// Поток B:
String dataFromA = exchanger.exchange("data from B");
```

### 11.5. Что такое Phaser? В чем его преимущества?

`Phaser` — гибкий аналог `CyclicBarrier` и `CountDownLatch`.

**Преимущества:**
1. **Динамическое количество участников**: можно регистрировать новые потоки "на лету" (`register()`, `bulkRegister()`).
2. **Поддержка многофазных задач**: после завершения фазы барьер обновляется для следующей.
3. **Termination**: фазы можно завершать досрочно (`forceTermination()`).
4. **Кастомное поведение**: можно переопределить метод `onAdvance()`, который вызывается при завершении каждой фазы, чтобы управлять логикой перехода между фазами.

**Пример: трёхфазный конвейер с динамическим добавлением участников:**

```java
Phaser phaser = new Phaser(1) {  // 1 = main зарегистрирован
    @Override
    protected boolean onAdvance(int phase, int registeredParties) {
        System.out.println("=== Фаза " + phase + " завершена, участников: " + registeredParties);
        return phase >= 2; // Завершить после 3 фаз (0, 1, 2)
    }
};

// Задача, выполняющая 3 фазы
Runnable task = () -> {
    for (int phase = 0; phase < 3; phase++) {
        System.out.println(Thread.currentThread().getName() + " выполняет фазу " + phase);
        phaser.arriveAndAwaitAdvance(); // Ждём остальных
    }
    phaser.arriveAndDeregister(); // Отключаемся от phaser
};

// Динамически добавляем участников
for (int i = 0; i < 3; i++) {
    phaser.register(); // Добавляем нового участника
    new Thread(task).start();
}

phaser.arriveAndDeregister(); // main отключается
```

**Когда использовать Phaser вместо CyclicBarrier/CountDownLatch:**
- Количество участников неизвестно заранее или меняется
- Нужна многофазная синхронизация с разным числом участников на каждой фазе
- Нужен контроль над завершением фаз (через `onAdvance`)

## 12. CompletableFuture

### 12.1. Что такое CompletableFuture? Какие преимущества дает?

`CompletableFuture<T>` — реализация `Future`, поддерживающая функциональный стиль написания асинхронного кода.

**Преимущества перед обычным Future:**
- Создание цепочек действий (callback-style) без блокировок
- Обработка исключений внутри цепочки
- Комбинирование нескольких задач (ожидание всех или любой)
- Явное завершение (`complete()`, `completeExceptionally()`)

**Реальный пример — сервис заказов с параллельными запросами:**

```java
class OrderService {
    private final ExecutorService exec = Executors.newFixedThreadPool(4);

    // 1. Проверяем пользователя и наличие товара параллельно
    // 2. Если оба ОК — резервируем и создаём заказ
    public CompletableFuture<Order> createOrder(long userId, long productId) {
        CompletableFuture<User> userFuture =
            CompletableFuture.supplyAsync(() -> userRepo.findById(userId), exec);

        CompletableFuture<Product> productFuture =
            CompletableFuture.supplyAsync(() -> productRepo.findById(productId), exec);

        return userFuture.thenCombine(productFuture, (user, product) -> {
            if (user == null) throw new NotFoundException("User not found");
            if (product.stock() <= 0) throw new OutOfStockException("Out of stock");
            return new OrderRequest(user, product);
        }).thenCompose(request ->
            CompletableFuture.supplyAsync(() -> reserveAndCreate(request), exec)
        ).orTimeout(5, TimeUnit.SECONDS)     // JDK 9: таймаут на весь процесс
         .exceptionally(ex -> {
             if (ex instanceof TimeoutException) {
                 return Order.failed("Order timed out");
             }
             log.error("Order creation failed", ex);
             return Order.failed(ex.getMessage());
         });
    }
}
```

Этот пример демонстрирует: параллельные запросы (`thenCombine`), зависимые цепочки (`thenCompose`), таймауты (`orTimeout`), обработку ошибок (`exceptionally`), и всё это без единого блокирующего `get()`.

### 12.2. За счет чего достигается параллелизм в CompletableFuture?

**По умолчанию**: используется `ForkJoinPool.commonPool()` (parallelism = `Runtime.getRuntime().availableProcessors() - 1`, но не менее 1). Также можно передать свой `Executor` в методы с суффиксом `...Async`.

**Важный нюанс — кто выполняет callback'и?**

Это частый источник ошибок. CompletableFuture имеет два типа методов:

| Тип метода | Где выполняется callback |
|---|---|
| **Без Async** (`thenApply`, `thenAccept`, `thenRun`) | Там же, где завершилась предыдущая стадия: либо в потоке, выполнявшем задачу, либо в потоке, вызвавшем `complete()` |
| **С Async** (`thenApplyAsync`, `thenAcceptAsync`) | Всегда в переданном Executor (по умолчанию `ForkJoinPool.commonPool()`) |

**Почему это важно — классическая ловушка:**

```java
// Плохо: медленная операция в thenApply может заблокировать общий пул
CompletableFuture
    .supplyAsync(() -> fetchFromDB())       // выполняется в commonPool
    .thenApply(data -> heavyProcessing(data)); // выполняется в том же потоке commonPool!

// Хорошо: явно передаём свой Executor для тяжёлой обработки
Executor heavyPool = Executors.newFixedThreadPool(4);
CompletableFuture
    .supplyAsync(() -> fetchFromDB())
    .thenApplyAsync(data -> heavyProcessing(data), heavyPool);
```

**Ещё одна ловушка**: если `CompletableFuture` уже завершён к моменту вызова `thenApply()`, callback выполнится в **вызывающем потоке** (например, в `main`), а не в пуле. Это может привести к неожиданной блокировке UI или event-loop.

### 12.3. Как создать асинхронную задачу без возврата результата?

Используется статический метод `runAsync(Runnable)`:

```java
CompletableFuture<Void> future = CompletableFuture.runAsync(() -> {
    System.out.println("Выполняется в " + Thread.currentThread());
});
```

### 12.4. В чем разница между thenApply() и thenAccept()?

- **`thenApply(Function<T, U>)`**: трансформация результата. Принимает `T`, возвращает `CompletableFuture<U>`.
- **`thenAccept(Consumer<T>)`**: конечное потребление результата. Принимает `T`, возвращает `CompletableFuture<Void>`. Дальнейшая трансформация невозможна.

### 12.5. Как создать CompletableFuture, уже завершенный с результатом?

```java
CompletableFuture<String> future = CompletableFuture.completedFuture("готово");
```

Полезно для: заглушек в тестах, кэширования, совместимости API.

### 12.6. Какой метод используется для обработки исключения и fallback?

**`exceptionally(Function<Throwable, T>)`** — аналог `catch`. Срабатывает только при ошибке, позволяет вернуть резервное значение.

```java
CompletableFuture
    .supplyAsync(() -> riskyOperation())
    .exceptionally(ex -> {
        log.error("Ошибка", ex);
        return "default value";
    });
```

### 12.7. Как объединить результаты двух независимых задач?

**`thenCombine(CompletionStage<U>, BiFunction<T, U, V>)`** — объединяет результаты двух независимых задач, когда обе завершатся.

```java
CompletableFuture<String> future1 = CompletableFuture.supplyAsync(() -> "Hello");
CompletableFuture<String> future2 = CompletableFuture.supplyAsync(() -> "World");
future1.thenCombine(future2, (a, b) -> a + " " + b); // "Hello World"
```

Родственные методы: `thenAcceptBoth()` (без возврата), `runAfterBoth()` (без данных).

### 12.8. Как дождаться завершения нескольких (N) CompletableFuture?

Для ожидания многих задач используются статические методы:

**`allOf(CompletableFuture<?>... cfs)`** — ожидает завершения **всех** переданных задач:
```java
CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> "A");
CompletableFuture<String> f2 = CompletableFuture.supplyAsync(() -> "B");
CompletableFuture<String> f3 = CompletableFuture.supplyAsync(() -> "C");

CompletableFuture.allOf(f1, f2, f3).join(); // Ждёт все три
// После allOf().join() все три завершены — можно забирать результаты
String result = f1.get() + f2.get() + f3.get(); // Не блокирует (уже готовы)
```

**`anyOf(CompletableFuture<?>... cfs)`** — завершается, как только **любая** из задач завершится:
```java
CompletableFuture<Object> firstDone = CompletableFuture.anyOf(f1, f2, f3);
Object result = firstDone.join(); // Результат первого завершившегося
```

> **Важно**: `anyOf()` возвращает `CompletableFuture<Object>` — тип размывается, нужно кастовать. `allOf()` возвращает `CompletableFuture<Void>` — после его завершения нужен отдельный `get()` на каждом Future для получения результатов.

### 12.9. В чем разница между thenCompose() и thenCombine()?

- **`thenCombine()`**: объединяет **независимые** задачи (параллельно).
- **`thenCompose()`**: используется для **зависимых** задач, когда вторая задача не может начаться без результата первой (последовательно).

### 12.10. В чем разница между thenCompose() и thenApply()?

Это аналог разницы между `flatMap()` и `map()` в Stream API:

- **`thenApply()`**: преобразует результат `T` в `U` (синхронное преобразование).
- **`thenCompose()`**: преобразует результат `T` в `CompletableFuture<U>` (асинхронное преобразование, "разворачивание" Future).

```java
// thenApply — вернет CompletableFuture<CompletableFuture<String>>
future.thenApply(result -> CompletableFuture.supplyAsync(() -> result + "!"));

// thenCompose — вернет CompletableFuture<String>
future.thenCompose(result -> CompletableFuture.supplyAsync(() -> result + "!"));
```

### 12.11. В чем разница между exceptionally() и handle()?

| Метод | Когда срабатывает | Принимает | Возвращает |
|---|---|---|---|
| `exceptionally()` | Только при ошибке | `Throwable` | Значение того же типа |
| `handle()` | Всегда (и успех, и ошибка) | `(T result, Throwable ex)` | Значение любого типа |

Аналогия: `exceptionly` — это `catch`, `handle` — это `catch` + `finally` с трансформацией.

### 12.12. В чем различие между thenRun() и thenApply()/thenAccept()?

- **`thenApply()`** и **`thenAccept()`**: получают результат предыдущего этапа.
- **`thenRun()`**: не получает никаких данных. Просто триггер, срабатывающий по факту завершения задачи. Принимает `Runnable`.

### 12.13. Что происходит при использовании не асинхронного метода после долгой задачи?

Использование не асинхронных методов (`thenApply`, `thenAccept`, `thenRun`) после долгой задачи может привести к выполнению в неожиданном потоке:

1. **Если задача еще выполняется**: код выполнится в том же рабочем потоке пула сразу после завершения задачи. Если внутри тяжелая операция — поток будет занят дольше.
2. **Если задача уже завершена**: код выполнится в вызывающем потоке (например, `main` или UI-потоке), что может "заморозить" интерфейс.

**Решение**: Используйте Async-версии (`thenApplyAsync`, `thenAcceptAsync`) для гарантии выполнения в пуле потоков.

### 12.14. Что такое Structured Concurrency (Java 21+)?

**Structured Concurrency** (JEP 453, прошла 4 раунда preview в JDK 21-24, **стала финальной в JDK 24** — JEP 480) — подход, при котором все дочерние потоки (задачи) привязаны к лексическому блоку кода. При выходе из блока гарантируется, что все дочерние задачи завершены (или отменены).

**Проблема «старого» подхода с `CompletableFuture`:**
- Дочерние задачи могут «утечь» и работать дольше родительского метода
- Нет гарантии, что все задачи завершены к моменту возврата из метода
- Отмена родительской задачи не отменяет дочерние автоматически
- Сложно понять, какие потоки к какому запросу относятся

**StructuredTaskScope** — основной класс Structured Concurrency:
```java
// Все дочерние задачи завершатся до выхода из try-with-resources
try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    Future<String> user = scope.fork(() -> fetchUser());
    Future<Integer> order = scope.fork(() -> fetchOrder());
    
    scope.join();           // Ждём завершения всех
    scope.throwIfFailed();  // Пробрасываем исключение, если была ошибка
    
    return new Response(user.resultNow(), order.resultNow());
}
// Здесь гарантированно все задачи завершены
```

**Ключевые преимущества:**
- **Безопасность**: дочерние задачи не «утекают» за пределы scope
- **Читаемость**: древовидная структура запросов отражается в коде
- **Отмена**: при ошибке в любой задаче остальные автоматически отменяются (`ShutdownOnFailure`)
- **Наблюдаемость**: дампы потоков показывают иерархию «родитель-ребёнок»
- **Интеграция с Virtual Threads**: каждая задача — виртуальный поток

**Политики завершения:**
- `ShutdownOnFailure` — отменяет все задачи при первой ошибке
- `ShutdownOnSuccess` — отменяет остальные задачи при первом успехе (гонка задач)

### 12.15. Как ограничить время выполнения CompletableFuture?

В JDK 9 добавлены два метода для работы с таймаутами:

**`orTimeout(long timeout, TimeUnit unit)`** — если задача не завершилась за указанное время, CompletableFuture завершается исключением `TimeoutException`:
```java
CompletableFuture.supplyAsync(() -> slowOperation())
    .orTimeout(5, TimeUnit.SECONDS)
    .exceptionally(ex -> "fallback"); // сработает при таймауте
```

**`completeOnTimeout(T value, long timeout, TimeUnit unit)`** — если задача не завершилась за указанное время, CompletableFuture завершается с резервным значением (без исключения):
```java
CompletableFuture.supplyAsync(() -> slowOperation())
    .completeOnTimeout("default", 5, TimeUnit.SECONDS);
```

> Оба метода появились в JDK 9 и особенно полезны для внешних API, где нет контроля над временем ответа.

### 12.16. Как создать уже завершённый CompletableFuture с ошибкой?

В JDK 9 добавлен статический метод **`failedFuture(Throwable ex)`** — CompletableFuture, который уже завершён с ошибкой:
```java
CompletableFuture<String> failed = CompletableFuture.failedFuture(
    new IllegalStateException("ошибка")
);
```

Полезно для: тестов, fallback-логики, адаптеров к API, ожидающим `CompletableFuture`.

Также в JDK 9 появился **`delayedExecutor(long delay, TimeUnit unit)`** — Executor, который выполняет задачу с заданной задержкой (без отдельного `ScheduledExecutorService`):
```java
CompletableFuture.supplyAsync(() -> "result",
    CompletableFuture.delayedExecutor(1, TimeUnit.SECONDS));
```

### 12.17. CompletableFuture vs Structured Concurrency: когда что использовать?

| Критерий | CompletableFuture | Structured Concurrency |
|---|---|---|
| Стиль | Функциональный (цепочки) | Императивный (try-with-resources) |
| Управление жизнью задач | Ручное (`cancel()`) | Автоматическое (scope) |
| Утечка задач | Возможна («забытые» Future) | Невозможна (scope гарантирует завершение) |
| Обработка ошибок | `exceptionally()` / `handle()` | `throwIfFailed()` / `ShutdownOnFailure` |
| Комбинирование | `thenCombine`, `allOf`, `anyOf` | `scope.fork()` + `scope.join()` |
| Наблюдаемость | Плоская структура | Древовидная (родитель-ребёнок в дампах) |
| Когда использовать | Простые цепочки, реактивный код | Серверные запросы с параллельными подзадачами |

**Простое правило**: если задача — «сделай А, потом Б, потом В» → `CompletableFuture`. Если задача — «сделай А, Б и В параллельно, дождись всех и верни результат» → `Structured Concurrency`.

## 13. Диагностика и проблемы

### 13.1. Что такое Thread Dump?

Thread Dump — снимок состояния всех потоков JVM в один момент времени. Как фотография пробки на дороге: видно, кто где стоит, кто кого ждёт, и из-за кого затор.

В дампе для каждого потока видно:
- Стек вызовов (какой метод исполняется, какой его вызвал, и так до `main` или `run()`)
- Состояние: `RUNNABLE`, `BLOCKED`, `WAITING`, `TIMED_WAITING`
- Какими мониторами поток владеет и какие ждёт

**Как снять дамп:**
```bash
jstack <pid>              # Утилита JDK — самый популярный способ
jcmd <pid> Thread.print    # Альтернатива jstack
kill -3 <pid>              # Linux: дамп в stdout процесса
Ctrl+Break                 # Windows: в консоли, где запущена JVM
```
Программно — через JMX: `ThreadMXBean.dumpAllThreads()` или `Thread.getAllStackTraces()`.

**Главные применения:**
- **Deadlock**: jstack сам находит взаимные блокировки и пишет «Found 1 deadlock» с указанием потоков
- **Зависания**: видно, что десяток потоков висят на `BLOCKED`, ожидая один и тот же монитор
- **Утечки потоков**: видно тысячи потоков с одинаковым стеком, которые не завершаются

### 13.2. Как можно предотвратить Deadlock в Java?

Deadlock — ситуация, когда два или более потоков заблокированы навсегда, ожидая ресурсы, захваченные друг другом.

**Классический пример deadlock'а:**

```java
Object resourceA = new Object();
Object resourceB = new Object();

// Поток 1: захватывает A, потом хочет B
new Thread(() -> {
    synchronized (resourceA) {
        Thread.sleep(100);
        synchronized (resourceB) {  // Ждёт B — но B у потока 2
            System.out.println("Поток 1: оба ресурса захвачены");
        }
    }
}).start();

// Поток 2: захватывает B, потом хочет A
new Thread(() -> {
    synchronized (resourceB) {
        Thread.sleep(100);
        synchronized (resourceA) {  // Ждёт A — но A у потока 1
            System.out.println("Поток 2: оба ресурса захвачены");
        }
    }
}).start();
// → Deadlock: поток 1 ждёт B, поток 2 ждёт A, никто не продолжит
```

**Способы предотвращения:**

1. **Порядок захвата ресурсов (Lock Ordering)**: Всегда захватывать мониторы в одном и том же строгом глобальном порядке. В примере выше — всегда сначала `resourceA`, потом `resourceB`.
2. **Тайм-ауты**: Использовать `tryLock(long timeout, TimeUnit unit)` вместо `synchronized` — если не удалось захватить блокировку за время, освободить уже захваченные и повторить.
3. **Минимизация критических секций**: Держать блокировки как можно меньше времени.
4. **Использование высокоуровневых синхронизаторов**: `ReentrantLock`, `Semaphore`, `CountDownLatch` вместо вложенных `synchronized`.
5. **Обнаружение Deadlock'ов**: `ThreadMXBean.findDeadlockedThreads()`.

**Решение примера выше с tryLock:**

```java
ReentrantLock lockA = new ReentrantLock();
ReentrantLock lockB = new ReentrantLock();

void safeOperation() {
    while (true) {
        if (lockA.tryLock(100, TimeUnit.MILLISECONDS)) {
            try {
                if (lockB.tryLock(100, TimeUnit.MILLISECONDS)) {
                    try {
                        // Критическая секция
                        return;
                    } finally { lockB.unlock(); }
                }
            } finally { lockA.unlock(); }
        }
        // Небольшая пауза перед повторной попыткой
        Thread.sleep(new Random().nextInt(100));
    }
}
```

**Три стратегии борьбы с deadlock:**
- **Deadlock Prevention (предотвращение)**: Устранение одного из условий возникновения deadlock'а (взаимное исключение, удержание и ожидание, отсутствие вытеснения, циклическое ожидание).
- **Deadlock Avoidance (избегание)**: Алгоритм банкира — система анализирует, приведёт ли выделение ресурса к deadlock'у, и блокирует запрос, если это опасно.
- **Deadlock Detection (обнаружение)**: Периодическая проверка наличия циклического ожидания с последующим принудительным завершением одного из потоков.

### 13.3. Почему неизменяемые (Immutable) объекты потокобезопасны?

Их состояние не может быть изменено после создания, поэтому:
- Нет необходимости в синхронизации для защиты от одновременной записи.
- Не может возникнуть состояние гонки (Race Condition).
- Не требуется синхронизация при чтении из разных потоков.

**Условия неизменяемости:**
1. Все поля — `final`
2. Класс объявлен как `final` (чтобы нельзя было переопределить методы)
3. Нет методов, изменяющих состояние
4. Если поля содержат ссылки на изменяемые объекты, они не должны быть доступны извне (защитные копии в геттерах)

Пример: `String`, `Integer`, `LocalDate`.

### 13.4. Что такое аннотация @Contended и зачем она нужна?

**`@Contended`** (JDK 8, внутренняя; JDK 9 — публичная через `-XX:-RestrictContended`) — аннотация, которая предотвращает **false sharing** (ложное разделение кэш-линий). JVM добавляет padding (отступы) вокруг аннотированного поля, чтобы гарантировать, что оно находится на отдельной кэш-линии и не делит её с другими часто изменяемыми полями.

```java
public class Counter {
    @Contended
    private volatile long value; // Гарантированно на отдельной кэш-линии
}
```

**Без @Contended**: два соседних поля в объекте могут попасть на одну 64-байтную кэш-линию. Если два разных потока меняют их на разных ядрах — кэш-линия постоянно инвалидируется (false sharing), производительность падает в десятки раз.

**С @Contended**: JVM добавляет 64+ байта padding до и после поля, гарантируя изоляцию.

**Используется внутри JDK**: поля `ThreadLocalRandom`, `ConcurrentHashMap.CounterCell`, `LongAdder.Cell`, `ForkJoinPool` и других высоконагруженных структур.

### 13.5. Что такое ForkJoinPool.managedBlock()?

**`ForkJoinPool.managedBlock(ManagedBlocker blocker)`** — механизм, позволяющий задаче внутри `ForkJoinPool` выполнять блокирующие операции без блокировки всего пула.

**Проблема**: Если задача в `ForkJoinPool` вызывает блокирующий метод (например, `take()` из `BlockingQueue`), она блокирует worker-поток. Другие задачи не могут использовать этот поток.

**Решение с managedBlock()**: Когда задача вызывает `managedBlock()`, ForkJoinPool создаёт **компенсирующий (compensation) поток** на время блокировки, чтобы общее количество активных потоков не падало.

```java
ForkJoinPool.managedBlock(new ForkJoinPool.ManagedBlocker() {
    @Override
    public boolean block() throws InterruptedException {
        return queue.take() != null; // Блокирующая операция
    }
    @Override
    public boolean isReleasable() {
        return !queue.isEmpty(); // Может ли разблокироваться без вызова block()
    }
});
```

**Где используется**: `Phaser.awaitAdvanceInterruptibly()`, внутренние операции `CompletableFuture`, блокирующие очереди в ForkJoinPool.

## 14. Исключения в многопоточной среде

### 14.1. Что происходит, если исключение выбрасывается в потоке (Thread)?

Если исключение не поймано внутри метода `run()`, поток просто завершается. Другие потоки и основной поток (main) продолжат работу. Информация об исключении будет выведена в `System.err`.

Исключение **не может быть проброшено** из `run()` в вызывающий код, так как сигнатура `run()` не содержит `throws`. Поэтому единственный способ узнать об исключении в другом потоке — использовать `UncaughtExceptionHandler` или механизмы вроде `Future.get()`.

### 14.2. Как UncaughtExceptionHandler помогает обработать исключения в потоках?

`UncaughtExceptionHandler` — это функциональный интерфейс с методом `uncaughtException(Thread t, Throwable e)`, который позволяет «перехватить» необработанное исключение перед завершением потока.

**Способы установки:**
- **На конкретный поток**: `thread.setUncaughtExceptionHandler(handler)`
- **Глобально для всех потоков**: `Thread.setDefaultUncaughtExceptionHandler(handler)`

**Типичные сценарии использования:**
- Логирование факта падения потока
- Отправка алертов (metrics, мониторинг)
- Освобождение ресурсов, связанных с потоком
- Перезапуск задачи в новом потоке

Если обработчик не установлен, исключение просто выводится в `System.err`, и поток молча завершается.

### 14.3. Что такое ExecutionException и когда она возникает?

**`ExecutionException`** — это checked-исключение, которое возникает при работе с `Future.get()`. Если задача внутри потока (переданная через `Callable`) завершилась с исключением, то при попытке получить результат через `get()`, выбрасывается `ExecutionException`, внутри которого (метод `getCause()`) лежит оригинальная ошибка.

```java
ExecutorService executor = Executors.newSingleThreadExecutor();
Future<Integer> future = executor.submit(() -> {
    throw new IllegalArgumentException("Ошибка в задаче");
});

try {
    future.get(); // Выбросит ExecutionException
} catch (ExecutionException e) {
    Throwable original = e.getCause(); // IllegalArgumentException
}
```

**Важно**: `ExecutionException` оборачивает оригинальное исключение, а не заменяет его — цепочка причин сохраняется.

### 14.4. Может ли main() завершиться, если в другом потоке вылетело исключение?

**Да.** Метод `main()` выполняется в своём потоке. Если упал соседний поток, `main` может даже не узнать об этом, если он не ждёт завершения этого потока через `join()` или `Future.get()`.

**Исключение:** если упавший поток был **единственным не-демоном** (кроме main), а main уже завершился, то JVM завершит работу. Если main ещё выполняется, падение другого потока на него не влияет.

Также стоит помнить: если падение происходит в потоке-демоне, JVM не будет ждать его завершения и может завершиться в любой момент.

### 14.5. Как CompletableFuture обрабатывает исключения?

В отличие от обычных потоков, у `CompletableFuture` есть функциональные методы для обработки ошибок в цепочке асинхронных вычислений:

- **`.exceptionally(ex -> fallback)`** — позволяет вернуть значение по умолчанию при ошибке. Принимает функцию от `Throwable`, возвращает значение того же типа.
- **`.handle((res, ex) -> ...)`** — даёт доступ и к результату, и к ошибке одновременно. Выполняется всегда (и при успехе, и при ошибке). Если `ex == null` — выполнение было успешным.
- **`.whenComplete((res, ex) -> ...)`** — похож на `handle`, но не изменяет результат (используется для логирования, cleanup).

```java
CompletableFuture<String> future = CompletableFuture
    .supplyAsync(() -> { throw new RuntimeException("Ошибка"); })
    .exceptionally(ex -> "Fallback value")    // Вернёт "Fallback value"
    .handle((res, ex) -> {
        if (ex != null) return "Ошибка: " + ex.getMessage();
        return "Успех: " + res;
    });
```
